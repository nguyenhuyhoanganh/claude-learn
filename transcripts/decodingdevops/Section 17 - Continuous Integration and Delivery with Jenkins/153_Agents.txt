# Agents

Hello and welcome. In this lecture we are going to talk about Jenkins agents. This also called us the distributed architecture in Jenkins, or in CI/CD, which is just a fancy name of some other computer executing the commands. Looks something like this. So you have the central master like Jenkins or any other CI/CD tool and other computers connected to it. And the master will connect to these agents, these computers via SSH TCP or any other protocol and sends the built instruction, like we have seen in the previous lecture, git clone command even install command. So those commands will be sent as the build instruction to the agents and the agent will execute those commands. Agent could be a Linux machine, a Windows, a Mac OS based on your requirement. So here we will have Jenkins as the master, the controller, that will be the central brain where we configure all the jobs and manages everything. Agents, nodes, or slaves will be the dedicated machines. Like we can configure one agent as Maven agent or Maven builder that executes only Maven commands. Now the word slaves should not be used, but I have kept it because few people still call it as a slave because that's how the concept developed, Jenkins master slave concept. But then the word slave was discarded and it should be discarded. So these agents or nodes will execute the jobs, the commands and the agent could be of different operating system. It could be a Linux agent, it could be a Windows agent. So if you want to execute dotnet or MSBuild, Windows based commands, you need to configure a Windows agent with the Jenkins master. Same way for Mac OS. So working is pretty simple. The master sends the signal, the build instruction using different protocol based on how you configured it. It sends the instruction and the agent just executes those instruction straightforward. So one benefit, you can clearly see that we can execute different kinds of jobs from one master, Linux based, Windows based or Mac OS based. The other is the parallel execution.
