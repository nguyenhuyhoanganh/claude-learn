Instructor: Hello and welcome to "AWS Load Balancers" lecture. In this lecture, we are going to see what is a load balancer, why do we need it, how the load balancer works, and in AWS, what are the different types of load balancers. Once we understand all this, we are going to actually do it. We will set up web servers or cluster of web servers, and on the frontend, we are going to set up a load balancer for these servers. We are also going to see AMI, Amazon Machine Images, and a launch templates. So really lot of things in this lecture, but let's first understand the load balancer. So, let's say we have multiple web servers. Now, why do we need multiple web servers? Well, in real time, one single web server or one server won't be enough based on the performance and also maintenance. When you have cluster of web servers running the same web service, you can anytime scale out. You can add more servers for the capacity, for better performance, or you can also scale in, reduce it, to save the cost. Also, you can update these servers and softwares on it without bringing down the entire user experience. But when you have cluster of servers, in this case cluster of web servers, you will need a single endpoint on which the user can access, and that single endpoint is provided by a load balancer. In short, a load balancer takes the request from the user on a specific port number, like 80 or 443, and forwards the request to the web server distributing this traffic. So, a load balancer will have an endpoint on which the user will access a frontend port. And when it receives this request, it is going to route on a backend port, which will be the port number of the server. So let's say if you have web server, commonly we use port 80, http port 80. Now, this will be the backend port for the load balancer, that will be the frontend port on which the user access. So, a load balancer is a device or a software. It could be a hardware computer or a software running on the computer that distributes the network traffic across multiple servers or application. Now, obviously we discussed this is to balance the load, have better performance, we can scale out and scale in whenever we want, and it acts like a proxy, it's not... The load balancer is not the real server. It provides the endpoint. It's a proxy in between the user and the server or servers. Now, if you're on AWS, you don't need to run a software load balancer. That means you don't need to launch an EC2 instance and install a load balancer software. AWS provides you Elastic Load Balancers and the different types of load balancers in AWS. Application load balancer, network load balancer, gateway load balancer, and classic load balancers. On AWS, the load balancers are called ELB in short, elastic load balancers, ELB. Let's first take a look at the classic load balancer. It's pretty simple. As we discussed, it takes the request on the frontend port, for example, 443, and routes the request to the backend server. So, it's simple. You create EC2 instances, set up your web service on them, and add these instances under a classic load balancer. You provide the endpoint of the classic load balancer to the user. It's ideal for a simple solution. It works on Layer 4. That means it does not understand the URL, it understands the IP address and the port number. Frontend port, backend port. Then, there is application load balancer. It is for http and https traffic. It works on Layer 7. That means it understands the URL and you can distribute the traffic based on the URL. So if the user access, let's say, hkinfotech.in, in it can send it to a group of server. But if the user says hkinfotech.in/videos, it can, based on the rule, route the request to a different cluster. So, it is intelligent routing and that's why it is used the most on AWS for http and https traffic. And in this course, most of the time, we will be using this application load balancer. The next one is the network load balancer. Now, this sounds like a classic load balancer but it provides a static IP address which is very important in some use cases, and it can handle millions of requests per second. And you must have understood now that it will be very expensive. It works on Layer 4, just like classic load balancer. And sometimes, it is put in front of application load balancer. Now, this will be out of the scope of this course because it is too expensive and it works same as classic load balancer. The last one is the gateway load balancer. This is also out of the scope and it is too early for you to understand this because for this, you need experience with AWS VPC networking. But just for your information, gateway load balancer works on Layer 3 and it is used to run and manage firewalls, intrusion detection systems. The best part about gateway load balancer is it does not do anything with the network packet, the source and destination IP address. All the information of the network packet remains same. It just forwards it to the appliance that could be a firewall and IDS. So for example, intrusion detection system, it needs all that information in order to scan the packet and understand for the security reason, of course. Now, AWS provides really good documentation for the load balancers, application load balancer, network load balancer, gateway load balancer, and you can come and read this later after you actually do the exercises.
