Okay. Let's talk about fuzziness. And I'm not talking about your fuzzy pet cat or something. This is more about dealing with typos and misspellings. So you may have noticed that most search engines can deal with a certain level of typos or misspellings and things like that on the part of the user. Here's how it works in Elasticsearch supports it, too. So a basic concept of fuzzy matches is a thing called the lunchtime edit distance, and it's a fancy name for a pretty simple concept. So the Levenstein Distance allows us to quantify common misspellings and typos. There are three different classes of these substitutions, insertions and deletions. So let's look at three examples here for substitution, two characters that would catch things where someone just types in the wrong a character by mistake. So for example, if someone misspelled Interstellar as Interstellar with an E instead of an A, that would still match if we were willing to tolerate Levenstein distance of one, because there was one character that was substituted for what it really should have been. Insertions are if I were to mistakenly insert an extra character that shouldn't have been there. If I went from Interstellar to Interstellar, you know, put in an extra s there that shouldn't have been there, that would still match if I were willing to tolerate 11 stone distance of one because one extra character was inserted that shouldn't have been there. Also, deletions work the same way. If I misspelled Interstellar to have one l instead of two, that could also match as well because that too has 11 striking distance of one. So basically every mistake, whether it's a substitution or insertion or deletion, will count as a value toward the Levenstein edit distance. One more thing to talk about is the auto setting for fuzziness. Sometimes you want different fuzziness tolerances depending on the length of the string.
