# Pagination

Okay. Let's talk about pagination. Imagine you're building a big website like Amazon or something and you want to return results to the end user one page at a time. It turns out that's pretty easy to do. All you have to do is specify a frame and a size parameter as part of your query. And the only tricky thing here is that the from field starts counting from zero. So let's look at this example where we have a query that returns eight different results or maybe even more than that, and we want to display only three results per page. So when I'm issuing a query for the first page of search results, I would pass in a front value of zero and a size of three. That means that I just want to start from the beginning and give me back the first three results for my query. So if I just want to display three results on a page, all I have to do is process the three results that I got back from that query. Let's say the user Paginated two, page two. They click on the next button in their search results and they go into page two of their results. That would issue a query that has a frame of three and also a size of three. So again, because we're counting from zero, if you want to start with result number four, you would issue A from A of three, which is a little bit confusing, but you get used to it. The size would still be three, indicating that they want to get a total of up to three results. So again, for a page two of these three results per page, I would issue a front parameter of three and again a size of three. And if I were to move on to page three of my results, I would have a frame of six with a size of three. In this example, though, I'd only get two results back because there's only eight results in total. This is how it would actually look like when you're actually issuing a request. There's actually a couple of ways to do it. One is that you can just specify the size and from as parameters on the actual URI. As you see in this first example. And that would actually just return everything in the index with a size of two items per page, starting from item number two, again, starting our counting from zero. We don't have a request body at all in this example, but implicitly there's a match underscore all in there. So we're just going to get every result in the index paginated the way that we requested. The second example works everything into the request body itself. So you can also specify from and size as parameters within your JSON body before you actually specify your query. That works as well. So if I wanted to do a search for sci fi movies and only see two results starting from result number two, which is really the third one, that's what that would look like. Pretty simple stuff. Now there are some caveats with pagination and these are not specific to Elasticsearch. This is kind of a problem with search engines in general. Deep pagination can really kill performance. So if you think about it, you can't really tell Elasticsearch, just give me back result number 10373 to 10383. That's not just going to have the overhead of retrieving ten results because under the hood it still has to retrieve and collect and sort everything in order to figure out what result number of 10,380 is. And it has to figure out all 10,300 results before that as well in order to give you what those results are for the range that you requested. So you can really start to kill performance when you start paginated and deep into your results. You should enforce an upper bound on how many results you'll return to your users. Otherwise, some nasty person will abuse your system and bring your system to its knees. And even websites like Google have upper bounds on how many results they return for this reason.
