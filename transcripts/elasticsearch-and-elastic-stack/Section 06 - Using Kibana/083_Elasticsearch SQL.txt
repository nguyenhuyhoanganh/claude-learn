# Elasticsearch SQL

Let's talk about a fairly new feature in Elasticsearch the ability to query your Elasticsearch index using Structured Query Language or SQL. If you've ever worked in a relational database before, this might make it a lot easier to interact with your data that's housed in Elasticsearch. This feature was rolled out in Elasticsearch 6.3. The very concept of issuing SQL queries on Elasticsearch is a little weird because SQL assumes strongly typed data, while Elasticsearch is a schema less document oriented model. In fact, they've been going through great pains to get away from users, drawing parallels between Elasticsearch indices and database tables. That's what motivated them to do away with the concept of types and type mappings in Elasticsearch. So in some ways, offering a SQL interface flies in the face of what Elasticsearch is trying to be, but in the end, it was just too tempting of a feature for them to not add it. It's not totally crazy to treat an index as a table. As we've seen, we can still define type mappings associated with an index as a whole, but there are certain structures supported in Elasticsearch that can't be represented in SQL, such as lists, which may contain different types. For example, you cannot try to display the genres field in our movies index from the SQL interface because it is a list. But apart from that limitation, it really is just straight up SQL and it's easy to use. You can just send a Post query to SQL that contains your SQL query, treating the index as a database table and it just works. They even provide a command line interface, just like you would see with a relational database that will allow you to connect to your Elasticsearch cluster and interact with it, just like you would with a relational database. The difference is that this database, of course, isn't really relational under the hood. It's document based and it's horizontally scalable. So unlike traditional relational databases, you can query a virtually unlimited amount of data using SQL this way. Overall, I find it exciting. Any feature that makes a new technology easier to use is fine by me. How this all works is surprisingly complex and highly optimized. We start by parsing your SQL query into an internal abstract syntax tree, or AST, at which point the syntax of your query is validated. The analyzer then matches up the tables, columns, and functions you're trying to use with the underlying indices and produces a logical plan for executing your query. This then goes to a query planner, which optimizes the query to remove any redundant operations, and finally produces a physical plan that is run by the query executor. The executor returns the results back to the client. As you'll see, we can gain some insight into what's happening by asking the SQL engine to translate a query into DSL or the equivalent JSON request that would actually be sent into Elasticsearch. If you want to develop an optimal complex query in Elasticsearch's own syntax, this can be a shortcut if you're handy with SQL. Let's head to our virtual server and try it out.
