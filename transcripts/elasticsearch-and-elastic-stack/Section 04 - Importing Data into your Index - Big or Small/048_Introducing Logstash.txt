# Introducing Logstash

So we can't really talk about importing data into elastic search without talking about log stash. Basically, it sits between your data and where you want to put it, and commonly, that will be between a log file being generated on a web server somewhere and elastic search in the context of this course. But logs stash is actually much more flexible than that, it can import data from pretty much anywhere, it doesn't just have to be local files, it could be coming from some other system entirely like Kafka or some distributed file system like S3, or HDFS, or something like that and it can put it, not just in elastic search, but other places at the same time like Amazon Web Services or to a Hadoop cluster or some database like Mongo D.B. You can actually import things from more than one source at a time, and output it to more than one destination at a time. So, it's a very powerful tool for pushing your data around. Basically ingesting data from some distributed fleet of things that are generating lots and lots of data, and funneling it into some system where it will be archived and analyzed, later on, and it's more than just plumbing, you can also do things like deriving structure from your unstructured data. So in the case of passing Web logs, for example, log stash can pass out those Web log lines and actually structure them into fields that have given names, it can filter out data as it passes through it, it can transform data, it can do things like anonymise personal data as soon as it sees it, or even exclude personal data entirely. That can be very important for legal reasons, if you want to make sure that you're not transferring personally identifiable information around your network unnecessarily, it's a big deal. You can also do geo location lookups. So for example, that can take a look at the access logs from a web server and automatically figure out where that log entry came from just based on the IP address that was logged. It can scale across many nodes, so part of the power of logs stash is that it can serve as sort of a buffer between these systems that are generating your data, and where you want to put it. So if you have a big load spike on your web server fleet, that doesn't necessarily have to translate into a spike in actually transferring that data into a elastic search or somewhere else, your fleet of log stash hosts just serve as sort of a buffer between those two systems. Logstash does guarantee at least once delivery, and if you want to take a look at the complete list of plugins available for log stash, you can follow that link there. But here's a sampling of just how versatile it is. You can see that it's not limited to local files, or even the elastic beats framework or anything else, there's a huge list of data sources that you can have log stash listen to or watch or transform, and then feed that data to other systems and pretty much anything you can imagine here, any protocol, you can just have data being published on http, or you can talk to some database, or S3, or whatever you want. There's a lot of common systems on here that you might find useful, and it also has a huge variety of output destinations as well. You can see that elastic search is just one of many possible destinations that you can send things to, you can send it pretty much anywhere you want to. So log stash is a very flexible and very powerful system, just can't say that enough. Now typical usage in the context of elastic search will look something like this. So the modern way of doing things, if you're going to be doing something like publishing Web logs into an elastic search cluster for visualization later on, would be to install a lightweight client called file beat on the individual web server hosts, and then file beat would send data to your log stash cluster, which would then buffer that up and pass things out and do geo location and things like that, and then send things into elastic search to be indexed. Sort of a legacy way of doing things would be to just use log stash directly on your actual web servers, and do all of the passing and imparting of structure and geo location actually on the web server hosts themselves. Obviously, that's not quite as resilient to the system, but that's kind of the old fashioned way of doing it where you just have a log stash listening to local files, actually listening directly to your access log, or your error log or both on each individual web host and then publishing that data directly into your elastic search cluster. Either way works. Again, the more modern way is to use file beat, which is part of the Beats framework to talk to log stash, then log stash is sort of an intermediate layer. If you don't have file beat, that gets back to what we call the elk stack elastic search, log stash, and kibana, where the 'L' stands for log stash, but these days, we actually use what's called the elastic stack instead because it's really more about using the beats framework than logs stash. We'll talk about file beat in the beats framework in more detail, later on in the course. For now, however, let's set up log stash on the individual host that we have.
