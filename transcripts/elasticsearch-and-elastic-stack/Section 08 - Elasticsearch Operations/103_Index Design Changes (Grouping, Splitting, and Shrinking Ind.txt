# Index Design Changes (Grouping, Splitting, and Shrinking Indices)

Boop boop. You've created the perfect design for your indices and they are happily churning along. However, in the future, you may need to reconsider your initial design. Maybe you want to improve performance, change your starting settings, adjust for growth, whatever the reason. Elasticsearch is flexible and allows you to change index settings. So let's learn how to do that. During the lifecycle of an index, it will likely change to serve various data processing needs like high ingest loads, query volumes of varying intensity reporting and aggregation requirements and changing data retention and data removal. Generally speaking, changes that can be performed on an index can be classified into these four types index settings, sharding, primary data, low level changes to the indexes, inner structure such as the number of segments or freezing, which we won't cover for the purposes of this lesson. However, if you'd like to read more about Advanced Index Design concepts, hop over to the Corelogic's blog at bit. Actually slash tune elk. Elasticsearch indexes have various settings that are either explicitly or implicitly defined when creating an index. There are two types of settings dynamic settings that can be changed after index creation, or static settings that cannot be changed after index creation. Dynamic settings can be changed after the index is created and are essentially configurations that don't impact the internal data index directly. For example, number of replicas. This is the number of copies the index has. Refresh interval is when the invested data is made available for queries. Blocks is disabling readability or readability of the index. Pipeline is selecting a pre-processing pipeline attached to the documents. Pipeline is selecting a pre-processing pipeline apply to all documents. We can update dynamic settings with a put request to the index slash underscore settings endpoint. Static settings, on the other hand, are settings that cannot be changed after index creation. These settings affect the actual structures that compose the index. For example, number of charts, the number of primary charts and codec defining the compression scheme of the data. Shards are the basic building blocks of Elasticsearch as distributed nature. It allows us to more easily scale up a cluster and achieve higher availability and resiliency of data. High availability. When we say that something has high availability, it means that we can expect the service to work uninterrupted for a very long time by spreading services and data across multiple nodes. We make our infrastructure able to withstand occasional node failures while still continuing to operate normally. The service doesn't go down, so it's still considered available. High resiliency is achieved by means such as having enough copies of data around so that even if something fails, the healthy copies prevent data loss or otherwise said the infrastructure resists certain errors and can even recover from them. Imagine having an index with multiple shards. Even if one of the shards should go down for some reason, the other shards can keep the index operating and also complete the requests of the lost shard. This is equivalent to high availability and resiliency. Furthermore, if we need to achieve higher speeds, we can add more shards by distributing the work to multiple shards. Besides completing tasks faster, the shards also have less individual work to do, resulting in less pressure on each one of them. This is equivalent to scaling up. Work is done in parallel faster and there's less pressure on each individual server. As we mentioned before, the number of primary shards is a static setting and therefore we cannot change that on the fly since it would impact the structure of the master data. However, in contrast to primary shards, the number of replica shards can be changed after the index is created since it doesn't affect the master data. If you want to change the number of primary shards, you either need to manually create a new index and re index all your data along with using aliases and read only indices. Or you can use helper APIs to achieve this faster. For example, post slash source index slash shrink slash target index name would lower the number of charts or post source index underscore split slash target index name would multiply the number of shards. Both actions require a new target index name as input. If we need to increase the number of charts, for example, to spread the load across more nodes, we can use the split API. However, this shouldn't be confused with simply adding more shards. Instead, we should look at it as a multiplication. The limitation to bear in mind is that we can only split the original primary shard into two or more primary shards. So you couldn't just increase it by plus one. Let's go through a few examples to clarify. If we were to start with two and multiply by a factor of two, that would split the original two shards into four. Alternatively, if we were to start with two shards and split them down to six, that would be a factor of three. On the other hand, if we started with one shard, we could multiply that by any number we wanted to. We cannot, however, split two shards into three because that's not an even multiple. The Shrink API does the opposite of what the split API does. It reduces the number of shards while splitting shards works by multiplying the original shard. The shrink API works by dividing the shard to reduce the number of shards. That means that you can't just subtract shards, but rather you have to divide them. For example, an index with eight primary shards can be shrunk to four, two or one. One with 15 primary shards could be brought down to five, three or one. When you change your primary index data, there are too many ways to reconstruct it. Now you may be thinking, why change the primary data at all? There are two potential causes for changing the primary data. One is that the physical resource needs change the or the value of your data needs to change. Resource limitations are obvious. When ingesting hundreds of docs per second, you will eventually hit your storage limit. Secondly, the value of your data tends to gradually decline, especially for logging in metrics use cases. Holding millisecond level info doesn't have the same value when it was fresh and actionable as opposed to being a year old.
