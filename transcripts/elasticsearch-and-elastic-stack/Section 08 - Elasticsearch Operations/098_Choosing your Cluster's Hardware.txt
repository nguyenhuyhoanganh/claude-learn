# Choosing your Cluster's Hardware

Another very immediate decision you'll have to make when creating your first elastic search production indexes is what hardware you want to run it on. Running it on your own desktop as well and good but in the real world obviously you need to run it on real hardware that can do real stuff. Now it's really not that complicated. The thing is with elastic search, Ram is most likely your bottleneck. So the more RAM you have, the better. That's the main thing you want to optimize for. Now the sweet spot, according to elastic search, is 64 gigabytes per machine. The idea is to allocate 32 gigabytes to elastic search itself and then another 32 gigabytes to the operating system and the disk cache so that Lucene can accelerate its disk IO operations to the fullest extent possible. Now you never want to go above 32 gigabytes for elastic search itself because then you start having to deal with these really long pointers to memory that can make things much less efficient. So ideally, you would want 64 gigabyte machines in terms of RAM, where half of that gets allocated to elastic search, and half is just left to the operating system to do what it will with it. You do not want to go under 8 gigabytes if you can help it, things start to get really inefficient below that number. Other considerations are the disks that you have. Obviously, faster disks are better, and if possible use SSD, solid state drives. It's also important to have the right IO scheduler setup on your SSD drives, specifically the deadline or no op IO schedulers and you ought to look up how to set that up on your particular operating system. But it can actually make a big difference, as elastic search is very Io heavy, and the faster the disk, the better. There's no need to have redundancy in your disks themselves, just used raid 0, because your cluster is already redundant, that's why we have replica nodes. You shouldn't be wasting resources and having replicated storage as well. Elastic search is not very heavy on the CPU, so if you need to save some money that's probably the place to save it. You do, however, need to be on a fast network because there is a lot of network traffic obviously happening as you're replicating data across all the different nodes in your cluster and routing the request to different nodes in different shards. Elastic search also recommends not using network attached storage if you can help it. So a lot of cloud based solutions tend to go toward that sort of system but whenever possible, you want to use local SSD instead as network attached storage can cause some problems, and in a lot of hardware provisioning environments, you're just given very simple choices between small, medium, large and maybe extra large configurations. I mean, that's how I used to work, and if you're using some cloud based platform that's obviously how it's kind of set up as well. Usually, you want to gravitate toward the medium to large configurations when you're given a choice like that, too big is going to be overkill and have more overhead, and like we talked about, if you have too much memory that can actually slow things down. Also don't have too many small boxes either because you know you end up not being able to fit enough indices or enough shards onto a single box, so you want to go with a medium to a large given the choice and well that's all there really is to it. So that's your general guidance, and if you follow these guidelines, you should have a fairly optimal hardware configuration for your shiny new elastic search index.
