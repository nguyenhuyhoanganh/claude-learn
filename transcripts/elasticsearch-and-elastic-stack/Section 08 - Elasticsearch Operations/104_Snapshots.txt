# Snapshots

So while elastic search is a pretty resilient system sometimes you want to be safe, and it's a good idea to actually store a backup of your entire index somewhere just in case something truly horrific happens. That's what snapshots are for; snapshots are a backup mechanism that elastic search provides, and it allows you to store backup copies of your index data to some network attached storage, to your local file system, to Amazon's S3 service to an HDFS cluster, Microsoft Azure, whatever you want. And it's smart enough to only store changes since your last snapshot. So it's just storing incremental backups to conserve resources. So instead of actually storing an entire new copy of all your data every time, it only stores what's changed and that allows you to make a series of snapshots very quickly and very efficiently. Here's a little bit of a cheat sheet on how it works. So the first thing you need to do is actually create a repository for all of your snapshots to live in. You need to set that up as part of your configuration for the cluster in your elastic search dot yml file. You need to set the path dot repo configuration setting to wherever your repository is going to live, and in our case it's gonna be on a local file system. We'll just throw it under a backup directory, that I throw in my home directory. In the real world of course, that might be somewhere else on a different machine or even a different data center. But for this particular example, we'll just put it there. Then you need to send a put command into your cluster; that says put underscore snapshot slash repository name. And within that, we need to specify the type of the repository. In this case it's going to be a local file system and that's designated by FS. It could be s3 or something else. And under the settings you need to specify the location that those repositories snapshots will live under. So in our case, that's going to be home user name backups, to match the path of the repository in my configuration. Slash repository name, which we're calling backup dash repo in this example. Here's another little cheat sheet of how to use snapshots if you want to create a snapshot of all your open indices on your cluster. You just send a put command of underscore snapshots slash repository name slash snapshot name. If you want to get information about that snapshot you can just swap that put for a get, and if you actually want to monitor the progress of a snapshot that's currently being stored, you can just do a slash underscore status at the end of that get command to get up to the second information about how that snapshot is going. Now if you do actually restore that snapshot at a later time, that's also very easy to do. All you need to do is close off all of your open indices using this post underscore all underscore close command, because obviously you don't want people writing data into an index while you're trying to restore it from a backup. So that will close it off to all new changes. And then all you have to do is say post underscore snapshots slash repository name slash snapshot name slash underscore restore, and that will start the process of restoring your index to what it was in that particular snapshot. Obviously this is the sort of thing that you want to practice beforehand.
