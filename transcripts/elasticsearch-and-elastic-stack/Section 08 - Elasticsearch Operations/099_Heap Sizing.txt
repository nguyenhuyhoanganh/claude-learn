# Heap Sizing

Another operational consideration that's very important is setting the heap size for elastic search because it's almost certainly not set to what it should be. Now the default heap size and elastic search is only one gigabyte. So if you're running on a machine that has 64 gigabytes of RAM, you're only using one gigabyte of that for elastic search, and if you have things set up by default, the other 63 gigabytes are not completely going to waste, your OS will use it for disk caching, but obviously this is not the ideal configuration. What you want to do instead is at least give half or less of your physical memory to elastic search. Generally, you want about half of your physical Ram dedicated to elastic search itself, and the other have to be given to your operating systems so that lucene and the operating system can just use it for caching purposes. Elastic search is very disk heavy so you want to make sure that has a lot of memory available to cache what's on your disk into memory, that's why you want to leave about half of your physical memory just to lucene in the operating system, and only allocate half of it to elastic search. Now if you're not doing any aggregations on analyzed string fields, you know we've seen examples of doing that earlier in the course, you can get away with even less than half of the memory for elastic search and that might be a good thing. The more memory you have for caching your data, the better, but obviously, you want to give a classic search what it needs to be efficient. Now smaller heaps are going to result in faster garbage collections as well, so it's not just about getting more memory into your cache, it's also about making sure garbage collection is running efficiently as well. Elastic search is built on top of Java, and one of the bad things about Java is that it manages memory using garbage collection, which means every once in a while it has to go through and clean everything up and that can actually give you a performance hiccup while that's happening. So if you have a smaller heap, that's a little bit faster to do. The way that you actually set your heap size, is a couple of different ways. One is by exporting an environment variable called ES underscore heap underscore size, and you can set that explicitly to whatever amount of memory you want to allocate to elastic search itself, say 10 gigabytes might be a good choice if you're running on a 24 gigabyte system or something like that, or you can also export yes underscore a Java underscore OPTS, and you can specify a specific Java fly there for the heap size. It's there if you want to do it add a little bit of a lower level and then kick off elastic search once that's been set, either way works, just make sure that you don't cross 32 gigabytes for your heap size on elastic search, once you get above 32 gigabytes then it takes a lot more memory to store pointers to memory because you can no longer fit that in four bytes. So for efficient performance, you don't want to go higher than thirty two gigabytes because performance will take a pretty big hit once you cross that line, this is why we said in the previous lecture that 64 gigabytes, it's kind of the sweet spot that allows you to dedicate 32 gigabytes to elastic search and the other remaining 32 gigabytes to Lucene and the OS, that's the most optimal configuration to have because you know obviously individual machines are expensive. Ram, again, is your real bottleneck so you want to use as much RAM as possible and each physical machine. So there's some guidance. Make sure that you do check your heap size and set it to whatever is appropriate for the hardware you're running on, because the default size of one gigabyte is almost certainly not what you want.
