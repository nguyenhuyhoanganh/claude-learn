# X-Pack Security

In real world settings, securing your data an elastic search will be really important. If any personal information is being stored in your cluster, and this probably includes the data and most weblogs, you probably have a legal obligation to protect that data from people who shouldn't see it. The licensed version of X PAC offers several built in security features, it offers access control to the cluster with password protection, role based access control and IP filtering. If you're running on a cloud based elastic search offering, the cloud it's on, such as AWS, may offer its own security features instead. Elastic search allows you to encrypt your data in transit using SSL and TLS, and has message authentication to ensure data is coming from where it should be. You can increase security further by separating client traffic from node to node communications at the network level to ensure the outside world can't access the underlying elastic search servers themselves. Finally, the elastic stack offers an audit trail so that you can see who is accessing your cluster and what they are doing, this can help you identify attempted attacks and data breaches, hopefully before they are successful, but in a worst case scenario, it may give you the information you need to track down the bad guys and prevent them from doing it again. When thinking about security privileges and authentication, you need to think about users, groups and roles. Users can be assigned to various groups and each group may have a set of specific privileges associated with it. For example, a group might restrict which indices documents or fields members of that group are able to see, if your organization already has an active directory or LDAP system in place, elastic search can integrate with that allowing you to use the user in groups you already have.
