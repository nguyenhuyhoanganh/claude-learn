# Command - Introduction

Now let's learn about command design pattern. Now, this is one of the powerful design pattern. It's quite easy structurally, it's quite easy to implement this design pattern, but it provides you with great power on how you can achieve functionality in your application. So let's look at command pattern and why we need it. So let's say we want to represent a request or a method operation or a method call as an object. Then we can use the command design pattern. So once again, let's say we want to call a method on an object and we want to pass on some arguments. We want to have this entire act of calling the method on an object and passing some arguments. We want to represent this entire act as an object. Okay. And you will see the benefit of doing that. This object, the command object that we are going to create will have the information about the method that we want to call it will have information about the object on whom we are going to call the method. And it will also have information about the arguments that we need to pass to this method call. And this is the power of command design pattern, because a simple method call that would have been a line that we type in our code, it now is an object and we can treat that operation now as an object. Okay, So advantage of this pattern is that we can now send this object to different parts of our code. We can add this object in a collection and we can send it even to a remote server, so on and so forth. Okay, So there is a great benefit and flexibility that you achieve by using this design pattern. We can even cue the command objects. So instead of calling a method immediately on an object, we can defer that method call using the command object. So we will cue them in a collection and we will have a separate thread, for example, whose job it is to take these command objects and execute them, which will result in a method being called on the object. Okay, so it's a quite powerful design pattern. And even if it sounds a little bit confusing, once we see and implement this design pattern in Java, you will get hold of all the concepts. So now let's look at the UML diagram of this design pattern. First of all, we have the command interface. Now, typically command interfaces are quite simple. They define the execute method and this method typically doesn't need any arguments. Okay? Because all the information that is needed by the command should be provided while creating the command. Okay, so this command interface is used by the receiver or the code which is going to actually call the execute on it. Okay. And that could be a different part of our application. So that code doesn't know what we need as arguments or what object is used by the command. So all that information is provided in the concrete command instance, and the interface itself simply exposes a single simple method execute. Next up, we have the concrete command. Now this is the implementation of our command. And this class object of this class has all the information about what should this command do? It will have the information about the actual object on which we are going to call a method. It will have the method information so it will know which method to call on that object. And it will also have the information about the arguments that we are going to pass in that method call. Okay. And that is all part of state of this concrete command. Next up, we have receiver. Now we are talking about the command object and we are saying that we are going to call a method on an object. So receiver is that object. Okay. So receiver represents an object on whom we are going to call a method from our command. Okay. So it simply has method that we need. Next up, we have invoker. So invoker is a code that actually calls our commands execute method because we are not going to call execute as soon as we create a command because that's not any much different than what we are doing without this design pattern. What we are going to do is we are going to create instances of our command and send it to different parts of our code. Now invoker is that part which gets the command objects and it calls the execute method on these objects one by one. Okay. Lastly, we have client. The client is actually responsible for creating the concrete command instance. So client will set the reference to receiver. It will send the information about arguments that there are needed to call a method to our concrete command instance. All right, so this is the command design pattern. All right.
