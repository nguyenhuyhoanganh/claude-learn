# Command - Implementation & Design Considerations

All right. Now let's look at some of the implementation and design considerations while implementing the command design pattern. So first of all, we can support, undo and redo functionality using our commands. Okay. And this makes it really important for applications which are user interaction driven. For example, let's say you have a workflow designer that you have that you have implemented and client is using your workflow designer designer in a browser. Then any command or any interactions will result in execution of some commands. You can queue those commands in a queue and then you can provide the undo functionality using that queued command objects. Now, if your command is really simple, that means if it doesn't support any undo functionality and all it does is that it simply hides the actual function call that happens and a fixed value for arguments. Then we can reuse this command objects. And this is especially true if you are designing an application, let's say a web application that is used by hundreds of users and they are all going to work on this workflow designer. Then on your server, maybe you can reuse these command instances if you don't have any undo functionality or the state that is stored in these commands is immutable. Now for commands that are going to be queued for long duration and this could be because there are actually there is actually an undo functionality. And to provide that functionality, we are going to hold on to the commands which are already executed. Or maybe because your system is under load in those situations, always pay attention to the size of state that is maintained by them. This point is really meant if you have undo functionality, okay, because if you implement this undo functionality, your command is going to end up saving the state of the receiver before the command was executed. Okay. We are going to see how undo functionality is implemented in the next few lecture when we discuss the memento design pattern. Okay. But in those situations where you have undo functionality, your commands state size will be very high. Now let's look at some design considerations here. The commands can of course be inherited from our existing command and they will use part of the functionality that is provided in the base command and add on to that functionality. Okay. So that way you can reuse whatever code or whatever logic is implemented in previous commands. Now on the other side of scale, you also have composition. So using composition you can create what are called as macro commands. So these are commands which compose, which are composed or created using two, three or multiple sub commands in them. And when you execute this macro command, then it will call the execute method on its sub commands one by one. And this is a powerful feature of using the command design pattern because you can then provide more aggregate operations to your user. Okay, So you may have a functionality that, that let's say takes a workflow, resizes it to the A4 paper size, rotates it on a in a landscape mode and then runs a print command, so on and so forth. So the opportunities to implement functionalities which are composed by using smaller commands is very high here. Lastly, for implementing the undo feature that I just discussed, you can make use of Memento design pattern. So we're going to see that design pattern and we are actually going to implement an undo functionality there. But it allows you this design pattern allows you to store the state of receiver in your command and you will then queue your already executed command in a undo queue. Okay. Um, okay. So these are some of the points that you should keep in mind while working with command design pattern.
