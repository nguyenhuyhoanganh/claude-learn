# Command - Summary

All right. Now let's look at the summary of command design pattern. Now, command pattern allows you to treat requests for operation or method calls as objects. So this allows you to send these objects to different part of your code where they are executed, or you can queue them and use a separate thread in order to execute these commands. The command typically invokes an actual operation, or it calls an actual method on receiver. So receiver is your regular object on which a method is called. But this command also contains the parameters or arguments that are needed to call that method. And client code is actually responsible for creating instances of command, and it provides that instance of command with the receiver object reference to that receiver object as well as any arguments or any request information. Now, commands can also implement undo feature, so using this feature you can undo previously executed commands. All right, so this is the command design pattern. Now let's look at the UML diagram of this design pattern. So to begin with, we have a command interface. Now a command interface is a simple interface that is used by the invoker. So invoker will simply call the execute method on the command. Then we have a concrete command which implements this interface and it provides actual functionality. So it will call an actual method on the receiver. So receiver is the object on which we are going to call a method from our execute implementation in concrete command. Concrete command also stores information about the arguments that are needed to call this method. Client is going to create object of our concrete command and it sets or it gives this concrete command instance reference to the receiver or the object on which this command will be executed. And it also provides the concrete command with the arguments that are needed to call that particular method. Invoker is a separate part of our code where this command is executed. Okay, so invoker can be a code which runs in a separate thread. Now let's look at a typical implementation of command design pattern. So in this example, we have a simple interface called as command, and we have a method execute. Typically, the execute method will not require any additional information from the invoker because all information that is needed to carry out the operation or carry out its functionality is provided to the command while creating instance of that command. So here is an example of a concrete command. So here we have a Add member command class which implements our command interface. And in the constructor of this class we take all the information that we are going to need in order to call or perform a certain operation. Okay, so we get hold of the receiver, which is a service. In this case, we get hold of any arguments that are needed to call that method. In the execute method, we are simply going to call a method on our receiver using the arguments that are given to us. Then we have a receiver. So this is simply an object of any class that you want. And command concrete command is going to call a method on this particular receiver object. Next we are going to see an invoker. So in this example we have invoker running as a separate thread. So you will see that we have a collection here with name pending commands and the client is going to add commands to be executed on this list. Okay. In order to use this design pattern, the client will create a concrete command instance, providing it with the information that is needed. It will also provide the receiver reference and any arguments that are needed by this command. It will then add this command to the list of this task runner so client can send this command anywhere in our application and that command will be executed by our invoker. All right, so this is the command design pattern.
