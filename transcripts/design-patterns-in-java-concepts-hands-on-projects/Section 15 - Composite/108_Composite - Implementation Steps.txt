# Composite - Implementation Steps

Let's look at the steps that are needed or that you would typically follow if you are implementing a composite design pattern. So we start by creating our abstract class or interface, which has the role of component. Now this defines methods that are common to both the composite node and a leaf node. Now, next up, we need to choose whether where we want to define methods that allow you to add and remove nodes from composite. You can put them in component, but then your leaf node will be forced to provide implementation for those methods. Or you can put it in composite, but then you have to make your client code aware that there is a separate class type called as composite, which has children. Whatever the decision you need to make, you need to make while you are implementing component. Next up, we implement the composite and we need to make sure that whenever an operation, actual operation is invoked on the composite, we are delegating that operation to its children. In lymph nodes. We need to handle the non-applicable operations. So, for example, if you have defined the Add remove operations in the component, then you need to provide some implementation in Leaf. And typically that means throwing an exception saying that we do not support this type of operation. Of course it is applicable only if you have defined add remove operations in your component class. In the end, the composite pattern implementation should allow us to write algorithms in our client code without worrying about whether it is a leaf node or composite node. There should not be a frequent check. Whether there is a leaf node, then do this. Whether it is a composite node, then do this. It should not happen at all in our code and that's the whole purpose of composite design pattern.
