# Composite - Introduction

All right, Now let's see how we can implement a composite design pattern in your application and why we need this structural design pattern. So first of all, why we need a composite and when can we use it? So whenever we have a tree structure of objects in our application, or you can call that a part whole relationship, that means multiple objects create or make up one whole object in those situations. If we want to treat all objects in that hierarchy or in that tree in similar way, then we can use the composite design pattern. So this design pattern allows us to treat objects in a tree structure the similar way, whether they have their own children or whether an object is a leaf node in a tree. And that's the benefit or that's the value that this design pattern provides you. Now, one thing to remember is this is not the simple composition concept that we know from object oriented programming, because a composition simply means that we have an object with a reference inside it which points to another object. That's the composition. Composite design pattern uses composition. There is no question about it. However, it goes one step further and it allows you to treat objects which have composition or which are using composition and which don't in similar way. Okay, so that's the value of this composite design pattern. Now, whenever you have a tree structure of objects in your application, you will be using composite in one way or another. Now let's look at the UML diagram for this design pattern. Okay, it's quite a simple UML diagram. So we start with component. Now this will be our base class, abstract base class or an interface, and it defines methods that are common to all nodes in a tree. And that's pretty simple to understand. What is strange or what is different here is that this design pattern demands that you also add methods that allow access to children in the base interface. Now, this is a point of some argument because a point can be made that having these children management methods in the base class or interface forces our leaf node to provide implementation when it cannot. Okay. And another side of argument is that the purpose of composite design pattern is to be able to treat our classes composite or leaf nodes in similar way. So that means we need to have these operations. Okay. So I'm going to leave the decision to you whether you want to keep the children management operations in the leaf node or whether you want to keep that operation or move those operations to class where it actually needs it. Anyways, the next class in the hierarchy in the design pattern is the leaf node. Now this is the class which represents leaf objects in our tree structure and it has implementation for the operation, the actual operation that is needed by client, and it provides the actual functionality in that method. Next up, we have composite. So composite is the one which has the children within it. Okay. So it is using the composition concept and it has also provided implementation for operation or the functionality that his client is requesting. Now in the implementation of operation, this class is going to delegate it to all of its children. Okay. So that's the difference between the implementation of core functionality between composite and leaf. Leaf is going to provide the actual functionality. Composite is going to delegate that to its children. Additionally, it defines methods or provides implementation for methods that allow you to add and remove a leaf node from composite. Lastly, we have client. Now this is the class that is going to use our composite design pattern. And the benefit of using composite is that client is always going to work with our component interface. All right. So this is the UML diagram for composite design pattern. And once again, to highlight the point is that component interface is defining methods such as add a child or remove a child. And a point can be made that these operations belong only in composite and it should be defined in composite only, not in the base class, because now Leaf has to provide implementation for these methods where we don't need those methods. But another side of argument is that since client wants to work only with the component interface and that's the whole purpose of composite, we need these methods defined in component. Okay, so I'm not going to take a side here. I'm just making you aware that a point can be made on both sides of arguments and it's up to you how you implement this design pattern in your code, because ultimately design patterns are nothing but guides on how you can do things. They are not rules. So you can of course change them as per your requirement. All right. So this is the introduction to composite design pattern.
