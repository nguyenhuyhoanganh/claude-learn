# Composite - Pitfalls

All right. Now let's look at some of the pitfalls of the composite design pattern. Now, it is quite difficult to restrict what gets added to our tree or our hierarchy. And if you have multiple types of leaf nodes, which is somewhat common in real life, then the client code again ends up doing runtime checks to see whether it is calling the operation on correct type of a node. So again, there is a chance that the whole purpose of having the composite is defeated because now the client is doing the runtime check because it is quite difficult to provide all of these operations in component class, in the base class or base interface because they may not apply to all of the nodes or all of the child classes. So that's a pitfall of composite design pattern. So you should be on the lookout where you are running into that kind of scenario. Now, creating the original hierarchy or original tree can still be a complex operation because the design pattern doesn't tell you how to build your tree. It simply tells what is the structure of your already built object tree. So it's up to you to implement, are up to you to create that original tree. And it can be quite a complex process. And if you are using caching for your nodes in order to save onto some memory or on the initialization cost, then the creation of this composite tree can be a complicated operation. So you should be aware that the design pattern doesn't tell you anything how you are. You are creating your tree.
