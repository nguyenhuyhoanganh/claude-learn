Now let's look at a summary of composite design pattern. So whenever we have a parent child relationship or a street tree data structure of objects in our application, then we can use the composite design pattern in order to simplify dealing with such a data structure. Goal of this pattern is to simplify the client code, which works with these data structures, and it simplifies it by allowing the client code to treat our composite object and leaf object in the similar way. So client doesn't need to worry what type of object it is currently dealing with. Now composites, which are the classes in our tree structure which have children in them, they will delegate the operation to the leaf node or children which are present in them, and the leaf nodes are the ones which actually provide the implementation or which provide the functionality which is needed by the client.
