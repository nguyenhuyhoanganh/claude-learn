# State - Pitfalls

All right. Now let's learn about some pitfalls of this design pattern. The first pitfall is very simple that we may end up writing a lot of classes depending upon the possible state values that our object can take. And all those classes will need unit testing as well. So it adds a lot of work for us when we are using this design pattern with a object which can have a lot of different state values. Second pitfall is that state transitions can become a little bit tricky to implement and this becomes more apparent and adds more complexity if there are multiple possible states to which we can transition from our current state. So we have seen example where we were transitioning from one state to another and there was only one possibility where our order state can go. But if we have complicated state machine to implement, then this transition can become quite complicated. And if our transition state transition is triggered by states themselves, then a single state ends up knowing about a lot of other states. Now, we may not realize all the possible values of our state when we are beginning our state design. That means the classes that we are going to write may need modification as our design and implementation evolves over the time. All right. So these are some of the pitfalls of this design pattern.
