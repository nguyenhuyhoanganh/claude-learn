# State - Summary

All right, now let's see a summary of state design pattern. So if we have an object whose behavior is completely tied up to its internal state and that state can be represented as an object, then we can use state design pattern. So in this design pattern, each possible state value becomes a separate class. And inside that class we define methods or implement behavior which is specific to that state value. Our main object, which is also called as context delegates. The actual operation to whatever its current state object is. And states will implement that behavior which is specific to that particular state value. The context objects. State change is also an explicit operation. Now we are aware that our object state is changing from one state to another, and this is possible because each state is a separate object. So we have this clear distinction about what state our object is currently in. Now state transitions that is moving from one state value to the next one is handled either by states themselves, so they will decide when to navigate or when to move to the next state. Or we can have our context object implement that state transition. Now one benefit of this pattern is that we can reuse our state objects if our state is not using any instance variables, and it is only providing the behavior which is specific to that state value. We can create our state object once and use it in every instance of our context object. Now let's look at a human diagram of state design pattern. So to begin with, we have here context, and this represents our object to which our client can interact with. So client will call operations on this context object. State of this object is now represented using the state design pattern, so it has an object inside of it which represents its current state. Now that object is an implementation of state interface. This interface defines operations which can provide a behavior which is specific to current state. And that interface is used by our context object. Then we can have one or more concrete implementations for the state interface. Each implementation of state represents a possible value that our context state can have inside of these concrete state implementations. We are going to provide the implementation for the behavior which is specific to that value. So at any given time, our context is going to have an object of any one of these implementations inside of it, and that object is going to represent our state. Now let's look at a typical implementation of this design pattern. So here we are going to take an example of order processing application. So we have an object called as order, which can exist in multiple states and we have an option called as cancel. So client code can request the cancellation of order and how that cancellation is handled or how the cancellation happens depends on our state of order object. To implement this, we create an interface which is our abstract state, and this interface defines operations which will be called by our context object. Then we will have one or more implementations of this interface. Each implementation represents a possible state value that our context can have. So our order can be in a state called as in transit. So we have got our payment of the order and now we have shipped those items, but they are still in transit. So this class in transit represents that particular state value. Each implementation of state interface will provide the behavior which is specific to this state. So in this example, since we are in transit, we are contacting courier service to stop the shipment and return objects or return items to our warehouse. And we are also going to contact the payment gateway to roll back the payment, which is done by user. All right. And it is going to return some cancellation fee that is specific to this state. If our order was in a different state, let's say it was in the delivered state, then we will have a different workflow when handling cancellation of our order. All right. So this is how we can implement state value specific behavior in these classes. Lastly, we have the actual context class. So this is our order and you will see that we have a field here called a state, and this represents the current state of our object. So if client code calls, cancel operation or cancel a method on this order object, it is going to delegate that operation to the current state object. All right. So we are going to call state dot handle cancellation and we have to handle the state transition as well. So there are a couple of ways. One way is that state themselves can decide when they will progress to the next state value or our context object can decide when its state changes to a new object. So in this example, we are handling state transition in our context object. So our order is deciding when to progress to next state. So we have a method here called as payment successful. And this calls a method which we have written called as processed. And this method triggers the next state or transition to next state. All right. So this is how we can implement state design pattern in Java.
