# State - Introduction

All right. It's time to learn about state design pattern. And it's very interesting design pattern, actually. And once you learn all about this design pattern, you will have a new way to extend your existing implementations. All right, so what is this design pattern? So this design pattern allows our objects to behave differently based on whatever its current internal state is. So you may be wondering, isn't this a very generic statement? Aren't all objects supposed to behave differently if we change their state? Well, the difference here in this design pattern is that it allows us to define that specific behavior which is specific to a state value in a separate class. In our regular classes, we would have some, if else, condition or some switch case which checks our internal state and behaves differently. Using this design pattern, however, we are going to implement that state specific behavior in a separate class. So if we have a requirement that, let's say our state has value A and we want to behave differently, and if the state value is B, we want to behave differently, we are going to create two new classes and we're going to have objects of those classes in our original object. So whenever an operation is called on our main class, it is going to delegate that operation to whatever its current state object is. Now state transitions. So how should we move from state one to state two? That is called a state transition and it can be triggered or it can be carried out by states themselves. So states will decide based on some input that they get when an operation is called on them, that it's time to switch to a newer state or a different state. Or we can have our main class, which has this state object, decide when to transition to a newer state. Now, a benefit of this pattern is that we can add new states and thus new behaviors for our object without changing our main class. So we will be creating new classes which will represent behavior that is specific to a new state value. All right. So we can keep extending the behavior or keep adding to the object's behavior without modifying the original class. Now let's look at the UML diagram of this design pattern. So we have here a context object. So this represents our main class or class, which client interacts with, and this defines some operations which our client calls and expects some functionality out of that. Now, this context has inside of it a state object, and that represents the state of this particular object. And it defines the state, defines the actual behavior when operation method is called. State themselves extend or implement a common interface. So here we have an interface called as state. So state defines operations which are called by our context. We can have one or more implementations of this state and each concrete implementation will implement that behavior, which is expected by context, which is specific to this particular value. So one thing that you should remember is that these concrete classes represent specific values that our state can take. Right. So if your state is, let's say, an integer, then these concrete values could represent concrete. States would represent values like one, two, three, so on and so forth. So important point is that these implementation or these classes, concrete state classes, represent values and behavior of this context that is specific to that particular value.
