# State - Implementation Steps

All right. Now it's time to look at some typical stamps that you would follow while implementing state design pattern. So our first job is to find out the distinct values of state in which our object can exist. So, for example, if you have, let's say, an order object, then your order can be in a new state, a paid state in transit, state and delivered state. So you identify the values for the state in which our object can exist and each state value becomes a separate class. And these classes will define implementation or define behavior which is specific to that value. So in our order example, we will have a different behavior for order cancellation. So we a paid order has a different workflow when it is canceled and a new order has a different workflow when it is canceled. So we are defining these specific behavior in each of those classes. Next up, in our main or context class, we are going to define some methods which are actually used by client. So our states are not exposed to the client. So client is not aware what is the current state of the object? It is simply going to call the operations or methods on our main class or context class. And these methods will simply delegate that operation to whatever current state object they have within it. Then we have to decide how our state transition is going to happen. So to state transition means progressing or moving from one state value to a next state value. So in our order example, we have, let's say our order is at New State, then we have to decide how that order moves from new state to paid state, whether the state themselves decide when it's time to move to the next state, or we can also take that decision in our context class. So we have to decide and implement that state transition in our code. Now, client is always going to interact with our main class and context class and it is unaware of existence of this state. And that is what gives this design pattern a power because we can keep adding new states and thus new behaviors. And our client doesn't need to change because a new state is added. All right. So that is the benefit of this state design pattern. And this is how we can start implementing this design pattern in Java.
