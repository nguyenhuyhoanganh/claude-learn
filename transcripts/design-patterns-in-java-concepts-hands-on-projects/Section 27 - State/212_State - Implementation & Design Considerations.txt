# State - Implementation & Design Considerations

All right, Now let's discuss some implementation and design considerations when working with state design pattern. So first point is that you can ask the client code to initialize your context with a specific state value. In some scenarios, it may be necessary that client is in a better position to tell our context that you should begin from this particular state. However, after that, the state or the context is going to decide what is the next state. So that is one way that you can initialize your context with a specific state value. Now if state transitions are done by state objects themselves. So if state implementation decides that it's time to move on to the next state, then it has to know about at least one state because that is how it is going to create the object of that new state class and set it into our context. But if we have this kind of implementation, then adding new states might require changing one or more state class implementations because our state class needs to know that this is a new state and we also need to provide some criteria in those state implementations that if this is the scenario or this is the input, then use this new class. Okay. So doing state transition or triggering state transitions from states themselves can require some additional changes if we add new states. Now, let's look at some design considerations. One thing you will notice that our state objects can be shared because they are defining behavior for that particular state value, which that class represents. All the other details that are needed by this behavior. For example, in our order example, we saw that we can keep our order number, the items in that order, some shipping discounts and all that in our context object itself. So our state can be easily shared between different objects of our context class. And to do that, we can use a flyweight design pattern. Right now, state design pattern is not the same as state machine. Okay, exactly. It is not. You can say that it is an object oriented implementation of state machine. State machine can be defined in loose terms as state transitions based on some input values, and they use some form of table data structure or to map those input values to different states. A state design pattern focuses on providing a behavior that is specific to a state value of that context object. So state machines only define the transitions. So they define that if these two values or if these two variables have these values, then this is the new state. State design pattern defines the behavior that if your state is X, then your behavior is so and so. If your state is Y, then your behavior is something different. So this is the difference between state design pattern and state machines. Again, I will say that you can call state machine or state design pattern as an object oriented implementation of state machine.
