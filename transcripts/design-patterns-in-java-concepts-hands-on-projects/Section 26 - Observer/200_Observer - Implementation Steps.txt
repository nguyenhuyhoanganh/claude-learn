# Observer - Implementation Steps

All right. Now let's look at a sequence of steps that you would typically follow while implementing observer design pattern. So we start by creating an interface that will be implemented by all of our listeners. And this is the observer interface. It's usually a very simple interface with a method which is called by our observable or subject to notify listeners that its state has changed. Next up, we can create an interface for our subject or observable. And this is needed only if we have listeners that are subscribed to multiple objects. However, in our if in our system there is only one object which is going to send the notifications, then we don't need this interface. We can directly use a concrete class and provide our functionality there. Implementing this subject or over observable means taking care of the registration and deregistration of observers. This is also called as attaching an observer or detaching an observer. Next, we need to make sure that we are notifying all the registered observers whenever our state changes and providing methods which can be used by our listeners to query state information. Okay. And typically you will find that the observables will have some sort of collection inside them, which is used to keep track of all the listeners which are subscribed to us. No concrete observers use a reference that is passed to them when our observable calls them to tell that they have a state change. Okay. However, if we are actually passing the changed state value in the notification, then they won't need to call back. They will get the value in the method itself as a parameter and use that value to do further processing. So depending upon our implementation, concrete observers will depend or will rely on this interface. On this reference that we pass in the notification method.
