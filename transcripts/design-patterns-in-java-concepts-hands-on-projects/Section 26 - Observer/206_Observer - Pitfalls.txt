# Observer - Pitfalls

All right. Now let's look at some pitfalls of this observer design pattern. So if we have every setter method triggering an update or sending a notification to all the registered observers, then it can be too much. If we have a client which is setting properties on our observable one after another. So this is a pitfall because we now have no control on how many notifications are being sent, because if anybody calls a setter, we're going to send a notification. We have no control in the observable on how many notifications are being sent. Second pitfall is that each update becomes expensive as the number of observers increase. And if we have one or more slow observers in that list, then each set recall on that observable can be a costly operation. The last pitfall is that in typical implementation of this design pattern, observers call back the subject or our observable to find out what change has happened in the object state. And this can add up to quite a bit of overhead if we have number of observers very high and number of notifications are also very high. Then we have this lot of chatter that is happening each time somebody changes a state of our observable. So all these pitfalls need to be taken into considerations and we have discussed some solutions in our implementation and design consideration sections so you can implement one of those suggestions to counter some of these pitfalls.
