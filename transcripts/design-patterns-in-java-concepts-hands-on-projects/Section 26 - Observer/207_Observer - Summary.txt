# Observer - Summary

All right, Now let's look at the summary of observer design pattern. Now, observer pattern allows us to define one to many dependency between objects where many objects are interested in finding out about the state change of a single object. So these observers will register themselves with the subject or the observable, as it is sometimes called. And whenever a state of that subject or observable is changed, it is going to send out a notification to all the registered observers. Now in the notification that is sent out, it is quite common practice. That subject is going to only send a reference to itself as an argument to these method calls. So that means if observers, if any of the observer wants to find out the actual value of a state variable, he needs to call back our subject to find out that value. We can also register observers for specific event only. So instead of having a generic single register method, we will provide register methods that correspond to different state changes. And using this approach, we can improve the performance of sending out the notification whenever a state change occurs. So if a state change is isolated to a particular variable, we would have observers listening for that particular state variable. So we are going to send notification only to those observers. Now, this design pattern is also called large Publisher Subscriber Design pattern. Now Java messaging or Jmz uses this pattern. But there instead of registering with the subject, we register with JMS Broker, which acts as a middleman between the single sender and multiple listeners. Now let's look at the UML diagram of this design pattern. So first of all, we have a subject interface. This is also called as observable, and it defines methods for registering and de-registering observers which are interested in finding out the state changes in the state of this subject. A concrete subject will actually have the state because our subject was an interface. So a concrete subject implements this interface and it also has a state. All right. And it will also maintain the list of registered observers who have called the attach method on it. And if any of its state value changes, it's going to use those list of registered observers and call their update method. So we have this observer interface which is implemented by all of the listeners in our system. This method defines a single method called as updated. And as we discussed, it is typical to send only the reference of the subject in the method call. So the concrete observers will implement this observer interface and they will either have a stored reference for the concrete subject or they will use the reference that is passed in the update method to call back concrete subject to find out any actual state value that they are interested in. All right, so this is how we can implement our observer design pattern. Now let's look at an implementation of a typical observer design pattern. So first of all, we have here a subject. So instead of an interface, we directly have a concrete class which acts as a subject in our observer design pattern. You will see that it provides methods for attaching or registering observers and detaching or removing already registered observers. It maintains a list of observers inside it. And whenever a property changes, for example, here we have a method called as Add item which results in change to its state. It's going to call all the register observers and call their updated method. So we have the order observer interface here, which is what we accept in our attach and detach methods. So this interface has a single method called as updated, which is called whenever our order changes and it has a single parameter of order. Then we have our concrete observers. So here we have a price observer which implements our order observer interface and it has implementation for the updated method. Now this observer is only interested in finding out changes in the price total price of this order. So whenever it receives an update notification, it's going to call the get item cost method on the order and based on the state value, it will take some actions on it. Now, there is one thing that is still remaining. It is how to attach our observers with the subject. So whenever we create an instance of our observer, we would have our subject instance ready and we can call the attach method and pass the created observer in the attach method. So using this code, we are going to attach our observers to this subject. And when somebody calls a method like Add item, it's going to result in update notification sent to all the registered observers. All right. So this is the observer design pattern in Java.
