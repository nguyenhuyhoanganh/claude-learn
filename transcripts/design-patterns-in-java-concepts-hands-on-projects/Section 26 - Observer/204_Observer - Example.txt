# Observer - Example

All right, Now let's look at some real world examples of observer. And this is similar to proxy pattern. You will find no shortage of real world examples of this design pattern. In fact, this is such a useful pattern that Java itself has support for this design pattern. We have observer interface which is part of Java util package and observer observable class in the same package which support basically this observer design pattern. Although I haven't seen people use these classes outside of some swing applications for some reason. Anyways. Another example of this design pattern. A great example can be found in the Java servlet applications. So in a web application where we use Java Servlets, we can create multiple listeners. So you might have worked, you must have worked with a couple of these. We have Http session listener, we have servlet request listener and there are multiple other listeners as well. I haven't listed all those so we can implement these interfaces in a particular class and then we can register those classes with servlet context. So there is a Add listener method on the servlet context where we can pass an object of these classes. So these listeners are notified when certain event occurs. So if we, if a request comes onto our web application, then a request object is created. So our servlet request listener will be notified of that. If somebody adds a new value to our Http session, then our Http session listener will be called. So this is an example of observer design pattern. One point I would like to highlight here is that in this particular implementation of servlet context and these listeners notification is sent based on the type of event that has happened and the interfaces that are implemented by our classes. So this is a way where servlet context has optimized its observer design pattern implementation. Okay. So if we have implemented Http session listener interface, then only our observer is notified of changes to Http session. If we have implemented the servlet request listener interface, then only we will be notified when a request is created and destroyed. Okay, so this is how the servlet context is optimizing its observer implementation so that it doesn't notify all objects or all observers for each event that happens in our web application. All right. Another example of observer can be found in spring framework, where we can implement the application listener interface to get notifications from spring beans.
