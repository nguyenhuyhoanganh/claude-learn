# Observer - Implementation & Design Considerations

All right, Now let's discuss some implementation and design considerations when working with observer design. So in some rare scenarios, you may end up with circular update loop. So what happens in this situation is that an update is done to our observables state, so it sends out a notification. That notification is received by an observer, which takes some action because of that notification. If that action results in update to our observable again. Then it is again going to send notification. And if the observer takes the exact same action again, which results in update to our observable, we are again going to get a notification and again we are going to take the same action. So you will see that we have an infinite loop here, the circular update loop. So always be on the lookout for this kind of problem, especially if you are implementing this design pattern where our observer calls on to some other objects which may indirectly change our observables state. The next point is that an observer object can listen for changes in multiple subjects or observables. Because this whole design pattern works on interfaces, so we have observer interface and observable or subject interface. In the observer interface, we have this single method updated and we are passing the observable or subject as an argument in that method. So while implementing this method, our observer can check which object has changed and depending on the type of object, it can take some different action. So our single observer is capable of listening to state changes in multiple objects. Last implementation point is that you should keep an eye on the performance of using our observer design pattern. And this is especially true if you are expecting high number of observers being registered to a single object, and if 1 or 2 of those observers are slow. That means if they take a lot of time while processing the notification, then it can break our entire system because it's going to result in pile up of pending notifications, or we can even miss some notifications. So if we are using some kind of multithreading or blocking queue. Then the problem increases in the complexity. So always keep your observer methods short and simple and as much as possible do not do heavy calculations in those methods. Now let's look at some design considerations. So one enhancement that we can do to this design pattern is to enhance this design pattern so that we reduce the number of notifications that are sent on each state. Update. So what we have seen so far in the UML diagram, as well as the example that we tried out, we had this single attach method where every observer was registering with our subject or observable, so there was no way to find out what this observer is interested in because in real life scenario you will find that we write observers which are interested in particular property of that object. So in our hands on example, we had Price Observer, which was only interested if the price or item cost changed in our order. We had a quantity observer which was only interested in finding out if the quantity of our order changed. So what we can do is that we can instead of providing a single attach method, we can define multiple methods and each method will correspond to a particular property in our object state. Or it can describe a particular event that happens on our object. So that way we have a mechanism to find out that, okay, if we have a registration property for a registration method for this particular property, all observers which are registering there will be notified only when that particular property changes. Okay. So this is some little enhancement that we can do in our observer implementation or observer design to reduce the number of notifications that are sent out. Of course, this is going to increase the complexity of our observable, but you get a good benefit out of that. Now you will find that we typically send notifications from our observable when someone changes its state and you will know that a good place to do that is basically our setter methods because that is a method that is used by external code when it wants to change state of our observable. But instead of writing notifications, instead of writing the code, which sends out notifications in each of these setter methods, what we can do is we can put that code in the in the client code, which is actually calling these setters. So, for example, let's say you have a piece of code that sets ten properties on our observable. In typical implementation, what will happen is that since it is calling ten setter methods, we are going to send out notification ten times because we have this code which sends out notification in each setter. What happens? What if we took that code which sends out notification and put that in the code which is actually calling those ten setter methods? So now we have a single notification that is sent at the end when all property updates are done. So what we can do is instead of sending notification ten times, we have now reduced that count to one. However, this is not that easy to implement because there are multiple places from where our observable can be modified. And additionally, we are taking the responsibility of sending notifications from our observable and giving it to our client code. And it can very easily forget to call or send out those notifications. However, in some special scenarios you can keep this point in your back of your mind that you can have this kind of change in the implementation of this design.
