# Chain of Responsibility - Summary

This is a summary of chain of responsibility design pattern. So whenever we want to decouple the center of a request or code, which sends requests to the object which actually handles the request, then we can use chain of responsibility design pattern. So this design pattern completely separates the code, which handles the requests from the code which actually is responsible for sending that request. Now we want to do this decoupling because it allows us to provide multiple objects chance to handle that request. Without chain of responsibility, we would have an object which is hardcoded in our code and we would be calling a fixed method on that particular object. So we are coupling the exact object that we are using. If we are not using chain of responsibility, if we use this design pattern. However, we don't have this coupling between the code which uses our handlers and the handlers object themselves. Additionally, we can also add multiple objects in our chain so that we don't have to know all these objects in advance. So that is an additional benefit of using chain of responsibility. Now a handler in our chain checks, if it can handle the request, if it cannot handle, then it will pass on the request to the next object in the chain. Now, we can still pass the request down the chain, even if our handler is able to process or work on that request. In fact, the filter chain, which is implemented in Java Servlets, follows this principle. There is nothing in this design pattern that says that you cannot pass the object if you are able to process the request. Okay, so you have the flexibility of designing or flexibility to choose what happens if a request is handled in one of the handlers. Now let's look at the UML diagram of this chain of responsibility design pattern. It's a pretty simple design pattern to implement. We have a handler interface here. It can be a base class, it can be a concrete class or an abstract class as well. Handler defines a method that is used by our client. Now this is the method which requests handles our request. Now, request can be a single object or single parameter that is defined in this method, or it could be multiple parameters as your handlers require. Handler can also optionally provide an operation that allows you to get access to the next object in the chain. Okay, so the next object is typically called as a successor. Now we have more than one implementation of this handler. Now each implementation provides a different type of functionality. While handling this incoming request, for example, we can have an authentication handler which performs authentication on incoming requests. We could have a validation handler that performs validation on the incoming object, so on and so forth. So we have these multiple implementations and we are going to create objects of these implementation and then we are going to chain these objects with one another. So we will give a reference of the next handler to one of the handler objects and create or build our chain. The client needs to know only about the first object in our chain and it will give the request to that first object. And then that object will decide what happens if it is able to handle the request or not. Typically, if the object is not, Handler is not able to handle that request or process that request, it will give it on to the next handler in the chain. All right.
