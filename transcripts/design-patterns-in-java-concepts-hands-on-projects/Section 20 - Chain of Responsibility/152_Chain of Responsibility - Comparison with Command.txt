# Chain of Responsibility - Comparison with Command

Now let's compare chain of responsibility with command design pattern. So in chain of responsibility, if Handler can't handle the request, it will pass it on to the next handler. So it's not mandatory for every handler handler to handle or process every incoming request. That's not mandatory. If it cannot handle it can pass it on to the next one with command. However, there is no chance to pass on the request to a new object or a different object. The command has to handle the request itself. With chain of responsibility. There is no guarantee that the request will be handled or some action will be performed as a result of that request. There is no guarantee built into the design pattern. In fact, the design pattern makes you explicitly aware that your request can go unhandled through entire chain. With command design pattern. It is assumed or it is assured that the command will be executed and the request will be handled, which represents that command. Now we don't track the handlers which handled our request. Okay, so there is no provision in design pattern or the design pattern doesn't lend itself easily to track which handler object processed our request. Okay. And so that means there is no way that we can reverse the actions of handler. There is no provision in the design pattern for that. On the in the command design pattern, however, they are trackable because commands themselves are distinct objects. So if you run same command twice on a request, you will have two command objects. So you can easily track these objects in some data structure like linked list and using that you can implement the undo functionality or you can reverse the effects that these commands had on your request object. All right. So these are the differences between chain of responsibility and command design pattern.
