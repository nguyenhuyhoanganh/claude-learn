# Chain of Responsibility - Introduction

All right, Now let's look at chain of responsibility design pattern. And the this is again, a behavioral design pattern. And design patterns in this category are quite interesting. And you will find yourself using the using these over and over again in your development. So what exactly is chain of responsibility? So whenever we need to avoid coupling between the code which sends some request for functionality and the code which handles or provides that functionality, then we can use the chain of responsibility. Okay, so it reduces the coupling between these two parts of our code. So what happens is typically the code which wants some request handler or which wants to call some method, it knows the exact method it needs to call and it knows the exact object which is going to provide that functionality. Okay. So this is why there is a tight coupling. So we get hold of an object and we call a method on that. So now our code is tightly coupled to that particular object. Now chain of responsibility solves this problem by giving more than one object chance to process the request so your code doesn't know which object is actually providing the functionality. You just know the first object in the chain and you send your request through that chain and anybody in the chain can handle and provide the functionality you desire. So we create objects which are chained together. Okay, so the chain is created by one object knowing which is the next object in the chain, okay? And that object knows which is the next one. And in this way we can create objects which are chained together and we give our requests to the first object in our chain. And if that object can handle that request, then it's fine. If not, then it can pass on the request to the next object in the chain. Okay. And one side effect of this particular design pattern is that there is no guarantee that your request will be handled. Okay. So your request could fall off at the end of chain. And this is the chain of responsibility design pattern. Now let's look at the UML diagram for this design pattern. It's quite a simple design pattern to implement. So first of all, we have a handler. Okay, so handler is an interface that is defining the functionality or a method to provide a functionality that our client needs. Okay. Now this handler interface can also provide a method that allows the implementer to chain or maintain a chain to the next object in link. Okay, so this is called as a successor. So a handler implementation will have a reference to the next object in this chain which is called as successor. Then we have a concrete handler so you can have more than one handler or one implementation of this handler and they will provide their own implementation in the handle request. So this method generally defines a generic functionality like process request or handle request. So these are the common names that are given to these methods. And what happens inside that method is decided by the implementation. Okay, so you may have a authentication handler and its handle request implementation will perform authentication on the incoming object. You will have a another implementation called as a format handler and that may format the content of the object or content of a string which is inside our object. Okay, so what happens in the handler request is decided by the implementation that we have provided. Okay, so we have more than 1 or 1 or more implementation of our handler and these handlers know about the next object in the chain. So our client uses the first object in our chain. So we give our client only the first object, and client will send its request to that first object, and the request will keep on traversing the chain unless until it is handled by one of the implementations. All right. So this is how a chain of responsibility design pattern is implemented.
