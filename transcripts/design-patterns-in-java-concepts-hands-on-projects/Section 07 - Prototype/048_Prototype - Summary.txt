# Prototype - Summary

Now let's look at summary of prototype design pattern. You will use prototype design pattern whenever creating a new instance of a class is a costly operation, or in some cases it may not be possible because the object is directly supplied to your code. In Java we implement this pattern using the clone method, which is defined in the object class. So we override that method in our class and provide the implementation to return copy of our existing object. Now objects which have majority of their state as immutable are good candidates for prototype. Having a large mutable state makes the cloning operation that much costly and difficult to implement. Now, when implementing clone method, you have to pay attention whether you need a deep copy or a shallow copy of your prototypes state. Also, we have to ensure that the clone that we are returning from our clone method is reset or initialized with appropriate state values so that the code that is using our cloned copy doesn't get affected by leftover values for state. Now this is the UML diagram of our prototype design pattern. There is nothing to it. We will have a prototype class which will implement or provide the clone method. Now clone method is provided by overriding it from the object class. In case your prototype is an abstract class, you will have more than one concrete prototype classes. Our client will directly use the clone method in order to create copies of our prototype. Now this is a Java implementation for our prototype design pattern. So here we have a game unit class, which is a prototype. Now this class implements Clonable. Now this is a marker interface that tells the outside world that our game unit supports the cloning operation. The clone method itself is overridden because it is provided in object class. Inside this clone method, we will make copy of our object by doing either a shallow copy which is done by the clone method in the object class. We will initialize the copy with correct default values and return that copy so that the outside code can use it. We will also provide the clone not supported exception in our throws clause so that a subclass of this prototype can decide whether it wants to support the cloning or not. All right. So this is all about the prototype design pattern.
