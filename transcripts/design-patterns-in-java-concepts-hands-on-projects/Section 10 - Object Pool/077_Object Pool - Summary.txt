# Object Pool - Summary

Now let's look at the summary of object pool design pattern. Now whenever you have a class and creating objects of that class is a costly operation and you need many such objects in your application. But each object is used for only a short duration. Then you can use an object pool. Typically objects in your application which represent an external resource like a thread, a class that represents a connection to an external resource like database or VMs or a class that represents something like socket. These are good candidates to be used or to be pooled in an object pool. Now let's look at the UML for the object pool design pattern. It's quite simple. We have an abstract reusable class, so this is the interface that is used by the client. And this interface defines the operations that can be performed on that particular object. Then we have concrete reusable. So this is the implementation of our abstract reusable class. This class has its own state and it implements the operations that are defined by the interface. It also provides an operation called as reset. Now, this is the operation that resets the state of this object into its default value. And this operation reset operation is called by the object pool so that whenever it returns this object to the client, it can be used without any problems. Next, we have our object pool class. Now this class will be creating or caching objects of our concrete reusable class. There are only two methods that are really important here. One method is called as get method, and this method will return a pool object from our internal cache. Then there is another method which is called by the client when it wants to release an unused object. All right. So these are the two methods which are used by our client to get the pool objects in and out from this class. Now, the objects that we are going to pool, they should provide this reset method that we just discussed. Now, one point to remember is that this operation should be efficient. Otherwise, the release call which the client is making is going to take a lot of time and it's going to add to the performance cost. Now, the pool must handle synchronization issues effectively so that your pool can be used by realizing some performance gains. Now, the reset operation that we just discussed, it should be called or it should be invoked before we add that object into the pool. Now the client code has a responsibility. Responsibility that it must release unused objects back into the pool. And a failure to do that on the client code part can break your entire system because you may be leaking resources because of that. Now, some classes like threads can handle these situations on their own because a thread can determine when its work is done and so it can return itself back into the pool. But many other classes don't have this luxury, and client has the responsibility to responsibility to explicitly release these objects back into the pool. Now, object pools are difficult to optimize because they are very sensitive to system load, which they face at runtime, so you should keep that in mind. Pools are a good choice when the object that is being pooled represents a fixed quantity of externally available resource, like, for example, thread or connection or the sockets that we can open. Lastly, if you create objects when the pool is empty, then we have to also write code that makes sure that we are trimming our pool back to a reasonable size.
