# Proxy - Introduction

All right. It's time to learn about proxy design pattern, and chances are very high. You have already used proxy in this video. We're going to see formally what a proxy design pattern is and how we can create a proxy of our own. So whenever we need a stand in for a real object, a placeholder object which can work in place of a real object, we can use proxy. And this particular behavior or this particular characteristic of proxy makes it suitable for wide ranging applications. And here are some of the important applications of proxy design pattern. We can have a protection proxy, so we'll have a proxy and that will control which operations can be called on the real object. Then we have a remote proxy. So we have a proxy object and we call method on that proxy object as if it was a real regular object. And behind the scenes, that proxy can handle the actual network communication with remote server. Then we have a virtual proxy which acts as a stand in for a real object, and that proxy can be assigned to references that can be added in a collection, and we can delay construction of the real object until it is absolutely necessary. Okay, so these are just some of the applications where you can use proxy design pattern. Now, one of the characteristics of proxy is that client is completely unaware of existence of a proxy. It works transparently. Client doesn't need to call any method to convert a proxy to real object. Everything happens transparently. Now, some examples of proxy design pattern in real life are lazy loading of collections by Hibernate AOP based method level security that is provided by Spring framework. Various frameworks that allow you to make web service stubs or optimized stubs. All of these are using proxy design pattern. All right, so now it's time to see what a UML diagram of proxy design pattern looks like. It's quite a simple design pattern. So to begin with, we have a subject. So this is an interface which is used by our client. Okay? There is nothing to it. It has some operations defined in it and which are implemented by the real object. Okay. So the real object is an actual object, actual class which implements these operations and we can create object of that class. Then we have proxy. So proxy also implements the exact same interface which is implemented by the real object, and it maintains a reference to the real object in order to provide the actual functionality. So proxy will work around that real object in order to provide its various features like authentication, checks, network, call all of that. All of those things are handled by proxy. Now, one point I want you to highlight right away is that not all proxies need a reference to real object. And the example that I used a moment before, which is the remote proxy, it simply works as an object and its method actually calls on to a remote server. Okay, so not all proxies need a reference to real object. They can work without one. All right, so this is the proxy design pattern. Next, we are going to see how we can implement this design pattern in Java.
