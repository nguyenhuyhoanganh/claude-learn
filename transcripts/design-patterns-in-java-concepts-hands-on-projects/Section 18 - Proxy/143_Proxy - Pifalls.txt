# Proxy - Pifalls

All right, Now let's look at pitfalls of proxy design pattern. The first pitfall is that the dynamic proxy feature, which is provided by Java, only works if your class is implementing one or more interfaces because Java creates a dynamic proxy by implementing one of these interfaces. So if your class is not implementing any interface, then it becomes impossible to create a dynamic proxy using Java's API. The next footfall is that if you have a requirement to carry out multiple things like auditing authentication, use a proxy as a standing object. And if you want to do all of those things on a same instance of a class, then it is better to do that in a single proxy instead of creating multiple proxies and then having to chain them together or using recursive composition. That becomes quite difficult to manage. And the reason for that is some proxies create the actual real object on their own, and it is decided by that proxy when the object is created. So if you have multiple such proxies in that chain or composition, then you may end up with multiple instances. So it's always a good idea to implement all of those things in a single proxy so that you have one single proxy for one instance of your class juggling multiple instances or juggling multiple proxies which work on the same instance is a lot of trouble. Now the static proxies. The static proxy design pattern that we saw where we wrote our own proxy class, implemented the interface and provided the functionality which is needed out of our proxy. This design pattern looks quite similar to decorator or adapter pattern, so if you are doing a code review, it becomes quite difficult. What exactly is happening here or which design pattern is used and it becomes more difficult if you don't already know what decorator adapter and proxy design patterns are. All right. So these are some of the pitfalls of proxy design pattern.
