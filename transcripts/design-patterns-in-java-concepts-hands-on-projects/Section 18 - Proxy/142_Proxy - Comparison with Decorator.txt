# Proxy - Comparison with Decorator

All right, Now let's compare the proxy design pattern with another structural design pattern called as decorator. Now, these two design patterns are often confused, especially static. Proxy and decorator look the same in the code, but they are actually quite different. So let's look at the differences between these two. So depending upon the proxy that we are implementing, we do not need access to real object all the time. For example, if you are creating a virtual proxy, which is a stand in for real object, or if we are creating a remote proxy, we don't need real object all the time. A decorator, on the other hand, always needs the real object, so it needs an object which it can wrap and provide functionality on top of that provided by the real object. The purpose of proxy design pattern is to provide features like access control, lazy loading, auditing. So we want to provide features which are not necessarily used by the client. So we are adding handling some cross-cutting concerns as you can see, like the access control auditing, these are the concerns which can be addressed or which can be provided using the proxy design pattern. A decorator, on the other hand, is mean to add a functionality which is visible and usable by the client. Okay, so this is the functionality which is which is needed by the client code. All right. So these are some of the differences between proxy design pattern and decorator.
