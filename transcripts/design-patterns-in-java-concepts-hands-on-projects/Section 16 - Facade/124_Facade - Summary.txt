# Facade - Summary

Now let's look at a summary of facade design pattern. So we will use facade design pattern whenever using our subsystem requires dealing with lot of classes and interfaces by the client code. So this code now becomes tightly coupled with these classes, which are part of our subsystem. Using facade design pattern, we can provide a simple interface which client can use, and that particular interface will provide the exact same functionality, or it will satisfy the use case which is needed by the client code. Now, facade is not simply a method forwarding okay facade methods. They encapsulate the interactions that are needed to achieve a functionality or to achieve a task, and they encapsulate that interaction so that the client code doesn't have to do those in its own. All right. So that's the purpose of facade. It will not simply do a method forwarding as we see in adapter now, facades are often used whenever there is a legacy code and we have a new system that is going to use that legacy code to avoid the coupling of our newer code. With the legacy system, we add a facade over the legacy code and that way our new code doesn't become tightly coupled with that legacy code, which can be improved or replaced in future releases. Now let's look at the UML diagram for the facade design pattern. It's quite a simple design pattern. Here we have a subsystem that has multiple packages, multiple classes and interfaces which are needed for getting a functionality out of this subsystem. Then we have our facade, which can be a simple class which defines a simple operation which encapsulates the interaction that is needed with these classes from our subsystem. All right, so this is the UML for facade design pattern. Now let's look at the implementation of a typical facade pattern. So here we have a new class called as email facade, and this is a facade over our email subsystem. It has defined a simple operation called as send order email, which accepts an order object and returns whether the email was sent successfully or not. Now you will see that this method, which is defined in facade, encapsulates all the interactions that are needed in order to send email. So it will interact with classes like factories. It will deal with classes like template stationary class. It will work with the mailer, which actually sends the email. So using the email facade, the client will only know about our single method center order email. It won't get coupled with these classes that are needed in order to achieve that functionality. All right, so this is the facade design pattern.
