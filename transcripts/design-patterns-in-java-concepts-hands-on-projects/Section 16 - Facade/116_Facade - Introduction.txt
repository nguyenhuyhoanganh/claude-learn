# Facade - Introduction

Facade uses structural design pattern, which is easy to understand. It's quite easy to implement and it's of great help, especially if you're working with legacy systems or code that is already present. So let's jump into it and see what facade does and why we need it. So if you have a client code which interacts with large number of classes and interfaces, which are part of a subsystem, and this interaction is because client code needs some functionality that is provided by this particular subsystem. All right. So client can carry out its work. That is fine, But what happens in this process is that the code becomes tightly coupled with our interfaces and classes which are part of our subsystem. So if you want to make any changes to those classes or interfaces, you have to make sure that your client is going to work with those changes. And facade design pattern tries to solve this problem. It tries to reduce the coupling of these classes and interfaces, multiple classes and interfaces in our subsystem with the code which is using that subsystem. So facade provides a very simple and unified interface to our subsystem. So we our subsystem may provide one or more functionalities or it can carry out one or more tasks which our code needs, and we expose that functionality in this single facade, class or facade interface and client can only interact with our facade. He doesn't have to know about all the inner classes and interfaces and it can still get the same result. Now one thing that you should remember is facade is not just a 1 to 1 method forwarding, so you are not going to have a method in the facade which simply going to call onto a particular method in your subsystem. Okay, it is more than that and we are going to see what we mean by that when we are trying out the example of implementing facade design pattern. Now let's see a UML diagram of this facade design pattern. First of all, we have the subsystem here, so this subsystem may comprise of multiple packages, classes and interfaces. These classes may have their own relationship between them. All right, then we have the facade. So facade provides a simple method definition which corresponds to use case which is needed by the client code. And this particular facade will interact with these classes from within the method. All right. So anybody who wants to use functionality of our subsystem will talk to facade and facade will in turn talk with all these inner classes and interfaces that are part of our subsystem. All right. So it's pretty simple design pattern to understand and you will see that it is of great use, especially if you have some legacy code and you want to provide a simplified interface to access functionality there. All right. Next, we are going to see how we can implement facade in our code.
