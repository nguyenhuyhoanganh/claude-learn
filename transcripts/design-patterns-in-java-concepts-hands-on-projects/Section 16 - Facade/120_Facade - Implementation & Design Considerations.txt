# Facade - Implementation & Design Considerations

All right. Now let's look at some of the implementation and design considerations when using the facade design pattern. First of all, the main purpose of facade should be that it should minimize the complexity of using our subsystem. Okay. So that should be your main criteria when you're defining methods and defining interactions that happen between the client code and facade. Next up, you can have an interface or an abstract class as a facade and you can have multiple implementations of that. So client can now use a subclass and that can interact with different subsystem implementations. For example, in our example we had email subsystem, we had a single facade called as email facade. Instead of that, you can have an email system that works, that uses, for example, Microsoft Exchange, another subsystem that uses something else, and you can have a facade as a interface and two implementations of that facade. Okay? So that is one way that you can implement your facade design pattern if your subsystem itself has different implementations. Now, facade is not a replacement for regular uses of your classes that are in your subsystem. Okay. So we saw that client was getting interacting with these classes and there was an issue of tight coupling and we added facade to that, solve that problem. But that doesn't mean our regular classes cannot be used outside of our subsystem. So do not make any assumptions while you are implementing classes in your subsystem that there is going to be a facade and then and through facade, only our client can interact with these classes. And this is true if you are making assumptions regarding the validation that can happen in a facade, do not make any such assumptions right. Your classes in subsystem as if they were they were going to be used as regular classes and then implement facade. All right. Now let's look at the design considerations. When working with facade. The facade is really a great solution if you want to reduce coupling between two different subsystems. All right? Because often you end up with classes in this subsystem which know too much about how the another subsystem is implemented. And facade is a great way to reduce that kind of coupling. Okay, so two systems subsystems can interact through facade and avoid knowing too much about another subsystem. Now main purpose of facade of course, is to provide simplicity of usage of our subsystem. However, your only concern is the tight coupling that happens between the client code and classes of our subsystem. Then you don't even need a facade. You can use another design pattern like Abstract Factory, which solves this problem of tight coupling between client code and classes of our subsystem. Okay, so facades purpose is to reduce the coupling, but it is also important to note that facade also serves to simplify usage of our subsystem so it serves these two important purposes.
