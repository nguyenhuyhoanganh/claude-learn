# Flyweight - Pitfalls

Now let's look at the pitfalls of flyweight design pattern. Now there is a runtime cost that is attached to using this design pattern because of the extrinsic state. So client has to either store the extrinsic state or it has to calculate it every time. It needs to use any operation that is provided by flyweight design pattern. So there is an additional cost when using this design pattern. Second pitfall is actually what makes this not that much usable for us. It is often difficult to find a perfect candidate for flyweight. Lots of objects that you are going to use in Enterprise Java application do not have this clean separation between extrinsic and intrinsic state. Either they are completely immutable or they are completely mutable. Now, one type of application that benefits greatly from flyweight and where I have seen flyweight used quite frequently is the graphical applications like applications that are graphical editors, word processors and video games. They can make use of this design pattern quite easily, and it is one of the most used design pattern over there, because having an intrinsic state which can represent the graphical representation of your object and having an extrinsic state like position location is quite common in those applications. All right. So graphical applications typically make use of flyweight in enterprise applications. It's quite difficult to find a perfect candidate for flyweight design pattern.
