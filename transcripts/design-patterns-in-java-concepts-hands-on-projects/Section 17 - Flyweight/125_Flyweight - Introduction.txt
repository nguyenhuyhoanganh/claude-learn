# Flyweight - Introduction

Now let's learn about flyweight design pattern. Now, this is again a structural design pattern. That means it deals with objects and arrangement of objects. So what problem this design pattern tries to solve? So let's say you are in a situation where you need large number of objects of a particular class. Now, maintaining these many instances of this particular class in memory can be a performance concern for you. Now, memory consumption and memory optimization has an interesting history. I started in late 90s where memory was something of a precious resource and we would use int and bitwise operation in order to save some memory. Then we had Dotnet and Java and suddenly memory was of no concern for us. As long as we don't leak memory, we could use any amount of memory that we would want. Of course, within a limit. But it wasn't the primary concern when designing or implementing applications. Nowadays, again, we have cloud, so we are using third party cloud providers like AWS, Google, Digitalocean And if you look at the price of an instance on any of these cloud providers, which has high memory configuration, you will see that it's pretty easy to lose your shirt there if you are not careful with how much memory your application uses and with advancement of microservices and everybody using microservices now where an instance can be started and stopped dynamically based upon the load on the system, we need to be extra careful about how much memory we are using in our service. All right. So it was a bit of a I went on a tangent there, but coming back to flyweight, what does this design pattern has to offer us? So flyweight allows us to share an object in multiple contexts because as you know, in real life, we rarely have an object which is needed hundreds of times. What we have in real life is hundreds of objects which have different values for their state. So what we do in flyweight is that instead of sharing an entire object, which is not even possible for a lot of scenarios, we divide the object state into parts. So we have an intrinsic state of an object. That means it is a state which doesn't change no matter where the object is used. And then we have another part of state which is called as an extrinsic state. And this state refers to the properties of our object or properties in our object state that are specific to the context or specific to where that object is being used. So once we divide our state in these two parts, what we do is we write a class which has only the intrinsic state, okay? And that way we can create one object and share it everywhere. The extrinsic state, however, is provided by the client code. The code which is actually using our object. It will take care of providing that object with the extrinsic state so we can carry out our operation. Okay. So it can store either store that extrinsic state where needed or it can calculate it each and every time it wants to use our object. Now, in order to get our flyweight objects, we typically provide some form of factory so that the client doesn't need to worry about how the object is created and how it should be shared. We take care of that in our factory. Now let's look at the UML diagram of this design pattern. Okay, so first of all, we have the flyweight, and this defines an interface that is used by the client and it defines a method or an operation. And these operations typically expect an extrinsic state to be passed to it. Okay. So our object cannot perform its operation unless it gets access to that extrinsic state. Then we have the concrete flyweight. So the concrete flyweight is a class which is implemented with only the intrinsic state in it. That means object of this class is a perfect candidate for sharing in any context. It will provide the implementation for the method or operation that client needs and it will use the provided extrinsic state in order to carry out that operation. We can also have unshared flyweight. That means some of subclasses of our flyweight can be classes which cannot be shared. And these classes, they will define the class with all the state that is required by its object. Okay. So these classes will have all the state. That means this will be a typical class that you write if you are not using flyweight. Lastly, we have a flyweight factory. Now this flyweight factory is used by the client and this factory takes care of giving out the shared object. And if it is a unshared flyweight, then it will create a new object of that class and return it to the client. So using this factory, we encapsulate away the knowledge that some of our objects are actually shared or cached and they are reused in multiple contexts. The client will basically compute or store the extrinsic state that is needed in order to call operation on our flyweight class object. All right. So this is a kind of difficult pattern to understand. And this intrinsic state and extrinsic state is something that you have to see for yourself before you can understand what this pattern is doing. So we're going to look at an example and we're going to see how this flyweight pattern works in real life.
