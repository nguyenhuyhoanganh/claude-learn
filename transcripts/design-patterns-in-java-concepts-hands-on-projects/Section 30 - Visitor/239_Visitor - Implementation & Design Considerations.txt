# Visitor - Implementation & Design Considerations

All right, Now let's look at some implementation and design considerations when working with visitor pattern. So the first point is that our visitor can work with objects of classes which do not share a common base class or common interface. Apart from, of course, the object class, which is the base class for all the objects in Java. So apart from that, we do not need to have a common interface like in our implementation. We saw that we have the employee interface and all the roles in our organization were extending or implementing that employee interface. So that is not required or that is not mandatory for visitor pattern to work, because ultimately what happens in a visitor is that somebody passes a visitor object to our object structure through the accept method and inside that accept method implementation. That class is going to call a specific method on the visitor interface. So having a common interface for all those classes is optional. However, the code which is actually passing our visitor object to these object structure, that code needs to be aware of these individual classes. Now often for visitor to work correctly or to provide an effective implementation, it will need internal state of the object that it is visiting. So we may have to provide getter and setter methods for the state of our object. Now let's look at some design considerations for this pattern. So one effect of this pattern is that related functionality is now grouped in a single visitor class instead of spreading across multiple classes. So in the implementation, we saw that we had this print visitor, so the functionality there was printing out information about a particular rule. So we had all that functionality which could print information about any rule in our organizational structure in that single visitor class. If we didn't use visitor pattern, the print functionality would be provided or implemented in each of those classes. So we would have a print in the programmer in the project, lead manager and vice president. So instead of having this functionality in four different classes, we had single print visitor which had that print functionality. So that results or that gives us a benefit that adding new functionality is as simple as adding a new visitor class. Now visitors can also accumulate state. So as they are visiting different objects in the object structure, they can keep on accumulating that state. So along with the behavior that we get with visitor, we can also have a state per object that is stored in our visitor. And we saw that in the appraisal visitor. So our appraisal visitor was carrying out the appraisal and storing the final rating within the visitor itself. So if we need a new state for a new behavior that is being added via a visitor pattern, we don't need to modify our existing classes. We can store that state for specific object in the visitor itself. The last design consideration is that visitor can be used to add new functionality to object structure, which is implemented using another design pattern. So we had a composite design pattern that we saw in the structural Design pattern group so we can add a functionality If we already have a composite with us, that means we already have an object structure. All we need to do now is to create a visitor and we will add new functionality to that composite In the similar way, we can provide the implementation for interpreting our interpreter design pattern. So in the interpreter design pattern, you will see that we had this abstract syntax tree that was built and we can use visitor pattern along with that interpreter to provide the interpretation of sentences. All right, so these were some of the points that you should keep in mind while working with visitor design pattern.
