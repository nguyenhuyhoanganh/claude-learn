# Visitor - Pitfalls

All right. Now let's look at some pitfalls of this design pattern. So first pitfall is that visitors need access to object state to provide a proper implementation of functionality. So that means the objects which are part of our object structure, we end up writing a lot of getter and setter methods in those classes. So. And the purpose of that is we are going to expose all the state which is present in our object so that visitors can work. But what it does is that it weakens the encapsulation of those objects because other classes which are not visitors can also start using those getter setters and in turn getting access to that object state. All right. So that is one pitfall. Supporting new classes in our visitor requires changes to all visitor implementations. Okay, so we saw that we had an object structure and there were objects of few classes there. If we decided that our object structure is now going to support a new object or object of a new class, then our visitors also need to provide that visit method which is specific to that new class. And since we are going to add that method in the visitor interface, we have to change all the implementations of visitor. Okay, so supporting new classes is a lot of work in visitor design pattern. Now, if the classes themselves change, that means the classes whose objects we are visiting. If those change. Then all visitors have to change because these classes work with the interface that is exposed by these concrete classes. All right. So visitors can change without affecting the objects. But if the classes which whose objects are part of our object structure, if those classes change, then our visitors have to change as well. Last point is that this pattern is a little bit confusing to understand and implement if you are studying it for the first time.
