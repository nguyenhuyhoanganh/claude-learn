# Null Object - Pitfalls

All right, Now let's discuss some pitfalls of null object design pattern. So the first pitfall is that creating a proper null object may not be possible in all scenarios. And this is because some classes may be expected to cause a change when an operation is called on them. And since we since our null object is not going to create or cause that change, absence of that change can cause other classes to fail. So if you remember from our implementation video, we had a service called as Complex Service that was using a storage service to store a report file. Now, we had created a null object for our storage service, so when we had given our null object to the complex service, whenever it's asked us to save the report on the disk, we didn't do anything because that is the job or that is how a null object works. But imagine if our complex service, after calling save for a particular report, is then going to call another service, let's say an email service, and it is going to pass that report, file name to that email service saying that email, this particular report file. Now, since we are using null object, the report file is not going to be present on the disk because our null object was doing nothing. Now email service is going to see that there is no file and it's going to fail. So this is an example where a proper null object creating a proper null object is not easy. You have to do a lot of thinking. And this is the next point here actually, that finding what means, what it means to do nothing is not that easy in every scenario, or it may not even be possible. And this is especially true if our methods in null object are expected to return other objects. Okay, so what should we do there? Well, whether we should use or return another null object that will be written from our method or whether we should use return a null value. So these are some of the questions that you have to answer before you can start using null object design pattern.
