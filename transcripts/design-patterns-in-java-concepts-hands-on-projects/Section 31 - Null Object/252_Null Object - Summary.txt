# Null Object - Summary

All right, Now let's look at a summary of null object design pattern. So null object pattern allows us to represent an absence of real object as a do nothing object. So instead of using a null value, we are going to return an object which will be another representation of absence of real object. So method implementations in a null object will do nothing. Okay. In case we are expected to return some value from the method, then these methods are going to return a sensible hardcoded default value. Now classes that use a null object, they won't be aware of presence of this special implementation that we have. Okay. Because the whole purpose of this design pattern is to avoid null checks in other classes and handling that absence of object case transparently. So this is the requirement here that the code which is using our null object should not have to do anything special when it is dealing with null object. Now null objects themselves do not transform into real objects in some point of time, nor do they use any kind of interaction to a real object, because those are the things which are done by proxy design pattern. Now let's look at a UML diagram of this design pattern. It's a pretty simple design pattern. So here we have a base class or an abstract class which has defined some operations and our client is using with this particular interface, which is defined by that abstract class. Then we have a concrete class which is the regular class, and this class has provided implementation for that operation. So if our client is using an object of concrete class, it is our regular scenario or regular flow of program execution. However, for some reason we do not want to provide a real object of concrete class. In that case, we create an object of our null class. So null class object of null class represents the null object in the implementation of operation. In this class, we are basically going to do nothing. So calling an operation on this null class object will result in no side effects. All right. So our client is going to be unaware whether it is working with an object of concrete class or a null object. All right. Now, let's look at a typical implementation of this design pattern. Again, it's going to be a simple implementation. So here we have a class called as storage Service, which has a method called as save, and an object of record class is given to this method and it is expected that the storage service will basically write out that report. Now let's say we are running in a test environment or a special scenario where we do not have access to permanent storage or we do not want to create that particular file. For example, we are running some test case. So in that case we can use a null object. So we have a null storage service which extends from our storage service. And inside the implementation of save method, we are basically doing nothing. We are not saving that report and this is what our null object means. The methods inside our null object are not going to do anything. All right. So this is the null object design pattern in Java.
