# Null Object - Comparison with Proxy

All right, Now let's compare our null object design pattern with proxy. Now, these design patterns are not related in the sense that proxy is used for something different and null object is used for completely different thing. However, these two patterns are often confused and that is the reason we are comparing them here. So the first difference is that null objects never transform or create a real object, or they do not even provide some indirection to the real object. Okay, so they do nothing and they do not transform or they do not become a real object. Bronx is, on the other hand, will transform or they will need a real object eventually. So for example, we had this virtual proxy where we were creating a standing object. However, for some operations we needed real object. So in that case, proxy is going to create real object. We had a authentication proxy, which was basically intercepting the call, and if the authentication is successful, it is going to pass that method call to the real object. Okay, so proxy is need real object to work. Now null objects do not act on behalf of the real object. Okay. Their job is to do nothing in case of proxy. However, if our real object is absent, then proxies will provide behavior that matches with what is expected of a real object. So in case of a virtual proxy example, if somebody calls a setter method on our virtual proxy and we do not have a real object, then we're going to cache that value. So in effect, we are going to our client is going to set a new value on the real object, because as soon as we create a new object, we are going to pass on that stored or cached value to that real object. So even in absence of real objects, proxies will provide a behavior that mimics or matches with the one provided by real object null objects. On the other hand, will do absolutely nothing whenever their operation is called.
