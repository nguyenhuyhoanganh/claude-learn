# Interpreter - Implementation Steps

All right. Now let's take a look at some of the steps that you would follow while implementing the interpreter design pattern. So we start by studying rules of our language for which we want to build our interpreter. And we define, first of all, an abstract class or an interface that represents our abstract expression, that is going to have, of course, the interpret method in it. Then each rule of our language becomes an expression. That means we are going to write a class that represents that particular rule. Now, while implementing the classes, we need to make sure that the expressions which do not need other expressions, they become our terminal expressions. Okay, so they become leaf nodes in our data structure. Then we create the non-terminal expression classes. So these are the classes which you can think of them as operators in arithmetic expression. So we have, let's say, an expression saying five plus two. So we will have a terminal expression that represents a number, and that terminal expression will have one instance with value five, another instance with value two, and the plus operator will now become our non terminal expression because it will contain these two terminal expression five and two in it. Okay, so terminal expressions will call interpret operation on its on their children. Okay. So if we have, let's say nested nested operations going on, then we can calling the interpret on children will resolve those expression and return their results. Additionally, the non terminal expression itself can perform some interpretation if it is applicable in the example of five plus two, our plus operator is going to first call interpret on our terminal expression. These expressions are going to return simply value which is stored in them. So these expression expressions will return five and two and our plus operator is going to carry out the actual addition. Okay, so this is how you will implement Interpreter Now building the abstract syntax tree using these classes. It can be done by the client itself or as you will find it easier, we can create a separate class that builds this syntax tree. Remember, the pattern itself doesn't tell you how to build that tree, so it's up to you how to do that. It also doesn't tell you how to parse the actual syntax or actual sentence. Sorry, how to parse the actual sentence. So you have to come up with your own solutions for those problems. Then client will use this particular syntax tree that we have built using these expressions and it will interpret a sentence, A context that we discussed previously is passed to the interpreter. Okay, so it will typically have a sentence which needs to be interpreted and it can also have additional fields which can be used by expressions in our tree and they can store some intermediate values or they can use it to pass some values, so on and so forth. So what happens in context is completely up to us. How to use that object is up to us. All right, so this is how you can implement this design pattern.
