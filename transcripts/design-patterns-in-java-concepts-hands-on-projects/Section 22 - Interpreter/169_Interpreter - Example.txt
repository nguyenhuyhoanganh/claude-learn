# Interpreter - Example

All right, Now let's look at some real life examples of interpreter design pattern. And one great example of this design pattern is the pattern class in the Java util dot regex package. We use this class to perform regular expression matching in Java. So this class has a static method called as compile method. So in the compile method we specify a string which represents our grammar rule and you will see that we use some regular expression syntax in order to specify our grammar rules and using those rules as well as some flags. A pattern object or instance is created by the compile method. So when this instance is created, an internal abstract syntax tree is built. All right. So as we saw in the implementation of Interpreter Design Pattern, we build our abstract syntax tree that is done in the compile method of pattern class. So once that is done, we can check any sentence against those rules using the matcher class. So we call another method on the pattern object called as matcher. And in the argument of matcher we specify a sentence that we want to test against the rules that we specified in the compile method. All right. And it returns us an object of matcher. Using methods. On this object, we can check whether our sentence matched and we can find out more information about the groups of our parts of our sentence which matched with these grammar rules. So pattern class in the Java util dot regex is a great example of interpreter design pattern. Another example that you will see is the expression language support that is present in JSP and JSF. So the classes which support parsing of L or the expression unified expression language that are present in the Java X dot l package. So vendors which implement or provide implementation for these specifications, JSP and JSF specifications, these vendors will implement some interfaces and classes that are present in that Java X dot L package. Some important classes that you would want to know is the expression class which serves as the base class for expressions that can be created. So L supports two types of expressions. There can be a value and a method expression. Then we have the expression factory and implementations or vendors will provide implementations of expression factory in order to implement support of L. There are also classes called as L resolver and there are multiple child classes of this class which complete the interpreter implementation. Okay, so in case you don't got any of that, you can simply remember that the expression language support in JSP and JSF is also an example of interpreter design pattern and so is the SPL support in the spring framework.
