# Mediator - Implementation & Design Considerations

All right, Now let's look at some implementation and design considerations while working with mediator design pattern. So first point is that your mediator should be able to identify which object has send it the notification for state change. That way, mediator itself can filter out that object while it is sending out the update to remaining objects. Because filtering out this in the object itself can be quite a complicated task. Okay, so if mediator can figure out, okay, this is the object which has sent me state change notification, it can exclude that object while it is sending out the update. If an object method, which is called by our meteor to tell that somebody else has changed, if that method is slow or if it is taking a longer time to process this update, then it can severely affect our mediator performance. In fact, you will find that this is a common theme wherever we are using a notification mechanism. We need to make sure that the method which is called from outside, like from mediator, these methods should do as much or as little work as possible and immediately return control back to the mediator so that we don't affect the performance of our whole implementation. We often end up with complex meteors. And the reason for that is quite simple, because we have taken the complexity which was part of each object, and we put that complexity inside our meteor as our interactions between objects become more intelligent. If they become more specialized, then our mediator implementation is going to become quite complex and it can become quite hard to unit test that mediator. So always keep an eye on how complex your mediator is getting. If it becomes quite complex to test, then you may want to consider other solutions to your problem. Now let's look at some design considerations while working with Mediator. So first of all, we can extend our mediator, and a benefit of that is that you can create different type of interactions based on where your mediator is running. So if your application can run on a cloud as well as on a private server, you can have different interactions because in cloud environment you may want to keep an eye on different parameters. For example, you may want to know if an instance is created so your mediator will have that object. When it is running on cloud, you will have a different implementation of mediator while you're running on a private server or a test server. All right. So you get the idea that you can customize your interactions by inheriting from mediator. Now, mediators typically tend to be specific to the objects that they are coordinating. So that means that abstract mediator may not be necessary if you don't see yourself needing a subclass. So if your application is always going to run on same platform, always going to interact with same set of objects, then you don't need an abstract mediator. Now, in order to find out about the notifications, we can use another design pattern called as observer. So we're going to learn about that design pattern after a few lectures. So you can use observer design pattern to implement the notification mechanism through which our objects notify our mediator. All right. So these are some design considerations for mediator design pattern.
