# Mediator - Implementation Steps

All right. Now let's look at typical steps that you would follow while implementing mediator design pattern. So we'll start by creating our mediator. Now, it can be an abstract entity. That means our mediator can be an interface or an abstract class, or it can be a concrete mediator. If we don't see a need for creating subclass of our mediator in future, whatever implementation we choose, we need to provide a method in this mediator which can be called by the participating objects to tell our mediator that its state has changed. Now this method will want to know which object has changed so that it won't send notification to the same object back again. And optionally, we can also ask for the actually changed value of the state so that we can send this value to other objects. If we don't accept the changed value, then another option is that the remaining objects will call back on the original object to find out what is the changed state and what is the changed value. Okay, so it can be implemented in both ways. You can give the state value in the notification itself so other objects don't need to call back or you can only send the object, which has changed. In that case, the other objects will need to call back on the original object. While implementing this method, we need to send out the notification to all the remaining objects that this particular object has changed its state. Now, one requirement for this design pattern is that mediator needs to know about all the participant participating objects. To achieve this, we can either have our objects register themselves with mediator as soon as they are created, maybe in their constructor, or we can have our mediator create the object itself and that way it will know which objects are part of the collaboration. Now, depending upon the implementation that we are creating our mediator for, we need to handle this change notify change loop, which can be an infinite loop if our object cannot determine whether its state value changed because of the mediator notification or because of some external source like user interaction or a another method call. If you have this kind of object where the value change is not clear from where the value change happen, then you need to find out or you need to take care of that, uh, infinite loop and you will see how this can happen, especially if you are working with UI controls in Java effects. All right. So these are some these are some of the steps that you would follow while implementing mediator design pattern.
