# Mediator - Comparison with Observer

All right, Now let's compare mediator design pattern with observer. We are going to look at observer after a couple of lectures, but let's look at the differences between these two. So intent of mediator is to encapsulate a complex interaction between objects. So we have seen that there is a group of objects, and whenever one object state changes, it notifies every other object in that collaboration. So there is a web of communication that happens between these groups and our mediator. Design pattern encapsulates that interaction. Observer, however, is used whenever we have one too many dependency or one to many communication pattern between objects. So in observer we have a single object whose state, if it changes, then there are multiple listeners who are interested to know about that state change. But the listeners themselves do not tell our original object if their state changes. Okay, so there is one directional flow. We have one object and whose state, if it changes, we notify multiple other objects and that's about it. In case of mediator, if any one of object changes its state, all the other objects are notified about that. So that's the difference between the intent or purpose in these two design patterns. Another difference is that mediator implementations are typically specific to objects being mediator mediated. That means mediator would typically use a method that is defined in that particular class to tell that object that some state has changed. Okay, so we are tightly coupled with that particular object or particular class whose object we are mediating in case of observer. The design pattern is entirely implemented using interfaces. So we have interface called as observer and another interface called as observable. Okay, so object who is who is being monitored will implement the observable interface and the listeners will implement observer interface. So the communication happens using methods defined in that interface. That means we can have any class implement an observer pattern and he will be eligible to listen to those events or state changes. Okay, so mediator is tightly coupled with the objects that it is interacting with. Observer works completely on interfaces. That means there is no tight coupling between the implementation concrete implementations.
