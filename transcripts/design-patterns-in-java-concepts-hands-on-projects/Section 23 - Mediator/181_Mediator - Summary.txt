# Mediator - Summary

All right, Now let's look at a summary of mediator design pattern. So whenever we have a group of objects that are communicating within themselves and notifying each other about state, then they are usually very tightly coupled because each object knows about all the other objects. Using mediator design pattern, we can add or decouple these objects. In this design pattern, each object only knows about the mediator object. So whenever that object state is changed, it notifies the mediator that this particular state has changed and the mediator will in turn notify all the remaining objects in the collaboration. That means an object only knows about mediator, and we still get the benefit of knowing any state changes that are happening in this group Now. Mediators are typically hard to implement as a reusable object because they are typically tightly coupled with the classes whose objects are interacting or part of this collaboration group Observer design pattern solves this problem because it uses interfaces in order to implement that one to many communication. Other mediators are very easy to implement. So if you have a very small group of objects, you can easily create a mediator and decouple those objects. And also it provides a benefit that we can extend the mediator and then we can specialize the interaction that happens between these objects. Now let's look at the UML diagram of mediator design pattern. So first of all, we have the actual mediator. We can implement it as an interface or abstract class, or we can create it as a concrete mediator class. The responsibility of mediator is that it should define a method that can be called by the objects in this collaboration to tell mediator that their state has changed. A concrete mediator is needed if our mediator is of abstract type. A concrete mediator will know about the actual objects which are part of this collaboration. So it will have references to those objects. And whenever one of the object tells mediator that its state has changed, it will use those references to notify these objects that somebody has changed their state. Then we have k-league. So K-league is represents a class who's objects are part of this group. So K-league knows about the media and it has a reference to this mediator and it will tell mediator whenever it state changes. We can have multiple concrete colleagues and these will be the classes which provide their own functionality so they can be very different classes. We can have a class which prints some message in the log, another class which can send notification over the network. And these two objects of these two classes can be part of same group which collaborates between them. The concrete colleague will get a notification from mediator whenever state of one of the object changes, and it should handle that notification and carry out or implement its functionality.
