# Decorator - Implementation & Design Considerations

Let's consider some of the implementation and design considerations that should keep in mind. Now, we have seen from the UML that we have a requirement that our decorator extends from a component class. Now we have concrete components which are our real objects that are also extending from this common base class. So if you have a large state defined in your component base class, then that entire state is going to get inherited in decorator objects. And typically decorators don't need that information because they are their own little thing to the behavior. All right. So try to avoid having a large state in your base class or base component. Now you should pay attention to equals and hashCode methods of decorator. When using decorator, you have a choice. And this is the decision that depends on the project that you're working. Whether a decorated object is equal to the same instance, but without decorator. Okay, There is no right or wrong answer here. Only point that you should pay attention is that you are taking the equals and hashCode into consideration, especially if you are planning on using collections like Hashset and HashMap. You should decide what happens to decorated object, whether it remains equal to and to the same object, but without decorator or not. Now, decorators support what is called as recursive composition. What it means is that we can have a decorator which contains another decorator within it. And this is possible because the decorators have same interface as the original object. So nobody can differentiate who is using this object, whether this is a decorator or not. Okay, so they behave similarly. So each decorator typically adds a small bit of functionality and you can end up with a nesting to a great level. So you will have a decorator which again contains a decorator and so on and so forth. And this type of code can become a little bit difficult to debug. All right. So keep that point in mind when you are implementing Decorator. Now let's look at design considerations. Now, decorators are very flexible and powerful than inheritance because inheritance is static by definition. So you have a class which provides the functionality, and if you create object of that class, you're always going to get that functionality. With decorators, you can dynamically choose which objects can get the new functionality and which don't. Okay, so this is the benefit of using decorator. So while you're designing, while you are designing your application, you can make use of this feature of decorators. Now, decorators should act as a kind of a skin over your object. That means they should provide small, meaningful functionality on top of your existing behavior. Do not try to change meaning of the operation in your decorator. Okay, so if your method and undecorated objects provides a certain type of behavior, decorators should not change the meaning of that behavior completely. All right. So these are some of the points that you should keep in mind while designing applications using decorator.
