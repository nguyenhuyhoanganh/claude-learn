All right. Now let's look at some examples of bridge design pattern. Now, one great example of this pattern is found in the Jdbc API. More specifically, when we are dealing with driver manager and the driver interface that is exposed or that is defined in the Jdbc API. So in our code we will deal or we will work with driver manager. So in your code you will call the driver manager and call the gate connection method and use Jdbc API in order to interact with the database. On the other side of the bridge, we have the actual implementations of the driver interface, which are specific to a particular driver vendor. So we have a different database, sorry, different class for connecting to MySQL database, a different set of classes for connecting with Oracle. And we can switch these implementations and our code can continue to work with our driver manager. So we have these two separate parts of inheritance hierarchies so our driver can change without needing to change our client, which is working with driver manager. On the other hand, if we change the driver manager how it is defined, our drivers don't need to modify. All right. So this is how we can use or this is how the bridge design pattern is used in the Jdbc API. Now one thing that you will quickly pick up is that bridge and adapter design pattern are quite common. In fact, if you look at the structure, the way they are implemented, you will find that they are quite similar. We in the adapter as well, especially in the object adapter, you will see that we also give an object to our adapter and it provides some methods and delegates to the object that it was given. So Bridge also works on the similar principle. But here we have an object of the implementation which is used by our abstraction hierarchy. The difference in these two patterns, however, is between the intent or purpose between these two design patterns. Adapter is written when we already have an existing class and we already have a requirement of adapting this class to different object. Bridge, however, is written from the ground up. So we when we are writing our set from map class, we are making the decision when we are writing this class that we are going to use an existing map object. All right, so there is a difference between the purpose between the bridge and adapter design pattern. However, if you look at the code, then it can be quite confusing to differentiate between adapter and bridge. So always remember that whenever you are thinking about design patterns, you should always think conceptually that is what is the purpose of this particular design pattern. Instead of going into how this pattern is implemented, that way you will avoid a lot of confusion while differentiating between these patterns.
