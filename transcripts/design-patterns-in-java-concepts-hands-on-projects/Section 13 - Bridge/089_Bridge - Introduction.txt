# Bridge - Introduction

Now let's learn about bridge design pattern. This is the structural design pattern and it is one of those design pattern that may take some time for you to understand what exactly this does, because it promises something which may sound completely impossible when you first hear it. So you might have noticed that typically when we use inheritance, our abstraction and our implementation are tightly coupled. Of course, because our implementation is going to extend or implement our abstraction depending upon whether it is an abstract class or general Java class or an interface. Now this bridge design pattern promises that it can decouple our implementation and abstraction so that we can change any one of those without affecting the another. So we can add an operation in our abstraction and we don't have to change our implementation. So how is that possible? Well, this design pattern achieves this feat by creating two separate inheritance hierarchies, and it may sound a little bit confusing, so please bear with me. We are going to learn what this means. But what it does, this design pattern, what it does is it creates two separate inheritance hierarchies. We have a hierarchy, inheritance, hierarchy for abstraction. So abstraction is what our client code uses. So our client is going to use any one of these classes from our abstraction, and we have a separate hierarchy of implementations. And these two hierarchies are connected using composition. Okay, So when you read about it or hear about it, it sounds quite impossible to do. And that's because you may be thinking about having an inheritance or sorry, having an interface change without needing change in its implementing class. That's not what this design pattern does when it is talking about abstraction. It is talking about the interface that is used by client. When it is talking about implementation, it is talking about class that can provide that functionality. Okay, think about adapter but not quite adapter. So let's look at the UML diagram, the formal UML diagram of this design pattern to begin with. We have an abstraction here, so this can be a class or an interface that defines the interface that is used by our client code. Now this abstraction also has a reference to whoever is providing that functionality. Next, if we have a refined abstraction, so refined abstraction is basically a child class of our abstraction, and it defines any specialized operation that we want. Then we have implementer. So implementer is basically a base interface or a base class that provides method that implement the functionality, which is which is expected by our abstractor. And what happens here is our implementer can provide implementation in smaller steps. An example could be your abstraction can define a method called as transfer money from account X to account Y. Your implementer will have methods that says check the balance of account X. Another method that checks whether account Y is active or whether it is dormant. It can have another method that actually takes the amount to be transferred from one account and deposit into second account. So the point here is that implementer won't have 1 to 1 matching methods that are defined in abstraction. Okay, so this class is completely on its own, which is defining some methods and using some combination of these methods. We can implement the function that is defined in our abstraction. Don't worry, this is going to get clearer as we learn about the actual hands on example. All right. To move on, we have a concrete implementer which again implements whatever is defined in the implementer, and it provides code, actual code that carries out that particular functionality. All right. So this is the UML diagram for bridge design pattern. And the name Bridge comes from this particular relationship, the composition that is present between abstraction and implementer. So implementer is present inside of our abstraction, and these are the two inheritance hierarchies. On the left side, we have an abstraction hierarchy, and on the right side we have implementation hierarchy. And these two are connected by this composition relationship. And this is what the bridge refers to. All right. So it may sound a little bit still unclear. Hold on to it.
