# Bridge - Summary

So this is a summary of bridge design pattern. We use this pattern whenever we want our abstractions. That is the interface that is used by our client and the implementers. These are the classes which provide this functionality. We want to separate or decouple them and bridge. Bridge achieves this decoupling by defining separate inheritance hierarchies for our abstraction. These are the classes that are used by our client code and the implementers which provide the functionality, the actual functionality. So we have these two separate inheritance hierarchies, and we use composition in order to connect these two hierarchies. And this connection using composition is what gives this design pattern its name called as bridge. Now, the implementers do not have to define the exact same methods that are defined in abstraction. What is common practice or what you will typically find in real life is that implementers have these primitive methods, methods that do small amount of work, and then we have abstractions which use these small methods in order to provide a functionality which is which corresponds to a use case that our client code wants to use. Now let's look at the UML diagram of the bridge design pattern. So we have an abstraction here, and this defines a base class or an interface that is used by client. It has methods that are used by our clients. We can also have refined abstraction, which will be the implementation of this interface or abstraction or abstract base class. Then we have implementer. So this is again an interface or an abstract class that defines methods which carry out a small amount of work. We have concrete implementers that implement these methods so we can have different algorithms or different ways in order to achieve this small amount of functionality. And we have these two inheritance hierarchies on the left side. We have an abstraction hierarchy, and on the right side we have implementers in their own inheritance hierarchy. We connect these two inheritance hierarchies together using composition. That means our abstraction class of our abstraction will have an object of implementer inside it. And using that object, our abstraction will provide the functionality that is needed by client. All right, so this is the UML diagram for bridge design pattern. Now let's look at the implementation for this design pattern. First of all, we have Fifo collection, so this is an example of our abstraction. It is an interface that defines two methods called as offer and poll that are used by our client code. Next up, we have a refined abstraction. So refined abstraction is nothing but an implementation or a child class of our abstraction. Now you will see that this abstraction is using a class called as Linked List. And this linked list refers to the implementation or one of the abstract implementers. So linked list has defined its own methods, and we have these methods which are defined in abstraction and they do not correspond to each other one as to one. However, in refined abstraction, we get hold of an instance of the concrete implementer, and we use that instance in order to provide the functionality that is needed by client. Lastly, we have an actual concrete implementer which implements these methods using one of the several choices it has available to provide implementation of methods defined in implementer. All right, so this is how you can implement bridge design pattern.
