# Strategy - Summary

All right, Now let's look at the summary of strategy design pattern. So this design pattern allows us to encapsulate an algorithm in a separate class. So the class, which is using these algorithms, which is also called as a context class, can now be configured with desired implementation of an algorithm. Now it is typically the responsibility of the client code, which is using our context object to configure it with any appropriate implementation of algorithm which our client wants to use. Strategy objects are given all the data that they need by the context object. Because remember, context object is the one which is actually going to call method or execute that algorithm. So context object is in a better position to give that algorithm everything that it needs. So this data can be passed as arguments to the method which is defined in our strategy, or it can pass itself so that the algorithm can query our context and get any desired information or data out of it. Strategy objects typically end up being stateless. Since we are giving all the information as argument to the methods, the object itself becomes stateless and that's why they become great candidates for flyweight design pattern. So if you remember from our discussion about that design pattern, you will see that we had some intrinsic state and an extrinsic state, and you will see that how that applies here. So we can have some constants in our implementation of a strategy which will be intrinsic state and the extrinsic state will be given to that particular strategy by our context. And combining these two, our algorithm will be able to perform its desired operation. Now, client code ends up knowing about all implementations of our strategy, and that is one of the drawbacks of this design pattern. So if we add a new implementation to our application, then our client code needs change because it now needs to create an object of that new implementation. Now let's look at the UML diagram of this design pattern. It's a pretty simple design pattern. So here we have context and this is the object which our client is using. So client is going to call operation on this context. This context is going to have a strategy object inside of it. Now client is going to provide our context with a strategy object before calling this operation. Now, this strategy object will be an implementation of this strategy interface. This interface defines few methods, which allows our context to run or use an algorithm. Each concrete implementation of this interface represents a separate algorithm. So when client calls an operation on our context, context is going to use the strategy object and execute the applicable algorithm so we can keep adding multiple implementations in our application. And each implementation of this strategy will be a separate algorithm that solves that particular problem. Now let's look at a typical implementation of strategy design pattern. So here we are going to consider an example where we have a collection of orders and we want to print that collection. To print that collection, we're going to have multiple ways to go about it. We can have an implementation that only prints summary of all those orders. So it's going to print, let's say, order number and the total cost of that order, we can have another implementation which is going to print all the details of each order in that collection. So you will see that these are two different algorithms which print our collection of orders to implement this problem or to solve this problem using strategy design pattern. We start by creating our strategy interface. So order printer is our strategy interface. It defines a method called as print and it takes a collection of orders. Then we are going to have the context object. So this is the object or this is the class which is going to use an implementation of order printer. So in print service you will see that we have a field here named as printer, which is a reference of order printer type. So this is the reference which is going to hold on to the strategy. In the constructor of print service, we are accepting an object of order printer type. So this is how our client is going to configure our context with a strategy object. Our printer print service is going to store that reference and whenever client calls print orders. Method on our context. It's going to use that strategy and print the collection of orders. Now, there can be multiple implementations for our order printer. So here we have one implementation which prints only the summary of all the orders in that collection. On the similar lines, we have another implementation called as detail printer and this implementation is going to print details of each order in that collection. So these are two separate algorithms which are implemented in two separate classes. But both of these classes implement our strategy interface, which is in this case, order printer. All right. So this is how we can use strategy design pattern in Java.
