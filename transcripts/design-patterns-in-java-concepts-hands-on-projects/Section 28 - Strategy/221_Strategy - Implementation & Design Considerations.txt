# Strategy - Implementation & Design Considerations

All right, Now let's discuss some implementation and design considerations when working with strategy design pattern. So we can implement our context class or main class in a way where strategy object is optional. So we are going to have some default algorithm that is used by our context if no strategy is given to it, and this can take care of problem where we have some client code which doesn't want to deal with this concrete strategy objects because in a typical implementation, our client gives our context class a object of one of the concrete implementations of strategy. So they these become now tightly coupled. So our client knows about all possible implementations of strategy to break this coupling. What we can do is we can have a default implementation of the algorithm in our context class. And if strategy object is not given to us, we're going to use that default one. If somebody gives us a strategy object, then we're going to use whatever is implemented in that class. Now you will see that strategy. Objects are typically given all the information that they need as arguments to its methods. And if you're running in a situation where a number of arguments are very high, then you can code your strategies to use an interface to get the data that they want and we can have our context class implement that interface and provide methods that will give information to strategy implementations. So the point here is that if you are running into a problem where your methods are getting longer because they need more information, we can simplify that using an interface. Now let's talk about the statelessness of strategy objects, because you will see that similar to state pattern, our strategy objects also become stateless and they will typically get all the information that they need in the method itself as arguments. So they become perfect candidate for sharing between context objects and we're going to see how we can share that. Then let's talk about some design considerations. So strategy implementations can make use of inheritance to simplify creation of child objects or creation of new algorithms. And we can do that by taking out common functionality in all algorithms and putting that in one or more base classes. All right. And using that base class, we can define common operations there so our child classes can be simpler to implement. Now, since we just discussed that our strategy objects can be stateless, we can use a flyweight design pattern in order to share our strategy objects between multiple context objects. All right. So these are some of the points that you should keep in mind while working with strategy design pattern.
