# Iterator - Implementation & Design Considerations

All right, Now let's look at some implementation and design considerations when working with Iterator. This is a pretty simple pattern, so there is nothing complicated here. The first implementation consideration is that detecting changes to the underlying collection is quite important. Okay? Because if our iterator instance is being used by a client and the underlying collection changes, then it means that we may be violating the contract with our client that is using it. And again, the same example somebody calls Hasnext, we say, okay, there is an object and before he gets a chance to call the next method on our iterator, somebody clears out the collection. Okay, so we need to throw an exception. In that case, that means we need to identify that our underlying collection has changed. Now, having our iterator implementation as inner class makes it quite easy to access that internal collection, which is part of our aggregate objects. Because remember, the reason we use iterator is because we don't want to expose our internal collection to outside world, because we want to have a freedom to change that internal representation or internal implementation of how we are storing objects. Okay, but an inner class gets access to those fields or the internals of our aggregate object and that way it can provide access to these its elements and it can also detect if the underlying collection has changed. Now let's look at some design considerations for this design pattern. You should always prefer iterator interface and you should always hide your actual concrete class from the client because that way you get a freedom of how you are implementing your iterator and you can change your concrete implementation without affecting the client who is using that interface. Now you should know that iterators have many applications which are not directly, which are not directly related to collections. For example, we may want to provide an iterator to iterate over contents of a file which is stored on our disk. Okay. So we can use the iterator design pattern so that our client can read these lines one by one. Okay. We can also use the iterator design pattern if giving access to someone to a network resource. Okay, so there are multiple applications for iterator design pattern which are not directly related to collections like arrays or linked lists and something like that. Okay. We can give wherever we want to give a sequential access to information, we can use this design pattern.
