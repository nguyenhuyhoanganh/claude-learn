# Singleton - Summary

All right. Now we will look at summary of Singleton design pattern. So we use singleton pattern whenever we want to ensure that there is only one instance of our class in entire application. To implement Singleton in Java, we start by making our constructor private. That way, nobody outside of our class can create instances of our of our class and nobody can extend our class. Then we provide a public static method in our Singleton class. In order to get our singleton instance outside. Now, when we are implementing Singleton, there are two broad choices available to us. We can implement our singleton as an eager singleton. An eager singleton is a singleton where we create our one instance as soon as our class is loaded by the class loader. On the other hand, we have lazy singleton, and here we create our instance when somebody actually asks for that instance. Now you should always try to use eager singleton because it avoids many complexities related to synchronization. However, you should note that creating or implementing eager singleton adds to your application startup time, because as soon as class loader loads your class, you will be creating your instance. And if the cost of creating the instance is high, then it will add up to your application startup time. So if you are worried about your startup time, then you should look into lazy Singleton. Now what we are going to do here is we are going to look at couple of ways in which we can implement a singleton. We will start by looking at an eager singleton. So here we have a class called as Eager Registry, which is our singleton class. You will see that we have made our constructor private. Then we have a private static final variable declared inside this class called as instance. And this variable is going to hold our singleton instance. You will also see that we are initializing the static field with the new instance or new object of our class. And this is what Eager Singleton is. So as soon as our class is loaded by the class loader, the class loader will initialize our static instance variable and it will create our instance of singleton class. Then we have a public static method called as Get instance, which returns this instance to outside code. So in this way we can implement an eager singleton in Java. Next. We have a lazy singleton which is using double checked locking. So again, here we start by making our constructor private. Then we again have a static, private, static variable called as instance, which you will see that we have declared the static variable with volatile keyword. So a volatile variable is not cached by threads. So if multiple threads are accessing this variable's value, then they will refer to the main memory in order to check what the current value of this variable is. And it is important that we use volatile qualifier for our singleton instance field. Then we have get instance method and this is where we are going to create our singleton instance. So we start by creating or checking our instance and check whether it is null. If our instance static instance is null, that means we have not created our singleton instance yet. Then we take a lock on our class using synchronized block, and inside this block we again do the check for Null. And this is what double check locking is. So we are checking twice for the value of our instance variable. And if it is still null inside of our synchronized block, then we create our singleton instance. And finally we will return our instance. So next time, if somebody calls our get instance method again, our first check will return false and we will directly return already created singleton instance. So this is how you can implement a lazy singleton using double check locking. Now there is another way that we can implement lazy singleton and that is using initialization holder idiom. So what is an initialization holder idiom? So let's look at it. So first we have our Singleton class here and again we have our constructor marked as private. But instead of declaring our static field inside of our Singleton class, what we do is we create a static inner class, a private, static inner class called as register holder. And inside of this class we declare our static variable, which is going to hold on to our singleton instance. And this static class register holder is what is called as initialization holder. Now, this instance will be initialized with our singleton instance as soon as our register holder class is loaded. But since this is a private class inside of our Singleton class, nobody from outside can refer to this class. In our get instance method, we are going to access this class and return the instance static variable. Now, as soon as get instance is called for the first time, we are going to access this class, which means the class will be loaded by class loader and then our singleton instance will be created. So this way we can take advantage of both of eager singleton and lazy Singleton implementations. By using the initialization holder idiom, we can avoid doing explicit synchronization and by using the initialization holder, we also defer creating our singleton instance until somebody calls our get instance method. All right, so this is Lazy Singleton using initialization holder. Now, next up, we will we can implement Singleton as an enum, so we will simply declare an enum. You will declare your instance as the enum constant and you will provide all the methods that you will typically implement inside of this class. Now, having your singleton as an enum takes care of all the problems that you need to take care of. If you are implementing Singleton as a regular class, as an enum, no other class can extend from our registry enum As an enum. Nobody can create a new instance of our registry enum. Additionally, having your singleton as an enum will take care of serialization of your singleton instance. So these are the implementations for Singleton. Now, there are actually very few situations where a singleton is a good choice. Singleton is one of those pattern which is overused and often misused. One of the situations where Singleton is a good choice is for storing the application configuration values. So in this situation, we will have a file on our file system which will store the configuration values used by our application. Now we can have a singleton which reads these values at the startup and provides access to these values from multiple parts of our code. Now, one important point is that these values should be immutable. That means you should not allow changing your configuration values from within your code because then you will run into the same problem of having mutable global state. Logging frameworks like Log Forge also make use of singleton design pattern. Then there is Spring framework, which is an IOC or inversion of control container and a great framework for enterprise development. Now inside spring we define our Java classes as spring beans. Now spring will instantiate these beans and provide them with dependencies. Now, by default, all the beans that are created by spring will be singleton. One benefit here is that since the instance creation is handled by spring, we do not have to modify our code in order to make it singleton. Spring also supports other types of instantiation models, but Singleton is the default way in which spring creates beans. All right, so this is all about the Singleton design pattern in Java.
