# Section Summary & Key Takeaways

Let's quickly summarize whatever we had discussed so far in this section. Kafka is an event streaming platform. Event is anything that happened in our business domain. Kafka is designed for high throughput and low latency. It's open source and distributed, highly available, horizontally scalable. The topic is a way of organizing, collecting and distributing the events in Kafka. It's an append only immutable structure. Topic is a logical abstraction. Topics are divided into multiple partitions for scalability. Each partition has something called offset. It's like array index. Offset is the position of a message within the partition. Each message we write into a Kafka topic can have a key. The key is used to determine the partition. By having a key and a partition, we solve two problems at the same time. Message ordering and scalability. We normally do not run one single Kafka server in production environment. We will be running a group of Kafka servers as a cluster. Each Kafka node will start with some roles. Either broker or controller or broker, controller. If the node has broker role, it will handle client request, read and write. If the node has controller role, it is not a controller. It means it's eligible to play the controller role. To become the controller, they have election process and they will elect one controller among them. If the controller dies for some reason, they will have the election process again to elect the new controller. We will have multiple brokers in a cluster, but at any given point, we will be having only one active controller. When the topic is divided into multiple partitions, the controller might select multiple broker nodes. You are the leader for partition 0 and you are the leader for partition 1 and so on. Single topic data is distributed in a cluster like this. The controller might select few other nodes to be the follower for the leader nodes. To replicate the data for high availability. Kafka will not keep the data forever. They have something called log retention policy. Using these properties, we can control how long we can keep the data. We can have hundreds or thousands of servers in our Kafka cluster. You might want to keep certain servers as your bootstrap servers for the initial connection for your application. Even if the data is distributed among multiple brokers in the cluster, our application does not have to worry about all those details. The bootstrap server will provide the entire cluster information to the Kafka driver. Then Kafka driver will handle the rest. In the real life, in our event-driven architecture, we will have multiple microservices like order service, payment service, inventory service, shipping service, etc. We will also be running multiple instances of these services for high availability and scalability purposes. When the order service produces order events, multiple applications like payment service, inventory service could be interested in consuming these events. We will be running all these instances as a group to represent one logical consumer. We call them consumer group. By doing this way, we can scale horizontally. Different consumers will be consuming messages from different partitions. So we will not do redundant processing. Message ordering is also guaranteed by using the event key. Kafka internally maintains some kind of ledger to track the current offset for each consumer group and for each partition. By doing this way, Kafka knows how many messages every consumer group has processed so far. If the consumers leave and join the group as part of restart, etc., it knows from which offset it has to deliver the messages to the consumers. Kafka also provides the options to reset the offset if required. For Kafka, every single consumer should be part of a group. When we ignore the group name, under the hood, Kafka will be creating one anonymous consumer group with a random ID. Then it will be placing the consumer in that group. Then it will start delivering the messages. Since the console consumer is more of a demo tool, testing, learning tool, so here normally we ignore the group name. But in real life, when we develop our production grade applications, we will be giving the group name.
