# Consumer Group Scaling Scenarios

Since, we have been talking about partitions, consumer groups, scaling etc. Let's quickly consider certain scenarios, so that it will help us to clarify certain questions. To better understand this, let's consider this setup. We have one producer and we have a topic with three partitions. Then here we have one consumer group called payment service and currently we have only one consumer running. When you have only one consumer, Kafka will be assigning all the partitions to that one consumer. So, it will deliver all the messages from all the partitions to this consumer. We already saw this, hopefully this is clear. Now let's imagine that one more consumer joined the group, maybe because of auto scaling. So Kafka will do partition rebalancing and it will assign one partition to this new consumer and the other consumer will be consuming two partitions. Now let's imagine that we have one more consumer in the group, again because of auto scaling. Now we have three partitions, we have three consumers, Kafka will do the partition rebalancing, so it will assign one partition to one consumer, very simple, right? Now let's imagine that we have one more consumer joining the group. What will happen now? In this case, this consumer will be sitting idle. This is expected because we do not have any partitions here to assign to that consumer. Remember that Kafka will not distribute one partition between two consumers. We have already established the rule that the message ordering is very important. So one partition as a whole, it will be assigned to one consumer. The consumer can get multiple partitions, that is not the problem. But the one single partition cannot go to multiple consumers. So if a consumer group has more consumers than partitions, some consumers will remain idle because there are no partitions to assign to them. Please remember this.
