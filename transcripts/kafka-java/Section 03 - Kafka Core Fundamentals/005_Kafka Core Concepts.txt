# Kafka Core Concepts

In this section, we are going to talk about Kafka Fundamentals. The whole section is about Kafka, no Java, no Spring Boot, we will be playing with the command line tools to familiarize ourselves with Kafka. The first few lectures will be theory focused, then we will have more hands on demos. Theory is important to have a proper mental model. Let's start with some core concepts first, we say event driven architecture, right? But what is an event exactly? How can we define that? Event could be anything that happened in our business domain. For example, Sam liked a tweet, that could be an event, Sam placed an order, that could be an event, user clicked on a link, that could be an event and so on. So event could be literally anything. People also use these terminologies like events, records, messages interchangeably, particularly developers use events and messages interchangeably. But the Kafka terminology for the message is record. So they all mean the same. Now let's talk about Kafka. Kafka is an event streaming platform. It can capture any event whenever it occurs in your application, then it stores those events safely in some place internally, so that they can be delivered to other applications. It can deliver the messages in real time for processing. It's open source and distributed. It's mostly developed in Java, some components in Scala. How does Kafka store the messages? Actually it's difficult to give a clear answer for now. We are going to discuss in a super high level, but eventually we will learn how things are working internally. Kafka has something called topic. It's a way of collecting and organizing data within a Kafka cluster. For example, how will you organize data in a relational database? You have tables, right? Similarly, we have topics in Kafka. Just to give an idea, we can have hundreds or thousands of topics in a cluster. It's not a problem if we have enough resources, but we will come to that later. Just understand that we do not have any hard limit on the number of topics. For example, these could be the topics for our application. Whenever end user clicks on a product link, we might want to write that into a Kafka topic called product view events. Similarly, whenever users place an order, it might go into the order events topic. For Uber-like applications, the driver's phone app might send the current location, latitude, longitude into the driver location topic and so on. Just a note here that it is not like we have to have the minus sign in the topic name. It is not like that. There is no such requirement. The topic name can include uppercase, lowercase letters, numbers, dots, et cetera. You can follow any pattern you like. Having some kind of consistency is good. Whatever works for you and your team, you can follow that for the naming convention. Let's assume that this is our Kafka server and we have some topics created. Here we have one microservice. It could be an order service developed using springboard. Whenever an user places an order, this application writes that information as event in this Kafka topic. This could be order events topic. Then we have another application developed using Python. It could be a payment service. This application consumes these messages and charges payment for the user. Simple right? Any application produces messages is called a producer. Similarly, any application which consumes the messages is called a consumer. Okay. Everything is good so far, but life might not be always like that. What will happen if this single Kafka node crashes due to out of memory error? What will happen to our application? How can they communicate? What will happen to all our orders? Let's discuss that in the next lecture.
