# Tuning Consumer: max.poll.records

This will be a quick lecture and let's talk about the consumer. How does the consumer application work? Whenever we send the messages to the Kafka server, does the Kafka server push the messages to the consumer or the consumer pulls the messages from the Kafka server? Is it a push or is it a pull? Okay. So the Kafka server will not push the messages to the consumer. The consumer has to pull the messages from the Kafka topic or broker. Sometimes people ask this question, the connection between the Kafka broker and the consumer is the persistent TCP connection. So why can this Kafka server not push the messages using the connection? Actually it is not very hard to push the messages, but there is a problem with this approach. For Kafka, each and every message has to be delivered to the consumer and it expects some kind of an acknowledgement from the consumer that, yes, I saw that message. If we skip the whole acknowledgement step, what will happen here is, let's say the producer is producing super fast, 100,000 messages per second, Kafka broker might be pushing all the messages to the consumer. But what if the consumer is not able to keep up with this producer speed? What if it's processing speed is just one message per second you're getting, right? So now we will be losing the messages, you're seeing the problem, right? So for Kafka, each and every message has to be processed safely and reliably. So it wants the acknowledgement from the consumer that, yes, I saw that message, I processed the message, something like that. So because of this reason, it will not push the message instead, it will ask the consumer to ask for the messages, only then it will deliver. How does the acknowledgement work and all, we can talk about this later, but the answer to the question is, it's a pull process, okay? As we had some properties at the producer side, we also have some properties at the consumer side, but the one important property is maximum pull records. Let's say the producer has produced 1,000,000 messages. Now when the driver, the consumer side driver asks for the messages, what do we want? Do we want all the 1,000,000 messages at once? That could be too much for the consumer, right? So the driver will say just maximum 500, not more than that. Of course we can modify this value, but this is the property the driver will use. If you have lots of messages, do not give me more than 500. So that is how it will ask the broker and the broker will give the messages. If it has thousands of messages, it will give only 500 maximum. If it has just only one message, it will just give one message. We also have a couple of more properties to discuss around this, but I will do when the time comes with the hands on, otherwise it will be too much theory. But hopefully you get the high level idea what is going on here.
