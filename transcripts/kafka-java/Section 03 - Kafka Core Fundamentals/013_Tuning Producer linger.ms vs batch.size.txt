# Tuning Producer: linger.ms vs batch.size

This will be a quick lecture and in this lecture I would like to introduce a couple of properties at the producer side. First of all we know that we have several property files. We have seen that. The Kafka node would be using one of these files. Broker Properties, Controller Properties or Server Properties. We have already discussed this. Then we have also noticed these files. Producer.Properties and Consumer.Properties if you remember. But these files they are not anything to do with the Kafka server. The Producer.Properties contains the configuration properties of the producer applications can use. Similarly Consumer.Properties is for consumer applications. The console producer is simply a command line tool. Under the hood it uses the Kafka client library, Kafka driver. Let's imagine that this small box represents the Kafka driver. When we develop our application using Java, Spring Boot later we will be including the same Kafka client library as well. So it is not like the Kafka console producer that itself goes and writes that messages to the topic. So it doesn't work like that. Instead the console producer whenever I type something and hit enter it will be giving all those messages to the driver. Similarly whenever Java, Spring Boot application when it produces messages it will do the same. It will be giving these messages to the driver. The driver is the one which will be sending these messages to the Kafka topic. Now the driver uses two important properties to send messages efficiently. Linger.ms and batch.size. The Linger.ms the maximum amount of time the driver can wait to collect the messages from the application. The batch.size is the maximum amount of bytes it can hold. That is let's say the Linger.ms is one second. So what if the producer application has produced one GB within one second. So it's not like the driver will be keeping all the one GB in memory. You are getting right. So what we are saying here is either wait for one second or wait to collect this much bytes. So whichever comes first. For example the driver might have collected just one byte. OK. But one second passed. So OK. In the one second I have collected only one byte. Let me send. So it will send like this. Or within 10 millisecond if the driver has collected 16 KB then in that case. OK. I have enough bytes to send. So it will send. So this is how it will work. Either this or this. Whichever comes first. So the timeout option for the console producer under the hood changes or uses this property. So this is why when we set the timeout is zero. Right. So the Linger.ms is set to zero. So the driver sends the messages immediately. Hope it is clear.
