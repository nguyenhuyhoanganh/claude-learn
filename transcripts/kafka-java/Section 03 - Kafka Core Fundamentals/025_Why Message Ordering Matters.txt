# Why Message Ordering Matters

In the previous demo, we saw that when we run multiple instances from the PaymentServiceConsumer group, only one instance was getting all the messages, right? The other instance was sitting idle. Why is that? First, we have to understand that Kafka is an even streaming platform. I had already mentioned that message ordering is very important for Kafka. It's an append-only immutable structure. It will keep on appending messages. And in the same order, it will be delivering the messages to the consumer. The basic expectation here is that applications need to process all the events in the order it happened. Now, let's consider this scenario for a minute. There is a banking application. A customer deposits $1,000 check. Then the customer immediately wants to withdraw $50 cash from his account. So, the application raised two events. One is $1,000 deposit event. Then $50 withdraw event. Now, let's imagine that there is a transaction service. It's a microservice and we are running multiple instances of this transaction service as consumer group. Kafka delivers the $1,000 deposit event to this consumer from this consumer group. So, this consumer is processing $1,000 deposit event. Let's imagine that this deposit process will take time to process. It will take like 10 seconds to process. Let's imagine like that. While it's processing the deposit event, what we are thinking here is this consumer is sitting idle, right? Why can we not deliver the next withdraw event to this instance? So, let's deliver this. This is what we are thinking. So, let's imagine that we are giving this withdraw event to this consumer. That's because it's sitting idle. So, what do you think that it will happen now? While this service is still trying to process the deposit event, before it inserts the record saying that the customer has $1,000, this instance will be trying to withdraw $50 from the customer account. But this will not be able to find the balance, right? So, this will throw an exception saying that this customer does not have any money. So, this customer cannot withdraw. So, you see the problem, right? This is why we have to process all the events one by one. This is exactly why Kafka does not randomly distribute the messages among instances. Now, it kind of looks like if I have a topic and if I have thousands of events, then I cannot horizontally scale. I have to have only one consumer to process all the events. It kind of looks like that, right? But actually, no. Kafka gives a solution for that. That is where the partition concept comes in. Let's discuss that.
