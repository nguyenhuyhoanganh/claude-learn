# Partitions and Message Keys

In the previous lecture we were talking about the importance of message ordering and why we had to process all the events in the Sequence one by one with this approach. We would be facing the scaling problem. So, how are we going to handle that? We have something called a partition. That is a topic is nothing but a logical abstraction A topic is made up of partitions These partitions are the actual physical storage units that is where the data is stored So we can say a topic is divided into multiple partitions When we create a topic we have to mention how many partitions we want like this So far we did not mention anything So Kafka assumed one partition To keep all of us on the same page. Let's imagine that we are creating our topic with two partitions then Kafka will be creating a topic with the two partitions like this This is partition zero and this is partition one. So what is the use of it? It's going to help us solving both the message ordering issue and the scaling issue at the same time Here we have something called a key message key So the events whatever we produce right that event can have a key Do not compare this key with our database primary key. So the key can be null The key can be anything here The concept here is that the message whatever we produce that can have a key We have the Java hash code function, right? Similarly Kafka has some logic. This is just a pseudocode to give you an idea So in a high level what Kafka does is that? They pass the key To find the hash code Then by using the mod operator Number of partitions we have in the topic. They find the partition number This is where the key is used If you check the Kafka source code, this is the exact function to find the partition but same concept Now let's take our bank application one more time we have created a topic with the two partitions like this Let's imagine like that. We produce Deposit events withdraw events, etc but this time we produce our events with a key and We choose the account number as the key Because of this reason what will happen here is that? When the application keeps on producing messages any Message related to the account number a1 is guaranteed to go to the same partition Even if we have multiple partitions like this There is no way that the deposit even for a1 to be here and the withdraw event for a1 to be here You are getting right So all the events related to one account will be part of the same partition So any event related to account a1 a everything will be part of this same partition Similarly, there could be another set of accounts like a2 a7, etc So they all will land on this partition. They will not go to this partition The idea here is that within that partition the order is guaranteed based on the key Now, how are we solving the scalability issues here? Since we have multiple partitions and the order is guaranteed within the accounts, right Kafka can happily assign the whole partition to this consumer and this whole partition to this consumer Now look at that we have parallel processing And these consumers can happily process all the events from their partitions one by one because of this approach There is no way that the withdraw event will be processed before the deposit event You are you are getting right? So there will not be any conflict like that This is how kafka solves both the scalability issue and the ordering issue at the same time There are a couple of things we would have noticed the offset Belongs to the partition not to the topic So this offset zero is for partition zero this after offset zero is for partition one So whenever we add messages the offset will keep on increasing um for that partition There is a very good chance that one partition might be having more messages compared to another partition that is Fine actually, but if you want proper distribution, we have to choose the key Carefully, for example, let's imagine that You have chosen date The current date as the message key so on this date If you are producing 10 million messages Where do you think all the messages will land? It will not be distributed between these two partitions. They all will go to the same partition, right? So we have to use Or choose the key Carefully a proper key which will return different different values for example If it's a bank account related event use the account id as the key If it's a user click event, then use the user id as the key If it's order event then use order id as the key and so on you have to choose your key carefully
