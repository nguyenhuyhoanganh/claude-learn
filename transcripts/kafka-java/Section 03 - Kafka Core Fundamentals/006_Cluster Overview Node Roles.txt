# Cluster Overview: Node Roles

In this lecture, let's talk about the Kafka cluster. Running a single Kafka server is okay for local development or lower environments like Dev or QA. But we do not do that for production environments. We will be running multiple Kafka servers together in a cluster mode like this. Here, each box represents one Kafka node. We call the whole setup a Kafka cluster. We use Kafka cluster to give us high availability. That is, any single node failure will not affect our application. In fact, I will be demoing this later in this course, you will see. Then, horizontal scalability. That is, our applications are going to talk to the Kafka server, send the messages to the Kafka server. How many requests can a single Kafka server can handle at a time? So we have to run multiple Kafka instances to distribute the load. The way in which it works is that they are like a family. They all talk to each other, and they all know about each other in the cluster. For example, this instance knows about this instance. Similarly, this instance knows about this instance, etc. Just a quick note that it does not really matter if you are running one instance or 100 instances from the application point of view. Our application code will remain the same. So from the application point of view, the Kafka cluster, it's a complete black box. Each Kafka node can play a role in the cluster. There are only two roles, broker or controller. Broker is responsible for managing the data, storing the data, handling the read and the write request. That is, the producer and the consumer application will be talking to a broker. We can have multiple brokers in a cluster. The controller is responsible for managing the cluster because in a large cluster, then there are multiple machines working together. Someone has to act like a manager and guide these nodes. Hey, you do this. Hey, you do that, etc. So it is done by the controller. At any given time, we will be having only one active controller in a cluster, and the controller does not handle the client communication. Let's say in our Kafka cluster, we have this node acting like a controller. As we said, at any given time, we will be having only one controller, and we can have multiple brokers. So now we will have the same question as we had in the previous lecture. If we can have only one controller, what if the controller dies? Again, it's going to be a single point of failure, right? Actually, no. All other nodes, they can elect another node. In this case, this node will be the new controller for them like this. This is how we assign the roles to a Kafka node. We have one property called process.roles, and the value can be either broker or controller or broker, controller. Let me explain this so that we are on the same page. Let's take this node. Let's imagine that this node is started with this property, process.roles broker. So what it means is that this node will act like a broker only. It is not eligible to play the controller role. So when the controller dies, this node will never be elected as the new controller. We are telling this node that you just focus on broker responsibilities. Similarly, this node might be started with process.roles equals controller. So what we are saying here is, hey, node, you forget the broker-related responsibilities. You can be a controller. So that is what we are saying. Normally, we do that in a big cluster. For example, we can have a cluster with 100 plus nodes. So because in a large cluster, the controller has lots of work to do.
