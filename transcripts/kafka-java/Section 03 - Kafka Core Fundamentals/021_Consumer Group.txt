# Consumer Group

In this lecture, let's talk about a concept called consumer group. This is very, very important, please listen carefully. In the previous lecture, we had a demo and we observed a behavior. That is, we have a microservice called order service and it keeps on sending messages to a Kafka topic called order events. There are two other microservices, inventory service and payment service. Both of them are interested in consuming these events because it has to process inventory and this has to process payment for the order. So both of them are interested in consuming these events and both of them receive the events. So this is perfect. This is exactly what we want. But in real life, I will not be running one single instance like this in production. I would be running multiple instances of inventory service, multiple instances of payment service like this. So the problem here is all the instances will be receiving the same order event. So we will end up doing redundant processing. This is bad. What we really want is only one instance should be receiving that order event. That is, if this instance has received that order event, then all other instances should not be receiving the same event. Similarly, if this instance has received one order event, then all other instances should not be processing the event, should not even receive that event action. So this is exactly what we want. Hopefully you and me, we are on the same page. I assume this is what you also want. Let's discuss how Kafka solves this problem. Kafka has a concept called consumer group. That is, we can run multiple consumers as a group to represent one logical consumer. So here I have four instances of inventory service, right? So I basically have four consumers. So we can configure in a way that these consumers or instances, when they join the Kafka broker or the cluster, when they join, they will tell that, hey broker, we are all part of the same group. We are all simply different instances of the same microservice. So this is how they will join. Kafka will understand this and it will see them as one single logical consumer. So when we have a consumer group like this with the multiple instances, Kafka will deliver the message to only one consumer within the group. So there will not be any redundant processing. It's very simple actually. So here we have the order service. Here we have our Kafka server. Now let's imagine that we are starting two console consumers, normally the way in which we do console consumer, bootstrap server, topic, et cetera. But this time we will be using the group name option, minus minus group. And you have to give some group name, any name you like. Similarly, let's start another console consumer like this. Okay. Same topic, et cetera. And here you have to give the exact same name. You have to use the same group name. So now the Kafka broker will understand that, okay, so I see two consumers belong to the same group. It will understand that. Now when the Kafka broker receives the order events, it will deliver the order events to one of the consumers within this group like this.
