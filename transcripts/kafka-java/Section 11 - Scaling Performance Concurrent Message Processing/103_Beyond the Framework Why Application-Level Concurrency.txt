# Beyond the Framework: Why Application-Level Concurrency?

In the previous lecture, we saw the concurrency feature provided by the framework. It was straightforward, just update this configuration property, there is no code change required. But as you also saw, the maximum number of threads we can create is limited by the number of partitions we have for the topic. If the producer is publishing messages very fast, then the processor may still not be able to keep up. So in situations like this, what can we do? Let me share a couple of ideas or techniques that we can apply at the application level. Whatever we had discussed earlier was framework level concurrency. Beyond this point, we are going to discuss application level concurrency. This is the concept, we are going to solve this problem by consuming records in batches like this. That is, we will be enabling the batch mode message consumption, we will be getting a list of orders from the order events topic, then we will be creating additional threads ourselves in our application level, and we will be doing message processing by using multiple threads, each message can be processed independently to produce the output delivery object. To make these ideas easier to understand, I am going to split the discussion into two scenarios. This is the problem context. The producer publishing speed is very high, but the processor or consumer processing speed is slow, and the framework level concurrency is not sufficient for us. So we have two scenarios. In scenario one, the message ordering is not important. Remember that not every single application in the world is the banking application or mission critical application. It is normal and many real world applications do not need message ordering. So in the scenario one, we do not care about message ordering. In the scenario two, we will assume that message ordering is important based on a custom key. So let's go through them one by one. First, let's focus on this scenario one. I am going to call this unordered concurrent message processing. That means message ordering is not important. In this approach, we will be updating our delivery processor to consume records in the batch mode. So instead of getting one record at a time, we will receive the list of order. We will then process this list using multiple threads. This is our process method. Let's imagine like that. And we take this list as our input parameter. And we process each item independently. Each order is handled by a one dedicated virtual thread. Since we have already said that this involves a network call, virtual threads are a good fit for this. They work very well for blocking IOT. Once all the processing is done, we will build the list of delivery object and we will be returning that. So the virtual threads are created inside the process method and they are destroyed once the delivery objects are built. For this approach, we can set the framework level concurrency to one. We do not need this because we are handling the concurrency ourselves in this case. So the pseudocode, the framework level pseudocode will look like this. This is our bean. So the framework will be creating a handler like this for the delivery processor. Then it will create the Kafka consumer object. It will connect to the broker. Then it will pull for records from Kafka in the batch mode. So whatever the record it receives, it will be giving them to the handler. Whatever the handler returns, in this case, the list of the delivery objects that will be sent to Kafka topics. Since we are not processing records one by one and instead of processing them concurrently, now the processor will be able to keep up even when the producer is very fast.
