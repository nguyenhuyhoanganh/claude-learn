# Performance Goals: Scaling Beyond Multiple Consumers

In this section, we are going to talk about concurrent message processing. First, I want to show you the concurrency capabilities that Spring Cloud Stream provides out of the box. These are useful, but you will see that the options are somewhat limited. Then we will look at a few practical techniques, how we can use multiple threads as part of message processing to speed things up and improve the overall performance of our application. For demo purposes, we need a concrete example, so I am going to reuse the same project setup we built in the content-based routing lecture. Here we have a producer that produces order events. Then, based on the product type, our processor will be building the delivery object, digital delivery or physical delivery. Then it will be sending that to appropriate topics. From there, our consumers, the digital delivery consumer or the physical delivery consumer will pick up the messages. This was the basic idea, right? Let's use this project. Now, I am going to add one constraint to make this more realistic. The processor is doing some work. It does not just simply forward the messages. It has to build the delivery object. And in the real life, it often means calling a database or calling another microservice and so on. But for this demo, we are going to simulate that the building one delivery object takes around like 200 milliseconds. 200 milliseconds might sound too much. Oftentimes, database calls will not take that much time. But let's keep it like this. With that condition, with that constraint, we will see how concurrent message processing can help us to handle higher throughput and improve the application performance.
