# Section Summary & Key Takeaways

Let's have a quick summary. By default, our consumer and the processor applications process every single message sequentially. There will be only one thread. To scale and do concurrent message processing, we can configure the concurrency value for the consumer or the processor binding as we see here. The framework will create multiple threads to connect to the Kafka broker which will kind of simulate multiple consumer instances from a consumer group. Each thread will be assigned one partition. So by creating multiple threads, we can process multiple partitions concurrently. This is a good starting point. But as you see, it depends on the number of partitions we have for that topic. If you only have a few partitions and the producer producing speed is very high, then we can also do concurrent processing ourselves. For the application level concurrency, if the message ordering is not important, then it's very easy to consume the records in the batch mode, then assign each item to a thread to process all of them concurrently. If the message ordering is important based on some custom key, then use that to partition the list into multiple smaller buckets like this. Then assign each bucket to a thread so that we could have concurrency across buckets, sequential processing within the bucket.
