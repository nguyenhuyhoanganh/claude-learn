# The Need For Event-Driven Architecture

Thank you so much for your interest in this course. In this course, we move beyond basic REST based communication. We are going to build a system that is resilient by design using Spring Cloud Stream and Apache Kafka. Let's talk about what is happening in our industry now. AI can generate a lot of routine code, exposing thread endpoints, mapping DTOs, generating boilerplate almost instantly. So if our role as engineers is only to produce that kind of code faster, we are actually competing with the AI which never sleeps. The real value of an engineer is shifting. Your value is not how fast you can write code. Your value is architectural integrity. Companies need engineers who can design systems that actually work in the real world. Systems that can scale, handle failures, unpredictable traffic, etc. That is why companies increasingly look for engineers who can think in terms of architecture, not just implementation. Let's look at how the backend systems actually look in most companies. On paper, they are called microservices. The architecture diagram looks clean, many small services, each with a clear responsibility. But in reality, most of these systems behave like distributed monoliths. Why? Because the services are connected through synchronous REST calls, forming a long dependency chain. Service A calls service B, which calls service C, which calls service D, and so on. The runtime behavior is still tightly coupled. So if we send a request, we will receive the success response only when all the services are available fast and error-free. Let's break down those problems one by one. Let's take a simple example, an e-commerce application. A customer places an order, the request first reaches the order service. Now because the system is split into multiple microservices, the order service cannot complete the request by itself. It has to call other services synchronously. It calls the product service to get the latest price. Then it calls the payment service to charge the customer. The payment service might internally call a fraud check service. If the payment succeeds, then the order service calls the inventory service to reserve the item. Then it calls the shipping service to arrange delivery. The shipping service may call the notification service to inform the customer about the delivery date, et cetera. So it looks like a single request to the customer, but it's actually a long chain of service to service calls happening in sequence. And the customer sees a success response only if every service in that chain is available and responds quickly. What if the fraud check service is slow? Then the payment service cannot proceed until it gets a response. The order service is waiting for the payment service response and the customer is waiting for the order service response. So a delay in one service starts propagating through the entire chain. In synchronous systems, the latency accumulates. And from the customer's perspective, the entire application feels slow, even though only one service had a problem. Now consider a different scenario. What if the fraud check service is not available? Maybe it's down. Maybe there is a network issue. Then the payment service cannot complete its work. So it fails. Since the payment failed, the order service cannot proceed as well. So the entire order flow fails. At this point, the customer's order request is lost with the 500 internal error. So the customer has to retry, assuming they will try again. Some customers might retry. Some customers will not. So a failure in one internal service becomes a failed user experience. In synchronous systems, the availability of the entire operation depends on the availability of every single service in the chain. Now let's say the business wants to introduce a new feature, a recommendation service. Based on the current order, the recommendation service will suggest related products to the customer. If you like product A, you might like product B and so on. Sounds simple, right? But in a synchronous architecture, this is not just adding a new service. The order service now has to call this new service as part of the flow, which means modifying the existing code, testing the entire chain again and introducing one more dependency that must be available and fast. Every new feature increases the length of the dependency chain and the risk of failure. So even small changes become difficult and risky because you are touching the critical order flow. In a tightly coupled synchronous architecture, adding new capabilities is not easy. It's dangerous. Let's quickly step back and look at the bigger picture. In synchronous microservices, communication becomes complex and tightly coupled. Every service depends on other services responding correctly and quickly. If one service slows down, the entire flow slows down. If one service fails, the whole operation can fail. If we want to introduce a new capability, we risk breaking existing flows. All the services must always be available. Otherwise, the request can fail. At that point, we lose the customer's order, hoping customers will retry. So even though the system is split into many microservices, the behavior is fragile. Now the question is, can we design systems without these problems? Yes, event-driven microservices. What if the services do not have to call each other directly? Instead of sending requests and waiting for responses, these services can publish events and move on. Other services can react to these events independently whenever they are ready. This is the core idea behind event-driven architecture. Communication becomes asynchronous and non-blocking. Services become loosely coupled. The system no longer depends on every service being available at the same time. In the next few slides, let's revisit each problem we discussed and see how this approach changes the outcome. In an event-driven system, the order service no longer coordinates the workflow. When the customer places an order, it simply publishes an event, order placed event, and immediately returns 202 accepted response and its job is done. The order service does not need to know which service will process the payment, which service will update the inventory, arrange shipping, etc. Each of these services subscribes to the order event and handles its own responsibility independently. So the complex orchestration disappears. Communication becomes simple. Publish once and react independently. That is how the event-driven architecture removes the need for complex synchronous coordination. In an event-driven system, a slow service does not block the entire flow. When the order service places the order placed event, it does not wait for anyone. If the payment service is slow, only the payment processing is delayed. Other parts of the system can continue independently. The system becomes non-blocking by design and the impact is isolated to that service. Similarly, in an event-driven system, a failure does not immediately break the entire operation. The order placed event is stored durably in the messaging system. If the payment service fails while processing it, the order placed event is not lost. The customer does not have to retry. The service can retry processing later when it recovers. So instead of failing the customer's request immediately, the system shifts to eventual completion. Failures are handled as part of the workflow. In an event-driven system, services are decoupled from each other. The order service does not know about other services. That means we can easily introduce a new service, like a recommendation service, without modifying the order service. The new service simply subscribes to the order placed events and it starts working. No changes to the critical workflow. This is what makes the event-driven systems easier to evolve over time. In an event-driven system, services do not have to be available at the same time. When the order placed event is published, it's stored in the broker. Other applications can process it whenever they are available. This removes the requirement for all the services to be up at the same time. Services interact through stored events, not live requests. So by shifting from synchronous communication to event-driven communication, we fundamentally change how the systems behave. Dependencies become looser, failures become manageable, scaling becomes practical, and adding new capabilities become safer. Event-driven architecture is not just a different way to communicate between services. It's a different way to design systems for the real world. There are some challenges as well, specific to the event-driven architecture. And in this course, we are going to learn how to build these systems step by step. To take this course, you should be comfortable with Java 8 or above, Spring Boot and Docker. But no worries, I am going to cover Kafka from scratch in this course.
