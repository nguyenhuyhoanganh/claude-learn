# Reactive Architecture: Sinks vs. Flux

In the previous lecture, we have updated the MovieService and the CustomerService to notify the RecommendationStreamService when we consume these events. Now, in this lecture, let's discuss how are we going to send the real-time recommendation. To send the real-time notification, in the traditional MVC applications, we have something called SSEEmitter object. You might have used, we need to create one SSEEmitter object per user. Using that, we can send the recommendation. It will work fine, but we have to be careful with our implementation that we have to manage the connection lifecycle. That is, in the real life, there could be thousands of customers, so we need to know when the customer closes the browsers, etc. So that, we need to dispose the SSEE object ourselves. This will complicate the implementation. This is why, if you read the Spring documentation, they will recommend Spring WebFlags or Reactive Programming in general for streaming requirements. Reactive Programming is all about non-blocking streaming communication with the back pressure. The connection lifecycle is handled automatically for you. It's kind of in-built. We do not have to do anything special there. However, it is not like you have to use Spring WebFlags. We can still use Spring MVC, but with Reactive Programming. We can easily achieve that, and this is what we are going to use instead of the SSEEmitter. You do not have to be an expert in Reactive Programming to implement this. I will try to keep this simple. If you have not used this before, it could be confusing at first. I will explain. No worries. But remember that the Reactive Programming is completely a different way of programming. So assuming you have no experience, I can only explain in a super simplified way. If you are curious and you need to learn thoroughly, I have a couple of courses. Please check them out. In Reactive Programming, we have something called Sync. And this is how we will be creating the Sync object. Imagine this Sync like a tube. So through this one end of the tube, we will be sending items like this. Whatever you want to send, you can send. Usually it will be owned by the application. In our case, our recommendation stream service will be owning this Sync tube. So using this one end, we will send some items. The other end of the tube, we call that Flux. Basically from the Sync, you create the Flux. This Flux represents the other end of the tube. This Flux can be subscribed by multiple observers. So in our case, our observers are customers. So multiple customers can subscribe to this Flux. So whatever you send via this Sync, our customers can receive them. So this is how we are going to implement the real-time recommendation. So this is a very, very high-level idea. Now let me explain these options. Sync is a class here. We have one factory method, Mini. So what it means is that I am going to create one Sync object using which I am going to send multiple items. So this is what we mean by Mini. There is also one option to send only one object via Sync. In our case, we will be sending multiple recommendations. So we use the Mini option. Then what is this Multicast? As I had mentioned already, the other end of the tube is represented by this Flux. This Flux will be subscribed by the subscribers. So they are observing. So when we say Multicast, there will be multiple subscribers. So whatever I am going to send, everyone will receive that. So that is Multicast. There is also one option called Unicast. That means the Flux can be subscribed by only one observer. But in our case, there will be thousands of customers. So when we have the new movie recommendation, when we send, everyone should get it. So we use the Multicast option. Basically, I am broadcasting to all. That's what it means. Then onBackpressureBuffer. In the reactive programming course, I have a dedicated section to talk about this for 30 minutes. But I cannot do that here. So in simple terms, whatever I send via this sink has to be delivered to all the customers via network. What if I emit too fast? What if I send too many items? So there is an internal queue to buffer the items. So what should be the queue size in that case? So I am saying that use a small queue. When I say small queue, in this case, small buffer size, the default size is 256. But it's configurable anyway. So these are the options I use to create the sink object. When you say sink.asFlux, then you create the other end of the sink object, which is Flux like this.
