# Services API & Event-Driven Architecture Discussion

Now, let's zoom in on the movie service. We will have one table called movie and it will have a few fields like this. Actually, we will have more fields than this, but this is enough to give you an idea for now. We will have a couple of endpoints like this. One get API to get the movie details by the given movie ID. Then one post endpoint to create or add a new movie. Now you might ask, what about movie update or movie delete, etc. Please remember that our goal here is even driven communication among microservices, not to spend time on how to develop REST APIs. So this is enough for us. Whenever we add a new movie, we will be inserting one record into the movie table. Then we will be emitting one event like this movie added event via movie events topic. Very simple. Similarly, for customer service, we have these table and these endpoints. One get API to get the customer details for the given ID. Then one patch endpoint to update the customer's favorite genre. That time we will be triggering this event via customer events topic. Here we are using patch endpoint because we are not going to replace entire customer information. We are going to replace just the genre. The recommendation service will have a couple of tables in its database. It will be consuming this customer genre updated event and it will update that customer one likes action movies and so on. It needs this information for generating personalized recommendations. Similarly, the recommendation service will consume this movie added event and it will update its table like this. Now you might be wondering the recommendation service basically has one table called movie as the movie service has. Similarly, the recommendation service has one customer table as the customer service has. Right. So is it not data duplication? Yes, but in the event driven architecture, we do that intentionally. We do this so each service can run independently. The recommendation service will still work even if the movie service is slow or currently down. That independence is what gives us the scalability and resilience. So it's a feature, not a flaw. This is an important concept in the event driven architecture, which helps us to implement other complex patterns like CQRS, even carry the state transfer pattern, etc. Even though the recommendation service stores the movie information in its database, the movie service will be the main source of truth for any movie information, not the recommendation service. For example, when we add a new movie here, we might be having information like title, IMDB rating, budget, revenue, some overview or the storyline about the movie, etc. But the movie service will not emit all these details. Instead, it will give only a very basic information that, hey, I have added this movie. I have given some basic information here via this event. If you need more details, just use my get API to get the more details. So this is how we are implementing. Hope it's clear. The recommendation service will have these two endpoints. One get API to provide movie recommendation for the given customer ID. Another get API to push the recommendations real time. We have two types of recommendations, newly added and personalized. When the users, when they access the application or login using the get API, we will see the home page with the recommendations. Then if the users are still active or still connected, then in that case, in whenever we have a new movie added, right? So that time we will be updating the recommendation and push it via this stream endpoint to the users. Cool, right? So this is how we are going to implement.
