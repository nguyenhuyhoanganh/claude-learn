# Architecture: How Offsets and Acks Work Together

In the previous lecture, we had discussed that Kafka expects an acknowledgment from the consumer. Now, you might be wondering, so far, we never explicitly acknowledged anything, right? For example, when we used the console consumer or in all the demos where we used Spring Cloud Stream to develop our consumer application, we never wrote any acknowledgment logic. That is mainly because both the console consumer and the Spring Cloud Stream, they do the acknowledgment under the hood. The console consumer acknowledges the messages as soon as they are printed on the screen. Similarly, when we use Spring, we will be exposing our Bean like this. Spring Cloud Stream will be creating an instance of the consumer like this. Let's call that handler. Then the Spring will be asking the Kafka broker for messages. In return, Spring will be getting a list of messages. The list could be empty if there are no messages. The list might contain just one message. Or the list could be containing 500 messages, depending on your maximum poll records configuration. Once the Spring gets the list of messages, then it will be delivering the messages to our handler like this, one by one. If the handler executes successfully without any error, the framework will be acknowledging the messages automatically for us. So in most cases, we do not manually acknowledge the messages. The framework will do it for us. Also, this acknowledging the message is a network call. So let's say Kafka gives us 100 messages. So we do not acknowledge one, two, three, four. We do not acknowledge one by one like that. Instead, once all the messages are handled, then Spring will be acknowledging the 100th message. So Kafka will update the offset as 100, 101, etc. Consider this scenario because I want this to be very clear. Let's say our topic partition has eight messages like this. This is our consumer application. Our consumer asks broker, hey broker, can you give me four messages? Maybe the maximum poll record is set to four. So that is how the consumer asks. Kafka broker will give those four messages. Now the consumer processes all the messages one by one. The consumer does not acknowledge one, does not acknowledge two, and does not acknowledge message three. But the consumer application sends the acknowledgement for message four. Okay. Now the consumer application crashed. Let's imagine like that. It restarts. Now it again asks for four messages. Okay. What are all the messages Kafka will deliver? What do you think? Will it deliver one, two, three, and five because it acknowledged four? No. Remember that when the consumer application acknowledges this message four, right? It's implicitly assumed that the consumer application has seen up to four. Okay. When you acknowledge four, that basically means that you have processed up to four. So that's what it means. Many people do not get this. So I want this to be very clear. So in this scenario, when the consumer asks for the four messages, Kafka will be giving five, six, seven, eight. Okay.
