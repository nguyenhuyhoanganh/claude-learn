# Manual Acknowledgement in Batch Mode

This will be a quick lecture. If we use the batch mode to consume our messages, that time spring will give us the message of list of key, right? We had said that in that case, the key header will be containing the list of keys, the payload will be containing the list of events and so on. So we might think that the acknowledgement header will be containing the list of acknowledgement objects, right? No, it's not true, we will have only one acknowledgement object. First of all, remember that this acknowledgement object is spring abstraction. It's not really part of the message which came from the Kafka broker. When we enable the manual acknowledgement, that time spring simply adds that in the header so that we could access this acknowledgement object. So in this case, for the list of events, we will be getting only one acknowledgement object. So this is how it will look like. So you get the message of list of key, you get the acknowledgement like this, you process all the events, then you acknowledge. So the entire batch is acknowledged. Now I'm sure you will be having this question. So let's discuss this. Let's say we are getting one list. That list contains 500 items. So I am accessing the acknowledgement like this, then I am processing all the 500 items one by one. I processed 498 items successfully, okay? The 499 item fails, some error is thrown. So we catch the exception, we do the knack. What will happen now? So this method will be invoked again and you will be getting the entire list once again. So in that case, oh my God, will I not do a redundant processing? I will be processing all the 498 items, right? Yes, we will do that if you are not careful. The issue is exactly same as the previous issue we had discussed just a few lectures ago. This is a design problem. It has to be addressed differently. Check the best practices section later.
