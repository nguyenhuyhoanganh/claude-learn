# Cluster Overview: Replication Factor & Partition Distribution

In this section, let's set up a Kafka cluster. That is the main goal of this section and also to understand how Kafka achieves high availability and horizontal scalability. Earlier, I had mentioned that we will not be running a single Kafka server in production. Instead, we will be running a bunch of nodes or instances together as a cluster to provide us the high availability and horizontal scalability. I made it sound like it was very easy. Just add a bunch of nodes and you will get high availability and horizontal scalability. Just like that. That is how I said, but it does not really work like that. Think about a completely stateless application like a typical Spring Boot microservice. Scaling this is very straightforward. You spin up more instances. Put a load balancer in front and that's it. Every instance is identical. Any instance can handle the incoming request. So here we do not have much challenge. But Kafka is very different. It's a stateful application. It stores data on the disk. So all the instances, they are not saying, for example, this could be order events. This could be product view events. This could be inventory events and so on. So if you want to produce order events or if you want to consume order events, you have to go and talk to this broker, not any random broker in the cluster. You have to go to specific broker. So this is where Kafka clustering becomes interesting and also more complex. We have seen this earlier, but I'm going to clarify or add a few more things. So I will have to start from the beginning one more time. Sorry about that, but I will not take much time. So it will be like a quick summary. We will have multiple servers running in our Kafka cluster. When we start the individual server, we specify the process.roles property like broker or controller or broker, controller and so on. So the broker role or the broker node handles the client read and write request. Most of the nodes in our cluster will be brokers. Some nodes will have the controller role. So when a node has the controller role, it does not mean that it is the controller. It simply means it can participate in the controller election. At any given time, we can have only one active controller in the cluster. The controller is responsible for managing the cluster. If the controller dies for some reason, then we will have another election among controllers and so on. We do all this, the broker role, controller role, etc. in the cluster to provide capacity and fault tolerance. It does not create high availability and horizontal scalability automatically. To better understand this, let's imagine a working cluster, a cluster with multiple brokers. Let's say I want to create a topic. As we know, topic is an abstract, topic is nothing but a collection of partitions. So I am creating a topic called product view events with one partition. So the controller will select one broker, let's say this broker and ask this broker to be the leader for that one partition. Now, as you see, I can have thousands of brokers in the cluster, but only one broker is responsible for that partition or that topic, right? If you take websites like amazon.com or any e-com site, thousands of people will be searching for products and viewing products, etc. So you might be running 100 product service instances in production to handle the load. All these instances will be sending the product view events to this one single broker because we have only one partition for the topic. Hopefully you see the problem. This is your Spring Boot microservice. This is a completely stateless application. So running hundreds of these instances is no big deal. So scaling this out, it's not a big deal at all, but all the hundred instances, they all will be bombarding this single broker to send their events because we have only one partition. Similarly, when you have only one partition in your consumer group, you can have only one consumer to consume the events. Again, this is something we have already discussed. So partitions are the fundamental unit of parallelism in Kafka. If you want to scale, you need to have more partitions. If I create a topic with the two partitions, then the controller might select two different nodes. One, this broker for partition 0 and this broker for partition 1 to distribute the load and the data. So brokers provide capacity, partitions provide scalability. But what will happen if this broker dies? Yes, it's not highly available. So partitions provide only scalability, but availability, that's a different problem. To provide high availability, we need to enable replication. There is something called a replication factor when we create a topic. So far, we had not used that because we need a cluster to do that. Now we are talking about a cluster, we are going to set up a cluster. It's time to talk about this. The replication factor tells how many copies of each partition should be present in the cluster. Let's say we create a topic with the two partitions and the replication factor 3, two partitions and replication factor is 3. So what the controller will do is that the controller will select two brokers to be the leader partitions. So this is the partition 0 leader, this is the partition 1 leader. So two brokers to be the leader for the partitions. Now the replication factor is 3. What it means is that one primary copy, so whatever the leader owns, we call that primary. Then it will select a few other brokers to be the followers for these leaders. That is the one primary, here we will have two more copies of this primary. So if this is partition 0 primary, here we will have the partition 0 copy and here we have partition 0 copy. So the data will be continuously replicated. Similarly, this is the partition 1 primary. Here the data will be replicated into other brokers like this. So when you say replication factor 3, that includes the primary as well. One primary, two copies, one primary, two copies. So this is how it works. When you say the replication factor is 1, then we have only one copy of the partition. We have only primary, that's what it means and there is no redundancy. In that case, if this broker dies, then the data is unavailable. So if you want to provide high availability, that is when this broker dies, one of the other brokers to become the leader for the partition, then you need to ensure that you set the replication factor for the topic to be greater than 1. So this is how we provide the high availability and horizontal scalability in Kafka. Brokers provide capacity, partitions provide scalability, replication provides availability.
