# Section Summary & Key Takeaways

In this lecture, let's quickly summarize whatever we had discussed so far. We learned how to build a Kafka producer application using Spring Cloud Stream. We will be creating a supplier bean like this. Then we will provide the configuration to define where the messages should go. If the supplier returns an event like this directly, then there is a challenge. The Spring Cloud Stream needs polar configuration to invoke the get method again and again to emit the data. The actual Kafka message can contain key additional metadata along with the payload. So if you want to produce the events with the keys, then we can use the message builder utility. Here we provide the payload key and other custom key value pairs to build an object like this. In this case, our supplier will be returning the message of T. The consumer can also consume the message of T like this if it wants to access the key topic name, partition information, etc. But it's completely optional at the receiving side. But when you produce items with the keys, you must return message type. As we had discussed earlier, Kafka stores everything as bytes. The producer application should serialize the message. The consumer application should deserialize the message. The Spring Cloud Stream will handle all these things for us. It can serialize the payload. It can serialize the custom headers, but it will not touch the key because the Spring Cloud Stream will not want to assume that type of the key. The value one can have different bytes representation depending on the type. It will be affecting the partition, message ordering, etc. So we have two options. Since Spring Cloud Stream does not want to do this, we have two options. Either we can provide raw bytes as key if you want, or we have to configure the serializer deserializer. So that the producer API will convert the key to bytes using these utility classes. Remember, you have to configure this both sides, serializer here, deserializer here. Stream Bridge is a Spring component from Spring Cloud Stream module. It gives us the ability to produce messages dynamically. Whenever we think that the supplier model is limited, we can use Stream Bridge. As Stream Bridge is a Spring component, you can auto wire wherever you want. Simply send messages when something happens, like when products are viewed, when orders are placed, and so on. Then we use the supplier type to create the Kafka producer application. The binding name will be derived. But when we use Stream Bridge, we will be using custom binding name like this to provide the configuration like topic names, etc. We can also send the messages directly to the topic. It's not a good practice for production application, but it will be useful for writing integration tests. We will do that later as part of upcoming sections. Reactive Programming Developers, for you, everything is same, there is nothing special. I feel like reactive programming simplifies a lot here. Our supplier will be returning a flux of T. So, the Spring Cloud Stream does not need any pooling. It can simply subscribe to the flux internally. Stream Bridge is a good utility, but we do not need that to produce messages dynamically if we use reactive programming. We will have the sync object like this. Using the sync object, we can simply emit data. From sync, we can create the flux and the Spring framework will be subscribing to the flux. So whenever you emit the data, the framework can receive that item and it can send that to Kafka topic. Very simple.
