# Limitation Of Poller Based Supplier

In this lecture, let's talk about a problem with the supplier type and discuss how we could solve this. We currently create our producer like this. Then we provide some binding configuration under the hood spring creates an instance like this. Then based on the polling interval, it invokes the get method again and again since the item provided by the supplier to the Kafka topic, we already know this. But what if I want to emit an item based on some condition, for example, whenever a product is viewed. In the real life, not all the events are poll based, our requirements could be like emit an item when a product is viewed by a customer, when the order is placed, or when the inventory drops below certain threshold and so on. The problem is if we create the bean like supplier of tea, then we cannot achieve the requirement because the supplier get method has to be invoked again and again to get the item. So instead of this, the supplier has to return a publisher which provides tea, for example, supplier of flux of order event. So the flux represents a stream. People who are familiar with the reactive programming can understand this easily. Basically, the order event producer returns a supplier that gives flux of order event. So internally, the spring framework can create the supplier like this. By using the supplier dot get method, it will be getting the flux. By subscribing to the flux, the spring framework simply observes. So whenever the flux emits the event, it can simply observe and send that to Kafka topic. The spring framework does not need any polling. The problem is easily solved for reactive programming developers that said this is exactly what we want. But what if I am not familiar with the reactive programming?
