# Goal: Scaling Consumers with Consumer Groups

This will be a quick section. In this section, I would like to demo scaling consumers with the consumer groups. This is the goal of this section. We are going to have a demo. In that demo, we are going to observe when we run multiple consumers from a consumer group, how partitions are distributed among the consumers, how we could scale message processing using consumer groups, what happens when new consumers join a group or existing consumers leave a group. This is what we are going to observe. This is something we had already discussed and kind of seen as part of the Kafka fundamental section using the console consumer, but we could not observe everything. Now as we know how to develop a simple producer and the consumer application using Spring Cloud Stream, we can demo this better. That is why we are doing this. This is the setup we are going to have. First we will create a topic with the three partitions. We will have one producer. That producer is going to produce thousand items per second, a lot of messages. We will have only one consumer from this consumer group. As you know, Kafka will be assigning all the three partitions to this consumer. We can see that. Now let's imagine your cloud autoscaler or Kubernetes autoscaler spins up another instance because of the load. When the second consumer joins the group, Kafka will do partition rebalancing and assign new partitions to these two consumers. Let's say third consumer joins the group because of autoscaler. Now Kafka will assign one partition to one consumer. So this is scale out scenario. When we scale in, that is when a consumer leaves the group, Kafka will be assigning or redistributing the three partitions among the available consumers. So this is the process. In the end, we can see how many messages the producer produced, then how many messages the consumers processed. The count should match. Simple demo.
