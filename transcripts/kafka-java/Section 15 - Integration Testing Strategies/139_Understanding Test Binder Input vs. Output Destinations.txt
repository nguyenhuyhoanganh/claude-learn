# Understanding Test Binder: Input vs. Output Destinations

Let's write our very first test using TestBinder. Let me quickly explain this to keep all of us on the same page. As I had mentioned already, the TestBinder is a test implementation of Binder API for testing our application. This is our Spring Boot test class. Imagine like this. To use TestBinder, we need to add this annotation. Otherwise, when you run the test class right, that time Spring will see Kafka Binder on the class path and it will try to connect to Kafka. So you have to add this to tell Spring that I am using TestBinder, not Kafka Broker. When you enable TestBinder, you have access to these two beans created for testing purposes. Input Destination, Output Destination. If you do not add this, you cannot inject this. So using the Input Destination, we can send messages to our consumer application. Similarly, using the Output Destination, we can receive messages whatever the producer application produces. Very simple, right? Here we have to give the topic names like order events, digital delivery, etc. Whatever the topic names currently you use in your application, right? So that is what we will be using here. The receive method has timeout. Our test cannot run forever, so we will give some timeout within this duration. If there is a message, we will get it, otherwise it will be null. And if you notice, the receive method gives us the payload only as byte array. It's a minor annoyance. So we need to deserialize this to Java object if we have to validate the object content, like if the order ID is 1, etc., if you want to validate, then we have to deserialize. Hopefully this is clear.
