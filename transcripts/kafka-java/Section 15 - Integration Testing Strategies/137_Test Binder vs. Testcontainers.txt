# Test Binder vs. Testcontainers

In this section, we are going to focus on integration testing. The goal is to understand how we test the message-driven applications. We will look at two approaches, the test binder and test containers. Both serve different purposes and we will see when to use what. The test binder is a test implementation provided by Spring Cloud Stream. It allows us to run our application without connecting to a messaging system. When we use the test binder, the messages do not go to Kafka. Instead, they go into an in-memory queue inside the same JVM. Whatever your application produces, it will be added to the queue. Your consumer application will be getting the messages from the queue. So, there is no broker, no network, no Kafka client involved. We do not need a heavy infrastructure to run the test. So, this makes a test fast and simple and is useful for testing our application business logic. So, the focus here is application business logic, not your broker or Kafka specific behavior. Test containers take a different approach. It starts a real Kafka broker inside a Docker container during the test. Our application connects to Kafka using the real Kafka configuration. It needs the Kafka binder. Messages flow through Kafka exactly as they would in the production environment. This will be useful to validate Kafka specific behavior. So, what should we use? Test binder or test containers? We will use both for different reasons. Most integration tests can be written using test binder. Using this, we can validate the business logic and these tests will be very fast and they will cover most of the application behavior. Then, we can add a small number of end-to-end tests using test containers. When you produce and consume messages with the keys, we have the key serializer and deserializer and all, right? Those are all Kafka specific. So, we can add a few tests to validate our application end-to-end with the Kafka broker using test containers. This will give us confidence that our application will work correctly with Kafka. If you are not sure which one to use and if you are confused between test binder and test container, the answer is always test container. Remember that here, we are testing against real Kafka broker. We are mimicking the production-like setup. So, you cannot go wrong with the test container. So, using test container is safe, actually, for your application. The test binder is fast and it's very simple. So, that is the only advantage we have here. But, this is not mimicking the production setup you are getting, right? So, having fast test and all, it's good to have. But, testing the real production setup, it must have. The quality is super important, right? So, we do not want to face any issues in production. So, you should definitely have a few tests using test containers for your application.
