# Section Summary & Key Takeaways

In this lecture, let's have a quick summary for what we did in this section. This section was mostly about writing integration tests for our event-driven applications. We discussed two different approaches, TestBinder and TestContainers. The TestBinder approach uses a test implementation of Binder API. It does not really connect to any messaging system or infrastructure. Instead, it uses an in-memory queue. Whatever the producer application produces goes to the in-memory queue. The test can get the messages from the queue and validate and vice versa. Very easy to write the test and validate your actual complex business logic. The TestContainers approach uses a real Kafka broker using Docker containers. We use the actual Kafka Binder as we do in production. My recommendation would be whether you use TestBinder or not, it's up to you. But you must have few tests using TestContainers. This will give us more confidence as we test against the real Kafka broker. To use TestBinder in your test, you have to enable TestBinder like this. Then you will have access to these two beans. Input destination, output destination. Using these, we should be able to send and receive messages in our test. Very easy, super simple. When we use TestContainers, writing tests might look like we have to do some additional steps. But those are all actually simple. Nothing complex. When we have the consumer application, your test will be acting like a producer application. And produce messages using stream bridge so that the consumer application will receive and you can validate that. Similarly, if your application is a producer, then you will be exposing a consumer bean in your test to consume messages and validate that. If you have a processor type application, your test will be acting like both producer and consumer. When you expose test beans as part of your test, please ensure that you are including the Spring Cloud Streams configuration in your test. Do not forget that.
