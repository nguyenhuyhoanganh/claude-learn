# Resilience and Developer Expectations

In this section, we are going to talk about error handling. The goal here is to understand how error handling works when we use Spring Cloud Stream. Specifically, we want to understand how retries work, how retry delays are applied like fixed delay and exponential back off and how we control which exceptions should be retried and which ones should not etc. First, I want to keep all of us on the same page. What do we normally expect from a message processing system when things go wrong? Because error handling is not about Spring Cloud Stream, it is about the system behavior. So let's imagine our consumer application like this and let's start with the simplest case. Here we are receiving a message. If the message is processed successfully, the expectation is very clear. We acknowledge the message and the Kafka commits the offset. End of story. Now, what if the message processing fails? Here we throw some exception. In that case, we should definitely not commit the offset because committing the offset would mean telling Kafka that the message is done. But it's not actually. The next question what we have to ask is when we throw exception, we have to check is this failure retryable or not. If it's a retryable failure, for example, temporary database issue, network glitch or some service being unavailable, etc. Then the expectation is simple that the message can be retried or message should be retried. When we retry, we usually will not want to retry aggressively. So we need some retry strategies like fixed delay or exponential backup to control how retries happen. We also cannot retry forever, so we need some maximum attempts to retry. But not all failures are retryable. Some messages could be bad, corrupt data, invalid input or missing required fields, etc. No matter how many times we retry, it's going to fail anyway. Right. So in those cases, we expect the system to skip retries and go to the next message. Finally, there is one more important case. What if a failure is retryable, but it keeps failing even after all the retry attempts. At that point, we do not want the system to be stuck forever. Right. So the expectation is stop retrying, acknowledge the message to move forward and publish the original message along with the error details to a dead letter queue or dead letter topic. So if you look at this table, this is not about spring configuration. This is the behavior we will want from our system. You might be wondering, why are we acknowledging the message in these cases? Or in other words, why are we committing the offset in these cases when the message is invalid or if the retry attempts exhausted? Think about this. This is Kafka topic. This is our consumer application. It consumes messages one by one. Let's imagine like that. So the messages like 1, 2, 3, they are all processed successfully. So we acknowledge the message. So we are good. But the fourth message fails. We can retry two times, three times, maybe 10 times. But can we retry forever? At some point, we need to move on to the next message. Right. So the only way to do that in Kafka is to acknowledge the message. In this case, acknowledging the message does not really mean the message is processed successfully. Instead, the consumer application says it's ready to move to the next message. That is when we acknowledge or commit the offset. This is how normally we implement this by taking the message which fails again and again and send it to another topic. Normally we call this dead letter queue or dead letter topic so that we do not lose this message. Simply accept the original message and move on. This is how we implement. So in the rest of this section, we will see how Spring Cloud Stream helps us implement each of these expectations step by step using retries, backup strategies, retryable exceptions, and the dead letter queue support, etc.
