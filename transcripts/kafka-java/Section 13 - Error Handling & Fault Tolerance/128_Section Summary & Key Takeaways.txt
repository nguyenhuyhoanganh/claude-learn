# Section Summary & Key Takeaways

In this lecture, let's quickly summarize whatever we did as part of this error handling section. Normally, these would be our requirements as a developer for a message processing application. If the message is processed successfully, we are good. If the message processing fails, then we might want to retry by using some strategy like fixed delay or exponential backoff. Not all the exceptions are retryable. So we need the ability to enable retry only for specific exceptions, not for all. After all the retry attempts, if the message processing still fails, we might want to move the message to another topic, which we normally call DLQ, so that we can manually inspect later. So these are the usual developer requirements when it comes to error handling. Spring Cloud Stream supports all these via configuration file. We do not have to write any code for these. We developers will be focusing on the business logic and expose our bean like this. Under the hood, Spring will be doing the error handling, the retry strategies and DLQs, etc. For the retry strategies like fixed delay or exponential backoff with maximum attempts, you provide the retry configuration at the binding level like this. The maximum attempts and the backoff initial interval, the initial wait time. Then you provide the backoff multiplier. If it's 1, then it's fixed delay. Some messages could be bad or invalid. So we might want to retry only in case of specific exceptions, not for all. We can configure that under retryable exceptions like this. Here, we have the ability to enable or disable retry for specific exception. We cannot retry forever. At some point, we might want to move to the next message. At the same time, we also might not want to lose the original message. In those cases, we use DLQ. It's a very simple configuration. You just say enable DLQ is true and provide a name for the DLQ topic like this. Once the retry attempts exhausted, Spring will send the message to the DLQ along with the error details and acknowledge the original message. Spring Cloud Stream provides another cool feature. We have the ability to pause and resume a binding whenever we want. We can use this feature to temporarily pause a consumer if the dependent services are unavailable.
