# Demo: Retryable vs. Non-Retryable Exceptions

In this lecture, let's talk about retryable exceptions. For that, this is the configuration we are going to use. As we had mentioned already, not all the exceptions are worth retrying. For example, invalid input, we do not have to retry this. As we know for sure that it will fail. In the configuration, there is a property called default retryable. Here it means, should we retry all the exceptions? So that is the configuration we provide. The default value is true. So Spring will retry for every single exception our application throws. We can keep this as true or false depending on our requirements. We can also provide explicit mapping depending on the exceptions like this. For example, for input validation exception, we are saying do not retry by providing false here. Similarly, for service unavailable exception, we ask Spring to retry by providing the value true here. So it doesn't matter what you have here, true or false. You are explicitly overriding the behavior for these exceptions by providing one mapping like this. So when the application throws input validation exception, it will never retry. When the application throws service unavailable exception, it will retry irrespective of this configuration.
