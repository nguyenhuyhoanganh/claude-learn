# Advanced Pattern: Dynamic Pausing and Resuming

In this lecture, let's talk about dynamic consumer control. This will be interesting, I guess. So far, whenever something goes wrong in our application, we relied on retries or DLQs, et cetera. This works well for message level problems. But sometimes the problem is not with the message. This could be an environment issue. For example, the database is down or the external service is unavailable. It might take some time for this to recover. It could be part of scheduled maintenance, et cetera. In situations like this, this is our consumer application. So retrying the message here again and again, it's not going to help much, right? What we really want is the ability to temporarily stop consuming messages, let the system recover, then continue from where we left off. The way in which it works is that we will have some health check manager for our dependent services like this in our consumer application. It's just an idea. The implementation does not have to be exactly as I say. Just use this as a concept. So it will check dependent services periodically, our health check manager. When it sees the external services not available or it's not healthy, then it will pause this binding, consumer binding, so that our consumer will not be receiving any new messages temporarily. Then the health check manager will continuously check periodically. When the external services becomes healthy, then it will resume the binding so that our consumer application will be getting the new messages. This is cool, right? How to implement this? It's very simple actually. Spring provides a bin for this bindings lifecycle controller. We have to inject, then use the change state method, provide your binding name and provide the state paused, resumed, et cetera. It's an enum state. So depending on the state you provide, your binding will be paused or resumed. Very simple. We can also get the current state of the binding using the query state method. When to pause, when to resume and all, it's up to us depending on the implementation.
