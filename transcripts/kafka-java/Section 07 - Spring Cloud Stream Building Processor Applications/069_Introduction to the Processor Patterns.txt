# Introduction to the Processor Patterns

In this section, let's build a processor application. That is the main goal of this section and also to understand different processing patterns. Among the event-driven application types, so far we have seen the producer and consumer applications. The processor is producer plus consumer. That is, it consumes events, it might react to that event by emitting another event. For example, the producer order service produces order events. Our processor consumes order events, then it might do some payment processing, then it emits payment events. Then that payment event can be consumed by this consumer, say reporting service, etc. This is the high-level idea. Now I want this to be very clear. It is not like the processor has to emit one event for every single event it consumes. It might emit zero or one or more events. It depends on your requirements. We have different processing patterns. The processor could be a simple one-to-one mapping processor. For every input event, there will be one output event. For example, for every order event it consumes, then it will be emitting one payment event. The processor could be a filter. That is, it will have a condition. If the condition is true, then it will be emitting an event. If the condition is false, then it will not be emitting any event. So zero or one. The processor could be a one-to-many mapping processor. This is one-to-one. This is one-to-many mapping processor. That is, for every input event, there could be multiple output events. You can easily relate this with our intermediate operators in the Java stream pipeline. Map is one-to-one. Filter is zero or one. The flat map is one-to-many. We can use this to understand. Very simple. Nothing special. The processor can also act like a router. We will be talking about that in the next section. Let's focus on this in this section.
