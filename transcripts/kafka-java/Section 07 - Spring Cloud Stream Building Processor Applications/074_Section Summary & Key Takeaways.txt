# Section Summary & Key Takeaways

In this lecture, let's quickly summarize whatever we had discussed so far in this section. We were focusing on building a processor application. A processor is a producer plus consumer. A processor can emit 0 or 1 or more items depending on your requirements. We have three different types of processing patterns like mapping processor, filter processor and one-to-many processor. It is like map, filter, flat map operators in the java stream or reactive pipeline. As part of one-to-one mapping processor, for every single input, you will have an output event. As part of filter processor, you will emit 0 or 1 event. In the synchronous style programming, we have to return null to filter out messages. As part of one-to-many mapping processor, we have to return a list of course, but we have to understand one small difference in the spring cloud stream behavior. The function contract is confusing here actually. So, if you return a list of T, spring cloud stream will assume that the whole list is the one output event. If you want this, yes, we can do this, but if you want to return multiple independent events, then we have to explicitly mention that it's a list of message of T. Only then the spring cloud stream will understand this. By the way, this behavior issue is applicable only for the synchronous style programming. For the reactive programming, whatever you emit in the reactive pipeline that will be sent. So, no issues there.
