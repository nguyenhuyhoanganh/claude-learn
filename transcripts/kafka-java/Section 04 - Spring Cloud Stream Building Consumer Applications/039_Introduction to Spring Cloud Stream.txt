# Introduction to Spring Cloud Stream

In this section, let's talk about Spring Cloud Stream. This is the goal of this section. Introduce Spring Cloud Stream, if you are new to this. If you had already used, please feel free to skip appropriate lectures. Set up a playground project for us to play with the Spring Cloud Stream. Develop consumer application. No worries, we will be doing a lot more as part of the upcoming sections, but in this section, focus on these first. What is Spring Cloud Stream? It is a Spring module like Spring MVC, Spring Data JPA. It is a framework for building message-driven or event-driven microservices. But why should we use this module? What is the advantage? I think you have the Spring Data JPA experience. You should be able to relate this easily. Let's say we have a relational database. Let's assume that we have a table called Order in this database. In our application, we will have a service class for the business logic. Then we will be creating one entity class called Order to represent the table record. Then we will be creating a repository called OrderRepository. This is what we will be using in our service class to query that table. Our service class will not have any clue if we use MySQL or H2 or Postgres database. Spring abstracts all the complexities. When the application starts, the Spring scans the class path and it decides, oh, we have the MySQL driver, so this application uses MySQL. That is how it works. In a very high level, you know that. In the future, if you have to move from MySQL to Postgres, it's a simple matter of replacing the driver and updating this property. If you understand this, then Spring Cloud Stream is kind of similar, but for messaging system. Like a relational database, here we have a messaging system like Kafka, RabbitMQ, Pulsar, etc. We will have the corresponding driver in our application. Then we will have some interfaces to abstract consuming messages and producing messages. Our service class or business logic will not have to worry about what the actual messaging system they use. In the future, if you need to move from one messaging system to another, just replace the driver and update a couple of properties. You are getting right. So this is how the Spring Cloud Stream works in a very, very high level. Spring Cloud Stream currently supports these Kafka, Pulsar, RabbitMQ, Amazon Kinesis and we also have the support for Google PubSub, etc. Please understand that this course focuses exclusively on Apache Kafka via the Kafka binder. My goal is not to use all these binders and show how they all work. Please do note that we are going to use Apache Kafka only. Remember that. When our application talks to Postgres database, we include the Postgres driver, right? Similarly, with the Spring Cloud Stream, if you are going to use Kafka as the messaging system, we will be adding Kafka binder. This binder is nothing but a driver. It's a client library to talk to Kafka. The name binder came up because it binds the application to the messaging system. One interesting thing about the Spring Cloud Stream is that we do not have anything like CRUD repository, okay? So here we will be using the Java 8 functional interfaces like supplier, consumer, function, etc. Very cool, right? To better understand this, first we have to quickly discuss the types of applications we have in an event-driven architecture. Normally, we say client, server for REST API, right? Similarly, in an event-driven architecture, we have producer, consumer and processor. Producer is an application which publishes events to a messaging system. For example, order service emits order events when user places an order. Simple, right? It does not know who is going to consume that event. Its job is to just send it. Similarly, consumer is an application which consumes the event. It could be a notification service. For example, in this case, this application might be consuming the order events and sends out order confirmation email. Then we have a processor. This processor is nothing but producer plus consumer. That is, like consumer, this application will be consuming events. It will be processing that event. Then it can emit new events. For example, let's say Netflix application. The user watches Matrix movie part 1. So, the application raised one event saying that this user watched Matrix part 1. So, this could be a recommendation service. So, it will be consuming that event, then recommend part 2 for that user and so on. Hope you get the idea. Now, let's see what interfaces Spring Cloud Stream uses for these applications. This is it. Very simple, right? If you have a supplier type being, Spring Cloud Stream assumes it is a producer. Supplier supplies values, right? So, in this case, order service emits order events. If you have a consumer type being, it's a consumer application. As you know, consumer interface does not return anything. It consumes order events. If you have a function type being, it's a processor. Here, it consumes order events and returns payment events. As you see, it's extremely easy to develop event-driven microservices using Spring Cloud Stream.
