# The Playground Project

As part of the GitHub repository, I have already shared the entire course project resources which you might be already using at this point. I have also included a playground project for us to play with the Spring Cloud Stream. You can refer to this project as part of the upcoming sections. Let me quickly explain certain things so we are all on the same page. It is a Spring Boot Maven project. It includes Spring modules like Spring Cloud Stream, Kafka Binder, testing modules etc. I try to keep the project up to date periodically. I update the versions and push the updated code whenever the major version changes. So no worries. I normally call these playground projects. What it means is that the whole idea of this project is for learning and experimentation only. To explore different ideas. To see how things work. That is the purpose. Do not compare this with the project which you normally push to production. You do that as part of your job. You are solving the actual business problem there, right? But here the goal is different. You are not solving any business problem. You learn how to solve business problems. Both are not same. So do not treat these projects equally. Once we understand the concepts then finally we will set up a proper Spring Boot project and build an application later to test what we have learned. Hope you get the idea. The project structure of this playground project could be very confusing at first. Please be patient for a couple of sections. You will see why it is organized this way and this is intentional. I will have one package called calmvinsguruplayground. Please feel free to change the package name as you want. Under this I will have several packages like this. Section 01, Section 02, Section 03 and so on. I will not have 99 packages. No worries, but I am just showing the high level format. This is how it will be. Each section is about learning or experimenting a concept. For example, how to develop a consumer, how to develop a producer, how to develop a processor, batch processing, parallel processing, error handling. Each section is independent. Section 02 does not depend on section 01. Section 03 does not depend on section 02 and so on. I might be adding readme files inside the package. I know it is not normal in our actual production grade project you will be having one readme file in the project root directory. But remember this is playground and it does not go to production. Since each section is about a concept, I want to have the corresponding information in that section. In the package itself just to keep things organized. Then under source main resources, we will have several spring cloud stream configuration YAML files. As I just said, we are going to learn a lot as part of each and every section and we cannot use one single application YAML. So I organize like this. By structuring this way, you can revisit this project anytime you like in the future. For example, let's say in the future you forgot how we did the batch processing. Then you suddenly remember, oh yes, I did this as part of section 03. So that now you can go through, check the readme file, check the corresponding configuration YAML files to get the complete idea. So this is why I am doing this way. Again, sorry if you find this annoying, but please be patient for a couple of sections.
