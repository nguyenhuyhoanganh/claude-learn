# Section Summary & Key Takeaways

In this lecture, let's quickly summarize whatever we had discussed so far. We were introduced to Spring Cloud Stream. It is a Spring module to build event-driven microservices. We have three types of event-driven applications. Producer, Consumer and Processor. Producer produces events. Consumer consumes events, but it will not return another event. Processor will consume the event and it will return another event. For example, the payment service could be a processor. It will be consuming the other event and it might return the corresponding payment event. Spring Cloud Stream abstracts all the messaging system related complexities and simplifies the event-driven application development process. It uses the Java 8 functional interfaces to represent the producer, consumer and the processor. We will come across these terms when we use Spring Cloud Stream. When we say binder, it's just a driver for the messaging system we use. When we say binding, we mean the mapping between our bin and the corresponding topic in the messaging system. We provide the configuration in our application YAML or properties files for the binding. These bindings have some naming format to provide the configuration. When we produce items, we will say vname out 0. When we consume items, we will say vname in 0 as the binding name to provide the configuration. The Spring Cloud Stream simplifies the actual code, but it uses heavy configuration. Since the Spring Cloud Stream is a framework and it supports multiple messaging systems, it has some common properties to provide the topic name, the group name, etc. We can also provide binder level configuration as we see here. When you provide consumer properties or producer properties at the binder level, then it's applicable for all the producer bindings, all the consumer bindings. But in some cases, if you want to override at the binding level, when you have multiple consumers, we can also do this way. The Spring Cloud Stream supports reactive programming, so we can also use publisher types like this. But when we develop the Kafka consumer application, in the traditional style programming, we will be using the Java consumer. But it's not recommended for reactive programming because when you use this, you will have to subscribe yourself. That is not a good practice. We should be letting the framework to do the subscribe. For that, we have to use Java function type so that we can return mono of void. Just because we use the Java function type, it does not really mean that it's a processor. To be a processor, we have to return an event. But in this case, we simply return mono of void. So we still consider this as a consumer.
