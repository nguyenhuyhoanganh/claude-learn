# The Business Case for Atomic Transactions

In this section, we are going to talk about Kafka transactions. First, let's start with something we all already understand. Database transactions. Imagine Mike wants to transfer $10 to Sam. Simple scenario, right? Our application receives a request like this. From the outside, it's just a single request. But internally, it's not a single operation. We need to deduct $10 from Mike's account and we need to credit $10 to Sam's account. So we have two operations for one single request. Now, these two operations must behave like one atomic unit. Either both succeed or none of them should happen. We do not want any intermediate state. For example, money is credited to Sam's account, but money is not debited from Mike's account. So that is exactly what a database transaction gives us, right? If everything succeeds, we commit. Otherwise, we roll back. We all know this. We have some experience with this. Now let's translate this idea into Kafka transactions. Let's take a simple processor type application. As you know, a processor consumes messages from one topic and might produce messages to another topic. To keep things simple and for better understanding, let's assume that our producer application produces money transfer requests like this. All those requests are going into a Kafka topic. Our processor application consumes these money transfer requests, does some validation. If everything looks correct, then it splits the request into two transaction requests and publishes them into another Kafka topic. Basically, it consumes this transfer request and splits that into two transaction requests. A credit request to credit money into Sam's account and a debit request to deduct money from Mike's account. So for one input message, we produce multiple output messages. Now this is the important part. We want this entire flow. Reading the input message, producing the output messages, then acknowledging the input message. They all should behave like one single atomic operation. We do not want situations like this. That is, the transfer request is consumed. The processor produces a credit request, but the processor crashed before producing the debit request. So the input message is also not acknowledged. So what will happen now? If the processor restarts, it will consume the money transfer request once again. And it will produce another credit request for Sam, right? So we will credit Sam two times. Similarly, in this case, the transfer request is consumed. The processor produces a credit request and the debit request. But the processor crashed before acknowledging the input message. So what will happen now? If the processor restarts, it will consume the transfer request once again. And it will produce another credit and debit request, right? So we will be crediting Sam two times and we will be deducting Mike two times. What we really want is this guarantee. Either everything succeeds, like consume, process, produce, acknowledge. So they all succeed. Or if anything fails, then the whole operation is rolled back as if nothing happened. That is exactly the problem Kafka transactions are designed to solve.
