# How Kafka Transaction Works

In this lecture, let's discuss how Kafka transaction works under the hood in a very very high level to keep all of us on the same page. The Kafka transaction gives an application the ability to do exactly what's processing. Let's first understand the setup. Assume this is our processor application. We have our Kafka broker and this is our topic. Internally, Kafka has something called a transaction coordinator. It is not a separate service, it's just a component. The transaction coordinator job is very simple actually. It tracks which transactions are open, which ones are committed, which ones are aborted. So, it keeps track of all these things using an internal topic called a transaction state. If our processor application wants to use Kafka transactions, then it has to explicitly say that. In the processor application, we will configure a transactional ID like this. This is extremely important. If we do not set this, then we are not using the Kafka transactions. So, the processor using the transactional ID, it will begin the transaction. At that point, it informs the transaction coordinator that I am having this transaction ID. I am going to produce a bunch of records in the transactional mode. Once I am done, I will either commit or abort the transaction. So, this is how the processor application begins the transaction. Then, it will produce the messages like message 1, message 2 like this. These messages are immediately returned to the topic as usual. Kafka internally tracks which message belongs to which transaction like this. So, these two messages belong to transaction 1. But, these messages are not yet committed at this point. Once the processor finishes its work and everything looks good, then it calls commit transaction. At that moment, the transaction coordinator writes a commit marker like this. This marker basically says the transaction is complete. The processor consumed a message. So, that is also acknowledged at that point. So, everything happens as one single unit of work. What will happen in case of abort? The application will be aborting the transaction in case of error. In that case, the transaction coordinator will be writing an abort marker like this at that time. And, this is very important. It does not behave like database rollback. Kafka cannot delete the messages. So, we have already said that the partition is immutable and append only. So, we can only keep adding messages. We cannot delete the messages. So, Kafka adds this marker to indicate that this transaction is aborted. These messages belong to a failed transaction. So, that is how it works. In some cases, the processor application might crash while processing the records. For example, the processor begins a transaction and it produces a couple of messages. Then, it crashed for some reason. The transaction coordinator will wait for some time. If the processor has not committed the transaction or aborted the transaction explicitly within this time. It is 60 seconds default. Then, the transaction coordinator will abort the transaction itself by placing this abort marker like this. In the real life, we might be running multiple instances of our processor application. So, there could be multiple concurrent transactions as well. So, let's not forget that. We need to ensure that the transaction ID is unique. Otherwise, the transaction coordinator cannot track. So, let's say we have two different instances producing messages in the transactional mode. So, they will be using different transactional IDs like this. Transaction Tx1, Tx2, something like this. So, whenever they produce messages, the messages will be added to the Kafka topic like this. And, Kafka will also be tracking which message belongs to which transaction like this. Let's say that one transaction is committed. The other transaction is aborted like this. So, Kafka will be adding markers this way. And, if you notice, the messages could be interleaved like this. Like one message from Tx1, another message from Tx2 and so on. So, in a very very high level, this is how the Kafka transaction works.
