# Producer-Side Batching: Efficiency & Throughput

In this section, let's discuss how to produce and consume messages in batches. This is especially useful for extremely high throughput applications where we want to produce and consume messages more efficiently. Sending messages to Kafka or consuming messages from Kafka are network cards. So we do not want to send 100 messages one by one. Instead, sending them as a batch, consuming 100 messages as a batch can significantly improve the performance. Let's first look at the producer side changes. As we know, Spring Cloud Stream is a framework. Under the hood, it uses the Kafka client library. The Kafka client library buffers records in the memory and sends them together based on the configuration. These are some producer properties we may want to tune. We already discussed some of these in the Kafka fundamental sections. The first one is the Linger.ms. This property controls how long the producer waits for more messages before sending the batch. If we consider this property alone and set it to say 10 seconds, then what will happen here is the application will be sending messages to the producer API, but the Kafka client will keep all the messages in the memory and it will try to accumulate more messages. Then it will send everything together after 10 seconds as a single batch. However, we usually do not configure this property alone. We will be using this together with the batch.size. Why? It's mainly because what if the application produces a large number of messages in a very short period of time? We do not want the client to keep everything in the memory and get an out of memory error, right? So we use these properties together. Batch.size defines the maximum size of a batch. This is the maximum wait time. This is the maximum batch size. So whichever condition is met first, time or size, the producer will be sending all the accumulated messages to Kafka. The default value for Linger.ms is 0 millisecond. That means send the message immediately. The default value for batch.size is 16 KB. I think this is okay. Then we also have compression. The compression is optional. By default, it's disabled. But if you are sending large messages, enabling compression can help. Now in practice, we can set the Linger.ms to 10 millisecond or 20 milliseconds. We could even set that to 1000 milliseconds if we want, but you will start noticing some lag at the consumer side. If you remember in our console producer demo, messages were produced very fast, but the console consumer received them with some delay. That was because the Linger.ms was set to one second in the console producer. With the 20 millisecond, you usually will not notice much lag in a high throughput application. Even 20 millisecond is enough to accumulate hundreds of messages. But remember that you should be tuning all these values depending on your application requirements. For batch.size, the default 16 KB is a reasonable starting point, but we can also set that to 32 KB or 64 KB. By default, the compression type is disabled. LZ4 is a good choice. It's very fast. It's recommended if you want to enable. We also have other compression types if you want to explore. This is how we will be producing messages in batches. We will be configuring these producer properties. Then in our application, we continue to send messages using stream bridge as usual. The Kafka client will be taking care of batching and sending them to Kafka in batches. No special code changes are required. The producer side is very simple if you configure these properties.
