# Module Summary

And that's it for this course module. In this module we did take a look at Environment Variables which are used to inject Dynamic values into your code. For example the name of the database or all the use them inside of your Workflow definition file as you saw. Now these values these Environment Variable values might differ from Workflow to Workflow and you can define them on a Workflow-Job or Step-level. We didn't see Step-level in action in this section but you can add this End key which we added to the Workflow or a Job also to a specific Step, if certain Environment Variables are only needed in that Step. Now as mentioned these Environment Variables are then automatically provided to the code that's executing on your Workflow-Job runner. And the Environment Variable values are also available inside of the Workflow definition file. So you can for example interpolate them when running commands or use the env context object to get access to the Environment Variables. We also have Secrets which are kind of related but where the idea is that you are storing values that must not be exposed as Secrets. And typical examples would be Database credentials API keys or things like that. Now these Secrets can be stored on Repository or Environment-level and for organizations also on Organization-level. And you can then reference these Secrets via the Secrets Context Object in your Workflow file as you also saw it in this course section. Last but not least we did take a look at these GitHub Actions Environments where the idea is that different Jobs can reference different GitHub Actions Environments. And in your Environments you can add extra protection rules to for example deny the execution of certain Jobs under certain circumstances. But you can also specify Secrets on Environment-level so that different Jobs can reference different Environments and get different values for their Secrets.
