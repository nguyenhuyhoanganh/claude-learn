That's all not needed because we're only interested in the issue that was created here. And GitHub Actions makes controlling Permissions for jobs quite easy and straightforward. Because you can add the special permissions key to a job, to control the permissions for various kinds of actions or areas a Workflow could be acting on. And in our case, it sounds like the issues area is the important one. So, therefore, here, we can go to our job, to this assigned label job, because permissions are managed on job level, to be precise, you can also add them here, on Workflow level, so that they apply to all jobs, but you either have to do it here or for a single job. So regarding the result, nothing changed but we could argue that the Workflow is now more secure because we restricted permissions. And why does this matter? Well, for example, if you would be vulnerable to script injection, those scripts would not be able to do things that are not allowed by Permissions. So you have some additional layer of protection, even though you should of course, try to defend against Script Injection in the first place. But even if you don't worry about Script Injection you also make sure that any actions, you are using are not doing things you don't want them to do. So even if you haven't analyzed all these actions deeply, which you should, if you don't trust the authors but if you haven't done that at least you know what they can theoretically do. And if you have a job that has these permissions, then an action wouldn't be able to suddenly start reading your code or do anything else. So adding Permissions does not protect you against all possible problems, but it is an additional layer you should consider implementing.
