# Using Actions In Workflows

Actions are another key building block, or component of GitHub Actions. And actually, when we got started with GitHub Actions and our first Workflows, I often said that we're creating an Action. But what I actually meant was that we created a Workflow, because technically there is a difference, and an Action is a separate feature offered by GitHub Actions. and an Action, in the context of GitHub Actions, is simply a custom or third-party application that performs a typically complex, frequently repeated task. Like for example, fetching the code from a GitHub Repository, and downloading it onto such a runner machine. Now, the alternative to such an Action is a command that's defined with help of the "run" keyword in your YAML file. And that's what we saw before, and what we used before, in this first Action. With this "run" command we simply execute typically simple shell commands on that runner machine, and these commands are, well, written down and defined by us. Now, of course, these commands could in theory also get more complex. You could also run more complex scripts with help of these commands. So it's not always necessarily a simple process that's started with such a command, but it will be a command, written by you, in your YAML file. Now, Actions on the other hand, are these predefined applications, which actually also can be distributed through a marketplace, because you can build your own Actions, and will do that later in the course, but you can also use official, or community Actions.
