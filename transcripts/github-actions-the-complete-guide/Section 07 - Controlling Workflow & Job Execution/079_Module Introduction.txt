# Module Introduction

At this point in the course, we're able to write solid GitHub Actions workflows that do a variety of things. But also at this point in the course, we're only able to write workflows that always execute from A to C. So from start to finish and that stop execution if an error occurs. If something goes wrong, if some step fails then the job to which the step belongs is canceled. All subsequent steps after the failing step are not executed and all other jobs are not executed. If one job fails because of a failing step. That's the default behavior and often that behavior makes a lot of sense. But sometimes you want to continue executing steps even if one step failed and some steps should even only be executed if another step failed. And it's not just about failing steps. In general we typically want more control over when exactly which job and step executes, because it's not the case that you always want to execute all jobs and all steps. Instead, often you have some conditions. And therefore discourse section is about controlling execution flow. It's about controlling when a job executes and when the steps in a job execute. You will therefore learn how to run your jobs and steps conditionally, and which tools GitHub Actions gives you to control that. And we'll then also dive into other execution flow related topics, like for example using an execution matrix or a job matrix, and you will learn what that is and when you might want to use that feature. And we'll explore the concept of reusable workflows, which can save us some time and some extra writing.
