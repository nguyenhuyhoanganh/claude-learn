# Module Introduction & What are "Utility Containers"?

Now did we know compose and we started building at least a little bit bigger applications and we know how to work with multiple containers. I wanna dig a bit deeper into what you can do with Dockers and containers. And I specifically wanna have a look at a concept which I like to call "utility containers." Now that's not an official term, utility containers is not an official term. It's just a term I like to use and after this module you will know why I like to use it. Because here's what I mean with utility containers.. Up to this point in all examples and all sections, we always worked with application containers that were Docker containers that contained our application, our application code and the environment we need to execute this application. And don't get me wrong, this is the primary selling point of Docker. This is one of the primary reasons of using Docker and it is the core idea behind Docker. So this is absolutely the most important thing with Docker. And the idea here was simple. We build our own Docker file, and then when we ran Docker run maybe with some extra options or with help off Docker compose this command in the darker file get executed, and our note API or react single page application or database started up. That is what we did this far. And again, that is the most important part of Docker that is what a Docker is all about. But this does not mean that we cannot leverage Docker for other things as well. And that is where utility containers come into play. With utility containers, I in the end mean containers which only have a certain environment in them. Let's say a node JS environment or a PHP environment something like that. And the idea here is that they don't start an application when you run them, but instead you run them in conjunction with some command specified by you to then execute a certain task. And this is a syntax we haven't seen this far. It's a syntax we haven't used this far. Adding an extra command after the image name when starting a container. But it is something we're going to have a closer look at in this module because this is something where Docker is also able to shine.
