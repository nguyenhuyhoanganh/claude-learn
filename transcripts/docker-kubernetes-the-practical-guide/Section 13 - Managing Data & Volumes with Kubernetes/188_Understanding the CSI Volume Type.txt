# Understanding the "CSI" Volume Type

Now there is one other type of volume, which I wanna draw your attention to. And that's the CSI type here. The CSI type is a bit different compared to the other types. CSI stands for Container Storage Interface. And this is a very flexible volume type. If we have a look at the other types, what you will notice is that, some of them are very specific. Like the AWS elastic block store volume type, which as the name suggests, allows you to use the AWS elastic block store service as a data storage solution. And something similar exists for some Azure services here. Now, there also are other types, like the NFS type, which allows you to connect to any network file system server. And you can learn more about all of that by for example, diving into the examples which you find here. But the CSI type, which I wanna focus on here, is special. This is a relatively new volume type, which was added by the Kubernetes team. And it was added to make sure that they don't have to add more and more built-in types for different cloud providers and different use cases. But instead, they expose a clearly defined interface. And then anyone can build driver solutions that utilize this interface. Now what do I mean by that? Let's say you wanna use the Amazon Elastic File System storage for storing your data. This is the storage solution we used in the deployment course section, when we used AWS ECS, you can use that if you would deploy your Kubernetes cluster on AWS. Now, if you wanna use this file system service to store your data, you could maybe make it work with the NFS type here because the EFS service is such a network file system, but that might take some extra work, and there is no AWS EFS built-in type. There's the elastic block store, but that's a different AWS storage service. So as I said, maybe NFS would work. But even better than that, thanks to this CSI type here, the AWS team built such a CSI integration for the AWS EFS service. And if you search for AWS EFS CSI, yes, it's abbreviation mania here, but if you search for that, you for example, find this GitHub project where you find more information and use such instructions for this CSI driver. And long story short, thanks to this CSI feature, it would be super easy to add EFS as a storage solution for your Kubernetes volumes. So the CSI volume type is a very flexible type, which allows you to attach any storage solution out there in the world, as long as there exists an integration for this CSI type like this AWS EFS CSI driver, and many such official drivers exist already. And theoretically, you could of course, also write your own ones. For example, if you bring your own data center or anything like that. Therefore, this CSI type is also important. We'll not use it here, because for this local playing around and this local development, we don't really need it. But later when we deploy this, and when we deployed to AWS to stick with this cloud provider, we will indeed utilize it to use the EFS service for storing our data.
