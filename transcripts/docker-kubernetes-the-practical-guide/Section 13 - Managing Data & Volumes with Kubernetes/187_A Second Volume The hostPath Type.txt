Now the emptyDir volume, which we're using here is a great basic volume, but it has one downside. What if we have multiple replicas, what if we have two replicas? But since volumes are closely attached to pods, that of course means that if a pod is not reached, if the request is sent to a different pod, our data is lost. Now, there are different ways of working around that. And we'll see a even better way later. But one way of working around that would be to switch away from the emptyDir, which creates a new empty directory per port to a different driver. And that would be the hostPath driver. This allows us to set a path on the host machine, so on the node, the real machine running this pod, and then the data from that path will be exposed to the different pods. So multiple pods can now share one in the same path on the host machine instead of pod-specific paths, which of course, are only useful if it's the same pod, which handles all requests. Now with hostPath, we won't solve the problem that we could have multiple node machines. And then the hostPath is still specific to one host machine. But at least it's a better approach than the emptyDir example from before, though I will say that if you work with just one replica, and you're just developing and trying out things, emptyDir, of course, is a great simple volume, which typically does the job. Nonetheless, let's switch to hostPath now. And switching is simple, we just replace emptyDir down there with hostPath and we now configure it differently. Instead of passing an empty object, we now can pass nested configuration keys here. And we need two keys. The first setting which we need to set is the path on the host machine where the data should be stored. So not the path in the container, but the path on the host machine. And therefore, of course hostPath is a bit like a bind mount. With bind mounts if you remember, we also were able to bind a specific path on our local host machine to a container internal path. We're now doing basically the same, our motivation now just is a bit different. Though, we could also bind a path, which contains some data which you wanna share with the container. Not the motivation I have here, but you could do that with hostPath as well. Because unlike empty Directory, hostPath will not always create a new empty path. But it will simply share a path and whatever is inside off that folder with this path in the container. So you could use hostPath to share some already existing data. It's not what I need here though, so I will simply share a data folder. So on the worker node they're running on because of this hostPath driver. And I hope I could make it clear where the difference is compared to emptyDir. Now, this is not pod-specific. This is of course useful, but it still has a couple of downsides. For example, as I mentioned before, here, in our local virtual machine, minikube, we only have one node, but if you had a bigger cluster with multiple worker nodes, which you typically do, the hostPath would still be node specific. So multiple pods, multiple replicas running on different nodes, would not have access to the same data, because only the pods on the same node have access to this data. Still here, we achieved an improvement and hostPath can be useful if you wanna make your data kind of independent from the different pods. If you wanna achieve a bit more pod independence, I would say, even though we'll find an even better solution later. I also again wanna emphasize that hostPath could also be useful if you wanna share already existing data into a container. It's not something we're doing here, but if that data folder would already exist, and would already hold some data, this data would also be available in this path in the container right from the start. And that's another difference to emptyDir which is always creating a new empty Directory, like the name suggests.
