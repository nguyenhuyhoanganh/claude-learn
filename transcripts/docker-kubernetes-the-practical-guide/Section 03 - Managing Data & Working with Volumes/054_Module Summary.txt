Because of that isolation concept often that's useful but sometimes it's not. Pretty much every application has some kind of data which should survive, which should persist. And that's why you can add volumes. Volumes in the end are folders on your host machine on which your containers are running, which are managed by Docker and which are mounted into a Docker container. There is this connection between the paths which are mapped inside of a container and the automatically created folder on your host machine. And whatever is written into that mapped path inside of the container, is also stored on the host machine therefore, and hence this data survives if the container is removed, because it's basically mirrored or copied to the host machine and not just stored inside off the container. Now specifically, we learned about two kinds of volumes. Named volumes are useful because of these survive if a container is removed and they offer, these are the volumes we need if we wanna store data permanently. Anonymous volumes are actually attached to a container and are removed if the container are removed and therefore they're not useful for storing permanent data, which should persist. They can still be useful to save temporary data make D container a bit more efficient because less data has to be stored in the container. More data can be outsourced to the host system. And we also saw that they can help us with bind mounts because bind mounts allow us to map local folders on the host machine, which we, as a developer know two internal folders into container for example, to expose our source code to the container. And then we can combine that with anonymous volumes to not accidentally override folders that already existed in the container, which we don't wanna override. Bind mounts are a bit like named volumes but the key differences that we know the path on the host machine, where the data is mirrored, and that we actually use that path to pass data into the container and to actually pass data into the container which we can change on the host machine. And where then the latest data is always available in the container. Like we did it with our source code. Now you can set up these volumes on the Docker run command with the -v option. Here we have a named volume and Docker automatically creates these volumes for you. If they don't exist yet, if they do exist it uses the existing ones instead.
