Now with Laravel and PHP specifically it's different, just installing PHP is not enough because in addition to that we'll then need a server. We can't build the server itself with PHP. Now here's our target setup, which we wanna achieve in this module therefore. And this will be a setup which allows us to build Laravel applications without installing anything on our host machine, except for Docker. Now, the idea is that we have some folder on our host machine that contains the source code of this Laravel PHP application. So that will be the folder which we can open on our local machine with an editor of our choice in which we can write Laravel code. Now this source code folder will then be exposed to one container, the PHP Interpreter container. That is a container which has PHP installed inside of it. And that's a container which will have access to our source code so that it can interpret this source code and generate a response for incoming requests. And that's the next building block besides this PHP Interpreter, which we need to run our code we also need that extra server. Because unlike with Node as I mentioned, we don't build that server with PHP. That's when we'll have a second container, which has Nginx in it, which is a web server and that container and this web server will basically take incoming requests then go to the PHP Interpreter and let that PHP Interpreter generate a response and send that response back to the clients who sent the request. Now for storing data, we're going to add a MySQL database and you could also use MongoDB databases in Laravel apps, but most commonly you work with SQL databases and therefore I'll show this setup. And of course our PHP Interpreter in the end needs to be able to communicate with MySQL database. Our level framework, which is just a bunch of PHP code in the end needs to be able to communicate here. So that's a number of container we need. Now these three containers, three individual separated containers which still will work together somehow are application containers. We don't use them for utility functionality, instead, these containers will stay up and running as long as our application is up and running. Because they are dealing with incoming requests with handling these requests and generating responses with help of PHP at Laravel and they are dealing with data storage in the database. But besides these three application containers our setup also needs a couple of utility containers. Because it turns out that in Laravel applications, there are three kinds of tools, utilities, which we need. For example, we need Composer. Composer is basically to PHP what NPM is to Node. In addition Laravel ships with its own tool called Laravel Artisian.
