# Hopping Window Event-Stream Processing Strategy

Welcome back. In this lecture, we'll continue learning about processing infinite streams of events and learn a new strategy which is called the hopping window. Before we talk about the strategy, let's get some motivation. As a quick reminder, in the last lecture, we learned about the tumbling window strategy, which divides time into fixed size back to back windows. One disadvantage we discovered is the dependency of the frequency of results that we can produce on the size of the window. So if the window duration is small, like a few seconds or a minute, then we can potentially produce results of processing a series of events very frequently. However, since each window covers only a short period of time with only a few events, it may not contain enough events to produce meaningful insights, Sites, especially if the volume of events is low or sporadic. We will have a lot of windows with nothing or very little data to aggregate. On the other hand, if we want to aggregate data from events over longer periods of time, like the error rate for the last hour or the number of website visitors in the last 24 hours, we will get much more meaningful insights at the end of each window. However, the results of an aggregation will be produced by the consumer of the events only once an hour or once a day, respectively. So, to increase the frequency of results of processing series of incoming events but still get insights over long windows, we can use another strategy called the hopping window. The hopping window is very similar to the tumbling window, in that each window has a fixed time interval. However, instead of being aligned back to back to each other, the hopping windows overlap with each other and are separated by a fixed hop, which is called the advance interval. This advance interval can either be smaller than the window duration equal to the window duration, or even larger than the window duration. If the advance interval is the same as the window duration, where in fact back to the tumbling window strategy. In other words, the tumbling window strategy is just a particular case of a hopping window where both the advance interval and the window duration are the same. Now, the real value of the hopping window is when the advance interval is smaller than the window duration. In those cases, no matter how long our window is, every hop interval, a new window is opened by the consumer, Tumor, and then one more hop interval later another window is created, and so on. That also means that after the duration of the first window has passed, every advanced interval, a particular window is closed, and every time a window is closed, the consumer aggregates the data from the events in that window and publishes the results downstream. For example, when window one is closed, all the events captured within its time boundaries will be analyzed and the consumer application will produce a sum, an average or some other single metric or summary, depending on the use case. Then one advanced interval later the second window will be closed, and the consumer application will analyze the events within its boundaries and produce another result based on those events. If the results are published by the consumer of the events to the UI or some other dashboard. We essentially get continuous monitoring or trend analysis with much more frequent updates than in the tumbling window. Also, if we want to increase the window size for whatever reason and analyze more events in every window, the frequency of publishing the events doesn't change because it's completely independent of the window size. Notice that unlike in the case of the tumbling window in the hopping window strategy, the same events can be part of several windows. For example, this event will be part of windows one, two, three, and four, which means the same data received by the event consumer will be processed as part of multiple aggregations. Now, hopping windows are very useful for near real time analytics, which is particularly good for UI dashboards. For example, for stock trading, we can show the user a lot of data such as the average, the median, maximum or minimum price of an asset in a relatively long window like a minute or hour. However, with a small advance interval, we can share fresh data every second with our users or third parties to help them detect shifting trends, make automated trading decisions, or display the data on an interactive dashboard. Similarly, if we are processing a stream of logs coming from our production machines and want to still see the hourly error rate, we no longer have to wait an hour to get fresh data. Instead, we can keep the window duration as one hour, Our, but set the advance interval to one minute. This way we can get an update on our error rate for the last hour in our system every minute, which helps us detect issues much earlier, but still keep the aggregation window very wide. Now let's talk about the advantages and disadvantages of the hopping window strategy. So as we just saw the main advantage of hopping window with an advanced interval smaller than the window size is that we can produce results of processing events much more frequently. This in turn is particularly great for interactive UIs and dashboards monitored by humans. And it also helps us respond to changes or important insights faster and more effectively. Now, of course, everything is a trade off and the hopping window is no exception. Just like in the case of the tumbling window, the consumer microservice or application aggregates all the events for the given window in memory. However, now, instead of maintaining only one window, we have multiple windows and many more events in flight, which increases the memory footprint of the microservice instance. Also, now we need to produce results much more frequently, which consumes more CPU resources, especially given that each event is processed multiple times since it's part of several windows. Also, since we produce results more frequently than in the case of a tumbling window, we also consume more network resources in the downstream microservice or application will also have more data to either store or process. Now, before we learn about the next strategy. I want to point out that in the hopping window, the hop interval doesn't always have to be smaller than the size of the window. If we set the interval to be larger than the window size, we simply revert to something similar to the tumbling window strategy, except now we have gaps between the windows. This is useful when the volume of events is very large but doesn't have high variability. For example, when we're getting a lot of sensor or user data, that doesn't change too often or we don't need to process all of it this way, we basically sample those events with fixed windows and discard the data coming in between. It's a far less common use of the hopping window, but I still wanted to mention it so you know it exists.
