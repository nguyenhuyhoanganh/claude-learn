# Session Window Event-Stream Processing Strategy

In this lecture, we'll learn another strategy for processing a stream of events. This strategy is called a session window. As always, let's start with some motivation. First, if we look back at all the types of windows we learned so far, in all the cases, the size or duration of the window has always been fixed. However, in certain scenarios, we want to analyze sequences of events that may span a variable duration. Typically also followed by a period of inactivity. For example, let's say we want to provide user behavior analytics for our company's website. To get such analytics and learn about user behavior, both on an individual user level and an aggregate level based on demographics, country, and so on. We need to track each user's activity throughout an entire session. For instance, in an online education website, a user May 1st log in. Then search for a course. Scroll through results. Click on a few options and then. Watch a promo video. Read the description and reviews and then add the course to the cart and check out. On the other hand, another user may log in, navigate to the course they already enrolled in, watch a few lectures, take a quiz, and then walk away from the computer. Notice that for each user, the session duration is variable and different from that of other users, so none of the previous strategies is going to work. So to capture those sessions, we can use a dynamic session window. In the session window strategy. Each session window starts with a particular type of event, which in this specific case is the user login. The size of the window is dynamic and is based on the continuous activity of the given user. When we reach some inactivity threshold. The session window is closed and the results are summarized. May be categorized as page views. Scroll, search or purchase, and then saved in a database or published to an analytics dashboard. The duration of the window is always going to be determined based on the time between the first event and the last event in the session. Now, session windows are very common and useful not only for analyzing human activity, but also in IoT. For example, anytime an autonomous vacuum cleaner works around your house, it will emit sensor events like location, temperature, or amount of dust collected to the cloud where a microservice may analyze and categorize those events. This can help us detect patterns over time, such as decreased efficiency and increased energy consumption, and it also allows us to suggest preventative maintenance to the users. Here again, the frequency and duration of each vacuuming session vary greatly. So session windows is the best choice. Another example can be data collected from drivers using a navigation app on their smartphones. This data can be used to optimize the route of all the other drivers that take the same or similar route. The driver or autonomous car can start a session by sending an event that starts the driving from point A to point B, while the app is getting directions from the navigation system. It is also sending data about its location and speed to a microservice or application in the cloud. This location data is then processed to continuously recalculate the best route and congestion areas to optimize the trips of other drivers. Here again, each driving session is part of a dynamic session window that starts as soon as the app detects movement and stops when the location update events contain the same location for some time, indicating an activity or when the user simply closes the app. Now, before we talk about the pros and cons of the session window, one important thing to point out is that this type of window is not global and is always created per user, key, device ID, and so on. So if we have three users in the system, we can have three overlapping session windows, one for each user. However, each user will have no more than one active session window open at any given moment. This is in contrast to the other windows that we saw previously, like the tumbling window, for instance. That can be global, which means that each window will be used for a group of different events from different sources, and once the window is closed, we can optionally group events by some key before performing any computations. Now let's talk about the pros and cons of the session window strategy. So as we just saw, the session window strategy is an excellent choice for real time analytics of user behavior while using an app or service. Due to the dynamic size of the window, we can logically group events based on activity instead of predefined time boundaries. Also, each event in the session window belongs to only one window, so we don't waste any computational power or processing the same events multiple times, which is a problem we had with the hopping windows and the sliding windows. One challenge of using the session window is defining the inactivity threshold. Of course, if we get a clear disconnection event from the user, then we can close the window immediately. However, in most cases we don't get such clear indications, especially from humans. Typically, people either just stop using the app or just switch to a different tab on their web browser without explicitly closing it on our end. We can never know if the user just got temporarily distracted, has gone to the other room to get their credit card to complete the purchase, or they completely switched off their focus to something else. So if we define the inactivity threshold to short, we may close the session window prematurely. Logically splitting what is actually the same session. And on the other hand, if we configure the threshold to long, then we will end up with session windows staying open for much longer, which consumes more memory inside our consumer application. Now, another downside or challenge of using the session window is that we open each window for a single user origin or event key, which means that we may potentially have many windows that we have to maintain. This can become an even bigger issue when the sessions are very long, such as in tracking the activity of users in video or audio streaming apps, or location tracking in navigation apps. One way to solve this problem is to run many instances of our streaming, processing, microservice, or application and dedicate one consumer instance to a set of keys corresponding to different users or event origins, and also with continuous monitoring of the memory and CPU utilization of those consumer instances. We adjust their number up to keep up with the demand or down to save on resources. See you guys in the next lecture.
