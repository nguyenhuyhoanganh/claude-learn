# Time in Stream Processing and Handling Late Arrival of Events

In this lecture we're going to conclude our discussion about event stream processing. Specifically, we're going to cover event timing and how to handle the late and out of order arrival of events. So let's talk about timing in event stream processing when we talk about event stream processing. For every event we can define three distinct timestamps. The first one is the event time. This is the time when the event actually happened and is present inside the event's payload. For example, if we're receiving temperature or location measurements from a mobile device or IoT device, the event time for each measurement will correspond to when the measurement was captured on the device. Now, once the event is published into our message broker and arrives at the consumer microservice or Our application. This timestamp is called the application time or arrival time. This timestamp is always greater than the event time, and the difference depends on many factors such as the latency in our system, the load in our system, or whether it's a real time event processor or a batch processor. Finally, we have the processing time, which is when the event is actually parsed and processed by our code. Typically, the arrival time and the processing time are very close to each other, and we can use them interchangeably. However, in case our microservice or application is overloaded, there may be a more significant gap between those timestamps. For the purposes of this lecture, we're going to ignore this difference and assume that they're always the same. Now, given this information, a good question you may ask is which timestamp should I use when deciding how to match events into windows? And the answer, not surprisingly, is it depends on the use case. In cases where we don't care about when the event actually happened, but need to act in real time, for example for monitoring or alerting purposes, we should prefer using the arrival time and not the event time. For example, let's say we're running a rate limiting microservice inside a credit card processing system that other customers can use to build their users. Now here, each event represents an API call from one of our many third party customers using our system. In this case, it's better to ignore the actual event time, which doesn't really matter here since we're using a sliding window which starts a new window for every incoming event when it arrives at our consumer. Another common situation where we prefer to use the event arrival time and not the event time, is when we either don't have the exact time stamp of when the event actually happened, or we cannot trust it. Consider a situation where we have a social media app that people download and install on their mobile phones. Users of that app can scroll through the content and leave likes or comments which are sent to our platform. Of course, besides the fact that the actual timestamp of those events is not really critical, we also have a few potential challenges. First of all, the clocks on the end user devices may be incorrect or out of sync with our clock. Additionally, in some parts of the world, a user may lose internet connection for a few minutes or even longer, which results in events being buffered on the user's device before the connection is re-established. Also, sometimes when the device is low on battery, some network requests can be delayed and then batched for higher efficiency. Now let's talk about when to use the event time instead of the arrival time or processing time. In situations where we do care about when the event actually happened. For accurate correlation or ordering, we should use the event time when deciding how to bucket those events to windows. For example, let's say we're analyzing real time trends in the stock market and grouping trade events into one minute hopping windows with a one second advance intervals. With the purpose of showing a moving average of the price of a particular financial asset. Now, depending on the volume of trades at any point, the load on our system may be different. That means that at different times, we may have jitter due to garbage collections of our application or buffering delays in our message broker, which means our event arrival time may vary greatly over time. However, what we really care about and what our users care about is the actual time of each trade, not when the event arrives in our system. And that's because they want to make the right buying or selling decisions based on those moving averages that show the trends in the financial market. Another example of when to use the event time instead of the arrival time, is when analyzing a sequence of events coming from the user's mobile devices for tracking and analytics purposes. For instance, let's say we have a language learning app that runs on our users devices, and we want to monitor and analyze how each user uses our app. So we can optimize their learning experience. In this case, our app can track different events like user login, starting a new lesson, completing a lesson, taking a quiz, and so on. However, as we saw before, while the user uses this app, the mobile device may not send each event immediately. Instead, due to connection issues or to increase energy efficiency, the events may be stored on the device and then sent in batches, sometimes much later than the event actually happened. So on the time scale it will look something like this. Now if our consumer microservice or application used the arrival time of those batches of events, then they would all appear as if they happened at the same time, not giving us any insight into the user's behavior. However, now we do care about the actual time and sequence in which the events happened on the user's device. So we can use a session window, for example, and get real insights about our user's behavior. Now let's talk about the last topic, which is handling the late arrival of events. As we just saw, in order for us to get an accurate analysis of events, we often prefer to use the event time instead of the arrival time. When mapping them to an event stream processing window. However, this comes with one challenge to illustrate it, let's take the example of Analyzing stock trading events just like before. But for simplicity, let's assume we're using a tumbling window of one minute. Now, if we use the event time to group events into the window, we may have a situation where the window that started at time zero has already ended. However, 10s later an event whose event time was zero minutes 57 seconds arrives at our system. So the question is what do we do with this late event since it happened before the current window was closed? It should be taken into the calculation and aggregation of this window. But because its arrival was delayed, that window has already been processed and its results published downstream. So the first option is, of course, to simply discard it. However, given that there's always some delay between the time when the event actually happened and the time it arrives in our system. We would be consistently throwing away many events. So the second option is to add a grace period to each window. In other words, instead of closing the window immediately when the window duration is met, we can add a period of, let's say, 15 seconds before actually performing the aggregation of events, publishing the results, and discarding the window. Any late event with an event time that falls within the boundaries of this old window that arrives within the grace period will still be added to this window. However, once the grace period ends, the old window is closed and its results are published. So even if more events come with event times that belong to the old window, they now will be discarded. So, using the grace period strategy, we give late events another chance to be added to the right window and therefore don't lose them, providing us with much more accurate analytics. However, the main challenge of this strategy is finding the right grace period. If the grace period is too long, then the results of the old window will be published only when that grace period is over, which introduces an extra and typically unnecessary delay. On the other hand, if we choose the grace period to be too short, then we are still likely to lose many more events whose event time still falls within the closed window but simply arrives too late. So another solution is called watermarking. With this strategy, we choose a value called a watermark. Then once the window is closed, we keep track of the maximum event time we have seen so far. Then for every new event, we first update the maximum event time if necessary, and then we check whether the maximum event time seen so far, minus the watermark value, is less than the upper bound of the old window. For example, let's say we define the watermark to be 10s and the current maximum event time seen is zero minutes 59 seconds. So as long as this condition holds, the window will remain open and new events will be added to it. For example, if a new event now arrives out of order with the event time of zero minutes 57 seconds, the max event time is still the same, so this event will be added to the old window. Notice that the arrival time of this event is irrelevant. Even if it arrived 20 or 30s later, it would still be added to the window. Also notice that if we receive an event whose event time still fulfills the condition but belongs to a new window, we're going to put it in the new window while still keeping the old window open. This is the whole point of having this watermark value, which allows for the late arrival of events that belong to an old window, even though the new window is already open. Now, as soon as we encounter an event whose event time minus the watermark value is greater than the upper bound of the old window, we close that old window and then aggregate the events and publish the results. And this new event will go to the new window. Now the advantage of the watermarking Marketing strategy is the fact that we don't have to define a strict grace period, which is tied to the arrival of the events. On the other hand, the disadvantage is that we can potentially delay the aggregation and closing of the window for a time that cannot be estimated ahead of time, which is in contrast to the case of the grace period strategy. So in conclusion, in this section, we'll learn about several strategies to process an unbounded or infinite stream of events. Each has its own advantages, disadvantages, and use cases. And finally we talked about the difference between event time, arrival time and processing time. And then learn a few strategies of handling late and out of order. Arrival of events.
