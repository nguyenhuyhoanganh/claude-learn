# Introduction to Event-Stream Processing and Tumbling Window Strategy

Welcome back. In this section, we're going to build upon our knowledge of event driven architecture and learn about processing unbounded or infinite streams of events. In this lecture, we'll start with some introduction and motivation and then discuss the first strategy for event stream processing. So let's first get some motivation for the problem we're trying to solve in event driven architecture. We typically have an infinite stream of events that we need to process. This data can come from external sources such as user activity in web applications, sensor data from IoT devices or third party services like payment notifications from credit card companies, and so on. However, our internal microservices also emit streams of events that other microservices subscribe to and process. Now, in many situations, reading and processing each event in isolation can be enough. For example, in a video on demand service, we can receive a new subscription event from the payment service after a user has been successfully billed for the next billing cycle. This event can be consumed by the user service to update the user's subscription status, the notification service to send an email confirmation to the user, and the recommendation service to create a special movie recommendations for the user. In this scenario, the subscription event can be processed on its own by each one of those services in complete isolation from other events that came before or after that event for other users or the same user. However, this is not always the case. In many cases, to get meaningful insights, we need to aggregate and analyze a series of events. Of course, analyzing an infinite stream of data is impossible. So instead, we use finite subsets of events by chopping those continuous streams into manageable pieces of data. By using finite windows, we can analyze streams of data almost in real time. This is especially critical for systems like anomaly or fraud detection. Pattern recognition, rate limiting, recommendation services, and so on, where each event on its own does not carry sufficient information. Those strategies of processing unbounded streams of events are very common and useful in microservices that consume events using event driven architecture, but also in big data pipelines, where the volume of incoming events is not only unbounded but is also very high. We'll talk more about big data later in the course. So let's talk about the different windows strategies for processing streams of events. Starting with the first and simplest windows strategy, which is called a tumbling window. Here the consumer application or microservice divides the time into fixed back to back non-overlapping time buckets, where each event falls into only one window. Once a window is open, every event that the consumer consumes will be added only to the currently active window. When we reach the end of the window, the consumer aggregates those events and produces the result as a single event. Whether this result is a sum, an average summary or anything else depends on the use case. Later this result can either be a new event emitted to another channel, a new data point in a user dashboard and alert that needs to be handled by an engineer on call, or a record stored in a database. Once the current window is closed, a new window is created for the same fixed duration and all the new events will now be bucketed into this new window. The old data from the previous window is discarded and doesn't consume any more resources. It's important to note that the message broker is generally unaware of the concept of this window. The consumer is fully responsible for buffering those events, typically in memory, until the window expires, and the results are either stored or published to another application or microservice. This implies that the longer the window is, the larger the memory footprint for this strategy. One example of using the tumbling window in real time processing is capturing and displaying the average measurement from a sensor, like the temperature of a weather tower, or energy usage of a vacuum cleaner in a ten second window, for instance. Since the volume of measurements can be quite high, aggregating them into a single average helps make sense of this data downstream, either by another microservice or a user. Also, with a small percentage of samples that clearly indicate an incorrect measurement due to dust particles or sensor defects. We can apply a technique called trimmed mean. This allows us to throw away, for example, 10% of the highest values and 10% of the lowest values, and calculate a more accurate average measurement for each window in later stages of event processing. We can then make better use of this much cleaner and lower volume data. Another example is log analysis coming from production machines. For example, we can use the tumbling window aligned to every round our look at the logs from production machines and calculate the error rate in the last hour. Then if we detect that in a certain window the error rate exceeds some threshold, we can send an alert to the notification service, which will send an email or a text to an engineer on call. Finally, in e-commerce systems where different sellers can sell their products to our user base, we can use the tumbling window to analyze the revenue in our system every 24 hours. At the end of each window, we group sales by seller and produce the total revenue from each seller in the last 24 hours. That data can then be stored in a database or published as an event which can be consumed by an email service and be sent to each seller daily. That 24 hour summary can also be sent to the payout service, which will calculate the payout to each seller. That accounts for taxes, transaction fees and platform fees, and then use a direct deposit to send the money to the seller's account. Now let's talk about the pros and cons of using the tumbling window strategy. The tumbling window is the simplest stream processing strategy. It's ideal when we need periodic discrete aggregations or reports with minimal memory and computational overhead, and also with clear time boundaries like round, minutes, hours, or days, for example. It's also ideal when we want to process each event only once, because it logically belongs only to one window. The disadvantages of the tumbling window are the low granularity of results. Essentially, the frequency of getting insights about our stream of data is limited to the size of the window, and we cannot get updates any faster. So if we look at the hourly error rate in our stream of events, if we suddenly have a high spike in errors, we will get that insight only at the end of the hour long window, which can already have serious financial impact. Similarly, in the e-commerce system, the seller will only get fresh data about their revenue at a particular time of the day when the 24 hour window closes and the results are published. However, until that window is closed, they will continue seeing the same old data. Another disadvantage is that it's not a good strategy for recognition of patterns or trends, because the windows have very clear boundaries, and each event falls into only one window. A series of events that may indicate a pattern of some sort may be broken into two discrete windows. For example, a pattern like this and the number of errors that crosses two one hour long windows will not be detected. The number of errors will be summed up or averaged separately in each window, which does not capture the trend that the number of errors increased drastically at some point in time, which may indicate a serious issue. Similarly, if the threshold for an alert is ten errors per hour, even though 11 errors have happened within the span of one hour because they cross that window boundary, this trend will not be discovered. So in the following lectures, we will learn about other strategies that can solve all those problems. So I'll see you guys in the next lecture.
