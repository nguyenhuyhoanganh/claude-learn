# Sliding Window Event-Stream Processing Strategy

Welcome back. In this lecture we're going to continue learning about event stream processing. And now we're going to talk about another very common strategy called sliding window. And just like before let's start with some motivation. First let's look at a scenario of a microservice that tries to detect fraudulent activity in a financial system like a bank or a credit card processor. For simplicity, let's assume that we have a rule that if a particular credit card is used more than ten times within a one minute window, we consider this as a suspicious activity. If that happens, we will trigger an alert to freeze the account, as well as push a notification and an email to alert the owner of the credit card. Now, if we try to use the tumbling window to detect those anomalies, we can quickly find an edge case where six transactions for the same credit card are captured within one window and another five fall within the next window. In this case, because the events fall within two separate windows, the fraud detection microservice will not detect the suspicious activity, even though we can clearly see that 11 transactions happened within a span of less than one minute. Of course, we can try to apply the hopping window strategy instead, which will ensure that events fall into multiple overlapping windows. So in this example, with a five second advance interval, those 11 transactions are captured by this window first and we successfully detect this anomaly. However, in this case we kind of got lucky if the 11 transactions are distributed like this instead. All those transactions still happen within a period of one minute. However, because our advance interval is not small enough, none of those hopping windows actually covers all those transactions. So in this case, we will still miss the fraudulent activity even with a hopping window strategy. Now, if we make the advance interval very, very small, then we're much more likely to detect such a pattern. But at the same time, if we also have periods of very little activity in our system, like at night when nobody is really using their credit cards, then those many overlapping windows are just costing us a lot of resources, even though they barely contain any events. So the solution to this problem is to use another windowing strategy, which is called the sliding window. In the sliding window strategy, each window is still of a fixed duration, just like in the case of the tumbling window and the hopping window in the most basic case. Every new event will start a new fixed size window. However, unlike in the hopping window, the advanced interval between consecutive windows is dynamic and is aligned to events instead of fixed points in time. The strategy is great when events arrive at irregular intervals. So if we have periods of very frequent events, we will have many overlapping windows, and if we have periods of low volume of events, we will have fewer windows, which saves us on memory and computational resources. Now, for any active window, as soon as its duration passes, and the first event in that window is about to fall out of scope, the results are calculated published, and the window is closed. So going back to our example of the fraud detection microservice, if we look at the same timings of events that we fail to detect as a suspicious activity using the hopping window, if we apply the sliding window strategy and look at the window that started with the first transaction. This window definitely catches the 11 transactions that are happening within the span of one minute. So at the end of this window, the fraud detection microservice will successfully identify this as a suspicious activity. Freeze the account and alert the user. Another very common use case for the sliding window is API rate limiting and throttling of users. Here, every time a user sends a request to our system or a third party application makes an API call to our system, we can emit an event to a message broker. Those events can then be processed by a rate limiting microservice that uses a sliding window to process each request or API call, and aggregate the events by origin. If the same user or the same third party client application exceeds its quota for a given period of time, we can update our API gateway or web application service to block any requests from that origin and respond with an HTTP to many requests status code. This can help prevent DDoS attacks on our website, but also make sure that all our third party client applications get an equal response time from our services. Now let's talk about the pros and cons of the sliding window. The sliding window processing strategy is an excellent strategy for real time monitoring and pattern detection. If we compare it to the hopping windows strategy, it's essentially a superior version where the advanced interval is flexible instead of fixed to a rigid time interval. Also, the advanced interval is essentially always as small as possible, because starting a window faster than the arrival of new events doesn't add any value. And if we start a new window less frequently than the arrival of events, we run into an issue where certain patterns are not detected. Just like in the edge case, we covered with the fraud detection microservice. On the other hand, the main downside of using the sliding window is that its processing and memory costs are the highest of all the previous strategies that we've covered, especially when the volume of events is very high, the memory footprint and complexity can become quite significant. For example, in this situation, if we use a tumbling window or a hopping window, the number of windows is going to be constant regardless of the volume of events. On the other hand, in the sliding window strategy, the number of windows is directly correlated to the number of events. So in this case we will have many more windows than in the previous strategies.
