# Testing Pyramid for Microservices - Introduction and Challenges

Welcome back. Now that we talked extensively about all the aspects of software development and design for microservices, let's talk about an even more important part, which is bringing those microservices to production. But before deploying a system to production and taking user traffic, we need to gain confidence through automated tests. So in this lecture, we'll start with a quick recap of how we used to test monolithic applications, and then we'll discuss how to apply the methods we used in testing a monolith to microservices architecture. And finally, we'll discuss some challenges of testing microservices and event driven architecture. So let's start by revisiting the process of testing a monolithic application. First, when we set up tests for a single application, we could group those tests into three categories unit tests, integration tests, and functional or end to end tests. This is commonly called the testing pyramid. At the bottom we have the unit tests. Each unit test tests a small logical unit such as a class or a module in isolation. Unit tests are the cheapest to maintain because they are small, easy to write and fast to execute because they are so cheap. We should also create the highest number of those tests compared to other tests, which is why they are located at the bottom of this pyramid. On the other hand, those tests give us the least confidence about our overall system because the only test each unit in isolation. And once we run the application, we have no idea if all those units will work together or not. The second type of tests is the integration tests. As the name suggests, those tests verify that different units and systems we integrate with, such as a database or a message broker actually work together. Those tests are bigger and slower, so we should have fewer of them than unit tests. However, after running those tests, we have more confidence in our system. Finally, we have the functional or end to end tests at the very top. Those tests run our entire system, which includes the UI, our entire application and the database, and they verify that it works as intended. From an end user perspective, each such test should test a particular user journey or business requirement and ensure that it matches the specification. We said for our application. Not surprisingly, end to end tests are the heaviest and most complex to set up and the slowest to run. This is why we should have the least number of them compared to the other tests. On the other hand, those tests give us the highest degree of confidence that our system will work as intended in production. Now, how do we translate this pyramid to a microservices architecture where microservices communicate with each other through the network or a message broker and each microservice has its own database? So as the first step, each microservice team should follow the exact same testing pyramid, which includes unit tests, integration tests and functional tests for its own microservice and database. Then we treat each microservice as a small unit that is part of the larger system and put it in a larger testing pyramid. So just like in the case of unit tests, testing each microservice in isolation is essential, but not enough to increase the confidence that all those microservice can actually talk to each other at runtime and work correctly together. We need to add another layer of integration tests. Those integration tests verify that every pair of microservices can talk to each other using the agreed upon API while mocking the rest of our system. To complete this pyramid, we need to add the system level end to end tests at the very top. Those tests, in theory, should run all of our microservices, databases, message brokers and front ends in a test. Environment and verify that all the individual components work together as expected. Now let's discuss the challenges with this new version of the testing pyramid for microservices. The first challenge is while those end to end tests provide us with the highest level of confidence, they are extremely hard to set up and maintain. It's also unclear what team should own this environment and when one of the microservice teams breaks their build. Even for a day, the entire test pipeline will be broken. This can result in all the teams being blocked and unable to make any releases to production. Alternatively, developers may just start ignoring those end to end tests and release their microservices anyway, which makes those tests just a liability with no benefit. It's also very costly to run what's essentially a duplicate environment to our already existing production environment, even if it's scale, is much smaller. In practice, some companies spend disproportionately too much effort building and maintaining those few tests while other companies decide to take the risk and simply don't bother investing in those tests at all. The second challenge of this new pyramid is that even running integration tests can be quite difficult and creates a point of tight coupling between teams. When the team that owns Microservice A that consumes the API of microservice, B wants to run integration tests, it needs to build, configure and run both microservices while Team A knows how to build and run their own microservice. It may be unclear to them how to set up microservice. B It's even more difficult if microservice B has many dependencies like a database or another service that needs to be run or mocked. But the team that owns Microservice B that provides this API also has a problem. This microservice may have multiple microservices that consume its API. So to ensure that the changes they make in their API did not break all those API consumers they need to build, configure and run all those consumers to execute their tests. As we can see, this can quickly get out of hand and slow down the development of our entire organization. The third challenge is when we use the event driven architecture to decouple microservices. In this case, we have a microservice that produces events to a message broker, and it actually doesn't always even know which microservices consume those events. This decoupling is one of the benefits that we wanted from the event driven architecture. However, now we can't really run any integration tests with those microservices. We also have the same problem when we are a team that owns a microservice, that consumes events from other microservices. Here again, we need to run those microservices and a message broker just to test that our microservice is able to consume those events and that the format of those events did not change without our knowledge. So now that we understand all those challenges, we're ready to learn a few alternative solutions that companies use to test microservices and event driven architecture. Then we'll learn how to apply that same pyramid to microservices. And later, we discussed a few challenges of using this traditional approach, which included the high cost and complexity of running a complete end to end environment. The tight coupling between teams to run effective integration tests and the complexity and overhead of testing event driven microservices. I'll see you guys in the next lecture.
