# Contract Tests and Production Testing

Welcome back. In the previous lecture, we discussed the testing pyramid and how to apply it to microservices architecture. We also discussed the many challenges of running end to end tests with microservices, which forces many companies to either invest disproportionally high efforts into them or abandon those efforts altogether. We also discussed the challenges of integration tests between microservices, which included their complexity and tight coupling between teams. So now that we identified all those challenges, let's discuss a few alternative solutions, starting with integration tests. The first solution that can help us with the complexity of integration tests is using lightweight mocking. Instead of running an entire microservice. Consider the scenario where we are a team that owns microservice A that consumes the API of microservice. B This team wants to run an integration test that validates the triggering. Certain functionalities with a set of parameters results in microservice. A sending a request to microservice B and that microservice A also correctly handles the response from microservice. B Normally to run this test we would have to build, configure and run microservice B with all of its dependencies. But now instead we simply mock the API layer of microservice B, which we integrate with and set it to send back a hard coded response if it receives the expected request. Similarly, the team that owns Microservice B, which provides the API to other microservices, can run a set of mock consumers instead of running real microservices. Then we can write tests that make those mock consumers, send us requests and test that our microservice returns the expected responses. The strategy reduces the coupling between the different teams, because if one team breaks their build, the other team can continue running its tests. It also reduces the overhead of running real instances of the other microservices. However, this strategy on its own has one major issue. The issue is that the contract between an API consumer and the API provider can get out of sync without those teams ever detecting it. For example, the API provider team may change its API update, its mock consumer and its tests and all their tests will pass successfully. But the communication about those changes may be lost or misunderstood by microservice A which consumes that API. So they either don't make any changes at all or make incorrect changes and their tests will also pass. Once those two microservices are released to production, though they won't be able to communicate with each other and cause an outage. To solve this problem, we can introduce a new type of test which is called contract test. Contract tests work by using a dedicated tool to keep the mock API provider and the mock API consumer in sync through a shared contract. When the API consumer team runs their tests, they are run against their mock API provider. During this test, they test that their microservice A sends the right requests and receives the right responses as expected. However, additionally, each request it sends to the mock API provider is recorded along with the expected response into a contract file. This contract file is then shared with the team that owns Microservice B, which provides that API using this contract. The team that owns Microservice B replays all those recorded requests to the real microservice B and verifies that the responses it gets are the same as recorded in the contract. And if microservice B has many consumers, it will take all their recorded contracts, create those mock consumers and run each one against their actual API implementation. This way, each team can run its own integration tests without dealing with the complexities of building, configuring and. Running other microservices. But the contract test tools guarantee that each team tests against the most up to date and correct contract shared between those microservices. We can also extend the idea of contract tests to integration tests between event driven microservices. Let's consider a concrete scenario of an e-commerce system. When a user sends a request to purchase a product, that request goes to the order service. The Order service then publishes an event to the payment service and responds to the user. The payment service consumes that event, builds the user, and then passes it as another event to the shipping service. For simplicity, let's focus on the integration tests between the payment service and the shipping service. In this case, the contract between the services is the event format that contains information about the user, the product, its quantity and so on. So instead of setting up a message broker and running both microservices, each team creates its lightweight mock of that message broker. So when the shipping service team tests its own ability to consume and parse the payment service event, that event is also recorded into a contract file. This contract file is then shared with the team that owns the microservice that publishes that event, which is the payment service. In this case, when that team wants to run integration tests between their service and the event consumer, they will trigger the function in their service that is supposed to produce that event. Then the contract test will verify that the event format and the content match the one that was recorded in that contract. And if it matches the contract between the two microservices are in sync and we can confidently release them to production. So as we can see, those contract tests can really simplify running integration tests for microservices that communicate synchronously and asynchronously, but still give us high confidence that when we deploy those microservices to production, they can communicate with each other, as we would expect. Now let's talk about end to end tests. While contract tests can be a great alternative to integration tests, they are not a substitute for end to end tests. So if setting up end to end tests is not feasible for our company, the alternative is testing in production. There are actually many ways to do it. One of them is using a gradual release using blue green deployment in combination with canary testing. A blue green deployment is a safe way to release a new microservice version to production using two identical production environments without any downtime during the release. The blue environment is a set of servers or containers that run our old version, and the green environment is a set of servers or containers that run the new version that we want to release. Once we deploy the new version to the green environment, no real user traffic is going to it. This is an opportunity to increase our confidence by running automated and even manual tests against those servers without impacting real users. After we run those tests, we can shift a portion of the production traffic to the green environment and monitor the new version for performance and functional issues. This process is called canary testing. If we detect an issue, we immediately direct traffic back from the green environment to the blue environment with minimal impact on users. On the other hand, if no issues are detected, we direct all the production traffic from the blue environment to the green environment and gradually decommission the blue environment since it's no longer needed. So now that we have an alternative for integration tests for microservices that communicate directly with each other and through a message broker in a form of contract tests, and we have an alternative to end to end tests in the form of production testing, using blue green deployment and canary testing of. Course, all those alternatives should be used only if setting up real microservices in development stage for testing purposes is too complex or expensive. In this lecture, we'll learn how to address the complexity of running integration tests and end to end tests in microservices architecture. We first introduced a simplistic approach for integration tests where each team simply mocks the other microservice. After that, we'll learn how contract testing solves that problem by keeping the contracts between the API consumer and the API producer in sync.
