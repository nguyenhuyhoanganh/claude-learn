# Distributed Tracing

Welcome back. Now, we finally made it to the third pillar of Observability. Distributed tracing with metrics and distributed logging. You may be questioning why we even need a third method for observability. So we'll start by discussing what distributed tracing is and why we need it. Then we'll discuss some terminology and what it actually takes to enable distributed tracing for microservices. And finally, we'll discuss some challenges of using distributed tracing in microservices and event driven architecture. So let's start with some motivation for distributed tracing. To illustrate why we need another solution, let's consider a real life scenario of an ordered transaction in an e-commerce system with microservices and event driven architecture. When a user places a purchase order for one of our products and that request goes to the API gateway. First, the API Gateway talks to the auth service to ensure that the user is indeed who they claim to be and is authorized to make the purchase. After that, the API gateway sends the Http request to one of the ordered service instances through a load balancer. The order service records the order in its database emits an event to a topic in a message broker and responds to the user. After that, we have a long chain of events and requests that go through several microservices third party services and message broker topics. And when the entire transaction is complete, the notification service sends a push notification and an email confirmation to the user. That email contains their payment receipt and order reference number and a URL where they can track their order. Now let's imagine that suddenly some users start complaining that they never received the email confirmation for their orders and other users start complaining that they do get the email confirmation. But many hours or even a day after they place the order. With so many microservices involved exchanging requests and asynchronous messages, the issue can be anywhere. So where do we even start the investigation? Do we start looking at the metrics dashboards of every microservice in this process? Or do we start reading through thousands of log entries of every microservice and message broker to find something that stands out? To make things even more difficult, every microservice runs as a group of instances on different computers behind a load balancer or a message broker. And every message broker itself is distributed and every database is also distributed. And on top of that, as part of the process, we also called third party APIs that we don't really have any visibility into. But that's not the whole story. In a real world system, we may have hundreds of microservices where each microservice plays some part in processing a request flow or a transaction. So if a complex and constantly evolving architecture, it's almost impossible for any human to remember what microservices are involved in processing a given type of request or in what order. So this is where distributed tracing comes in. Distributed tracing is a method of tracking requests as they flow through, the entire system starts at the client's device all the way through our backend services and databases. As the request is being traced, we collect critical performance information about the time each part of the system is processing it. This allows engineers to visualize the entire flow and understand all the components involved in processing the request and the time it took for each component to do its work. This is extremely valuable for troubleshooting bugs or errors that lead to wrong behavior or performance bottlenecks. Usually distributed tracing is not enough to tell us what exactly is happening, but it is enough to narrow down our search to a particular component or a communication problem between two components. After we know where the. Problem is happening. We can use the other pillars of observability, which are the logs and metrics to debug the issue further. So now that we understand what distributed tracing is and why we need it, let's go a bit deeper into how it works. When the initial request is being made, we generate a unique trace ID and place it into an object called a trace context. This trace context object contains key data about the entire trace as the request flows through the services. That context is propagated usually through Http headers or message headers inside events. But just passing the tracing context from service to service is not enough for the application instances to collect the tracing data. We need to instrument them using a tracing instrumentation library or SDK. Those tracing libraries come in different programming languages, so even if we have a polyglot system, we can still get a complete trace. Now, as soon as the service instance receives the trace context, it collects the necessary data and propagates the context for the next service. So at the end of the transaction, each service instance that was involved has its own measurements and data which can be later aggregated by the trace ID to visualize all the parts of a transaction and how long each part of the transaction took. The trace is broken into logical units of work which are called spans. Those trace spans can be coarse grained like the processing of a request by a service or a query by a database. Or they can be manually created by the developers using the instrumentation library. So different units of work within a service can be visualized and measured separately. This way, if one logical part appears slower than usual, we can investigate it further as a potential performance bottleneck. And if an expected span is missing, we may have a bug that we need to debug further to increase the granularity even further. We can connect related pieces of work together by organizing spans in a hierarchy with a parent child relationship. So for example, if after inspecting the trace, we notice that a particular service is slow to process a request, then we can expand that span and see its children and their children spans. There we can see that one of the database queries was surprisingly slow, which we can now investigate by looking for particular log messages in that service. Now let's talk about the other aspect of distributed tracing, which is how data gets collected from the services and aggregated into those visual views. There are many solutions and vendors with different architectures and deployments, but the general approach is as follows. Once the trace data is collected inside each service instance, it is pulled by an agent which runs as a separate process on the same host with each service instance. Those agents then send the tracing data to a central queue or topic in a message broker. Then all the tracing data that comes from many services is analyzed, aggregated, indexed and stored in a database by a big data processor. Later, a developer can query and visualize this data using a tracing UI in the browser. Now, as we would expect with all those benefits of distributed tracing, it also comes with a fair amount of challenges that are important to be aware of. The first challenge, which we already mentioned, is the fact that we need to manually instrument our code to collect data that can be used for distributed tracing. In most cases, it's not a big effort, but it requires us to depend on and load a certain library. Learn how to use it correctly and sometimes manually add that instrumentation where needed. Otherwise, if we don't do this manual work when we actually need to investigate the production issue, our spans may be. Too broad, lacking granularity or important data or in extreme cases, if we use the instrumentation library incorrectly, our traces may become broken and therefore useless. The second challenge is cost. First of all, we need to run an agent on each microservice host which consumes its own CPU and memory. Then the collected data must be sent over the network, which requires additional bandwidth. Then we need to run a big data pipeline with its own infrastructure to process those tracing logs that come from different services. This big data pipeline, of course, comes with its own maintenance cost, but by far the biggest challenge is the cost of storing those traces in a database and retaining them at least for a few weeks. So that developers can find them if they need to debug an issue. If we think about it with a few millions of requests per day where each request flows through a few dozen of microservices, the volume of data we need to store can be very high. So to keep the storage and network costs in check, most companies use sampling on the client side, which sometimes means we can get one trace out of a thousand or maybe even 10,000 requests. This lowers our storage costs, but with such a high sampling ratio, sometimes it makes it hard to find a trace that can reproduce a particular issue we're trying to debug. But in addition, we also have another problem, and that's the size of the traces and the amount of information contained in each one of them. Typical microservices and event driven architecture deployments usually involve so many components that a single trace can be so big that it's hard for a human to look at, let alone use it. In my personal experience, some of the traces I've seen in production were so big that the tracing UI couldn't even load them, so it was impossible to even look at them. And many companies are experiencing similar issues. But despite all of those challenges, distributed tracing is a very powerful and essential debugging tool for microservices and event driven architecture, and it's often attributed to restoring the confidence among developers that if issues happen in production, they have the tools to troubleshoot it and find the root cause. And finally, we talked about the challenges of distributed tracing, which included manual code, instrumentation, high costs and the amount of data that we need to visualize and store.
