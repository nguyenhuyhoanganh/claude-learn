# Introduction to the Three Pillars of Observability in Microservices

Welcome back to another lecture. As professional software engineers, we all know that no matter how well we test our code, bugs, failures and performance issues will always happen. Such issues are particularly hard to troubleshoot when we have a high scale distributed system comprised of dozens or even hundreds of microservices. So in this section we will talk about the important topics of observability and monitoring in a microservices architecture. In this lecture we will start by learning what observability means and how it's different from monitoring. Then we'll talk about why observability is so important for microservices architecture. And later we'll introduce the three pillars of observability, which we will discuss more in detail in the following lectures. So what is observability and how is it different from monitoring? On the surface, monitoring and observability are very similar concepts. Both provide tools that collect data and give us insights into our system in production and both allow us to detect issues when they happen. However, there is a subtle difference. Monitoring is the process of collecting, analyzing and displaying a predefined set of metrics, and by attaching alerts to those predefined metrics, we can find out when something goes wrong. However, monitoring tools and dashboards on their own usually will only tell us that something is wrong, but will not be sufficient to tell us what is wrong and how to remediate the problem. On the other hand, observability enables us to actively debug, search for patterns, follow our microservices inputs and outputs and get insights into the behavior of our system. This allows us to follow the flow of individual requests, transactions or events across the entire system, discover and isolate performance bottlenecks and ultimately point us to the source of the problem, as opposed to just telling us that the problem exists. While monitoring is very important for any type of system, observability is primarily critical for microservices architecture. If we have any issues in a monolithic application, it is usually relatively straightforward to debug or profile it, since it all happens within the boundaries of a single application. In the worst case, we can always ssh to an instance of our monolithic application and inspect its logs or instrumented and get an idea of what part of the code is causing the performance issues or is throwing the exception. On the other hand, in microservices architecture, a single user request may involve several microservices and databases. Those can even communicate with each other asynchronously through a message broker. Now, if an issue happens at some point in the transaction, it can be very difficult to find in which service it happened. Additionally, all those services run as a group of instances on different computers, so the challenge is even more amplified. And if this isn't enough, statistically, most issues in microservices architecture actually happen at the API boundaries of those microservices and not in the microservices code itself. So having insights into the flow and content of the data coming in and out of each microservice and tracing the path of requests across microservices is absolutely critical. Typically, having just one type of signal is insufficient to debug a problem in microservices and event driven architecture. So when we talk about observability, we typically refer to three types of signals which are commonly referred to as the three pillars of observability. Those pillars are distributed logging metrics and distributed tracing logs are append only files that record individual events happening within an application process. Container database instance. Or server. Those events are usually represented as strings of data in a structured or semi-structured way. Additionally, those events are accompanied by metadata such as the timestamp of the event, the request that triggered it, the method class or application where the event happened and so on. Metrics are regularly sampled data points represented as numeric values such as counters, distributions or gauges. Examples include counters of the number of requests per minute or errors per hour, distributions of latencies or gauges that represent the current CPU or memory usage, cache, hit rate and so on. The third pillar is tracing. Traces represent the path a given request takes throughout several microservices and the time each microservice takes to process that request. Traces may include additional information such as request headers, response status codes and so on. So when we get an alert about an issue or manually detected using our monitoring dashboards, we can use all three forms of signals to further debug the issue using a combination of those signals. We can trace individual requests, isolate the issue to a particular microservice or API and narrow it down to an individual method or even line of code that causes that bug or performance bottleneck. Using those signals, we can also get enough insight into how to address that issue, which can involve a rollback, a hotfix or changes to the infrastructure like adding more service instances, diverting traffic to another region or data center and so on.
