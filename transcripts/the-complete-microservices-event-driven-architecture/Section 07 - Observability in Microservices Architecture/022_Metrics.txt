# Metrics

In this lecture, we will continue discussing the three pillars of observability. And now we're going to talk about the second pillar, which is metrics. Well, first, get a quick introduction to metrics and why we need them. Then we'll illustrate the problem of collecting too many metrics that aren't necessary. And finally, we'll learn the five types of signals that we should focus on for maximum observability and minimum complexity. So let's start the discussion about metrics. Metrics are measurable or accountable signals of software that help us monitor the system's health and performance. Typically, metrics are regularly sampled, so we can see a continuous progression which helps us monitor our system and detect anomalies when they occur. And they usually come in numerical values. So we can easily quantify them and set alerts based on their direct or derived values because they are just numbers. Out of all the three pillars of observability, they're the easiest to collect, visualize and organize into dashboards. For example, a dashboard may show us all the vital signals for a given microservice, or it can compare each metric with its historical value from before the release or before a significant change in the system. Those production dashboards are a critical tool for us when a production issue needs our attention, instead of having to search and read through hundreds of log lines, we can look at one or maybe two dashboards and get a clue about what's happening. However, as a team that owns a microservice, a good question. We should ask is what signals we should measure, collect and monitor, and why wouldn't we just collect all of them? So the first important thing to understand is that the number of different signals we can collect is pretty large on the resources level. We can measure so many signals that may or may not be helpful in different situations, but additionally, we can instrument our application and measure anything starting from the number of requests we receive per minute to the number of times a given critical piece of logic is executed. However, collecting anything that we can measure is a big anti-pattern First of all, collecting and storing so many signals from each server or container can become very expensive, especially when we're talking about a large scale system with dozens or even hundreds of microservices. But even if we can bear the cost and we can handle the storage of all those metrics, we have another problem, which is information overload. Let's say we get a page in the middle of the night that our website is not loading or is very slow for most of our users. Now, given all the metrics that we're collecting, which dashboard should we look at? And even if we know which dashboard we should look at, the number of graphs on each dashboard won't fit on the screen. So it's really hard to find the anomalies. And even if we do notice a change in some metric, it will be very difficult to understand which one is the symptom and which one is the cause. So instead we should be smart about what metrics we collect. Luckily, we can leverage decades of experience from companies that have been running large distributed systems and microservices in production and focus on five types of signals which will give us the most information and the least amount of noise. Those five categories are based on two sources of knowledge. The first is Google Searches for Gold and Signals, which focuses on user facing metrics, and the second is the use method by Brendan Gregg, which focuses more on system resources. So combining those two resources, we have five categories of signals which are traffic errors, latency saturation and utilization. Traffic is the amount of demand being placed on our system per unit of time. For example, we can measure the number of Http requests per second or minute for a microservice that receives Http traffic. We can also measure the number of queries or transactions per second or per minute for a database and for a message broker. We can measure the number of events it receives and the number of events it delivers to consumers. Now, in some cases where a single request to a microservice results in many outgoing requests, it's also worth measuring the number of outgoing requests and incoming requests separately. And that's because open connections to other services consume system resources and can directly impact the microservice performance. The next category of metrics is errors. When we measure errors, we're interested in the error rate and the type of errors we're getting. If possible. The error rate can be a great signal to set an alert for because if the error rate suddenly goes up, the users are most likely already impacted. Now, in the case of an increase in the number of exceptions in our application, we may not be able to show the error type as a number value, so we defer this information to logging. However, if we start receiving an Http response status code that is different from 200 from a service we depend on, we can use that as a metric which will be very helpful in troubleshooting production issues. Also in latency sensitive systems, we can even count successful responses as errors if they exceed some latency threshold that we set ahead of time for an event driven microservice. We can also measure the rate of events that it fails to process. And the reason for the failure, if it's possible to classify in a message broker, we can measure error signals like the number of events it failed to deliver to customers. And on a database side, we can measure the number of aborted transactions, disk failures and so on. The next type of signal is latency. Latency is the time it takes for a service to process a request while this metric seems pretty straightforward, there are a few things to consider to measure it correctly. The first important consideration is not to look just at the average latency, but always consider the full latency distribution. For example, let's assume that our microservice receives about 1000 requests per minute. 95% of those requests are processed within 50 milliseconds and 5% of them take 5000 milliseconds to process. So our average latency per minute is around 300 milliseconds, which is pretty good for a user facing system. However, this completely hides the fact that 5% of our users must wait five seconds for the whole website to load. So if. We have a million users per day visiting our website. 50,000 of them have such a terrible user experience that they will likely switch to our competitor and never come back. So if instead of looking at the average latency, we look at the 95th percentile of the latency per minute, we would see that a significant portion of our requests takes an unreasonable time to complete, likely indicating a performance bottleneck in our system. The second important consideration is separating the latency of successful operations from the failed operations. If we mix the successful and failed requests, we may get the wrong data. For example, if we fail fast when we can't process a request, we may artificially make our total latency look better than it is. Similarly, if we take too long to respond with an error, we may not know about it. If those errors happen rarely and successful requests take very fast to process. However, if we collect and monitor the latency of both successful requests and fail requests separately, we can get a good view of what our clients experience in both scenarios. Now let's talk about the fourth type of signal, which is saturation. Saturation measures. How overloaded or full a service or a given resource is. This metric is very important for any type of system that has a queue, whether it's an external queue, like a message broker or an internal queue within our microservice or CPU. When we have too many things stored in a queue, it can indicate that our system cannot keep up with the current demand. For example, if the topic in a message broker that stores purchase orders from users keeps growing, that likely indicates that we have some scalability issue in one of the services that consume those events. Another example is if the network queue of requests that go to a database keeps growing, this may indicate that our storage device is not fast enough and we need to upgrade our hardware. Or alternatively, if it's an in-memory database, we may have to shard it and distribute the data across multiple database instances. Finally, if some work inside a microservice instance keeps growing, it may indicate that some part of our logic is too slow and this microservice instance may soon crash with an out-of-memory exception. Finally, having visibility into saturation can also explain why our users suddenly have longer latency or requests from other services are timing out. The last type of signal is utilization. Utilization is used for measuring how busy a particular resource is over a period of time. This typically applies to resources with limited capacity like CPU, memory, disk space and so on. It's important to point out that in most cases we will see performance degradation before we reach 100% resource utilization. So it's important to set alerts before we reach that critical level. For example, if we see that the CPU utilization on our microservice instances is growing and approaching 80%, we need to scale out and add more service instances. Otherwise, as we cross the 80% CPU utilization, we may start experiencing higher latency and other issues. Similarly, if we see that our database is getting close to running out of storage, we need to add more database instances to keep up with the growing amounts of data. It's also important to point out that we want to measure the utilization with higher granularity and not just an average over several minutes. Otherwise, we'll miss the short periods of high utilization that may be attributed to performance bottlenecks or other inefficiencies in our processing. Now, in conclusion, I want to point out that those five types of signals aren't necessarily the only ones that we need to collect. So depending on the business. Logic and the specifics of our system or microservice. We can collect additional metrics that can add even more to our observability. But those five types of signals are the most common that apply to any system and give us the most value by tracking them. In this lecture, we'll learn about the second pillar of observability for microservices and event driven architecture. We first got the motivation for metrics and mentioned that this is the easiest type of signal to collect and visualize, which makes it the most valuable for debugging production issues. And later, we talked about the five categories of signals that we should measure and collect. These were traffic errors, latency saturation and utilization.
