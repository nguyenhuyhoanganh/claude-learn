# Distributed Logging

Hey, welcome back. In the previous lecture, we got the introduction and motivation to observability in microservices and event driven architecture. We also learned about the three pillars of observability, which included distributed logging metrics and distributed tracing. Using those three types of signals, we can get deep insights into our system, effectively debug issues and find the source of the issue or the performance bottleneck. So now let's talk more in depth about the first type of signal which is distributed logging. First, we'll have a quick recap of what we mean by logging, and then we'll talk about some important best practices for distributed logging in microservices architecture. So let's start with a quick reminder of what logging is. Logging is one of the simplest way for developers to provide insights into the current state of an application. A log line can represent an application's event like receiving a new request or an action like performing a database query or starting a complex processing operation. It's also a way to record exceptions and errors in a method accompanied by the set of parameters that led to that issue. This information can be invaluable to debug and fix the issue and add enough tests to prevent such bugs in the future. However, in microservices architecture we can have thousands of instances of different microservices collectively producing millions of log lines per day. So when we have an issue, it's not practical to search and inspect those log files one by one. So the first important best practice for making sense of all this useful, but very large amount of data is to collect them into a centralized and highly scalable logging system. The system needs to parse and index those logs so they can be easily searched by patterns or text and grouped or filtered by attributes like host, microservice, time period or region to make searching easy. It's also a good practice to follow a predefined structure or schema and the same terminology across different events within a microservice and across separate microservices. Log lines should be easy to read by humans, but also easily readable by machines so they can be efficiently parsed, grouped and analyzed. This can be extremely important when we're faced with a time sensitive issue affecting users and we need to get the root of the problem very quickly. Some examples of log structures include log Fmt, which structures logs as key value pairs. Json and XML. The next best practice is assigning a log level or severity to each log line, depending on the framework or system we use. There are different log levels and different number of log levels, but the most typical ones are trace, debug info, warn, error and fatal. This adds an additional dimension to our logs that we can slice and filter on, which helps reduce noise and alert fatigue. For example, exceptions and user impacting errors need to be logged as error and fatal respectively. This way, if we're an engineer on call and we get an alert in the middle of the night that something is going wrong, we can search the logs and filter only by error and fatal, find out what exactly happened and take the right actions. We can also have automated tools that periodically search and group events on those levels of severity. Then those tools can either alert us or create a ticket and automatically assign it for somebody to work on. Similarly, events that indicate potential problems such as high processing time of outgoing requests or receiving requests containing unexpected values must be logged on the warn level. This way we can filter on this log level and potentially prevent future issues by addressing those events. Finally. Olya, if we're a developer digging into a very complex issue and we need to follow every event that happened on a particular application instance, we can look at the information on all log levels, which includes the most fine grained details logged on the debug and trace levels. The next best practice for logging is using a unique ID, commonly called correlation ID for every user request or transaction and adding this ID to each log line corresponding to an event or step and processing that request or transaction. Since each microservice instance is likely processing multiple user requests concurrently. This helps search and filter only the events related to the request in question. It can also help us to see the sequence of events for a given request across multiple microservices. The next best practice is providing as much contextual information to each log line as possible. For example, if we log an error or exception, we want to add the stack trace to debug and understand how we got to that line of code. But we also want to include the parameters that led to this error. If we have a database query that took a very long time to complete logging the exact query and the content of the request that triggered it can help us understand how this issue even happened and what measures we can take to mitigate it. Some common data points we want to include in almost every log line include the name of the service that emitted the log event, the host name where that event happened, the user ID or some other identifier that can add more context to who initiated the operation and of course, the timestamp of when that event happened. On that note, we also have two important considerations to keep in mind. First, we should only log information that is critical for debugging because on a large scale system, storage and processing can be very expensive. The second consideration is we should never log sensitive or personally identifiable information such as usernames, passwords, Social security numbers, emails, credit card numbers and so on. While having this data can be sometimes helpful in troubleshooting. This poses a huge legal risk for the company in the case of a security breach. It also adds a lot of complexity regarding security and data retention, compliance and generally it's just unethical. Nobody wants a random engineer troubleshooting an unrelated issue in production to have access to our personal information, especially if they don't need that to fix the issue. In this lecture, we'll learn the first pillar of observability for microservices, which is distributed logging. After getting a short introduction, we talked about some very important best practices that can make a huge difference when there are production issues that we need to debug. Those best practices included collecting all the logs from every microservice instance to a centralized system that can store and index those logs so they can be easily searched and organized. Then we discussed the importance of following a common log structure in microservices. After that, we mentioned the benefits of setting log severity for each log line. And finally we talked about adding important information such as a correlation ID and additional contextual information, which can be very critical to figure out what exactly happened and how to fix it.
