# Container Orchestration and Kubernetes for Microservices Architecture

Welcome back. In the previous lecture, we'll learn the benefits of deploying microservices inside containers in production. Some of the benefits we mentioned included full parity between development testing and production, environment portability between different cloud environments and better hardware utilization, which allowed us to restart microservices faster and reduce the infrastructure costs. However, we encountered a big challenge in deploying, configuring and managing thousands of containers, running hundreds of microservices, potentially even across different cloud providers. So in this lecture we will talk about the solution to this problem, which is container orchestration. We will first talk about what container orchestration is, and later we'll go more in depth by learning about a very representative architecture of a container orchestrator. So first of all, what is a container orchestrator? A container orchestrator is a tool that manages the entire life cycle of all the microservice containers within our system. It is like an operating system. But for microservices architecture deployed as containers, the responsibilities of a container orchestrator include automation of deploying new microservices or new versions of microservices as containers, managing resource allocation among existing containers to ensure each container gets the right amount of CPU memory and storage to work properly. Monitoring the health of containers self-healing, which allows automatic deployment of new containers to maintain a healthy number of instances of each microservice. Then we have automatic bin packing, which means scheduling containers in an efficient way to provide optimal utilization of existing hardware. Then we have load balancing between containers of the same microservice scaling services out or in by adding or removing containers based on the traffic and the resource utilization. And finally managing the discovery and network connectivity between services, containers and the outside world. Now to illustrate how a container orchestration tool works, I'll use the architecture of Kubernetes, which is one of the industry's most popular but not the only container orchestration tools. However, since this is not a DevOps crash course on Kubernetes or any other specific technology, I will spare you all the complex and unnecessary details so we don't lose track of the big picture. A typical container orchestration cluster will have a control plane deployed on at least one machine. We'll call that machine a controller node. The remaining VMs or servers that run our microservice containers are called the worker nodes. The controller node runs all the processes that orchestrate and manage our cluster. For example, we'll have an API process that takes commands from us to add a new microservice, update an existing microservice version or configuration and so on. All the information about the cluster configuration and current state is stored by another process, which is a key value store database. Now if we need to add another container of a particular microservice, we have another process that monitors the worker node resources and decides where to schedule that new container. Then we have another process that monitors the health of our worker nodes or state changes within those nodes. For example, if one of the nodes becomes unavailable, it will detect that and ensure that the scheduler process reschedules those microservice containers to the other worker nodes. Another process we have manages all the cloud provider specific logic, such as deleting unresponsive worker nodes by the cloud provider, adding cloud, manage load balancers, routing them to our containers and so on. Now, in addition to the processes running in the control plane, each worker node also runs a set of agents to manage the containers running. Only on that note. That includes the container runtime, which is the software engine running the containers on each host. Then we have the agent that monitors and actually starts the microservice containers when it receives the appropriate command from the control plane. And we also have another process which acts as a proxy. This proxy maintains a set of network rules to route network requests between containers of different microservices and balances, the load on containers of the same microservice. Additionally, sometimes we need to run each instance of our microservice with a sidecar process. This sidecar process can be a logging or monitoring agent or an in-memory cache. However, typically putting more than one process inside a single container is not a good practice. So Kubernetes allows us to bundle several containers inside another logical unit, which is called a pod. So this pod is the smallest runtime unit that Kubernetes will manage for us, even if that pod only contains a single microservice container. Finally, the description of all the microservices architecture, their configuration with container images they use and how much CPU memory and storage they need is described in a declarative and human readable way. This configuration also contains all the information about the services our microservices connect to outside the container cluster. Those include cloud functions, databases, object stores, message brokers, and so on. This configuration is stored, maintained and updated in a version control system just like any other code. Once a change is made to that configuration, it will be sent to the control Planes API server, which triggers the relevant updates within the cluster. Now, in a typical microservices architecture, we'll have hundreds of such worker nodes running thousands of containers belonging to different microservices. So for higher availability and to keep up with everything going on in the cluster, we'll also typically have multiple replicas of the controller nodes inside the control plane. And for even higher availability we can have multiple clusters with the same or different configurations running on different cloud regions or even cloud providers, and we can route traffic to each region using a global load balancer. This routing can be based on criteria such as physical proximity to the user, the load or health of each cluster and so on. As we can see, this container orchestration architecture can be pretty complex to set up and manage. It requires expertise from DevOps or site reliability engineers, and it also requires dedicated resources just for orchestration purposes like those controller instances. However, as we already learned earlier in the course, all that investment pays off because its cost is amortized across all the teams and microservices that we manage. So if our decision to migrate to microservices was correct, the benefits of running microservices as containers managed by the container orchestrator will outweigh its cost and complexity.
