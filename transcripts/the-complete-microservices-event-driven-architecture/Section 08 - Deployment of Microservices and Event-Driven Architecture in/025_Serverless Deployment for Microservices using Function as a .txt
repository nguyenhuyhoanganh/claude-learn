# Serverless Deployment for Microservices using Function as a Service

Welcome to another lecture. In the previous lecture, we covered a few infrastructure options for deploying and running microservices. In this lecture, we'll continue this discussion and talk about another type of deployment which is more event driven than the ones we mentioned in the previous lecture. This deployment is called function as a service. First, we'll discuss the problem this deployment type solves for us. And finally, we'll conclude by discussing its benefits and drawbacks so we can use this in production correctly. To illustrate the problem we're trying to solve, let's look at two real life scenarios. Let's consider a use case where we're running a business in the entertainment industry, primarily focused on on demand video streaming, where users can rent or buy movies and shows. However, as an additional offering, we also partner with different performance venues to help them sell tickets to their live events. So we have a dedicated microservice for that purpose that handles ticket reservation requests for live events only. However, those live events happen only occasionally once a month at most. And when we open the reservation, most requests come within 30 minutes to an hour. During this time, we sell 99% of our tickets, and then we're left with only a few cheap tickets that are not so good and a few very good tickets, but are very expensive. Those remaining tickets may be sold some time between the initial launch and the event date, but the traffic will not be very significant. Now, if we look at the traffic pattern to this microservice, we see that it's not getting any traffic most of the time. However, if we deploy it on a cloud VM or even a dedicated host, we will be paying for renting that hardware during all this downtime. At the same time, when we open the ticket reservation for the new live event, we get a very high traffic spike. So we also need to configure and pay for a load balancer and maintain autoscaling policies for this microservice, which adds even more to the infrastructure costs for running the service. Now let's consider another scenario. Let's say we're a digital advertisement company. One of our microservices allows our customers to run a monthly or quarterly report that provides insights into their reach and return on investment for their advertising. This report can be triggered directly by an administrator working for the advertisement company or by a cron job that runs once a month or once a quarter and triggers the report for them automatically. With even 1000 customers, we will only get about 1000 such requests or events per month or quarter. So running this microservice all this time just for the events that happen so rarely is not very cost efficient. But in addition to the infrastructure costs of running this microservice, there is also a lot of development cost for maintaining it. If we're a team that owns such a microservice, in addition to the business logic, we also need to maintain a lot of boilerplate code that handles Http requests or events from another source that triggers this logic. And we also need to maintain scripts to build package and deploy our microservice binary with all its dependencies to a cloud server. So we have to invest all that effort just to handle very little traffic or events that happen so rarely. So this is where the serverless offering called function as a service comes in. Function as a service is a cloud offering that allows us to architect our system in a fully event driven model, not only from the software perspective, but also from the infrastructure perspective. As part of this offering, we need to provide the cloud vendor with only two things the type of requests or events we wanted to handle and the logic we want to execute when that event is triggered and. That's it. Only when that event is triggered will the cloud provider take our code. Package it, deploy it to a physical hardware and execute it. In response to that request or event. Additionally, if our traffic has a seasonal pattern where we may have a lot of requests coming in in a very short period of time, in that case, the cloud provider will also handle the horizontal scalability to allow us to handle this traffic spike without the need to maintain any autoscaling policies or configuring a load balancer. Now, the pricing model for this offering is based on the number of requests our Microservice receives, as well as the time and memory required to handle each request by running our logic. However, as long as that request or event doesn't arrive, we don't pay anything. So the clear benefit of this offering for microservices that handle events that happen very rarely is. We can save a lot of money on infrastructure for seasonal workloads with rare but very high traffic spikes. The cloud provider also handles all the operational overhead of scaling that service. And the last benefit for both of those types of workloads is that it allows us to save a lot of development cost and overhead as the cloud provider takes care of building, packaging and deploying our microservice for us. Of course, just like anything in software engineering and particularly in software architecture, there's always a trade off. So the main downside of the serverless deployment is if our traffic pattern changes, our infrastructure costs may increase significantly. For example, if we start with a microservice that handles very seasonal or low traffic, but over time we start getting more and more requests, our code will run more and more frequently. And also as our business logic becomes more complex each time we receive a request or event handling, it will require a lot more time and memory. So as a result, it will become more expensive for us to use this offering than deploying it on a cloud VM or even a dedicated host. The second downside is the performance of a serverless deployment is much less predictable than deploying our business logic as part of a microservice we fully control. So serverless deployments like function as a service are not the best option for latency sensitive workloads. And finally, this is also the less secure type of deployment, because not only are our code runs in a multi-tenant environment, we also completely expose our source code to the cloud provider. Now let's compare this deployment type to what we've already learned so far. If we use the function as a service for the correct workloads. This can be the most cost efficient way to deploy our microservices. But if we use it incorrectly or not for the right workload, it can also be the most expensive option. And when it comes to security and performance, this is also the least optimal option of all the deployment types.
