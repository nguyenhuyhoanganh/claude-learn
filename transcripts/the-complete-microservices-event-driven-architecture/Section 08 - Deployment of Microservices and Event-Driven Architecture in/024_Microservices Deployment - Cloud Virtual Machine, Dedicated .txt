# Microservices Deployment - Cloud Virtual Machine, Dedicated Hosts and Instances

Welcome back to another lecture. Now that we know how to develop architect test and troubleshoot microservices in production, there's just one important step we need to cover, and that is how to deploy and run microservices in production. So in this lecture, we'll start the discussion by covering a few infrastructure choices for running microservices, primarily focusing on the cloud environment. The first and most common way to deploy microservices in the cloud environment is by using cloud virtual machines. As a quick reminder, a virtual machine is an isolated environment, running on top of a real physical computer. That virtual machine acts like a virtual computer with its own operating system and virtual resources like CPU memory network interface and storage. Those virtual resources are allocated, managed and mapped to real resources on the physical hardware by another layer of software called the hypervisor. By using virtual machines, the cloud provider can split each physical server into multiple VMs, and the amount of resources allocated to each VM depends on how we configure it and how much we pay for that. VM. Now each VM can run a different instance of the same microservice, a separate instance of a different microservice, or even a different instance of a microservice belonging to a different cloud provider customer. This property is called Multi-tenancy. By default, when we rent or reserve cloud VMs, we have no control over who else is running their VMs on the same host that allows the cloud provider to split all of their servers very efficiently, fully utilizing their hardware. And that in turn allows them to offer us competitive prices, which makes cloud VMs a very attractive option for running microservices. So the main benefit of using cloud virtual machines for running microservices is the affordable and flexible pricing. The typical pricing model for cloud VMs is pay per use, meaning that we pay only for the cloud VMs that we're renting for the duration that we're renting them. And the rate for each VM depends on the CPU memory and network bandwidth capabilities that we request. Now, one of the downsides of this deployment is security risks due to Multi-tenancy. Theoretically, when we run two VMs on the same server, those two VMs are completely isolated from each other. So if those two VMs are running two database instances, for example, and each instance belongs to a completely different organization, a security breach in one of the VMs should not pose a risk to the other VM. However, this is the real world. The hypervisor is a software that was written by human beings and all the security and configurations are managed and updated also by human beings. So in practice it is possible for a hacker to gain access to a VM that was poorly secured by the engineers of another organization. And because the cloud vendor allocated our VM to run on the same host that that hacker may manage to hack into our system and steal our data. Of course, the cloud vendors and the companies that own the hypervisor make great efforts to prevent this from ever happening. So the probability of this actually happening is very small. However, due to compliance issues in certain industries, we may be unable to tolerate even that small risk. Examples of industries that cannot afford to run some parts of their services on cloud multitenant environments include banking, healthcare and government or national security related services. The second drawback of running microservices in multi-tenant environments is potentially lower performance due to an issue called Noisy neighbor. Just. In the previous case. Theoretically, the hypervisor should be able to completely isolate and allocate each hardware resource to each VM as configured ahead of time. However, in practice, not all resources can be accurately allocated. For example, if the host has 16 CPU cores, then the hypervisor could easily split those 16 cores evenly. So each VM gets dedicated eight cores. However, the network bandwidth of a network card or the access to an internal bus that transfers data to and from a storage device can't be easily rationed in an accurate way, and many other physical resources inside the computer cannot be split because after all, it is one physical server. Additionally, the hypervisor itself may consume some CPU and memory for its own use. So while there isn't a lot of data to suggest that running a very intensive workload on one VM can significantly impact the performance of the other VM. In theory and in practice that impact is still there. So for very latency sensitive services like high frequency trading systems, gaming or video streaming multitenant deployments with cloud VMs is not the best option. That brings us to the next cloud deployment option, which is single tenant. Dedicated hosts or instances. In this deployment, we can ask the cloud provider to run our VMs on servers that are dedicated to the account that belongs to our organization. This means that the only tenants will have on the same host are instances of our microservice. Other microservices or databases belonging to our organization only. So if we're in an industry that doesn't allow us to share infrastructure with other companies, this is a great but obviously slightly more expensive alternative. The obvious reason why cloud vendors charge more for such deployments is that they cannot allocate their hardware as efficiently as the multi-tenant deployments. Additionally, some cloud providers even allow us to rent or reserve an entire host just for our organization. That gives us direct access to the host's hardware resources, which can completely eliminate the noisy neighbor effect and reduce the overhead of virtualization by the hypervisor. Of course, the main downside of this type of deployment point is that it's much, much more expensive than multi-tenant cloud VMs. So in conclusion, the multi-tenant VM deployment provides us with the best pricing but doesn't provide us with the most optimal security or performance. If we need additional security, we can rent a single tenant dedicated instances which are a bit more expensive, but guarantee that only VMs of our organization can run on the same physical hardware. And if we also need the most optimal performance and want to eliminate the possibility of a noisy neighbor, we can rent dedicated hosts, which is also the most expensive option.
