# Containers for Microservices in Dev, Testing and Production

In this lecture, we're going to talk about containers for developing testing and deploying microservices to production. This deployment type is one of the most popular and the most trending ways to package and deliver microservices to production. And we'll see why in a moment. But first, let's have a quick overview of what problem we're trying to solve using containers. Then we'll talk about the benefits of containers for microservices over the deployment on virtual machines. And finally, we'll talk about the challenges of running microservices in production using containers. So let's start by describing the problem we're trying to solve, starting with the development environment. One common problem that companies have is the lack of parity between the development environment and the production environment. For example, imagine you're a developer on a team owning a microservice. You set up all the development environment for that microservice locally with a local stand alone database version. You also installed all the dependencies for that microservice specific to your operating system. However, the production distributed database is configured differently than the local in-memory standalone database, and also the operating system in production is not the same as the one on our development laptop. So all the locations for the configuration files and the versions of the dependencies are entirely different. So when we finished developing our feature on our machine and tested it locally, everything worked fine. But when we deploy it to production. Things break or don't work as expected. One solution is to develop and test our microservice in a production like Virtual Machine on our development computer that solves the development and the production parity problem but introduces a new problem. Our host operating system needs to run another software layer called a hypervisor type to this, hypervisor runs just like any other program on our host operating system, but internally runs another entire operating system with its own kernel. This kernel manages all the guest operating system application processes, file system, security, networking, memory and many other components. And it also uses device drivers to interact with virtual hardware which is emulated by the hypervisor. So as we can see, we have so much unnecessary overhead just to run a single microservice instance. Now if we want to run and test the integration of a few microservices, we need to run multiple virtual machines, each with its own copy of an operating system and its own kernel. This creates so much overhead that everything will be extremely slow and inefficient, which makes the development and testing very hard. So containers solve this problem by isolating only what we want to isolate and sharing everything else. When we package our microservices into containers, each container image includes our microservice binary, the command to run it, and all the dependencies it requires in complete isolation from any other microservice. When we run one or multiple instances of that image, each container has its own isolated file system, network interface and runtime. However, the OS kernel drivers and everything else that we don't need to isolate are shared among all the containers. So the overhead of running a few dozen of containers on our machine is minimal, making it perfect for developing and testing microservices. Now the benefits of containers don't end at the development and local testing stage. Now we can run those containers in our continuous integration pipeline, which may run a completely different operating system and use completely different hardware than our development computers. But all of that doesn't matter because container images are completely decoupled. From the OS and the hardware so we can create them once and run them on any hardware and operating system that supports containers and container runtime. Now, since in this section we're focusing on deploying microservices to production, we can take it even further. When we look at our previous approach using cloud VMs, each microservice instance was deployed on its own VM and scheduled by the cloud vendor to run on one of its servers. This approach had many advantages that we already mentioned in the previous lectures. However, it also has a few big disadvantages that are very similar to what we already discussed in the context of a development environment. If two microservice instances run on the same cloud server, for example, each VM runs an entirely separate instance of the typically the same operating system. This duplication means we lose valuable memory, CPU and storage resources to the operating system and also deploying and starting. Each VM can take minutes until we're ready to accept traffic. But in addition, we have another issue, which is the cloud vendor lock in and lack of portability. The problem is when we create a VM image for a microservice, the format of this image may be cloud vendor specific and also the configuration that describes the type of VM we want to rent is also cloud vendor specific. So if we get an attractive offer from another cloud vendor and we want to migrate completely, we'll have to do a lot of work creating those new images and even a bigger problem is when we have a multi cloud environment, when we run some services on one cloud provider and another part on another cloud provider, or in a hybrid cloud environment where one part may run on a public cloud vendor and another run in our private cloud environment for higher security or performance purposes. For those situations, Cloud VMs will be a nightmare to manage across different environments. So all of this can also be solved by using containers. Once again, all we need to do is create those container images once and then deploy them on generic cloud VMs or even directly on dedicated hosts. In any environment, the only requirement is to install the container runtime, which is the software that runs those containers, and we're good to go. So to summarize the benefits of running microservices inside containers in production, we have better portability between environments, and that's because we can build a microservice image once and then use it in development, QA staging and production on any cloud provider hardware or operating system. We also get faster startup time because containers usually take only a few milliseconds to deploy and run. And we can also save a lot of money on infrastructure because the hardware utilization is much better when we use containers. So instead of renting multiple smaller VMs where we lose part of the CPU and memory to the operating system, we can rent larger VMs or even dedicated hosts. When we deploy the containers of different microservices on a single VM or host, we can better utilize the hardware because they share the same operating system kernel. And in many cases what this means is that we can run more microservice instances for the same amount of hardware than when we use cloud VMs. However, unlike in the case of the development environment and the continuous integration, we now have another problem to solve before we can take full advantage of containers in production. The problem now is we have two layers of abstractions that need to be glued together. On the one hand, we have the cloud infrastructure abstraction, which includes the cloud, VMs or hosts, which we need to rent and autoscale based on the traffic or load on our system. And we also have other cloud managed services like. Databases, message brokers distributed logging and so on. On the other hand, we have the containers obstruction, where we have dozens of container images representing different microservices. Each microservice image needs to be deployed as a group of container instances on that infrastructure. Then we need a way to discover and connect all those microservice containers to each other and the managed services through the network. We also need to manage the scalability and availability of each group of the containers so we can add more instances if the traffic goes up and remove instances when the traffic goes down. Additionally, we need to automatically replace containers that crash and a way to update all of the containers of the same microservice. When a new version of a microservice is released. Doing all of that manually for hundreds or even thousands of container instances potentially running in different cloud regions or even different cloud providers is an impossible task. So in the next lecture, we'll learn about the solution to this problem, which is introducing another layer called container orchestration. But before we do that, let's quickly summarize what we learned about containers. Since we learned a lot of new important concepts in this lecture. We first introduced the problem of the lack of parity between the development or testing environment and the production environment. This lack of parity may result in things working fine in development or QA, but breaking when we deploy microservices to production. Then we illustrated the issue of running a VM to solve this lack of parity, which introduces a very big overhead. And that's because for each VM we need to run an entire copy of an OS with its own kernel and the hypervisor program. After that, we introduced an efficient solution to this problem using containers. We then extended that solution beyond just the development and testing environment. And we saw that by deploying microservices as containers, we could increase the portability of our microservices across different cloud providers, reduce startup time for each microservice and even more importantly, cut down on the costs associated with infrastructure. Finally, we realized that there is a challenge in deploying and managing microservices as containers in production, which we will solve in the next lecture.
