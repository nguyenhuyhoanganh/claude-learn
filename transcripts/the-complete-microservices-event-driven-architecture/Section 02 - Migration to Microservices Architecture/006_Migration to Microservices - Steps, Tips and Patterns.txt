# Migration to Microservices - Steps, Tips and Patterns

In this lecture, we'll learn how to perform the actual migration from a large legacy monolithic application into microservices. We'll first talk about where to start the migration to ensure that it doesn't get derailed or abandoned. After that, we'll discuss how to prepare for the migration. Then we'll talk about how to execute the migration using the strangler fig pattern. And we'll conclude with one important tip to ensure that we don't run into issues while performing the migration. Assuming that we have a very large code base that we want to modernize and migrate to microservices. Where do we even start? The most common approach that developers think of is a big bang approach. In this approach, the team maps out the desired boundaries for the future microservices architecture there and then try to get the management on board to stop any development of any features and fully focus on the migration on the surface. It seems like a good idea if the entire team fully focuses on the migration and doesn't get distracted by developing new features, they can finish it much faster. At least that's the theory. However, this approach is actually the worst in terms of productivity and impact on the business. And let's see why. First of all, having too many developers working on the same project, which in this case is the refactoring of the monolith to microservices, is a bad idea. Remember that we mainly want to migrate to microservices because our team already has productivity and communication issues, which is what we call organizational scalability. So having too many cooks in the kitchen for such a big project will create a lot of friction. The second reason is estimating the effort for such a big project is very hard. And just like in the case of many big and ambiguous projects, we often encounter technical problems that we didn't foresee. So if we promised our managers to finish the migration within four months and we're still not done, five months later, the managers will start getting nervous. This puts the entire process in danger of abandonment. To minimize the additional time loss, finally, stopping any development of new features for many months can be very detrimental to the business. Product managers may get bored and have no way to get promoted, which in turn may cause them to leave the company. Salespeople have nothing to sell or pitch to new clients and users who really couldn't care less about our internal architecture may start thinking that our business is in trouble or our company is unwilling to keep improving their experience. So the better approach is incremental and continuous. In this approach, we identify the components within our code base that can benefit the most from migrating to a separate microservice. The best candidates for the migration ranked by the benefit we get from migrating them are the following areas with the most development and frequent changes. Components that have high scalability requirements that cannot be met as long as they are part of the big monolithic application. And lastly, the components with very little technical debt and good logical separation that fits the scope of a microservice. We prefer to migrate code with the most development because migrating code that nobody ever touches doesn't give us any value because it isn't the source of our problem. On the other hand, the logic that is constantly evolving is the biggest source of merge conflicts. It's also the biggest contributor to releases of our entire monolithic application. So if we separate it into a microservice, those changes will not require us to redeploy a new version of our code. That didn't change. Separating those parts of the code into a separate microservice will help us disentangle its business logic from the rest of the application, making it much easier to reason about and less likely to introduce. Use a bug. Once we migrate that part of the code into a microservice, we identify the next candidate and we keep going continuously until we reach a point where the original monolith is gone. Or what's left in that monolithic application never changes. So there is no point in touching it any further. This incremental approach has several important benefits. First, we don't have to set hard deadlines to complete the migration, even if it takes a year or longer to reach the final point. We are constantly making visible and measurable progress. Our business is also not disrupted, and even if the migration of one part of our code exceeds the estimates, it will be in the order of days or weeks and not months or years. Now let's talk about how to prepare for the migration. After identifying the component or the components we want to migrate, we must ensure we have good test coverage. This step is critical because with good code coverage we have the confidence that we did not break any functionality during our refactoring. Otherwise, the entire migration may get delayed if things start breaking. Once we have good test coverage, it's important to define a clear and well thought out API for the component we plan to migrate and in the last step we also want to isolate the component by removing any interdependencies to the rest of the application. Finally, let's talk about how to perform the actual migration and learn about a very popular pattern, which is called the Strangler fig pattern. The Strangler fig pattern introduced by Martin Fowler is inspired by a plant that starts as a small vine growing on top and alongside an existing old tree. And then over time, it spreads and grows, completely consuming the old tree taking its place. The idea behind this pattern is first to introduce a proxy in front of the monolithic legacy application. It simply allows requests to pass through. This proxy is commonly called the Strangler facade and is typically implemented using an API gateway. If you're not familiar with an API gateway, it is an off the shelf open source or cloud based component that routes requests based on the API they intend for. Once the newly created microservice has been thoroughly tested and deployed, we divert the traffic for that API from the monolithic application and send it to the newly created microservice instead. This change happens completely transparently to the users and minimizes the risk of issues during the migration. After monitoring the performance and the functionality of the new microservice for some time, we can remove the old component with the same functionality inside the monolithic application. Then we repeat the same steps with the new microservice until we're either left with an empty old application or with a legacy application to support old clients. Before we conclude, I want to cover one last tip to ensure a smooth and bug free migration when migrating an old monolithic application to a new and modern microservice architecture, it's hard to contain the excitement and start using newer technologies and programming languages right away. However, the best approach, while migrating to a new architecture, is to keep the code and the tech stack unchanged as much as possible. The reason for that is that every additional change we make is a potential source for a bug, and the migration itself is already complex and risky enough. So we don't want to add even more risk factors to the process. Once the migration is complete and the new microservice is running and stable, we can easily start refactoring it to use newer technologies if necessary. In this lecture, we'll learn how to prepare and execute the migration of a large monolithic application into microservices. We first learn how to identify the components that can be the best candidate. Dates for the migration. Those candidates included areas that changed the most frequently require the higher scalability or have the least amount of technical debt. Then we discuss the process for preparing and executing the migration, which involved adding test coverage, defining the API, isolating the components from the rest of the application, and finally using the strangler fig pattern to chip out the different parts of the monolith and migrate them into separately deployed and managed microservices.
