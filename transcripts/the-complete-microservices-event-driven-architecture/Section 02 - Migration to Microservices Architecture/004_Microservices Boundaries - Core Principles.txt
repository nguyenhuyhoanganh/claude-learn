# Microservices Boundaries - Core Principles

Welcome back. In this lecture, we will learn about the first set of principles for setting the boundaries between microservices. To arrive at those principles, we'll go over a few approaches to splitting an existing monolithic application. Those case studies are not just thought experiments, but real situations of companies that went through the migration and encountered issues in the process. So each case study will help us define a new core principle for a successful migration to microservices architecture. Let's start with introducing the system. We're going to migrate, as we mentioned in the introduction, just breaking a large code base into an arbitrary set of microservices and handing each one to a separate team will not necessarily benefit us. Let's imagine we are a successful e-commerce company running an online store for thousands of customers. The architecture of our system is a typical three tier architecture with a monolithic web application in the middle tier. This web application takes web requests from users running our front end code in their web browsers and mobile apps in the presentation tier. In the data tier, we have an internal database storing all of our transactions, products, reviews and inventory information. And we also have an external payment service that belongs to a third party which handles all the user credit card billing. Our monolithic architecture served us very well for many years, but now our code base has grown too large and complex. The binary size of our application requires very expensive hardware and the development team is well past the two pizza rule. So there are a lot of inefficiencies. We heard that splitting this application into different services would solve all of our problems, but we aren't sure how to split this application into microservices. So one approach to split this monolithic application would be to follow the internal logical layers that already exist in the application code base. We can take the store front layer of our code base that handles user requests, security and permission validation and also serves the HTML JavaScript and CSS to the web browser and split that into its own service, which is maintained by a separate team and is physically deployed as a separate runtime unit. Similarly, we can take all the business functionality that handles checkout discounts, seasonal sales products and reviews and run that as another microservice. And finally, we can take the data layer of that application and run that as a separate service as well. On the surface, this approach looks like a good idea. It takes advantage of the already existing logical separation within our application, so very little refactoring is required. However, this approach doesn't work very well and will not give us too much benefit. The reason for that is simple Every new feature we develop within this application will likely require an API change, a business change and a data change. That means that every microservice in our system will be involved in every feature requiring careful planning and release coordination between the different teams. So we will not gain any benefit for our organizational scalability using this approach. In fact, all we did here is just to make this three tier architecture into a five tier architecture which is still not considered microservices. So breaking the monolithic application into more tiers is not going to work. This leads us to the first principle for setting the boundaries between microservices. This principle is that each microservice needs to be cohesive. Cohesion means that elements that are tightly related to each other and change together should stay together. If all the logic that changes together stays within the boundaries of the same microservice, each team can truly operate independently. In our case, the migration. Failed because each one of our microservices was not cohesive enough to improve our organizational scalability. Now let's look at another approach many companies took when migrating to microservices, which is based on technology boundaries. After careful consideration, our developers came up with some interesting observations. They noticed that if they split our store front end layer and rewrite a particular part of it to NodeJS with JavaScript, they could reduce the number of lines of code by 50%. And if they write the other part in Java, they can improve the performance by 20%. Also, part of our product recommendation engine that requires heavy computations was already separated into a library written in C plus plus. At the same time, they also noticed that if we rewrite another part of our product recommendation to Python, we can use some powerful machine learning libraries that can improve our performance even further. Finally, if we split part of our data layer that talks to the external payment service to Golang, we can also improve both the performance and the number of lines of code in that module. After splitting those modules into separate microservices, we saw great performance improvements. However, those boundaries are purely technological. So outside stakeholders who are unfamiliar with the code base, like product managers requesting new features or support engineers, reporting bugs from users don't know what subteam needs to get the incoming tasks. For example, when a product manager needs to update the product recommendations engine, it is unclear whether they need to talk to this team, that team or maybe both. The same goes for API changes. The responsibilities of those two storefront services are not clearly defined, so it's unclear. For example, for a front end developer which codebase needs to change for a given feature? And also the terminology of each API is very convoluted because we're mixing different contexts such as users, products, bank accounts and so on, all in the same API. Also, the last microservice, which still has most of the business logic, has too much responsibility and is still too big. So it has the potential to become yet another monolithic application. This leads us to the second core principle, which is the single responsibility principle. Based on this principle, every microservice should do one thing and do it exceptionally well. If each microservice in our system follows this principle, there is no ambiguity about where a new piece of functionality needs to go and which team needs to own it. It also helps create a very clear and easy to follow API for each microservice because all the terminology entities or identifiers are bound to a particular context. For example, if we need to talk to the user service, it's clear that the ID here is the user ID, the name is the user's name and so on. But if we talk to the product service, the same terms will mean completely other things and there won't be any ambiguity around them. Okay, let's have one more attempt based on a real use case of a company migrating to microservices. This time we notice the micro part in the microservices architecture, which gave us the impression that splitting the monolithic application into tiny services would give us the best benefits. The smaller the better. So we can follow a slightly more coarse grained approach and use every package or module and make that into a micro service. Or given that our original monolithic application was written in an object oriented language, we could split every class into an independently deployed and managed microservice. So now in production we started seeing even bigger problems since every request from a user. Triggers a chain of communication between many microservices and each microservice needs to talk to even more microservices to do its work. The performance became really bad and troubleshooting issues became almost impossible. And that's because our microservices are tightly coupled with each other, and each operation requires a lot of communication between different services. So the third principle will learn from this attempt is that microservices should be loosely coupled. Loosely coupled microservices have very little to no interdependencies. This means that each microservice can perform its functionality with minimal communication with other microservices. It's also important to call out that the size of each microservice is not that important, and it's a common misconception that microservices have to be as small as possible. In Thatrillionegard, the term microservices is somewhat misleading. As long as our microservices are cohesive, follow the single responsibility principle and are loosely coupled with each other, the size doesn't matter. And we also shouldn't expect that all of our microservices are the same size. Naturally, some will be smaller and other larger, and that's completely okay. So those three principles are the prerequisites for a successful microservices architecture. In other words, if we don't follow them, we will likely encounter issues sooner or later. However, they still don't exactly tell us how to actually split the monolithic application into microservices that follow those principles.
