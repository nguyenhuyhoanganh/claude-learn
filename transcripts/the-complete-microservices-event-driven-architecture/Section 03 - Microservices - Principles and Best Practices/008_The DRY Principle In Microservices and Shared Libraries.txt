# The DRY Principle In Microservices and Shared Libraries

Welcome back. In this lecture, we'll revisit the dry or don't repeat yourself principle in the context of microservices. We'll first have a quick reminder of what the principle means and why it's so important. Then we'll discuss the challenges of following this principle in microservices architecture. After that, we'll discuss a few alternatives to shared libraries. And finally, we'll conclude with a short discussion about sharing and duplicating data across different microservices. So let's start with a quick reminder of the dry principle and its importance. Generally speaking, dry is one of the most fundamental principles of software engineering. If you find yourself repeating the same logic or the same data, you should always consolidate it in a single place as a shared method, class or variable. On a broader scope, if you have a more complex logic used in multiple projects or applications, it's a common practice to package that logic into a shared library that other applications can import and use. This way, if we need to change that logic, we can do it in one place and we don't need to look for its duplicates across multiple codebases. It also reduces duplicate effort so the work of a single engineer can be reused by others. However, the dry principle does not always hold true in microservices architecture, particularly in regards to shared libraries. So let's talk about a few reasons why shared libraries can be a big problem in microservices. The first problem is tight coupling. As a reminder, one of the core principles of microservice Mrs. was loose coupling. However, when multiple microservices share and depend on the same library, changes to this library become a source of tight coupling and friction between the different microservices and the owners of that library. If there are API changes to that library, those changes need to be communicated to the other teams that own those microservices. Then those teams will have to make the appropriate changes in their codebase so they can continue using that library. But even if the API doesn't change, the microservices will still need to be rebuilt, retested and redeployed with every change in the shared library. Additionally, a bug or a vulnerability in that shared library can potentially impact all the services that use it, which defeats the purpose of isolating those microservices as separate runtime units in the first place. Another issue when using a shared library is commonly referred to as dependency hell. Let's consider a situation where a particular microservice uses a shared library. Library A and that library internally uses another library. Library B Additionally, the microservice is using library B directly in another part of its codebase for a different purpose. Let's also assume that the microservice and Library A are using the same version of Library B at a given point. Now let's imagine that a new important update has been made to Library A that our microservice urgently needs. However, the new version of Library A now depends on another a newer version of Library B than the one we're using directly. So depending on the programming language and runtime we're using, in some cases, we cannot depend on two versions of the same library simultaneously. If that's the case, we are forced to make an unnecessary code change where we're using library B directly just to be able to update to a new version of Library A. In other cases where we can depend on and load two different versions of the same library, we are essentially breaking the dry principle. And that's because our microservice application now has to load two versions of the same library where most of the code is identical. Another downside to this is that it also increases the size of our binary and the time it takes to build and test it. So now that we understand the problems of shared libraries and the challenges that come with the dry principle and microservices, let's talk about some alternatives we can use. The first scenario we will discuss is when complex business logic is shared across multiple microservices. This situation may indicate that we incorrectly set the boundaries between those two microservices. Remember that one of the core principles of microservices is the single responsibility principle. So if we followed the decomposition by business capability or sub domain, we would very rarely get into this situation. And if we do, we may want to consider re-evaluating those boundaries. So one option would be to adjust the boundaries of those two microservices. So only one of them contains that business logic and the other doesn't. Another option is if that business logic is complex enough, we can separate it into its own microservice. Another situation is when we need to share common data models for communication between two microservices. In this case, having shared libraries or files is actually a good practice. The reason for that is when it comes to communication between two microservices, we actually want the two teams and codebases to be codependent. If the API or data model of one microservice changes to the point that they cannot communicate. We actually do want to break the tests as soon as possible. Otherwise if we don't share but duplicate that code, the tests of each microservice may pass successfully and not detect the change in the other microservice. So those two services may not be able to communicate at runtime and we'll find out about it only in production. Another approach to this problem is using code generation tools based on data models or schemas. In this case, we have a shared schema representing the common data model for the communication between the two microservices. Depending on the type of API language or technology, we can use the appropriate tool to generate the corresponding data models and boilerplate code for the communication between the microservices. So as long as the code is generated based on a shared interface definition and that generation is performed as part of the build and test process, it's a very common and safe approach when it comes to other code, such as utility methods that frequently change. It's better to duplicate it across different microservices instead of trying to reuse it in a shared library. This way, each microservice can have its own more specific implementation of that logic optimized for its own use case. It also makes it easier to migrate the entire code base of a microservice to another programming language if we want to. Now, if we absolutely don't want to duplicate a particular code, we have two additional options. The first option is using the sidecar pattern. Using this pattern, we package and deploy the shared functionality as a separate process. But instead of running it as a separate service, we run this process on the same hosts as the microservice instances. This way we can share the same functionality across multiple instances of the same microservice and across multiple instances of different microservices. The microservice instance can communicate with its co-located sidecar process using the familiar and standard network protocols. Since the sidecar process and the microservice process are running on the same host. The performance overhead of this communication is relatively smaller than if we ran. Processors on different hosts. On the other hand, this overhead is still higher than if we ran that logic as a shared library inside the microservice process. So as the second option, in the rare cases when we have very generic and stable code that rarely changes such as logging, retrying and pattern matching, it's okay to use shared libraries as long as they don't lead us to this dependency hell. So those libraries should be as self-contained and independent of one another as much as possible. But this should be used as a last resort. As a final note, inside the code base of each individual microservice, we should still follow the dry principle as usual. So code duplication within a microservice is unacceptable and we should always extract common logic into a single method class or module. Now let's switch gears and talk about data duplication on the surface, Duplicating the same data across microservices and storing them into databases may seem like a waste of storage space. However, as we already saw in the previous lecture, it is sometimes necessary to improve performance if we don't want to pay the price of sending a request to another service for data, then we can store a cache of that data inside of our microservice. In microservices architecture, this approach is completely acceptable as long as we remember two things. The first one is only one of the microservices needs to be the clear owner and source of truth for that data. That means that only one microservice can get, write, update or delete operations to that data and the other microservices containing a copy of that data get only read operations. The second thing to understand, which we also mentioned in the previous lecture, is when we duplicate data across microservices, we can guarantee only eventual consistency. So for situations where strict consistency is important, data duplication should be avoided. For example, in the case of an online store, we may want to store the rating or even a few recent reviews of a product in their product service. This way, when a user wants to look at a product, we can quickly load that data from the product service and we don't have to wait to load all those reviews and ratings by communicating with the review service. However, if we do it, we may be in a situation where our rating for a given product is not the most accurate and the reviews will show in the recent review section may not actually be the most recent anymore for a user who is considering buying this product. This inaccuracy is not that important and eventual consistency is completely acceptable. On the other hand, when it comes to the most up to date inventory information for a given product or the account balance of a given user, we absolutely need to guarantee strict consistency. So in this case, we would not duplicate that data in any other service other than its owner. In this lecture we revisited the dry principle for sharing libraries and data in microservices architecture. We first reviewed the benefits of following this principle in general and learned that it doesn't always apply in microservices architecture. We then mentioned some of the challenges of sharing a library across microservices, which included tight coupling the need to rebuild, retest and redeploy all the microservices with every change in the library and dependency hell. After that, we learn a few alternatives to shared libraries, such as separating complex business logic into a new microservice using the sidecar pattern code generation and duplicating some code across microservices. And finally, we talked about duplicating data across microservices and learned that it is sometimes inevitable and completely acceptable for performance reasons. However, its import. To remember that our data consistency will be impacted. And it's also important to remember that only one microservice needs to remain the clear owner for each piece of data.
