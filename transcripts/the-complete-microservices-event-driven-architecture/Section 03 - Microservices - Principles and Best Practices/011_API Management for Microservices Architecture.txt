# API Management for Microservices Architecture

Welcome back. In this lecture we will talk about API management in microservices architecture. As always, we'll start with the problem we're trying to solve when it comes to managing APIs. And then we're going to introduce the solution, which is the API gateway pattern. Finally, we'll address a common source of confusion between a load balancer and an API gateway. So let's start with the problem we're trying to solve when it comes to API management. Let's imagine we're a health care and fitness company providing digital services to clients and businesses like fitness centers, hospitals and doctors offices. On the client side, we have different devices like smartwatches, mobile phones and other wearable accessories interacting with our platform. And in addition, we have other companies, servers, developers, computers and health care professionals using our system using their web browsers internally. We have dozens or even hundreds of microservices exposing different APIs specific to their functionality. So the first problem is the different API endpoints exposed by different microservices are hard coded across many frontends and client SDKs. This tightly couples the client side code to the internal implementation of our system. For example, let's imagine that at some point we decided to break an existing microservice into two microservices and now each microservice exposes a new API with IDs and endpoints relevant only to this microservice. Because of this tight coupling, we will also have to refactor for all the client side code in each type of device to use the new APIs. Another problem is different microservices may expose their API to different customers. For example, we may expose public API endpoints externally to clients, private API endpoints that we expose only to internal services and partner APIs that we expose only to our partner businesses and developers. While internally we may do our best to follow specific API guidelines, it's not always possible to do it with external APIs. For instance, in the same service we may have to expose one type of API for a partner company that only supports some old API technology, but then we have to expose the same API functionality to two other companies that support newer but different API technologies. Additionally, we may have different versions of the same API for different tiers of clients with different capabilities based on their subscription level. For example, we may have one API which is more limited and is designed for free users while we offer another API with more capabilities to our premium customers. As we can see, this jungle of APIs in microservices architecture can quickly become very hard to manage and reason about. Another problem is controlling and monitoring how each client or device interacts with our system. For example, when loading a dashboard of their daily workout, a particular user's mobile app may have to make several API calls to different microservices, but because the front end makes those calls to our services directly, monitoring and tracking them across the entire system is very hard. Finally, a lot of boilerplate code such as authorization and rate limiting has to be implemented in each microservice. This leads to a lot of effort, duplication and wastes a lot of time for each team that could instead focus on something else. So to decouple our client side from the internal architecture and make API management easier, we can use the API gateway pattern. The API gateway pattern is particularly useful for microservices architecture and is used by many companies. This pattern places a component called API gateway at the entry to our system. And is responsible for all the API management. In the simplest scenario, the API gateway simply routes requests for a given API endpoint to the relevant microservice that handles it. But in most cases it can do much more than just request routing. For example, let's assume we have three clients that need to interact with the same microservice to get some data. But the API technology and data format for those clients is different the API gateway and then transform the requests to one canonical API type and data format that the microservice supports. And similarly, it can make that translation in reverse for the response back to the client. This makes the API definition and management much easier and cleaner for each microservice service because they don't have to care about the different clients and their specific limitations and they can all follow one standard and utilize the same infrastructure in addition to routing the request to the corresponding microservice. The API gateway can also perform traffic management and throttling. For example, if we have a business partner that is trying to query data from our system at a very high rate, we can easily throttle it at the API gateway level, preventing it from putting too much load on the microservice or microservices that need to handle those requests. We can also delegate the authorization and TLS termination to the API gateway. So microservices don't have to worry about it and can focus on their business logic implementation. Also, since all the traffic from our front ends is going through the API gateway, we can easily monitor it. This enables us to detect issues in particular APIs or analyze the behavior of our clients and partner companies. Finally, certain implementations of an API gateway can even allow fanning out a given request to multiple services and aggregating the results before sending it back to the client. This is a critical feature for multiple reasons. The first reason is that it even further decouples the clients from our internal implementation. For example, internally we can refactor those services by either combining several microservices into one or splitting an existing microservice into multiple microservices. And all of those implementation details will remain completely transparent and to the user. This feature is also very important for mobile devices and Iot because each network request on their end directly affects their battery life. Now, before we conclude, I want to discuss a common source of confusion between the role of a load balancer and an API gateway for microservices architecture. Both a load balancer and an API gateway route Incoming network requests from a client to a single destination. However, this is where the similarities between the two components end. The main difference between the two components is their purpose. The role of a load balancer is, not surprisingly, to balance the traffic load on a group of servers. Those servers usually run identical copies of the same application in the context of microservices, we typically have a load balancer in front of each microservice where each microservice is deployed as a group of identical instances. On the other hand, the purpose of the API gateway is to be our system's public facing interface. Its role is to route requests, not to servers, but to services based on different criteria. Now, besides the difference in their purpose, there is also subtle difference in their functionality. The load balancer aims to be as simple as possible to reduce the performance overhead. It also usually performs health checks on its backend servers to ensure it doesn't route requests to unresponsive servers. And it also usually offers different load balancing algorithms for different workloads. On the other hand, the API gateway is expected to provide many other features such as throttling, monitoring, API versioning and management protocol and data format translation and so on. So because those components have different purposes, they're usually used together in microservices architecture. In this lecture, we'll learn about the problems of managing APIs in microservices architecture. To solve those problems, we'll learn a pattern called an API gateway. Using this pattern, we placed an API gateway service at the entry to our system, which helps us route requests to different microservices. Additionally, it provides us with many features like throttling clients that send us too many requests handling authorization and TLS termination protocol and data translation monitoring and much, much more. Finally, we addressed a common confusion between a load balancer and the API gateway, which, as we learned, serve completely different purposes and are both used in microservices architecture.
