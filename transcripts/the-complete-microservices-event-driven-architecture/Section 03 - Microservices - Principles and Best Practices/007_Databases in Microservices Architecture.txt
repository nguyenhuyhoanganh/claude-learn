# Databases in Microservices Architecture

In this section, we will cover the key principles and best practices for microservices architecture. By following those principles will benefit the most from using microservices and minimize the chance of running into scalability or other issues. Those principles and best practices come from years of experience shared by software architects and technical leads running large scale systems using microservices. So in this lecture we will talk about the first principle and best practice the database per microservice. We will first learn the motivation behind this principle. Then we'll learn the benefits of sticking to this principle. And finally, we'll discuss the repercussions of following this best practice. So first, let's see what this principle of database per microservice means and why it's so important. Let's consider the architecture of an insurance company that follows the traditional three tier architecture. In this architecture, we have one monolithic application in the middle tier and one database in the data tier to increase the organizational scalability. We migrated this monolithic application into a set of microservices with a separate team owning each microservice. Assuming we set the boundaries of those microservices correctly, we should expect the teams to operate independently with very little friction on the surface. Allowing the different microservices to share data makes perfect sense. For example, the reporting service needs to access the policy purchasing data, claims data and the customers data so we can make the requests much more efficient by allowing the reporting service to access that data from the database directly instead of paying the performance overhead of getting the same data through the appropriate services. However, with this seemingly logical performance optimization, we have a problem. For instance, the policy purchasing service gets a lot of traffic from potential policy buyers, so the team owning that service decided to take advantage of its autonomy and replace its slow legacy relational database with a read, optimized NoSQL database. However, because the reporting service code is tightly coupled to the database technology, where the policy purchasing data is stored, the changes must happen in both code bases and be released simultaneously. Similarly, the team owning the claims service decided to change the schema of their claims table by renaming some columns, removing some old and unused columns and adding new columns. Now, because the reporting service depends on that old schema, those two teams will have to meet and discuss the new schema that works for both of them and also again, make those changes to both code bases and release them simultaneously. Finally, the team owning the customer service decided to add a lot of fine grained security around what user data can be read or modified based on different organizational roles. However, those security policies are outside the scope of the reporting service team's domain, so the customer service team will have to communicate those changes, including all the implementational details to the reporting service team. And once again, both teams must change their code base, write their own tests, and simultaneously release and monitor those changes. So as we can see, despite all the migration effort, our teams are still tightly coupled and require a great deal of communication and coordination, which is negatively impacting the organization's productivity and development velocity. So this is where the database per microservice principle comes in. With this principle, each microservice team fully owns its data and does not expose it directly to any other service, even at the cost of additional latency. When. Ever a given service requires data from another service. The request has to go through the target service API. Additionally, each service needs to abstract the actual database technology or structure at the API level. This way, if the team owning that microservice decides to change the database, technology or schema, that change will be completely transparent to the consumers of their API. Additionally, if the changes also require some changes in its API, we can always offer two versions of the API for a few weeks or even months. This allows the other teams enough time to make their changes and update to the new API without the need for any coordination. So now that we understand both the motivation for the database per microservice principle and the benefits it provides us with if we follow it, let's talk about a few downsides and repercussions. The first obvious downside that we already mentioned is added latency. Sending a network request to another service. Parsing the request. Querying the database. Sending it back and parsing the response definitely adds a performance overhead compared to a simple query against the database directly. When this performance overhead becomes an issue, it's okay to cache or even store some of the data that belongs to another microservice in the database, that belongs to the microservice that needs that data. However, it's important to ensure that the source of truth remains the only microservice. If we do cache or store data that belongs to a different microservice, we of course lose strict consistency and have to settle for eventual consistency. And that's because the data we store locally may be stale until we get the new data from the source of truth. The next downside is the loss of the ability to join data from multiple tables when all the data was represented as tables within the same database. We could easily perform the joint operations based on a common key or column. When we split that data across different databases, some of which may not even be relational databases. We can no longer perform those join operations. The only way to achieve the same functionality would be to pull data from two different databases, then transform the data so both objects are represented in the same format and then perform that join programmatically. The next downside is the loss of transaction guarantees. Going back to a single database, we could modify rows in multiple tables as part of a single transaction and guarantee that the change to all those tables would be atomic. Performing a distributed transaction across multiple services is very hard. So even though it's theoretically possible, it's almost never used. But no worries. Later in the course, we will offer a few patterns that can solve all those challenges very efficiently. In this lecture, we'll learn about a very important principle and best practice for microservices architecture, the database per microservice. We first got the motivation for this principle by exploring a real life scenario in that scenario. We saw that sharing a database between microservices tightly couples, their code bases and their teams causing a lot of coordination overhead. This overhead is eliminated when each microservice owns its own data and exposes it only through an abstraction API. Finally, we mentioned the challenges and drawbacks of using the database per microservice principle. Those challenges includes worse performance, more complex join operations, and the loss of transaction guarantees. I'll see you guys in the next lecture.
