# Structured Autonomy for Development Teams

Welcome back. In this lecture, we'll discuss another set of best practices for successful microservices architecture. We will first talk about some potential problems with giving too much freedom to each microservice team in deciding its tech stack tools and processes. Later, we'll talk about three tiers of setting the boundaries of autonomy for each team. And finally, we'll talk about a few factors that influence the boundaries of those tiers of autonomy. So first, let's address a common myth and misconception about the freedom that each microservice team should have. The myth is that the biggest benefit in microservices is that each team has full autonomy in choosing their technology, stack tools, databases, APIs and frameworks. This is one of the biggest pitfalls of companies that migrate to microservices. And let's see why. The first reason is the upfront cost of infrastructure. When we had our monolithic application, we put a lot of effort upfront in deciding how to structure our code, what frameworks we should use and how to organize our classes in modules, packages or libraries. We also set the guidelines on where to store our tests and how to run them. We spend a lot of time setting up and customizing our build tools, scripts and continuous integration pipeline and to make troubleshooting production issues easier. We also invested in monitoring and alerting tools, which, as I'm sure you already know, is not a trivial effort. Finally, a lot of work went into designing our database schema and configuring it so it can be scalable, fault, tolerant and highly available. Now imagine that after migrating to microservices, each team has to pay that same cost and duplicate the same effort for their code base and ecosystem. To make things worse, maintaining all the infrastructure is an ongoing effort. So if we have a dedicated DevOps, QA or QA team, those engineers will have to migrate the infrastructure of hundreds of microservices where each one is entirely different. Those teams will quickly get overwhelmed and have to hire more people to keep up with this jungle of technologies. But besides the setup and the maintenance cost, there is also a cost associated with learning and becoming productive using all those tools. So any new developer at a given team needs to read through the documentation, learn and memorize all their best practices and get used to following them in their daily work. But what if they need to work on another team's code base to make a small change? Or if they simply need to look at another team's code base and find out how they set up a particular integration test or what their configuration looks like. It would be a huge learning curve that just doesn't scale in a large organization. On top of that, there is another problem which is non uniform API. Let's not forget that all those microservices at the end of the day need to come together as one system and communicate with each other to accomplish a common goal in an easy and performant way. So imagine what would happen if every team that owns a microservice could define their own API in their own style, following their own best practices. Now a frontend engineer who has to pull data from different microservices would have to learn the API of each team and integrate with it, calling endpoints with different naming conventions and using many different API styles and technologies will make the front end code a huge mess that is hard to read and maintain. Similarly, if a particular microservice needs to communicate with several different microservices in different scenarios, we run into the same problem. The engineer who needs to add or update that API call will have to spend hours reading the other team's documentation to ensure that it adheres to. Their best practices. So as we can see, by allowing each team full autonomy, we are actually making things a lot worse and introducing a lot of overhead. At this point, you may ask, but Michael, isn't the whole point of migrating to microservices is to allow each team to be fully independent? Well, yes, but the key to a successful microservices architecture is the balance between team autonomy and structure. The term I like to use for this is structured autonomy. That means that each team is still autonomous, but only within certain boundaries which can be grouped into three tiers. The first tier is the most restrictive and describes the areas that should not be under each team's jurisdiction. Instead, those areas should be uniform across the entire company. That includes infrastructure such as monitoring and alerting, as well as continuous integration and continuous delivery technologies and scripts. This way we can invest a lot of effort up front into adopting or building this infrastructure from scratch, and that effort is going to be amortized across the entire organization, making this investment cost effective. Another area that should be uniform across the company is guidelines and best practices for defining public and internal APIs. This should be obvious because external clients don't care that they're sending requests that end up in different microservices inside our system and also internally. It makes integration and communication between different microservices belonging to different teams very easy. The final area that should be standard and uniform across all the microservices is security and data compliance policies. The reason for this is also pretty obvious. If one of our microservices gets hacked, our entire system becomes vulnerable. And if one of our microservices violates some local privacy or data retention policy, our entire organization will be held accountable. And nobody cares externally that it was the fault of only one small microservice. Now the second tier allows each team some freedom, but within some boundaries. This includes choosing a programming language and database technologies. After all, each runtime and data storage solution is optimal for a different use case. However, the set of approved programming languages and database technologies should be restricted. This is to prevent a complete jungle of technologies which can happen if every team can choose the latest and most fashionable programming language for their microservice. It also takes a lot of time for each company to develop expertise in managing, configuring and running each type of runtime and database in production. Most big tech companies use no more than a handful of programming languages across hundreds of microservices. The last year gives each team complete autonomy. This includes the release process, schedule and frequency. This way, each team can work entirely independently from other teams, release new features based on their workload priorities and requirements, and develop their own custom scripts and tools specifically for their needs to make their local development and testing more efficient. Also, each team should fully own its own documentation and onboarding process for new developers. It's important to call out that those are general guidelines and best practices and the actual boundaries of those three tiers. Different slightly in each company. One factor is the size and influence of DevOps or SRB teams. Typically, companies with more dominant DevOps or SRE teams lean more towards common standards, which makes their life managing the system easier. Another factor is the seniority. Of developers hired in the company. Generally, the more senior the developers are, the more freedom they prefer in setting up or building their own infrastructure. Another factor is the company's culture. For example, some companies just stick to one programming language like C sharp Java or Python, for example, and don't allow any freedom in choosing other programming languages. The benefit of this restriction is that they can hire developers once and move them between different teams with very little overhead. In this lecture, we'll learn about the disadvantages of giving too much freedom to each microservice team. Those disadvantages included upfront and ongoing maintenance costs, a steep learning curve and overhead and a non-uniform API. To solve those issues. We introduce the concept of structured autonomy, which allows teams to still be independent but within a set of boundaries. And finally, we talked about a few factors that can influence those boundaries, such as the influence of the DevOps or SRE team, the seniority of the hired developers and the company's culture.
