# Use Cases and Patterns of Event-Driven Architecture

Welcome back to another lecture. In the previous lecture, we saw some of the great benefits of using event driven architecture. However, as we would expect, it's not a silver bullet that we can use to solve every design problem. So in this lecture, we'll discuss the use cases where event driven architecture is a good choice for us and in which use cases we should prefer the traditional request response model. And finally, we'll conclude the lecture by discussing two event delivery patterns for event driven architecture. So in what use cases should we use event driven architecture? The first use case is for actions that can be classified as fire and forget and are asynchronous by nature. In those cases, the sender does not expect any data in return immediately or none at all. A good example is when a user sends us a request to generate a report that may take a few minutes or even a few hours to be complete. So in this case, our event is the generate report action and the result will probably be sent to the user's email when this action is complete. Another example is a user leaving a review on a product they purchased. In this case, the user doesn't expect any data in response now or in the future, and all they care about is whether the review was accepted into our system or not. Another use case is reliable delivery. This is especially important in financial transactions where we cannot afford to lose messages in transit or need to guarantee certain delivery. Semantics. We'll talk about delivery semantics more in depth in a later video. A great example of this use case is an online store where a user orders a product. In this case, if we tell the user that they will receive their order, we absolutely must keep our promise or send them an email explaining why that order was not fulfilled. But we cannot have a situation where a user submits an order, but we simply lose it in transit because some server crashed in the process. Another example would be a money transfer between accounts or banks. This is also a situation where this action requires certain reliability and we cannot afford to lose it. Another use case for event driven architecture is for infinite streams of data or events. An example of this use case can be location data from mobile devices or sensor data from Iot devices such as vacuum cleaners or autonomous cars. In this case, we have a continuous and infinite stream of data coming into our system, and we need to either analyze, aggregate, transform or store that data in real time. The next use case is for anomaly detection or pattern recognition. In this case, each event can represent a fact such as the current value of requests per second coming from each one of our servers. Each data point in isolation is not that interesting or useful. However, when we put those events in a sequence, we can gain interesting insights. For example, if the request per second for a given server or service increases, we know we should scale our system out and add more servers. Otherwise our system will not be able to handle the incoming traffic for much longer. Similarly, if the requests per second suddenly drops to zero, that indicates some hardware or software failure that we need to immediately investigate. The next use case is when we have a service with some kind of state change that it wants to broadcast to any other service that is interested in that event. This scenario is a perfect choice for event driven architecture because the producer of the event doesn't need to know anything about who is consuming it and how they use it. An example of this use case can be a user clicking on a digital ad on our website. Once our ad service receives that request from the front end, it can emit this event to a message. Broker. The message broker will then broadcast this event to all the services that are interested in it, and each service has its own role in the process, which is completely unknown to the producer and the other consumers of that event. The final use case is for buffering messages to tolerate traffic spikes in our system. In this scenario, we can tolerate a sudden spike of events coming from one service by placing a message broker and communicating a synchronously between the services. Let's consider a situation where we have a social media company and all of a sudden a global event triggered a massive number of posts or comments on a particular post. In this case, we can buffer those events in a message broker and deliver those events to the other services. Only as fast as those services can handle them. So now that we talked about the use cases that are a good fit for event driven architecture, let's talk about a few scenarios where we would prefer the synchronous request response model. The first use case is when we need to respond to the user or send their service immediately with some data. For example, if a user requests a web page containing all the products in a given category, we need to respond to this request synchronously and immediately. Otherwise the user will just leave our online store. Another sort of catch all scenario where we shouldn't use event driven architecture is when the interaction is very simple and we won't get any benefit from using the event driven model. The thing is configuring and running a distribute or cloud managed message broker does come with some complexity and cost. So if in this case we simply want benefit from this model, it's just not worth it. Now, in reality, a typical microservices architecture combines event driven architecture and the simple request response model. It's also better to start with just a simple request response model initially and then upgrade the critical parts to the event driven architecture where needed. Now let's briefly talk about two patterns for delivering events in an event driven architecture. The first pattern is called event streaming. In this pattern, the message broker is used as a temporary or permanent storage for events. The consumers have full access to the log of those events, even if they have already been consumed by the same consumer or by other consumers. New consumers that join later also have access to those old events and can replay them from any point they want. The event streaming pattern is a great choice for use cases of reliable delivery, and that's because the message broker retains those events either indefinitely or for a period of time, which allows us to recover those events and audit them if we need to. It's also a perfect choice for the use case of pattern or anomaly detection because the consumer needs access to all the past events in a given window. The second pattern is the publisher subscriber or pub sub. In short, in this pattern, the consumers subscribe to a particular topic queue or channel and receive only new events after subscribing. In this case, subscribers don't have access to old events, and as soon as all the current subscribers receive the event, the message broker will typically delete it from its queue. Also, if a new subscriber registers for events for the given topic, they will be notified only of new events. This pattern is ideal when the message broker is used only as a temporary storage or a broadcasting mechanism. After the subscribers consume the events, they're typically transformed and stored permanently in a database or are passed to another service. All the scenarios that fall under the fire and forget broadcasting buffer. Worrying or handling an infinite stream of events. Use cases are a good fit for this pattern. In this lecture, we'll learn some of the most common use cases for event driven architecture. Those use cases included fire and forget reliable delivery, an infinite stream of events. Anomaly detection and pattern recognition. Broadcasting and Buffering. We also mentioned that not all scenarios are a good fit for event driven architecture. So in situations where we need an immediate response containing data, or if the use case doesn't justify the complexity and cost of maintaining a message broker, we should use the traditional request response model. Finally, we talked about two event delivery patterns. Those patterns are the event streaming and the publisher subscribe pattern.
