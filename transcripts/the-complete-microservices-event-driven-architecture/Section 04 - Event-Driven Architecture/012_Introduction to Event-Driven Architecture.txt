# Introduction to Event-Driven Architecture

In this section, we will learn another architectural style called event driven architecture and how to combine this style with microservices architecture. In this lecture, we will first get the motivation for event driven architecture and why we need it. Then we'll talk about the fundamental concepts of event driven architecture. After that, we'll compare the request response model to the event driven model to illustrate its benefits. And finally, we'll demonstrate the use of the event driven model in a real life example. So before discussing the event driven architecture, let's get some motivation with a real life scenario. Let's consider a video on demand subscription service that we designed using microservices architecture. After the user tried our service for free, they're ready to proceed and upgrade to a paid subscription on our platform, which gives them access to all our movies and many other advanced features. On the backend side, we have the following microservices participating in this transaction. Our subscription service holds all the information about each subscriber, including their subscription level, expiration date and so on. We also have a payment service that, given a user's credit card information, communicates with third party companies to build a user. We also have the recommendations and personalization service the service creates and maintains a unique profile for each paying customer to provide them with their watch history and recommendations for future movies based on what they watched in the past and also based on their country. It's important to note that for a user to be fully subscribed to our service, we must ensure that each one of those services receives and successfully processes the subscription request. So let's explore a few ways to implement a user subscription and identify the issues with each approach. The first approach would be to send a request containing all the user's information to the subscription service. This subscription service will update its own database and then send the request to the payment service. The payment service will contact the third party credit card processing service and once it gets a confirmation, it will update its own database and send the request to the recommendation service. Once the recommendation service successfully updates its database, it will send a response to the payment service. The payment service will then send a confirmation response to the subscription service, and the subscription service will respond to the front end showing the user a successful subscription page. The benefit of this approach is when the user receives the response, there is a 100% guarantee that the subscription was successful and all three services received and processed the request. The downside, of course, is that this long chain of communication is holding the user for a very long time. This makes a poor user experience even if everything goes smoothly. But in the worst case, if the third party service that the payment service talks to is experiencing performance issues, or if the connection from the payment service to the recommendation service breaks and requires a retry, the entire transaction will make the user wait even longer. On the other hand, we cannot afford to cut corners and decrease the waiting time by sending a confirmation to the user from the subscription service before it actually gets the response from the payment service. And that's because if we do and the payment service fails or never receives the request, the user will not be correctly billed. Similarly, if we try to optimize and make the payment service respond before the recommendation service responds, we risk having an inconsistent state. This state may happen if the recommendation service crashes right before or during the processing of the request, so the user will not get any of the recommendations and personalization features as part of their subscription. So as we can see with a request. Response model. The more microservices we have in the chain, the longer we have to hold the user to guarantee a consistent state. And in many cases, the latency is too long to be acceptable. So let's try a different approach. Instead of making a chain of requests, we can use an orchestration pattern using the orchestration pattern. We add a single service orchestrating a complex business transaction involving multiple operations or services. This orchestration service may technically be combined with the API gateway, but let's assume it's a separate service. In our case now, a request will go to the orchestration service and then be broadcasted to the relevant services in parallel. This parallel execution will reduce the overall latency from the sum of latencies of all the services to only the maximum of the slowest service in the transaction. However, in some cases this latency can still be too high. For example, consider a situation where the third party credit card processing service is having performance issues because another company or companies are sending it too many concurrent requests. While this situation is not in our control, it can negatively impact our system and our customers. Additionally, this orchestration service is now tightly coupled to all the services that must be involved in the transaction. So if the API of one of the services changes or if we need to add another step to the subscription in the form of another service, we will need to update this orchestration service to call that service. Finally, if we have temporary issues with just one of the services in the transaction, all the subscription requests will have to be denied losing US valuable customers. To solve all those problems and many others, we can use the event driven architecture approach. The main concept in event driven architecture is an event. An event represents a fact, an action or a state change that happens in our system and is always immutable, unlike a request which is ephemeral. An event can be stored indefinitely within our system, and unlike a request that is meant to be consumed once and only by one server, an event can be consumed multiple times by different services. Now there are three entities that participate in the exchange of events in an event driven architecture. The producer who produces the events, a message broker which stores and routes those events and the consumer or consumers that receive and process those events. The message broker allows us to route a single event to as many consumers as we want, and it also adds a level of redundancy to our system, because once the event is produced into the message broker, it won't be lost and we can always retrieve it from the message broker. Now let's talk about the two fundamental differences between the request response model and the event driven model. The first difference is whether the communication in the request response model is synchronous. The communication in the event driven model is asynchronous. That means that in the request response model, when the sender sends a request, it must wait for a response, even if it contains no useful information. If the response never arrives. This usually indicates a problem. On the other hand, in the event driven model, the publisher doesn't need to or expect to get any response from the consumer or the consumers of the events. And the delivery of the event to the consumer is outside of its control and responsibility. This allows the producer to move on to processing its next task instead of waiting for a response it doesn't actually need. The second fundamental difference is the inversion of control in the request response model. When the sender sends a request to the receiver, the sender needs to first be aware of the receiver and it. Also has to know exactly how to call it API. If the sender needs to send a request to multiple receivers, it needs to be aware of all of them and also send the request to each one using their own unique parameters. This makes the sender depend on the receivers of the request. On the other hand, in the event driven approach, the publisher doesn't care and may not even be aware of the consumers of the events. This completely decouples the producer from the consumers, as we recall from the previous lectures. The loose coupling of microservices is one of the main qualities we desire in our microservices architecture. This is why event driven architecture and microservices architecture go together with so much success. So now that we understand the core concepts of event driven architecture, let's apply it to our user subscription scenario. One of the ways we can apply the event driven architecture for our use case is to place a message broker between each one of our services. Once the subscription service receives the request from the user, it stores it in its own database and emits an event to the message broker. At this point, it can send a response to the user who doesn't need to wait for the rest of the transaction to complete. Now the payment service will consume that event, process the payment store, the result in its own database, and then emit a new event for the next service. Notice that the user is unaffected, even if this part took a very long time. Next, the recommendation service will consume the event process it, and optionally produce another event to a notification service. The Notification Service will consume its own event and send the user a confirmation email or a push notification to their mobile app. And that's it. In this lecture, we got the motivation for event driven architecture using a real life scenario. In this scenario, we saw that no matter how we organize our microservices with a synchronous request response model, our services are still tightly coupled and the user is still at risk if one of our services is slow to respond. Later, we discussed the fundamental concepts of event driven architecture, which revolve around the event representing a fact, an action or a state change. That event is emitted by the producer and is routed asynchronously by the message broker to one or multiple consumers. And finally, we demonstrated how to use this model in our real life example.
