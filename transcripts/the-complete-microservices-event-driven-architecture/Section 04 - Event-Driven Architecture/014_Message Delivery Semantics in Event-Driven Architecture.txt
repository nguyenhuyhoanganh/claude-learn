# Message Delivery Semantics in Event-Driven Architecture

In this lecture, we'll talk about message delivery, semantics in microservices and event driven architecture. First, we will illustrate the problem using the simple request response model communication. After that, we will expand our discussion to event driven architecture. Then we'll introduce the utmost once and the at least once delivery semantics. And finally, we'll talk about the exactly once delivery semantics, some misconceptions about it and how we can achieve it in an event driven and microservices architecture. So first, let's discuss the problem we're trying to solve. Let's imagine we have a very simple communication between two microservices. In normal circumstances, the sender sends the request to the receiving service. The receiving service accepts the request, processes it, and updates its database. After this processing is complete, it sends back a response to the sending service. However, in some cases we may be in a situation where the sender never receives that response. This situation may happen if the receiver never actually received it in the first place or it didn't receive it but crushed right before it successfully processed it. Alternatively, it may have successfully processed it and stored it in a database, but the response did not arrive at the sender. So since from the sender's perspective, we can never tell the difference between those two cases, what should we do? Should we resend the request running the risk of the receiver, getting the same message twice? Or should we not resend the request and risk losing that data? When we deal with the event driven architecture, we have the same problem of event delivery, but it's a lot more complex than the request response model because now we have three entities, the publisher of the event, the message Broker and the subscriber or consumer. The publisher may produce an event for the message broker, but that request may be lost in the network, or the message broker may receive the event and store it in its logs. But the acknowledgement to the producer is lost in the network. And similarly, the message broker may send the event to the subscriber, but the subscriber never receives it or it does receive it, but it crashes right before it processes it. Alternatively, it could successfully process it, but the acknowledgement is lost in the network. So to deal with this situation, the publisher, the message broker and the subscriber need to agree on the delivery semantics of events ahead of time before discussing those delivery semantics. It's important to point out that the concepts discussed in this lecture are universal and are not specific to any message broker technology. However, the specific configurations and guarantees vary from one technology to another. So the first delivery semantics is the at most. Once in this situation we are okay with potentially losing some data but want to avoid duplication of events when we use the at most once delivery semantics, we tell our publisher that if it doesn't receive an acknowledgement from the message broker, it should not resend the event. So in the best case, if the event did arrive at the message broker, but the acknowledgement simply got lost in the network, we won't lose any data, but if the message broker did not receive the request or crashed before it stored the event, we may lose that event entirely. Additionally, we tell the subscriber to acknowledge that it received the event before it processes it or stores it in a database. This way, if the consumer crashes and restarts after it processes the event, we are still okay because that event was already processed. But if it crashes and restarts after the acknowledgement, but before it gets the chance to process it, the subscriber will not receive that event again, effectively losing it. This delivery semantics is perfect for scenarios where some data loss is okay and we can. Extrapolate what we need from other events. For example, if we were a ride sharing service and each driver sends us their location updates which we store as events in our message broker, we can afford to lose some of those events, but having the same location update stored twice or processed twice will just not be a good use of our resources. This semantics also provides us with the least overhead and the lowest latency. So for real time processing of events such as logs or metrics from hundreds of cloud servers, this delivery semantics will be the most cost effective. Now the second option is the at least once delivery semantics. In this semantics, we agree that if a publisher does not receive an acknowledgement within a given period of time, it will resend the event to the message broker. This guarantees that we never lose the event, but it could result in the same event being stored in the message broker more than once. Similarly, we configure the subscriber to first process the event and only after it's done should it send the acknowledgement back to the message broker. This way if the subscriber receives the event but crashes before it processed and stored it, it will receive the same event again after getting restarted after it received it the second time. It can process it again and then send the acknowledgement. Once the message broker receives that acknowledgement, it will no longer deliver it to the same subscriber. However, if the subscriber crashes right after it processed it, but before it sent the acknowledgement it, or if the acknowledgement is lost in the network, the message broker will redeliver the same event to the subscriber after it restarted. This will result in the subscriber processing the same event twice, but the event will never be lost. This delivery semantics is a great fit for use cases where not delivering certain events may result in losing valuable data, but some data duplication is acceptable. A good example of such a use case is if we want to send a user a push notification to their mobile phone about their product being shipped. If in some rare cases we push the same notification to the user twice, it's not a big deal. But we definitely do want to notify the user at least once. Another example is when the user leaves a review for a product on our website. In this case, since the review is tied to the user and we already have logic in place that allows the user to leave a review for a product only once, that review will simply get overridden or ignored by our system. Besides potential duplication of events, this delivery semantics has one more downside and that's increased latency. When the publisher sends the event to the message broker, it will have to wait for a given period of time before it can resend the same event. This can be in the order of milliseconds or even seconds, depending on the configuration we set for the publisher. Similarly, the message broker will have to wait and allow the subscriber some time to send back the acknowledgement before concluding that the subscriber either crashed or the acknowledgement was lost because of this potential added latency. This delivery semantics is not a good choice for real time or high throughput systems. Now let's talk about the third delivery semantics, which is the exactly once delivery. This is the most desirable delivery semantics, especially for use cases that involve financial transactions. Unfortunately, this is the most difficult to achieve and has the highest overhead and latency. It's also worth pointing out that not all message broker technologies offer the semantics, while the ones that do offer it don't always offer what you'd expect. So whatever technology you decide to use, it's important to read the documentation carefully, especially if you're handling financially impacting data. The General. Idea behind achieving this delivery semantics is as follows. First, on the publisher side, before sending an event to the message broker, we need to obtain what's called an item potency ID or a unique sequence number. Some message brokers can generate this for us internally, while in other cases we may have to use a separate service. The idea is to get a unique identifier we can send to the message broker along with the event. Now, if the publisher sends the event to the message broker and never receives an acknowledgment, it follows the same procedure as the at least once delivery semantics and resends the event with the same ID. Now, if the message broker is the one that provided this unique ID, it will check if an event with the same ID is already in the log. If it is, the event will be ignored and if it isn't, it will be added to the log. This achieves the exactly once delivery semantics on the producer side, which means we never lose events and we never end up with duplicate events. Now, is there anything we can do on the subscriber side? Well, let's consider the most typical scenario where the subscriber gets some financial event, like a billing event or a money transfer between accounts when the subscriber receives it. It needs to process it first, maybe by doing some validations or calculations, and then it needs to put it in the database. In those situations where the event subscription includes a state update outside the message broker, the message broker cannot guarantee more than at least once. Delivery semantics even when it advertises that it supports exactly once delivery semantics. In this case, once the subscriber receives the event from the message broker, it will have to process and store it in the database and then send an acknowledgement to the message broker that the event was successfully processed. Once the message broker receives that acknowledgement, it will not be delivered again to the same subscriber. However, if the subscriber crashes after it processes it and stores the event, but before it sends the acknowledgement, the message broker will continue waiting. And then after the waiting period expires, it will assume that the event was not delivered. So when the same instance of the subscriber is restarted or a new instance is launched, the message broker will send the same event to the subscriber, which will result in processing the same event twice. To avoid that duplication, we need to manually add an item potency ID to the event record in our database. If the same event is processed again, we can programmatically reject it. If another event with the same item potency ID is already present in the database. I want to emphasize that this is something that we need to do ourselves and it's not something that the message broker can do for us to achieve the exactly once delivery semantics. We can leverage the same idempotency id that comes with the event from the message broker. In this lecture, we'll learn about the potential of unreliable event delivery and what problems it can create. If we don't design our system upfront to handle it, then we introduced three event delivery semantics to implement in an event driven architecture. The first and simplest was the at most once delivery semantics where we can potentially lose some data, but data duplication will never happen. The second and more complex was the at least once delivery semantics. In this approach we introduced more latency to the system, but guaranteed that events are never lost. However, processing the same event multiple times is possible. And finally, we talked about the exactly once delivery semantics, which has the most latency and overhead, but guarantees that each event is produced once to the message broker and is consumed and processed only once by the subscriber.
