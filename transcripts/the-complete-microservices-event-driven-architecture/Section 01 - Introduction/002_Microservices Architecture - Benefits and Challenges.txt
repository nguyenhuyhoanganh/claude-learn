# Microservices Architecture - Benefits and Challenges

Welcome back. Now that we got the motivation for why we need microservices architecture, we're ready to talk more in depth about microservices. So in this lecture, we will first get an introduction to what microservices architecture is. Then we'll learn about the benefits we get from microservices and how they solve the scalability issues of a monolithic architecture. And finally, we'll discuss the challenges of using microservices, which we will learn to address throughout the course. So first, what is microservices architecture? Unlike the three tier architecture where the entire business logic is concentrated in one application, microservices architecture organizes our business logic as a collection of loosely coupled and independently deployed services. More importantly, each service is owned by a small team and has a narrow scope of responsibility. The benefits of this much more fine grained, logical and physical separation are as follows. First, with this architecture, we gain a much higher organizational scalability, since each service contains only a subset of the overall functionality, each service's code base is much smaller. This allows each developer to load their entire code base in the IDE much faster. Building each microservice also becomes much quicker because the size of each binary is greatly reduced. Testing and reasoning about each service in isolation also becomes much easier because there is much less logic to understand, run and test. This simplicity of the code base in each microservice also speeds up the onboarding process of new team members so our company can grow quickly and stay efficient. All of this increases the development velocity of each team, which means we can deliver more features and gain an advantage over our competitors on the system Scalability. We also get a lot of benefits in the monolithic application. Each instance had to run the entire code base, which required very powerful and expensive hardware to run each instance. When we break our code base into microservices, each service is much smaller, so each instance of a microservice consumes less memory and CPU. As a result, it can run on cheap and widely available commodity hardware. Each team can also make the best judgment on what technologies would benefit their service, and they can also respond to technology changes much faster by refactoring their already small code base or even rewriting their entire service altogether if necessary. Finally, we also get much higher stability for the entire system since each microservice is deployed as a separate runtime unit. The blast radius of a bug memory leak or a performance issue is a lot smaller. In most cases, a bug in one microservice will impact only that microservice or just the ones that depend on it on some level. But the rest of the system can continue operating just fine. So essentially micro services break that scalability barrier we hit with a monolithic architecture. I like thinking of those architectural styles as two gears in a mountain bicycle or a car with a manual transmission. The lower gear is the monolithic architecture which allows our company to quickly start moving from zero and grow to a certain threshold. But if we want to keep growing, we need to switch gears and migrate to microservices architecture. So now that we understand the benefits of microservices and how they solve the scalability problems of the monolithic three tier architecture, there's one big caveat. If things were so simple, our course would end right here. It would go back to your code base, break it into small services and everything. It would be easy and smooth from here on. Unfortunately, things aren't as simple with all the benefits that come with microservices. This architecture style also comes with many challenges and overhead. Starting with the technical challenges, the first big problem is that we took a centralized system and broke it into a highly distributed one. Now, Method calls that used to have very predictable behavior, success rate and latency became network calls between two different applications that may be running on different computers. Those network requests are now traversing an unreliable network where packets can be lost or delayed, causing unpredictable latency issues and even errors. Besides the network itself, the nature of distributed systems is that each component is inherently unreliable. For instance, a request from one service instance can hit a process that either crashed because of a hardware or software failure or is being restarted as part of a routine maintenance from a testing perspective. We also now have new challenges. On one hand, testing each microservice in isolation became faster and easier. On the other hand, now there is no guarantee that once all the services are deployed, they will work correctly together as a whole. And when we make a change in one service, we have far less confidence that once the new version of our service is deployed in production, it won't break another service. This situation may lead to the creation of very complex and slow integration tests that can slow our productivity. And given that each service may belong to a different team, it's very hard to decide who owns those integration tests to begin with. Finally, with microservices, it's much harder to troubleshoot performance issues and bugs. Let's consider a situation where a request from the user goes through a chain of microservices and in the end the user either didn't get the correct information, got an error or got the correct information, but with a long delay. How can we know which microservice is responsible for this bug or performance issue? If this isn't enough, we may also experience challenges in organizational scalability. One of the biggest questions is how to set the scope of responsibility between microservices. If we set the boundaries incorrectly, we may actually have more organizational overhead than benefits. For example, if every change in the system requires careful coordination between different teams, then we aren't better off than what we had in the monolithic architecture. Similarly, if every team uses a completely different technology stack tools and best practices, we may have a lot of duplicated effort and also confusion when looking at the codebase of a different team. So as we can see, if we aren't careful and don't consider all of those challenges, we may easily be in a much worse situation after migrating to microservices than staying with a big monolithic application. This situation is commonly referred to as a distributed monolith, or less formally a big ball of mud. The good news is that many companies have already gone through all those growing pains. So after many years of software architects of those companies sharing knowledge, success stories and mistakes, we can now establish a set of principles and best practices for a successful microservices architecture. So if we follow those industry proven principles, the benefits we get from microservices far exceed their overhead and complexity. So now that we fully understand both the benefits and challenges of microservices, we are ready to dive in and learn how to design scalable and reliable systems using microservices architecture. On the other hand, we also mentioned some of the challenges and overhead that come with microservices. Those include the complexities and unpredictability of running a highly distributed system with unreliable components, talking to each other through unreliable network and the risk of decreased organizational scalability.
