# CQRS Pattern

Welcome back. In this lecture, we're going to learn another very powerful pattern where we use the combination of microservices and event driven architecture. This pattern is called seekers. We will first get a general introduction to the seekers pattern. After that, we'll discuss two types of use cases where this pattern can greatly benefit large scale systems designed using microservices architecture. And we'll conclude with a real life example of this pattern. So first, what is seekers? Seekers stands for Command and Query Responsibility Segregation in a typical system that involves data stored in a database. We can group the possible operations on that data into two types. The first type is a command. A command is an action. We perform that results in data change in our system. This actions involve inserting new records such as adding a new user updates to existing records, such as editing a product review or updating the user's phone number or deletion of records, such as when a user deletes their own comment on a social media platform. The second type of operation is a query in contrast to the command. A query only reads data from the database but does not change it. Our system can return the data as is, or perform a simple transformation on that data, such as sorting or aggregation. However, the data inside the database will never change due to this query. An example of a query would be a request to show all the products in a particular category be sorted alphabetically. So when we use the command and query responsibility segregation in microservices, we essentially separate or segregate the code and storage for the command part of our system. From the query part of our system, All the data changes go through the command service to the command database and all the queries go through the query service to the query database, which contains a copy of the data in the command database to keep the command and query databases in sync. We use the event driven architecture. All we need to do is add a message broker between the command service and the query service. And every time the command service receives a request that changes the data within its database, it also publishes an event to the message broker that the query service can consume. When the query service receives that event, it updates its own database with the new data. Now the pattern has two main use cases in microservices architecture. The first is the separation of concerns between the command and query workloads in terms of business logic and performance. When we use secures to segregate the command part of our system from the query part into separate databases and services, we can get a lot of benefits. First, this pattern goes really well with the single responsibility principle for microservices. This allows us to have all the complexity of managing, write permissions, input validation and other business logic in the command service while keeping the query service clean, simple and focused on the best way to present the data to the user. This separation of concerns on the service side also allows us to easily evolve each part independently, using the optimal data model in our programming language for each workload. Also, if only one part for example, the command service was updated with new business or validation logic, the query side doesn't have to be retested or redeployed since it did not change. In addition to the separation of concerns, we also get very critical performance benefits. Since all the command type operations that involve data changes go to the command database. We can use the most optimal structure schema and database technology for write operations. Similarly, since all the. Worries. Go to the query database. We can organize our data to make those queries very efficient. Specifically for the read operations. And we can also pick the most optimal database technology and configuration for read intensive workloads, which is typically very different from the ones used for write intensive workloads. So basically we can optimize our system for both types of operations, which is otherwise impossible to do. If we had only one database for both command and query operations. In terms of scalability, we also get a lot of benefits. For example, we can adjust the number of instances for each microservice depending on its traffic. And since we're dealing with distributed databases here, we can also adjust the number of database instances for each database, depending on the rate of operations we get of each type. Now, the second type of use case for the secure pattern is solving another problem of applying the one database per microservice principle, which is the loss of the ability to join data from different sources. When we had all our data tables in a single relational database, we could relatively easily perform joint operations between multiple tables. However, once we split our monolithic application into microservices, each with its own database, we no longer have that option. So now to join data, we first need to make an API call to each one of those two microservices. Then each microservice will get the relevant data from its own database and return it to the caller. And only after we parse and format the data from the two different microservices can we join it programmatically into one view. This programmatic join operation can be very difficult, error prone and most importantly, very slow. Of course, this overhead is absolutely unacceptable if we're trying to show that data to the user in a browser or in a mobile app. So to solve this problem, we will follow the pattern and create a new microservice with a read optimized database. This microservice will receive only query requests for the specific data that we're trying to fetch to the user inside the database of this new query microservice. We will store a join view for all the data from both microservices, formatted in a way that is optimal for reading or querying to create and update this view. We're going to add a message broker where both Microservice A and Microservice B can publish an event whenever they change their data. So now any time the data changes in microservice A, for example, that microservice will also publish an event to the message broker. The query service will pick up this event and update its join view with the new data. Similarly, if a data change request goes to microservice B that microservice will also publish an event for the query service after it updates its own database. And once the query service consumes that event, it will update its view to reflect the new change. Now all the query requests that require a joint view of the data from Microservice A and Microservice B will simply go to the query service and the query service can simply return it with very minimal formatting or filtering. So the performance gain for the user or any other service that calls the API of the query service can be quite substantial. One important caveat with seekers, as we can now only guarantee eventual consistency between the right and the read operations. This means that in that time gap between the data change and the command database and the update in the query database, certain read operations will see the old data. So now that we understand the principles and the benefits of the seekers pattern, let's demonstrate it with a real life example. Let's say we have an online service that provides crowdsourced reviews about businesses such as restaurants, bakeries, car. Dealerships, hair salons and so forth. For example, a user who dines at a sushi restaurant can use our app to leave a review and a rating describing their experience at that business. Later, based on all those different users reviews. Anyone can open our app and search for a sushi restaurant in a given area. Our system in turn will return a list of restaurants sorted by the number of reviews, average rating and the relevance to the keywords the user typed in their search. Assuming we use decomposition by domain, we should have the following microservices. However, this current architecture has a few problems and conflicting requirements. First, on the command side, we have very strict business rules on who can add or update a business information. For example, we need to ensure that only verified owners of a business can update its website, address, phone number and other details. Similarly, when a user visits a business and wants to leave a review, we can have a lot of validation and business logic in the review service. For example, we need to authenticate the user to ensure it's not a bot that leaves fake reviews. We also want to ensure that the user hasn't left a review of the same business recently, and if they did, then we should prompt them to update their existing review instead of leaving a new review. We also need to validate that the review text and the star rating are valid inputs. From the performance perspective, we have very infrequent changes to the business information. On the other hand, we do get many reviews daily, so the review service needs to have a database schema and technology optimized for write operations. Now let's look at the query part of our system. When a user searches for a business such as a sushi restaurant or a vegan bakery, we need to join data from both the business service and the review service. This means that the business service will have to return all the businesses in the system that match the search query, and then we'll need to request all the ratings for each business so we can calculate and sort those businesses accordingly. As we can see, this operation can be extremely slow and inefficient. To solve this problem, we can use the secure rest pattern and introduce another service called the Business Search Service. The service will have a special read optimized database with a text search engine specifically designed for text queries. Now, any operation to add a new business or update an existing business in the business service will be published as an event to the business search service. The Business Search service will index its name, description and metadata and place it inside its database. Similarly, any time a review is submitted and approved by the review service, it will also be published as an event to a corresponding topic in the message broker. The Business Search Service will join that review with the corresponding business and recalculate its new average rating. So now any search request will go through the query part of our system to the business search service. The service is completely free from validation authentication or any complex business logic. All it does is parse and format the request and pass it to the text search database, and once it gets the result, it will return a list of all the businesses sorted by the relevance to the search query and rating based on a predefined formula. In this lecture, we'll learn another pattern that combines microservices and event driven architecture using seekers. We got a lot of benefits for microservices, such as separation of concerns, higher performance for read and write operations and higher scalability. Additionally, by using seekers, we solve the problem of joining data from multiple microservices, which otherwise would be very hard. To achieve in an efficient and performant way. And finally, we demonstrated the use of secure in a real life example.
