# Saga Pattern

Welcome back to another lecture. In this lecture, we're going to learn the first pattern where we can use microservices and event driven architecture. This pattern is called the saga pattern. First, we'll understand what problem we're solving with this pattern. Then we'll introduce the pattern itself. And later we'll discuss two ways we can implement this pattern using microservices and event driven architecture. So let's start with defining the problem we're trying to solve. As a reminder, one of the main principles of microservices is the one database per microservice. This separation allowed us to decouple the teams and codebases of every microservice in the company. Using this principle, the database is just an implementation detail of that microservice team and it's not exposed externally to other microservices in any way. This allows each team to use the right database technology for its workload and more importantly, iterate and make changes to it without having to communicate or coordinate those changes with anyone. However, as we all know, everything in software architecture is a trade off. So this decoupling introduces a new problem. When we had a single monolithic application with all its data stored in one database, we had the ability to modify multiple data sources atomically as part of a single transaction. As a reminder, a transaction guarantees that to an external observer, a sequence of operations appears as a single operation, even though those operations may involve multiple records within a table or, more importantly, multiple records within different tables. This is what we usually refer to as acid transaction guarantees, where acid stands for Atomicity consistency, isolation and durability. However, once we migrate to microservices, we no longer have one database to perform that transaction. Even worse, each operation in this workflow is performed by a separate microservice, which is only aware of its part of the workflow. So the saga pattern helps us perform a distributed transaction that spans multiple microservices and databases. In this pattern, we perform the individual operations that are part of that transaction as a sequence of local transactions in each database. Each successful operation triggers the next operation in the sequence, and if a certain operation fails, the saga pattern rolls back the previous operation or operations by applying compensating operations that have the opposite effect from the original operations. Generally, there are two ways we can implement this pattern. The first one is using a stateful workflow management service that orchestrates the entire transaction. The sole purpose of this service is to orchestrate the transaction in the correct order and also apply the compensating operations in the opposite order if things go wrong. To illustrate how this works, let's take a scenario of a vacation booking service that offers vacation packages to customers. A vacation package includes a round trip flight to the vacation destination, then a hotel reservation and a car rental for the given dates. This is a perfect example of a transaction because we can allow only one of two scenarios. We either have all the components of the vacation booked and paid for, which makes the transaction a success, or if any of the parts fail, the entire transaction is considered a failure and is aborted. So let's see how we can implement it using the workflow management service within our microservices architecture. Within our vacation booking service, we can have the following microservices. The payment service, the flight reservation service. The hotel booking service, the car rental service and the order service. So the first thing we need to do is define the workflow order within the workflow or. Restoration Service. The operations within that workflow are also commonly referred to as activities. In this case, we can do it in many ways, but let's say that first we secure the payment from the user for the entire value of the vacation. Then we booked the flights to and from the vacation destination. Given the seat preferences the user provided us with. Then we proceed to book a hotel room for the user. After that, we make a reservation with the car rental company for the given dates, and finally we add an order entry for the given user with all the details about the booking. After we define the order of the activities for the successful scenario, we also need to add the compensating operations for each activity in case we need to undo it. So for securing the payment from the user, the compensating operation would be crediting the user's account or credit card with the same amount. The compensating operation for the flight reservation is to cancel both flights to and from the vacation destination. The operation to undo the hotel reservation is to cancel the hotel booking and to undo the car reservation. We also need to simply cancel it. And although the last operation in the sequence is to insert the order details, we still need to define the compensating operation to it in case we want to add more operations to this flow sometime in the future. So the compensating operation in this case would be to update the order to a cancel status. Now the second step is to implement the workflow operations, to call the API of the corresponding microservices. And the third step is to implement the compensating operations to call the same or different APIs of those microservices depending on the operation. And that's it. Now, when a user wants to book a vacation with us, they will make a request to our web app service. The service, in turn, will show the user the available options and prompt them to provide their preferred dates and other details. Once the user submits the vacation booking request, that request will go through the API gateway to the Workflow Orchestration service. The service will initiate the workflow step by step. If everything goes smoothly, then the user will get billed. The flights get booked, the hotel room gets reserved, the car rental at the vacation destination is arranged, and once the order service gets updated, the orchestration service will return a confirmation to the user. However, if anything goes wrong, that entire transaction will have to be aborted and the user will get an error. For example, let's say that by the time our workflow orchestration service makes the request to the car rental service, that company runs out of cars in that location for the given dates. At this point in the workflow, the user already got billed for the entire vacation, including the car and the flights and the hotel were already booked as well. So the Workflow Orchestration service will apply the compensating operations, calling the hotel reservation service and the flight service to cancel those reservations. Then it would credit the user's credit card with the full amount. And finally, it will send an error to the user explaining what went wrong. The second way to implement the saga pattern is by using a purely event driven model using the event driven architecture. In this method, we remove the workflow orchestration service and delegate the management of the entire workflow to the microservices themselves. To accomplish this, the communication between the microservices is done asynchronously through events published to a message broker to compensate for the removal of the orchestration service. Now each microservice needs to know its role in the workflow. In other words, it needs to know where to send an event when the operation is successful and where it needs to send an event in case it fails. And it needs to trigger the compensating operation for the previous service in the sequence. To demonstrate this, let's go back to the vacation booking service example. Now, when a user places an order request for a vacation, this request will go straight to the payment service since the workflow is now asynchronous. The payment service will respond to the user immediately. Then it will initiate the sequence by billing the user and then sending a corresponding event to the message broker. The flight reservation service will pick up this event and make the reservation and upon success it will emit another event to a topic to which the hotel reservation service is subscribed. Assuming all the operations are successful, this entire chain of events will eventually lead to the order service getting the event that all the reservations are completed so it will update its own database. However, notice that since we used an asynchronous event driven architecture for the response to the user about the success or failure also needs to be asynchronous. So for that we will publish another event to another service called the Notification Service. The service will notify the user about the outcome of the workflow through an email or a push notification. Now let's see how this saga pattern implementation works. If one of the operation fails and we need to roll back the transaction using the same example as before. Let's say that the billing of the user, the flight reservation and the hotel reservation were all successful. However, for whatever reason, the car reservation service could not make the booking in this case to trigger the rollback of the transaction, it will publish an event. To another topic that indicates a failure in making the car reservation. Now, the hotel reservation service, which is subscribed to this topic, will pick up this event and know that it must now undo its operation by cancelling the hotel reservation. And then it will publish another event to another topic to which the flight reservation service is subscribed. Similarly, the flight reservation service will cancel the flights and publish another event that the payment service will pick up. In a similar way, the payment service will credit the user and emit another event to the topic that the notification service is subscribed. This event indicates that the entire transaction is a failure. Now the Notification Service will pick up that event and notify the user with all the details and the next steps that they need to take. So now that we learned about the Saga pattern and its two implementations, let's quickly summarize what we learned in this lecture. In this lecture, we'll learn about the loss of asset transaction guarantees when migrating from a monolithic application with a single database to microservices architecture. We solve that problem by using the saga pattern, which breaks the transaction into a sequence of operations that each microservice performs upon success and a sequence of compensating operations performed in reverse order upon failure. Later, we discuss the two ways we can implement this pattern. The first one used a stateful workflow management service to orchestrate the transaction, and the second uses the event driven architecture where each microservice is responsible for triggering the next operation in the sequence or triggering the compensating operation in the previous microservice in case of a failure.
