# Event Sourcing Pattern

In this lecture, we're going to talk about event sourcing, a very useful but commonly misunderstood architecture pattern. So before we talk about event sourcing, let's talk about the typical way that applications handle data. Generally, the data applications store in the database reflects the current state of our business, and any modification to that data reflects the new state which overrides the previous state. Of course, if we want to find out the previous state after it's overridden, we can always find its backup. In most cases, unless something went wrong, we don't care about the previous state or states of our data. We only care about what we have right now. Also, all the interactions we have with that data usually happens with Crud operations which are create, read, update and delete. And for most use cases, that's all we need. Let me give you an example. Let's assume we have an online store and inside our NoSQL database we have a collection of products for each product. We store information such as its name, description, price and links to images. Now, if at some point the owner of the product updates the product's price or description, the user doesn't care. All they care about is the current most up to date price and description, and the history of those changes doesn't have any value. However, this isn't always the case in certain situations. We need to know not just the current state, but also every step that led to it. For example, if we are an online bank, imagine what would be the reaction of a client if we gave them their current balance and nothing more. No explanation on how they got to that balance. No data on withdrawals, deposits, fees or credits. Just the current balance and nothing more. That would be inconceivable, right? And the reason for that is the way that balance was calculated is important not only for visualization, but also for auditing and potential corrections. Another example is if we're a merchant that sells products in an online store. Now imagine that all we have at any given moment is just the current state of the inventory that we have of our products in the warehouse. So imagine that we log in the system in the morning and see that there are currently 200 items of our product available. Then we log in again in the evening and we see that now we have only 100 items of our product left. So did we make 100 individual sales? If we did, that's great. But what if we sold 200 items and we got 100 returns? In that case, it's not that great, right? Or maybe we just get a one time bulk purchase from some company. So in this case, to just having the current state is not enough. And we want to see the steps that led to the current state. This is where the event sourcing pattern comes in in the event sourcing pattern. Instead of storing the current state, we have only events. Each event describes either a change or a fact about a certain entity in our system. Most importantly, events are immutable, meaning once we get an event into our system, it never changes. And the only thing we can do is to append new events to the end of the log. Now, to find the current state of an entity, we simply apply or replay all the events that happen to that entity. This is basically like version control, but for data where each event represents only the delta or the change from the previous state. For example, in the case of a bank account, instead of maintaining and updating the current balance, we store only the transactions that move money out or into the account. And by replaying those transactions from the day that the account was opened, we can always get to the most up to date current balance. Now the benefit of storing those transactions instead of the balance is that now we have the complete history of how we got to that balance, which we can report on and audit. For example, by looking at those transactions, we can notice duplicate transactions that may have had. Happened by accident. We can also detect suspicious patterns that may indicate fraudulent activity that the client was not aware of. Finally, we can also provide insights such as advice for better spending habits or maybe recommend another product such as a savings account or a credit card. Now, before we talk about the few added benefits of using this pattern, let's talk about a few ways that we can store and represent those events. The first way is to simply store each event as a separate record in a database. For example, in the online store use case, we can store every new order in the orders table, and every time the order status changes, we append a new row with the same order ID, its new state and the timestamp. So now we can see the entire history of each order transitioning from one state to another state over time. The advantage of storing events in a database is the ability to perform queries and analytics over entire data sets. For example, we can track the progression of one order, but we can also analyze and track the progression of multiple orders at once and get certain insights. For instance, we can find out that on a particular date we got a lot of new orders which may be correlated with a sale. Or we can also find out that on a particular day we had a lot of ordered delivery problems. Another way we can store events is using a message broker. In this case, instead of storing the events in a private database that belongs to a service, we can publish them for anyone to consume. Unlike most databases, message brokers are particularly optimized for handling a lot of events and also make it easy to maintain the order between different events at the same entity. On the other hand, performing complex queries on streams of events in a message broker is a bit harder and less intuitive. Now an added benefit to using event sourcing is write performance. Without event sourcing, if we have a write intensive workload, we get high contention over the same entities in a database which typically results in poor performance. As an example, we may have an inventory table that keeps track of current inventory of each product. If we have many concurrent updates to the same table or even worse, the same product, then all the updates will contend with each other and slow down each other as well as all the readers of that table. On the other hand, if we move to event sourcing, then each purchase or return becomes an append only event, which doesn't require any database locking and is a lot more efficient. Now let's talk about a few efficient ways we can use to read and reconstruct the most up to date state of an entity based on events. Of course, replaying all the transactions in someone's bank account Every time we want to show the client their balance is not very efficient. So to address that, we can apply a few strategies. The first strategy is taking snapshots at certain points of the events log. For example, we can take a snapshot of the account balance for each user once a month. So we still have all the history for all the transactions since the beginning of time if the user wants to access it. But now using those snapshots as a user can get their current account balance by only replaying the transactions from the last snapshot instead of the time the account was opened. The second optimization is using a pattern that we already know very well, and that is seekers using seekers. We can separate the part that appends events to our system and stores them from the query part, which stores the current state in a read optimized database That read only database can be even an in-memory database which makes reads even faster. Now, as always, every time a new event comes in, if we store those events in a message broker, then the query service can subscribe to that same topic or channel and pull the data straight from it. And the command side doesn't need any special. Database. The combination between secure and event sourcing is very popular in the industry. Since we get the best of both worlds. We get the history and auditing. We get fast and efficient, writes and we get fast and efficient reads. Of course nothing comes for free and if we use event sourcing with secures, we can get only eventual consistency, which again may or may not be good enough depending on your use case. In this lecture we'll learn about the event sourcing architecture pattern. Using this pattern instead of storing and updating only the current state of an entity, we store the changes or facts about this entity using events. We later talked about two ways we can store those events, which is a database or a message broker where each approach comes with its advantages and disadvantages. After that, we mentioned a very nice side benefit of higher performance when using the event sourcing pattern for write intensive workloads. And finally, we'll learn about a powerful combination of event sourcing and secure, which allows us to enjoy the benefits of auditing, performant writes and efficient reads.
