But let's summarize what we learned. So a CIDR is an IP range. A VPC is a Virtual Private Cloud, which is working for IPv4 and IPv6. Subnets are tied to an AZ in which we define CIDRs and we have public and private subnets. So how do we have a public subnets? Well, we attach an internet gateway and then we create a route from the public subnet to the Internet Gateway. And this provides IPv4 and IPv6 internet access if enabled. Now, route tables, they would be edited to have route to the Internet Gateway, the VPC peering connections, the VPC endpoints and so on. And they're key to make sure that the network flows within our VPC. The bastion host was a public EC2 instance that we were able to SSH into. And then this instance had SSH connectivity to other EC2 instances in private subnets. That's what it's called the bastion host. NAT instances was an EC2 instance deployed in a public subnets to give internet access to EC2 instances in private subnets. It is old, it is getting deprecated and you need to disable the source destination check flag for it to work and also edit your security group rules. So we have the NAT Gateway instead, which is much better. It's managed by AWS. It provides scalable internet access to private EC2 instances when the target of your request is an IPv4 address. Next, NACL. So they were network SELs and they were firewall rules to define inbound and outbound access at the subnet level. And don't forget about ephemeral ports in its... NACL are stateless, so inbound and outbound rules are always evaluated. Whereas security group rules are stateful. That means that if the inbound is allowed, the outbound is automatically allowed and vice versa. And security group rules are applied at the EC2 instance level. VPC peering was very helpful to connect two VPCs together and only if they have non-overlapping CIDRs. And VPC peering connections are non-transitive so if you need to connect three VPCs together, you need to have three VPC peering connections. VPC endpoints are allowing us to get private access to AWS services such as Amazon S3, DynamoDB, CloudFormation, SSM, whatever you want within the VPC. We saw they were gateway endpoints for Amazon S3 and DynamoDB, and all the rest was interface endpoints. VPC Flow Logs was the way for us to get some log level of metadata around all the packets within our VPC, and we had some information around the accept and the reject traffic. Now, VPC Flow Logs can be created at the VPC subnet or ENI level, and they can be analyzed and sent to Amazon S3 and then analyze using Athena, or they can be sent to CloudWatch logs and then analyze using CloudWatch Log Insights, for example. Now, to connect your VPC to your data center, you have two options. The first one is a site-to-site VPN. And so to do a site-to-site VPN is a VPN connection over the public internet in which we have to create a virtual private gateway on AWS and a customer gateway on your data center. And then you establish the VPN connection. And then we saw that if we establish multiple VPN connections using the same virtual private gateway, then we could use VPN CloudHub to have a hub and spoke VPN model to connect our sites. The second way to connect to our data center was to use Direct Connect. So this time the connection is completely private, it does not go over the public internet, but it's takes time to establish. You need to hook your own data center to a Direct Connect location for it to work. So it's more involved, but it's more secure and has a more stability onto the connection. Now, Direct Connect Gateway was to set up a Direct Connect to many VPCs in different AWS regions. We saw PrivateLink or VPC endpoint services, which is to connect privately to services within your VPC that you have created yourselves to customer VPC. And the cool thing about it is that it doesn't require VPC peering, or the public internet, or NAT Gateway or round tables. It's only used or mostly used with a Network Load Balancer and an ENI. And this allows us to expose a service within our VPC to hundreds or thousands of customer VPCs without exposing your network. ClassicLink is to connect EC2-Classic instances privately to your VPC, but it's going to be deprecated very soon too. Transit Gateway is a transitive peering connection for your VPC, VPN and Direct Connect, and it has this fact that everything can flow through it, which is quite nice. Traffic Mirroring is to copy network traffic from your ENIs elsewhere for further analysis. And then we looked at IPv6 in your VPC. We saw how to enable it, and we saw that there was the existence of the egress-only Internet Gateway, which is just like a NAT Gateway, but for IPv6 traffic out to the internets. So that's it.
