# Aurora

So now let's do a summary of Amazon Aurora. So Aurora is a compatible API for two database engines, which are PostgreSQL and MySQL. And Aurora is a very special service because storage and compute are separate. So about storage, while the data is stored in six replicas across three Availability Zones by default, and you cannot change this, therefore Aurora is highly available. On top of it, in case there is any issue with the storage, there's going to be a self-healing process behind the scenes happening. And if you wanted to increase the storage, no problem. There is auto-scaling out of the box. Now, for the compute side, this is the actual database instances, they are clustered, and they can be across multiple Availability Zones if you wanted to. And then if you had read replicas, you could auto-scale them as well to increase the capacity in case the load increases. Now, because you have a cluster of database instances, then you need custom endpoints to know where to write and read, and therefore you have a writer endpoint and a reader endpoint. You have the same security, monitoring, and maintenance features as RDS because, end of the day, they use kind of the same engine. And you need to know the backup and restore options for Aurora, which cannot summarized here, but I suggest you go back to that lecture if you ever need to. Now, some extra features you need to know about Aurora. The first one is Aurora Serverless. This is when you have unpredictable and intermittent workloads, but you don't need to do any capacity planning, so it is quite helpful in that use case. If you wanted to have a global database, you could use Aurora Global, and that gives you up to 16 database read instances in each region where your database is replicated. And the storage replication happens in less than one seconds usually across regions, which is something you have to look out for at the exam. And in case you have a problem in your primary region, you can promote a secondary region to become your new primary. If you wanted to perform machine learning on top of Aurora, you could use the Aurora Machine Learning module to use SageMaker and Comprehend. If you wanted to use a testing database or a staging database out of your production database, for example, you could use the Aurora Database Cloning feature to create a new Aurora cluster out of the existing one, and it is much faster than doing a snapshot and restoring it. So the use cases for Aurora are the same as RDS, but you have less maintenance, more flexibility, more performance, and more features out of the box, which makes Aurora a great choice.
