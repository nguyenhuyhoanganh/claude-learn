# Using Type Definition Files

As a quick side, one of those benefits of type definition files is that we can now hover over Faker right here, and then if you're on Mac, hold down the Command key, or if you're on Windows, hold down Control. You'll notice that Faker turns into a clickable link of sorts. If we now click on that, we'll be taken to the type definition file that we just installed. Notice how the name of this file is index.d.ts. Traditionally, type definition files are named with extensions of d.ts, so anytime you are looking at some TypeScript code and you see a file with that extension, that means you are looking at a type definition file. Inside this file, you're going to find nothing but TypeScript syntax. So the only goal of this file right here is to describe the different types of values, functions, objects, classes that exist inside the Faker library. So there will not actually be any functions that are implemented inside of here. Instead, you will only see a definition of the different types that are available. If you look inside of here, we can actually use this as a source of documentation. So for example, remember at the official documentation we looked at just a moment ago, we saw that address module. So this right here is telling TypeScript that this library has a address object, and inside of it are different functions like zip code, city, city prefix, and so on. We can see the different arguments that each of them take and the different type of values that they return. So like I said, we can use this file as kind of an alternate forms of documentation to understand how to work with a given library. So in this case, we can take a look at address. If we scroll down, you'll notice that latitude and longitude are two functions inside there, and we're going to use those functions to actually randomly generate our latitude and longitude. As a quick side note, I want you to notice how latitude and longitude return strings, which is actually kind of not very typical. Usually, we reflect latitude and longitudes as numbers. So having them being returned here as a string means that might be a little bit unexpected, and we might have to deal with that at some point in time. But we'll just see what happens. Now, here's a really quick note for you, something that really gets people hung up when they are getting started with TypeScript. At this point, we have only told TypeScript that our class user is going to be of type or going to have a property called location. And location will have an object with a latitude and longitude property. Here's the important thing. When we create an instance of a user, location does not get automatically set to be an object. So in other words, if we did right here a console.log of this.location, we would see undefined or null. It will be an object that has these two properties. But we have not done any initialization for that yet. There is no object here when we create our user. So we are responsible for initialization of this object and the properties inside of it. So in other words, that means to initialize latitude and longitude, we have to say this.location is going to be an object with properties lat and lng. So we can try to initialize both lat and lng. Once again, in my opinion, this is kind of an oversight by the faker people, and they really should be returning numbers here. However, that's just how they built this library. So we need to take the strings that are returned by latitude and longitude and convert them into numbers. To do so, we have to do an actual conversion process. Like this is a string right here. So we have to convert that into an actual number. So to do so, we're going to take a very JavaScript style approach. We're going to use the parse float function. Just to make sure you are aware, the parse float function is built into JavaScript. It takes a string and returns a number that has a decimal value attached to it. So we can call parse float on both, and the error goes away because now we are assigning a number to both latitude and longitude. All right, so that's pretty much it.
