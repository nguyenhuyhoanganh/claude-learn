# Annotating the Return Type

As I just mentioned, we're going to spend this video and the next couple to figure out some ways we can use TypeScript to better check the code we have written out inside of a reducer. So we're going to start off kind of long series of refactors here and eventually get to a better place. The first thing we want to take care of is the fact that right now we can very easily change the data property that we are returning to any other kind of value if we do so. As I mentioned, our application is probably going to break, but TypeScript is not saying that anything is currently going wrong. So to fix this up, we're going to apply a return type annotation to our REDUCER function. We're going to say that we're always going to return a value of type repositories state from this function no matter what. So to do so after my argument list, I'll put in a colon and then repositories state. So now TypeScript is aware that whenever we return anything from our reducer, it must always match up with this interface. So data, for example, must always be an array of strings. If I now try to change data to a number, I get an error. If I try to change it to a string, I get an error. The only possible thing I can assign to data is either an empty array or an array of strings. Well, that seems like a very small change, but it definitely adds in a little bit of a check just to make sure that we are doing the correct thing inside of our reducers.
