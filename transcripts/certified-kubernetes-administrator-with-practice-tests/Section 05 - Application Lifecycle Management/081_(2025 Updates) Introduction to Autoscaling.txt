Okay, now before we begin to look at auto scaling, let's look at what scaling means in the first place. So let's forget about Kubernetes and containers for a second and go back to the plain old servers and apps hosted on physical servers. And so if you go back in time, millions and millions of years ago, when we used to host applications on physical servers with a predefined CPU and memory capacity. And what happens when the load increases and we run out of existing resources on the server. Now we need to scale up the server. So we took down the application and added more CPU or memory resources to it and then powered it back up. And that's referred to as vertical scaling. So as you just saw we basically increase the size of the existing server vertically. We added more resources to our existing server by adding more CPU and memory components to it vertically. And so that's vertical scaling if the application supported running in multiple instances. Another thing we could have done is that we could have avoided having to shut down the server. Instead, we could have added more servers to it and shared load between them. So running more instances of the application by adding more servers is known as horizontal scaling. So that's the basics of scaling. Vertical scaling refers to adding more resources to existing application, and horizontal scaling refers to adding more instances or more servers to your system. Now let's look at how that relates to Kubernetes and the container world. When we first introduced Kubernetes, we mentioned that one of the major purposes of a container orchestrator like Kubernetes is to host applications in the form of containers and scale up and down based on demand. So there are two types of scaling in Kubernetes. What we just spoke about, uh, is scaling workloads. So you can scale the workloads by adding or removing containers or pods onto the cluster. Another approach to scaling is scaling the underlying cluster itself. So that's adding or removing more servers or infrastructure to your cluster. So there's scaling workloads and scaling the cluster infra. So let's look at what these mean. So there are two ways of scaling these individually. So there's horizontal and vertical scaling. So when it comes to the cluster horizontal scaling refers to adding more nodes to the cluster. Vertical scaling would mean increasing the resources on existing nodes in the cluster. When it comes to scaling workloads, horizontal scaling would mean creating more pods and vertical scaling would mean increasing the resources allocated to existing pods. Now let's talk about the how. So there are two ways of scaling. There's the manual way and the automated way. The manual approach of horizontally scaling cluster infra would be to manually provision new nodes and then use the kube adm join command to add new nodes to the cluster. When it comes to vertical scaling, I don't think that's a common approach used for Kubernetes, because it would result in having to take down the server and applications running on them, and then add more resources to the servers and bring it back up, which is not something that we would want to do. So since most of the infra these days are virtual machines, you could easily provision a server with higher resources, add it to the cluster and remove the older server. So I don't think vertical scaling is that common of an approach. Now when it comes to horizontal scaling Rescaling workloads. The manual approach would be to run the kubectl scale command on the workload, to scale up or down the number of pods, and to vertically scale resources associated to a pod. You would usually run the kubectl edit command to go in to that deployment or stateful set or replica set, and change the resource limits and requests associated with the pods. Now we will dive into more details about each of these in the upcoming lectures, but I'm just giving you a high level overview right now. Now, now that we have discussed all of this, finally let's look at the automated approach. So for horizontally scaling the infra we have what is known as the Kubernetes cluster autoscaler. For horizontally scaling the workloads we have what is known as the horizontal pod autoscaler or HPA. And for vertically scaling workloads we have the vertical Pod autoscaler.
