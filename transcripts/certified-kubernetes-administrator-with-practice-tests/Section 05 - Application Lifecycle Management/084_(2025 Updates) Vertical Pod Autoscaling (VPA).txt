Let's now look at Vertical Pod Autoscaler in Kubernetes. So let's start by looking at how I would manually scale a workload vertically. Say I am a Kubernetes administrator and I'm sitting on my machine looking at a cluster, I'm tasked to make sure that there is always sufficient workload to support demand for this application. And from the deployment configuration, I see that this pod requests a 250 milli cores of CPU and has a limit of 500 milli cores of CPU. And this means that 500 milli cores is the max capacity it gets. So as I did before, I would run the kubectl top pod command and monitor the resource consumption of the pod. And so remember that you must have the metric server running on the cluster to be able to monitor as the same as before. And when it reaches a specific threshold, what I'm going to do to scale the pod vertically is run the kubectl edit deployment command. So I would run the edit deployment command and I would change the resource requests and limits assigned to the deployment under the pod template, under the container section, and then save it. And what would happen is it would then kill that pod and create a new pod. So that's basically how I would do it manually. Now of course we don't wanna do it manually. We have the vertical pod autoscaler for that. So similar to the horizontal pod autoscaler, the vertical pod autoscaler continuously monitors the metrics and then it automatically increases or decreases the resources assigned to the pods in a deployment and thus balances the workload. So let's look at it in a bit more detail. So unlike the horizontal pod autoscaler, the vertical pod autoscaler do not come built-in, as such, we must deploy it. So we first apply the vertical pod autoscaler definition file available in the GitHub repo. And then when we run the kubectl get pods command in the cube system namespace and search for VPA, we'll be able to see that there are multiple components deployed. So there is the admission controller recommender and the updater service, which should be running. So let's look at what those are. So the VPA deployment consists of multiple components. So there's the VPA admission controller, the updater, and the recommender. So the recommender is responsible for continuously monitoring resource usage from the Kubernetes metrics API, and collects historical and live usage data for pods, and then provides recommendations on optimal CPU and memory values. And the recommender itself does not modify the pod directly, or it only suggests changes. The updater detects pods that are running with suboptimal resources and evicts them when an update is needed. So it gets the information from the recommender and monitors the pod. And if a pod needs to be updated, it evicts them. That means it just terminates the pod. And the admission controller intervenes the pod creation process and uses the recommendations from the recommender again, to then mutate the pod spec to apply the recommended CPU and memory values at startup. And this ensures that the newly created pods start with the correct resource requests. So basically the VPA recommender collects information, the updater monitors or gets the information from the recommender, compares that to the actual pod. And if the pod is beyond the threshold, it kills the pod and whether it kills or not, that depends on the policy that we'll talk about in a bit. But ideally it would kill the pod and then the admission controller intervenes because when a pod is killed, it's automatically, the deployment will automatically recreate the pod. And when it does that, the admission controller intervenes and updates the resources so that the pod is now, now comes up with the new size. So let's look at how to create the VPA service using a definition file. So not that there's no imperative match to create a VPA because it's not a built-in component like the HPA. So here we have the API version, which is autoscaling.k8s.io/v1. The kind is vertical pod autoscaler. The name is my app VPA. The target to monitor is the my app deployment. The container policies define what is monitored, in this case, the CPU minimum allowed and max allowed are configured. And finally we have the update policy mode. So there are four modes that VPA operates in. The off mode, which only recommends changes but does not do anything. So in this case, it's only the recommender working, the updater and the admission controllers are not working. In case of the off-mode. The initial mode that only changes pods on creation, not later. So in the initial mode, the recommender recommends a change. And when the deployment is scaled for some other reason and the new pods come up, the admission controller intervenes and changes the pods resource definitions. The updater in this case does not work to kill or take down the pod. In case of the recreate mode, that's where the updater component jumps in and evicts an existing pod, when its resource consumption goes beyond range specified. And finally, we have the auto mode that updates existing pods to the recommended numbers. Again, for now, this behavior is similar to recreate. So as of now, the auto mode works exactly as a recreate mode. But when the support for the in-place a bit of Pod resources is available, the one that we spoke about in the previous video, then that mode would be preferred over recreate. So that auto-mode is really built for the future. When the in-place update of pod resources are available in stable version of Kubernetes, that would be the default behavior. But as of now, the auto-mode works exactly like recreate, where if a pod goes beyond a range, it is killed. Now VPA will monitor resource usage and suggest adjustments. So we can check the recommendations with the command kubectl describe VPA followed by the name of the VPA. And here you can see the recommendation. So VPAs are just increasing CPU to 1.5. So we spoke about horizontal and vertical pod autoscalers, but you must be wondering when to use which. So let's kind of compare the two. First, let's look at the scaling methods. So VPA as we know works by increasing CPU and memory for existing pods or by just recreating those pods with the new resources. Whereas HPA adds or removes pods based on demand. This means that the VPA optimizes individual pod performance, while HPA focuses on distributing the load across multiple instances. Next, in terms of pod behavior, VPA restarts pods to apply the new resource values. This means downtime is involved when scaling up vertically. In contrast, HPA keeps existing pods running and simply spins up new ones ensuring continuous availability. Now what about handling traffic spikes? So HPA is clearly the winner here. It instantly adds more pods when demand increases, making it the preferred choice for applications that require rapid scaling. And VPA on the other hand cannot handle sudden spikes efficiently because it requires restarting pods typically, which introduces delays. Now, from a cost optimization perspective, both have advantages. So VPA prevents over provisioning by adjusting CPU and memory allocation to match actual usage. And HPA on the other hand, avoids unnecessary idle pods ensuring that resources are used efficiently without keeping excess instances running. So when should you use which? So VPA is best suited for stateful workloads and CPU or memory heavy applications such as databases, JVM-based applications, AI and other workloads that require fine-tuned resources. So let's say some applications require like heavy CPU in the initial startup, but then later does not require that much CPU. What you could do is that's where VPA comes in. So you could start the instance with a lot of CPU and once that initialization process is over, I mean when it needs to reduce the CPU memory usage allocated to it, then that's where VPA comes into picture. The HPA is ideal for web applications, microservices and stateless services such as web servers, message queues, and API-based applications where quick scaling is needed to handle fluctuating traffic. So in summary, VPA is about optimizing resource allocation for individual pods, while HPA is about scaling the number of pods dynamically based on demand. So choosing the right autoscaler depends on the workload type and how you need to scale your application. All right, that's all for this lesson.
