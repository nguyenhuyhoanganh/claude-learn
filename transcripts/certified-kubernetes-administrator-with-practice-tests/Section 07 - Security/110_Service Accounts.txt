# Service Accounts

Let's now look at service accounts and Kubernetes. So there are two types of accounts in Kubernetes. A user account and a service account. As you might already know, the user account is used by humans and the service accounts are used by machines. A user account could be for an administrator accessing the cluster to perform administrative tasks, so a developer accessing the cluster to deploy applications. A service account could be an account used by an application to interact with the Kubernetes cluster. For example, a monitoring application like Prometheus uses a service account to poll the Kubernetes API for performance metrics. An automated build tool like Jenkins uses service accounts to deploy applications on Kubernetes cluster. The second example I've built a simple Kubernetes dashboard application named my Kubernetes Dashboard. It's a simple application built in Python, and all that it does when deployed is retrieve the list of pods on a Kubernetes cluster by sending a request to the Kubernetes API and display it on a web page. In order for my application to query the Kubernetes API, it has to be authenticated. For that, we use a service account. A service account is an identity of a service. A service account is associated with what is known as a token. A token is what is used to authenticate a service account into the Kubernetes API. So if you have ever authenticated to an API using a curl command or something that you probably know of, the tokens that are passed through as a bearer token in the header of the call. This token is what proves the identity of the service account to the API endpoint. Well, think of it this way if the service account was an ID card that we create for a service, the barcode that's usually on the ID cards is the token. It's the barcodes that are scanned by the gates to prove the identity of that user. And it is this token that we will input in our Kubernetes dashboard later. Now let's look at how service accounts and tokens work in Kubernetes. By default, when a Kubernetes cluster is set up, it creates a service account by the name default in all namespaces. To list existing service accounts, use the kubectl get service account command. To see the details of an existing service account, run the kubectl describe service account command. Whenever a pod is created, the default service account is automatically attached to the pod. As you can see in the output of the describe pod command. Here, there's a service account field, and you can see that the service account associated with this pod is named as default. So the service account gets mounted as a projected volume within the pod. So think of a projected volume as a dynamic directory created inside the pod by Kubernetes automatically. So it is mounted at the location. VAR run secrets Kubernetes service account. Now if you list the contents of this directory, you'll see the token available as a file, and if you look inside the contents of the file, you'll see the actual token itself. The default service account comes with a lot of limitations, so if you have specific access requirements we must create a custom service account. So how do we do that. To create a service account run the command cube CTL create service account followed by the service account name which is dashboard Dash S.A. in this case, you can also of course create the service account the declarative way using a YAML file like this. So API version is v1, kind is service, account name is dashboard SA. And to view the service accounts we run the kubectl get service account command. This will list all the service accounts. And we can see that our newly created service account is listed here. And to list the details of a create a service account, we run the kubectl describe command and it lists its details. As you can see, when you first create a service account, it creates a blank service account with most of its properties set to none. So to associate a newly created service account to a pod, we pass the name of the service account using the service account name field in the pod spec. This time when we describe the pod, we see the dashboard service account associated with it. And if you look inside the mounted directory, you get to see the token file. Kubernetes automatically creates a short lived token and mounts it inside a projected volume. The Kubelet also automatically rotates this token, and this token is bound to the life of the pod, so it is automatically expired when the pod is deleted. Now, if you don't want a token to be automatically created and mounted inside the pod for the service account, you may add the auto mount Service account token key to false like this. You can define this either at the service account level. That way, no pods that use the service account gets the service account token auto mounted. Or you can define it at the pod definition level so that irrespective of whether this property is set at the service account level, it won't be auto mounted for that pod. Now, for the pods or applications hosted within the Kubernetes cluster. This setup that we've discussed so far works fine. So the pod is associated to a service account, and as soon as the pod is associated to a service account, Kubernetes automatically creates a token with an expiry date and automatically mounts that within the pod at a particular location, which the application running inside the pod can refer to and read from to make API calls to Kubernetes. At times, you might want to create a token that is to be used outside of the cluster, maybe to paste into the Kubernetes dashboard that we had created, or to put it into a CI CD tool or an external monitoring tool. In such cases, you can create a token by running the kubectl create token command. It creates a token and shares it on screen. And this token is not stored in any secret. By default, all tokens are valid for one hour to extend the validity. Add a duration flag to the command. Like this. If you decode this token using the jq and the base64 utility, you'll see an expiry defined as well as the service account name and other details that are associated with this token. This token can then be used as an authentication bearer token while making a Rest call to the Kubernetes API. For example, in this simple example using Curl, you could provide the bearer token as an authorization header while making a Rest call to the Kubernetes API. In case of my custom dashboard application, we can copy and paste the token into the tokens field to authenticate the dashboard application. So let's summarize what we have learned so far. Service accounts are used by other applications or services to interact with Kubernetes. Tokens are associated to service accounts to prove their identity. To create a new service account, run the kubectl create service account command followed by the name of the service account. To use the service account from an external application, such as a CI, CD tool or monitoring tools. We first create a token and specify the name of the service account associated with that, and it prints the token. On screen, you can then copy the token and paste it or use it in the external application. Now for internal applications or pods that are within the cluster, every namespace has a default service account. The default service account is automatically attached to the pod on creation. To attach a service account to a pod, we use the service account name field, and when a service account is attached to a pod, Kubernetes automatically creates a token and mounts as a projected volume, automatically rotates the token and automatically expires the token when the pod is deleted.
