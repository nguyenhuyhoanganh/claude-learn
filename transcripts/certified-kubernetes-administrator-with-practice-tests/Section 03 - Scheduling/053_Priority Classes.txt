# Priority Classes

Let us now look at priority classes in Kubernetes. We know that Kubernetes runs different applications as pods with different priorities. For example, the Kubernetes control plane components itself run as pods within the cluster. And we know that no matter what, those workloads need to always run. Similarly, we may have priority databases, critical applications running on the cluster as well as lower priority workloads like background jobs. Now, we need a way to make sure that higher priority workloads always get scheduled without being interrupted by lower priority workloads. This is where priority classes come in. Priority classes help us define priorities for different workloads, so that higher priority workloads always get priority over lower priority ones. If a higher priority pod cannot be scheduled, the scheduler will try to terminate a lower priority workload to make that happen. So priority classes are non namespaced objects, meaning they are not created within a specific namespace. They are not attached to a specific namespace. So once they're created, they're available to be configured on any port in any namespace. So how do you define priorities? We define priorities using a range of numbers. So a priority can be defined as high as 1 billion and as low as -2 billion or around that number. A larger number indicates higher priority. Now this range that I've just mentioned is for the applications and workloads that are deployed on the cluster. So the positive 1 billion to -2 billion number is for the applications or workloads that are deployed as apps on the cluster. But there is a separate range that's dedicated for internal system critical parts such as the Kubernetes control plane components itself. So they always get the highest priority. and they have priorities up to 2 billion. So to list existing priority classes run the kubectl get priority class command. And as you can see it currently lists the system cluster critical and system node critical classes. And as you can rightly see the value of these are close to 2 billion or around that range. Let us now see how we can create a new priority class. So to create a new priority class, we create an object with the API version set to scheduling v1 kind set to priority class metadata with a name. And then we define the value. So this is the number that we just spoke about. And finally we have the description. The description is an optional field. Once this is created we associate this priority class to a pod by using the priority class name property inside the pod definition. Like this. When that pod is created, it is assumed to have this priority that is assigned to the priority class. Now, what if I don't specify a priority class name for a pod? By default it is assumed to have a priority value of zero. However, if you'd like to modify that, you must create a new priority class and assign the global default property to true. So this is an optional field, but it can be used to define the default priority of all the pods. If there are no priority class name explicitly defined in that pod's configuration file. So as such. Also note that it can only be defined in a single priority class, so the global default property can only be defined in a single priority class because you cannot have multiple default values, right? So you cannot have more than one priority class with the global default property set to true. Now let us look at the effect of pod priority. So say there are two workloads with two priorities coming in to be scheduled a critical app with a higher priority of seven, and a jobs app with a lower priority of five. Now, since the critical app is of higher priority, it gets, uh, placed first. And if there is still resources available. And there are no more resources available on the cluster. What happens? Do we evict existing workload or does it wait? What does it do? Now that behavior is defined by the preemption policy defined in the priority class assigned to the new workload. If the preemption policy is not set, its default value is set to preempt a lower priority. This means that it would kill the existing lower priority job and take its place. But if you do not want it to kill or evict the existing workload, and instead wait for the cluster resources to free up, then you must set this preemption policy to never. So a preemption policy set to never makes the pods in that priority class non preempting, meaning that they cannot preempt other pods and it will wait to be scheduled in the scheduling queue. However, they will get a higher priority in scheduling over other lower priority pods that are also waiting to be scheduled. So there's the priority still applies only when it is going to be scheduled. But will it kill the existing preempt, or will it evict the existing workload or not?
