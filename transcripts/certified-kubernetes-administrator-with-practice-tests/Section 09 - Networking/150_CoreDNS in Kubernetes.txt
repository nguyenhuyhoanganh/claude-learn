# CoreDNS in Kubernetes

In this lecture we will discuss how Kubernetes implements DNS in the cluster. In the previous lecture, we saw how you can address a service or pod from another pod. In this lecture we will see how Kubernetes makes that possible. Say you were given two pods with two IP addresses. How would you do it? Based on what we learned in the prerequisite lectures on DNS? An easy way to get them to resolve each other is to add an entry into each of their Etsy hosts file on the first pod. I would say the second pod web is at 10.244.2.5. And on the second pod. I would say the first pod test is at 10.244 .1.5. But of course, when you have thousands of pods in the cluster and hundreds of them being created and deleted every minute, this is not a suitable solution. So we moved these entries into a central DNS server. We then point these pods to the DNS server by adding an entry into their Etsy resolv.conf file, specifying that the Nameserver is at the IP address of the DNS server, which happens to be 1.96.0.1. In this case, every time a new pod is created, we add a record in the DNS server for that pod so that other pods can access the new pod and configure the resolv.conf file in the pod to point to the DNS server, so that the new pod can resolve other pods in the cluster. This is kind of how Kubernetes does it, except that it does not create similar entries for pods to map pod name to its IP address, as we have seen in the previous lecture. It does that for services for pods. It forms hostnames by replacing dots with dashes in the IP address of the pod. Kubernetes implements DNS in the same way it deploys a DNS server within the cluster. Prior to version 1.12, the DNS server implemented by Kubernetes was known as kube DNS.
