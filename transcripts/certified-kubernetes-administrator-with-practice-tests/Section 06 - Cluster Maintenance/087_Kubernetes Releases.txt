# Kubernetes Releases

In this lecture we will see how Kubernetes manages software releases. So what do we know about API versions in Kubernetes so far? We know that when we install a Kubernetes cluster, we install a specific version of Kubernetes. We can see that when we run the kubectl get nodes command. In this case it's version 1.33.0. Now let's take a closer look at that version number. The Kubernetes release versions consists of three parts. The first is the major version, followed by the minor version and then the patch version. Now, while minor versions are released every few months with new features and functionalities, patches are released more often with critical bug fixes. Now, just like many other popular applications out there, Kubernetes follows a standard software release versioning procedure. Every few months it comes out with new features and functionalities through a minor release. The first major version, version 1.0, was released in July of 2015, and later releases followed through 2016 to 2018, to version 1.20 in 2020 and version 1.33 in April 2025. Now, apart from these, you will also see Alpha and beta releases. All the bug fixes and improvements first go into Alpha release. Tagged Alpha. In this release, the features are disabled by default and may be buggy. And then from there they make their way to beta release where the code is well tested. The new features are enabled by default in this release, and finally they make their way to the main stable release. You can find all the releases in the releases page of the Kubernetes um repository or the documentation page. You download the Kubernetes dot tar dot js file and extract it to find executables for all the different components. The downloaded package, when extracted, has all the control plane components in it, all of them of the same version. Now remember that there are other components within the control plane that do not have the same version numbers. The etcd cluster and the core DNS servers have their own versions as their separate projects. The release notes of each release provides information about the supported versions of dependent applications, like etcd and core DNS. Check out the links for some references. So that was a quick overview of the release versions. In the upcoming lecture, we will discuss how to upgrade from one version to another.
