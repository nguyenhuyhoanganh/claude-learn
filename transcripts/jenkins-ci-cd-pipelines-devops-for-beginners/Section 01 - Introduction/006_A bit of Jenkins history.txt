# A bit of Jenkins history

Let's briefly clarify why Jenkins has freestyle jobs and pipeline jobs, and why do we need them. And what you need to know is that initially, Jenkins only supported freestyle jobs, so they were just simply called jobs. And these are jobs that are configured through the UI. So essentially you would go and click ourselves and make different configurations here and to get these jobs to run. Now however, over time, as the needs of CI CD evolved, the limitations of freestyle jobs became more apparent and essentially newer and better approaches emerged. Now, as I said, freestyle jobs are configured through the Jenkins UI, which means that this job configuration isn't actually stored as code. It is stored somewhere internally in Jenkins. Now why this is a problem. One of the problems with the freestyle approach is that this configuration is impossible or very difficult to track. For example, you may have someone coming and making changes to this job, and there is no way to keep track of which changes have been made, who has made the changes, and so on. You may end up with jobs that don't work anymore and you don't know exactly what's the problem. Has someone changed something about the job or is it some other reason while the job is no longer working? Now this contrast a bit with the pipeline as code approach and principle, which is actually this other type of job that we have created, where if you are taking a look at it, there's still a few configurations that you can make, but there's almost nothing in compared to what the freestyle job allows you to do. So the idea here, with this approach, with a pipeline style, is that the entire configuration is stored as code. And when we're storing the pipeline configuration as code, it means that we can keep track of changes like we do when we're writing any other code. Now, apart from this, modern CI, CD often requires complex workflows with parallel execution, different stages, and, generally speaking, a more sophisticated flow control. Freestyle jobs have a more linear execution plan, and quite often, to get the freestyle jobs to do what you need them to do, you need to install a lot of plugins. I mentioned the beginning. Jenkins is heavily plugin based, so with freestyle, jobs really need a lot of plugins. You need to research which plugins you need, you need to add them, you need to configure them and so on. So in a nutshell, today we should no longer create Jenkins jobs using the freestyle approach. Now the freestyle approach is absolutely fine. You can quickly get something. Running, doesn't require advanced knowledge about jobs and many other things, and they're okay. But this is not what we want to use nowadays in terms of DevOps and, you know, following modern principles. Organizations out there may still use freestyle jobs, and that's okay because they are like legacy jobs. And typically you could still work with them. You can still manage them. But if you want to create new jobs, the goal of this course is to teach you the most up to date practices. So this is why in this course, we will not be focusing on freestyle jobs. We'll only be focusing on pipeline jobs.
