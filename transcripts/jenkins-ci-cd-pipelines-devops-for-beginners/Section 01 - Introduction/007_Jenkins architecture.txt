# Jenkins architecture

So what is really happening behind the scenes when we run a Jenkins job? Let's talk for a minute about the Jenkins architecture. At a minimum, the Jenkins architecture contains the Jenkins server, which is also called a controller, and a Jenkins agent, sometimes also called a node. The Jenkins controller manages the execution of the pipeline and saves the results. The controller knows what needs to be done, but does not do it itself. When a stage needs to be executed, it will find a Jenkins agent to actually run the job. It is like in the James Bond movies. MI6 is managing the operation, but a poor James Bond needs to be on the field and do the actual work. So a Jenkins agent is a program that executes a stage. A working Jenkins setup must have at least one agent, but they are often more of them to help distribute the load. The agent will retrieve a set of instructions from the controller. Run all the commands specified in the stage and then report back the result of the execution to the controller. In our simple Jenkins installation. The same instance will play both the roles of the controller and the agent. But this is something that is typically an anti-pattern for production systems. We really don't want to overwhelm the controller with work. So hope this overview helps you understand how things are intended to work.
