# Deployment Strategies: Why use a staging environment?

After our project has gone through the stages of build and test, we have automated the deployment to Netlify. Let's assume that the current address where we can view our website is the production environment. This means it's the address that leads to the latest version of the website, the version that the whole world can see. But what if we make a mistake and there is an issue with the deployment? The production end to end tests will inform us, but it might be already too late. Customers could have already seen our broken website, which would be embarrassing. So is there a way to test the deployment before actually deploying? Don't worry, this isn't a philosophical question or a chicken and egg problem. To test the deployment. We typically use the same commands that we would use for the production environment, but apply them instead to a non-production environment. A non-production environment is an environment used internally for development purposes. Quite often pipelines will have a staging environment. So what? The staging environment. A staging environment is a non-production, usually non-public environment that is very similar to the actual production environment actually so similar that it should be identical with the only exception that it is typically not public. We often use automation to create these environments and ensure they are nearly identical to production. This practice is known as infrastructure as code, and essentially eliminates the need of any manual intervention to create this environment. We use a staging environment as a pre-production environment. Pre-production means before the production, before the live environment. Essentially, we try out our deployment on the pre-production system. If it fails, the production environment is unaffected and we have plenty of time to troubleshoot the problem and figure out what's going on. And if it works, there is a very, very good chance that the production deployment will also succeed.
