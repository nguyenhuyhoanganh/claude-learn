# The past and future of Jenkins

Jenkins is by no means a new tool. I've been using Jenkins for over a decade now. The Jenkins project was started in 2004, and back then it was called Hudson. In its early days, Jenkins quickly gained popularity due to its ability to automate the build and development process, which was a game changer for software development teams back then. It offered extensive plugin support, allowing developers to customize and extend its capabilities to fit their specific needs. Jenkins provided a reliable and flexible solution for CI CD, which made it a staple in many development environments. While today Jenkins remains a powerful and flexible tool, it is important to note that other CI CD tools have emerged offering a more modern experience. You can say that they took what was working in Jenkins and fixed what didn't. Platforms like GitLab and GitHub have built in CI CD features that are directly integrated with git repositories. This integration provides a seamless experience, as developers can manage their code and the development pipeline within the same platform. They also better integrate with Docker containers, which can be used much, much easier in the pipeline. Security and maintenance are other aspects to consider. Honestly, how many times throughout this course were you prompted to update Jenkins or its plugins to do security issues? In terms of security, GitLab and GitHub have better measures in place as they are managed platforms. The company behind them regularly updates the platform and fixes any security issues without you needing to take action or even knowing about this. It is true that managed platforms can be expanded with plugins, but from my experience, I personally prefer a streamlined development process over the flexibility that plugins offer. The simplicity and efficiency of working with a single, unified platform often outweighs the potential benefits of extensive customization and all the security implications. In terms of the security of your own application. Both GitLab and GitHub offer features like easy to set up, security scanning, dependency analysis, and enhanced secret detection, providing an additional layer of protection for your code. Moreover, these platforms offer a more intuitive, modern, and user friendly interface compared to Jenkins. They allow developers to easily set up and manage their pipelines using YAML configuration files. This has the advantage of being a bit more readable and easier to manage than Jenkins groovy based pipeline script. So while Jenkins continues to be a reliable tool. Platforms like GitLab and GitHub are offering a more modern, secure and integrated CI CD experience. They present the current generation of DevOps tools perfectly suited for modern development workflows. I definitely recommend dedicating time to learning at least one of these platforms. I've added links to short tutorials in the resources folder. And if you think I'm throwing dirt at Jenkins, I want to start by saying that I have great respect for Jenkins. I love Jenkins, and I want to express a big thank you to the incredible team and all the contributors to the Jenkins project. Jenkins is truly an open source project with an amazing community. Many people have volunteered countless hours to make Jenkins better despite some issues. Jenkins is still an amazing tool and a powerful workhorse in many organizations. Keep in mind that Jenkins has been created many years before GitLab or GitHub existed. So I'm just encouraging you to broaden your perspective and see what else is out there. Therefore, investing time in learning and experimenting with these platforms can significantly benefit your DevOps learning path.
