# Revisiting the Jenkins architecture

Let's revisit the Jenkins architecture and better understand what is happening behind the scenes when we run a Jenkins stage that uses a Docker image. The Jenkins server, which is called a controller, will find an agent suitable to run the stage. This means that agent needs to be configured to use Docker. The controller will clone the git repository and store all the files in the workspace. Next, the agent will start the Docker container and map the workspace directory with a Docker container. This is super important because a it ensures the container has access to the project files, and b it ensures that any workspace files created or modified by the container will be available in the workspace, even after the execution of the stage has completed. Now, the main reason why we use Docker containers is to be able to use a specific tool, or a set of tools, or a specific build environment. In our case, we are using a NodeJS runtime to build the project. Once all the commands have been executed, the Docker container is terminated and Jenkins continues with the rest of the pipeline. Now, this is why previously I said that it's super important that the files are mapped between the container and the workspace, because once the container is terminated, we no longer have access to it. So having this file synchronization is super important. We can also take a look inside the logs in Jenkins and see exactly how this looks like. So for example here we're going to see that the first part is essentially cloning the git repository. And only then is the stage being started. So you're going to see here that the agent will have this docker run command. So this is when the container is being started. Essentially here it's checking that this container is running. Then the commands will be executed. And then here towards the end we're going to have the command docker stop. So the container is being stopped. And then any volumes created by this container will be removed. So hope this overview helps understand how things are intended to work when we're using Jenkins with Docker containers.
