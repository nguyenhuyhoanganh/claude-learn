# How to structure a pipeline?

For this lecture, I've enabled here the build stage so that we can have a discussion about how to structure a pipeline. So why does this pipeline look as it is right now? The way I've structured this pipeline is just an example. There are no fixed rules. Typically we start with build and then we test, and then we move on to other steps. But there are various considerations that we need to take into account. Different programming languages and technologies may require a different order. You are smart people, and I'm sure that you will be able to apply these ideas to whatever you are doing. However, I just want to discuss at least two guiding principles or two general ideas that you should always consider. First of all, are dependencies between the stages. So what do I mean by that? So for example, here in the build stage we are doing two things. We're not just building the project, but we are also installing dependencies. For example here to be able to run the unit tests we also need a project dependencies. We don't need the build artifacts. So there's no direct connection between the build artifact and the unit tests. They can run in parallel. There's no problem about that. But if we would run the build stage and the unit test stage, both of them in parallel, both of them will need to install the dependencies of the project. And this, of course, may take significantly more time than maybe running them one after the other. Apart from this, there are also direct dependencies. Here we are building something and then we're taking that build artifact and running the end to end test. So for this scenario we really need the build artifact. We cannot run this test before we have executed the build. So these are some examples of dependencies between them. Now if stages have dependencies then they need to be executed one after the other. We cannot run all this in parallel because we have dependencies between them. If something has a dependency to something else, it cannot be executed in parallel. For example, the build stage and the end to end test. They cannot be executed in parallel because there is a dependency, and when tests first need to build artifact to be able to start the server and run those tests. But in our case, the unit tests and end to end tests, they have no dependencies between them. So it is absolutely fine to run this in parallel. So we have established that there are dependencies between the stages and we need to get our things together. So we need to understand what depends on what. And based on that we can structure our pipeline. But apart from this, there are also other guiding principles we need to take into account. For example, a principle would be to fail fast. So what do I mean to fail fast? Generally speaking, and mostly through experience, we need to understand what are the most common reasons why a pipeline would fail. So, for example, if we know that our unit tests, they tend to fail a lot, we could simply first run the unit test, install the dependencies, and only then run the build. It really depends which is faster. Running the unit tests this only takes a few seconds or running the build. This takes maybe one minute. Installing dependencies is also part, so the build probably will be faster than, but the unit test will be slower. Various considerations, but the main idea is that we want to fail fast because this saves us resources. If we're spending a lot of time running the build, and then only then we notice, oh, actually, the unit tests are not working. We have wasted a lot of time building something that we don't need. Apart from this, just because we can run things in parallel, it doesn't mean that we should run everything in parallel. For example, let's say that the unit tests they can be executed in 15 seconds, and the end to end test can be executed in five minutes or maybe longer. So there's clearly a big difference between them. We can run them in parallel. But for example, if the unit tests fail after 15 seconds, the end to end test will still continue running. So that again will consume a lot of resources. If, for example, we move the unit tests outside of the parallel execution and run only them individually because they only take 15 seconds and they fail, then we don't need to start the end to end test and wait 15 minutes of processing time on our Jenkins instance just for nothing, essentially. Another example is linting. So linting is something that you typically want to do as fast as possible. If you're not familiar with linting, linting is essentially done by a tool which is called linter, and that linter will check the code for stylistic or programming errors, not necessarily in the sense of a unit test, but you would take a look at the code and see, oh, this is actually not a good idea to write your code like this, which is essentially like a static analysis. These linters are super fast, and they are generally considered like the first step in terms of quality assurance of the code. We haven't added a linter here, but a lot of projects have a linter, and it's pretty much executed in a similar fashion. How we run the unit test or something like that. So having a linter early in the pipeline is super useful, because if you have a linter later down and will go through the build and redo other tests, and then the linter will say, hey, you have forgotten a comma somewhere, this is doesn't adhere to the kind of stylistic standards that we want our code to be. So for that reason, again, wasted a lot of capacity just to have that linter very, very late in the pipeline. Now I'm going to end by saying that there is no replacement for experimentation and trying things on your own. Just try to give you here some general ideas so that you understand why we're doing things in a specific way. But if you're unsure if something works or not, just give it a try. See what is happening. See what are the impacts of the different changes, because that's the best way to learn. Doesn't mean that this is the only way to build this pipeline. You need to try it on your own and see if there are ways to optimize it. In the end, we want to have a pipeline that doesn't take a lot of time to run, but that is also efficient for the most cases that it will fail. How likely it is to fail because some stylistic issues, because someone has forgotten a comma compared to running the full blown end to end test.
