# Tagging the Docker image

In this lecture, I want you to talk about tagging a Docker image and to understand why tagging is so important. Currently, every time we're building our image, we just name it My Jenkins app and it's going to override any previous build. And with our Jenkins pipeline, we're going to create a lot of builds. And what we want with Docker Image is to be able to uniquely identify different Docker images that we have created, because we may want to deploy a different version of the application. And it also helps us manage updates rollbacks. In a nutshell, we need to keep track of what we are building. So for that reason, in this lecture, we're going to see how we can properly tag our images. And what do I mean by properly tag? Well, our image already has a name, but it doesn't really have a version. And in our configuration here we already have for example, this variable react app version. And all we have to do is copy this And after the name of the app in the right column dollar sign. And we're going to reference this variable. So now our tag is going to be much more specific because it gives the name of the application. And it also gives the version of the application we'll be able to identify which build has created which image. Another thing that I don't like is to have within commands configuration. So what we're going to try to do is define here a new variable. I'm going to call this variable app name. So our app name let's say it is Learn Jenkins app. So what we want to do next is use the app name and reference it here. So I did a rename from my Jenkins app to what do I have here. Learn Jenkins app doesn't really matter so much, but you understand the idea. We don't want to have configuration here. We are using environment variables and now we are better tagging this. Let's go ahead and run the pipeline and see if it works. The image has been successfully built. Let's take a look at the logs to see what is different this time. Expand the last command, and what we're going to see here is that there are no errors, so everything is working properly. Let's zoom in a bit because that's a super important part here. This is a variable. And it has been replaced here with the name of the application. And here is what we would call actually the tag. So now we can uniquely identify this image because it has here also the build number. And if we're taking a look at the builds that we have we're going to see here that this is the build number 109. So now there's a direct correlation between the build number and the Docker image that we have. So now we know from this build we're going to deploy the Docker image tagged 1.0. 109. And we can exactly identify the build that has generated that image.
