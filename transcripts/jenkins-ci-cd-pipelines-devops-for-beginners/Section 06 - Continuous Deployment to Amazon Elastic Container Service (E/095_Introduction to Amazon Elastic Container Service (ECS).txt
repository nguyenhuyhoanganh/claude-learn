# Introduction to Amazon Elastic Container Service (ECS)

When we use a cloud provider like AWS, we can rent virtual machines that have a dedicated CPU, memory, and disk storage, and we can use any operating system we desire. The underlying service that makes it possible in the AWS world is EC2. But this also means that we are in charge of managing that machine. We need to ensure that it is secure and that all software running is regularly updated. This can include operating system applications, libraries and so on. This is often too much overhead, especially for running some types of applications. But there is a way to take away this complexity and focus only on the application we want to deploy. The Amazon Elastic Container Service, or simply ECS, allows us to deploy and manage an application running in a container in the AWS cloud without us having to worry about the actual virtual server that runs the application. ECS can be configured as a fully managed container service, meaning that AWS handles the setup, maintenance, and scaling of the environment needed to run our containerized application. This is a great way to reduce complexity, and is possibly one of the easiest ways to deploy a containerized application in the AWS cloud, since it works with Docker containers. ECS doesn't really care about the underlying technology, allowing us to run Python, Java, NodeJS, PHP and many other types of applications. So this really gives us a lot of flexibility. We will be running a web server application that serves our simple website files. So this time, instead of just uploading some files to AWS, we are providing the entire application, which is self-contained in a Docker container. I'm sure you will be amazed by how easy it is to run an application with ECS. However, at this point I need to warn you again about potential costs. While running this service for a few hours or days will probably only cost you a few cents. If you let this service run for a month, you may get some unexpected charges on your card. Even if you're not actively using a service, once you have created it, it uses resources in the cloud. So stop any services as soon as you're no longer using them. This is the point where I have to hit you with a few technical terms. But don't worry, we'll explain everything throughout. We will first create a cluster and then create a task definition and run a task in the cluster. Makes sense right? No, I'm just kidding. Let me explain a bit more. In ECS, a cluster is essentially a group of servers that run containerized applications. Think of them as a team of virtual servers working together to manage and deploy our container. We need a cluster to provide the resources and the environment where our containers can operate, ensuring they have the necessary compute, power, storage and networking to run smoothly. So why not use a single virtual server on EC2? You may ask? Using a single EC2 instance or application could mean that if that instance fails, for whatever reason, the application will go down, which of course affects reliability. Additionally, a single instance limits our ability to handle increased traffic or workload, again impacting scalability this time. A cluster, on the other hand, ensures that if one instance fails, others can keep our applications running, thus improving reliability and allowing us to easily add more instances to handle higher loads. Improving scalability. Now, once we have a cluster, we need to tell the cluster what to do to give it a job or a task. This is done with a task definition. A task definition in SQS is like a blueprint describing the application we want to run. It tells ECS how to run the containerized application, including details like which Docker image to use, how much CPU and memory to allocate, and allows us to configure environment variables, network settings and much, much more. So task definition is the way we tell ECS what it should do, and it provides all the necessary configuration information for ECS to correctly launch and manage the containers that make up our application. So once we have created the cluster and we have a task definition, we can run that task definition in the cluster essentially deploying our application. So just to recap we need a cluster and a task definition. We execute the task definition in the cluster putting the virtual servers to work meaning deploying and running our application.
