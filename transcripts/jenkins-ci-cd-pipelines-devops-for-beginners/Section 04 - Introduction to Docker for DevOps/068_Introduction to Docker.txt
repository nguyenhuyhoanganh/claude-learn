# Introduction to Docker

Quite often, virtual machines are used to run applications. A virtual machine, or simply VM, is a computer that runs entirely on software, like a computer inside another computer. This is why the term virtual. This way, a big server can host multiple virtual machines, and each VM thinks it is a standalone computer with dedicated computing power, memory, and storage. Each VM also has its own operating system, meaning that one VM can run Linux, while another one, windows VMs, run on top of a hypervisor like VMware, not directly on the host OS. The hypervisor manages the VMs and allows them to operate independently. Now that we know that smaller software components call them microservices, if you wish, have advantages over monoliths. This also creates the problem of having different requirements. If multiple teams work on different components, they might have different opinions on which technology to use for their particular component. While one team may choose to do a component in Java, another team might like to use Node.js as it just makes more sense for that component. So different software components will generally have different requirements. Some may require different and even conflicting versions of libraries or tools. What if a legacy component only runs with Java eight, and a newly developed component that wants to take advantage of the newly released features of Java uses Java version 22? So obviously they cannot really coexist on the same machine. If in the past you could run a monolith on a virtual machine, having a virtual machine for each component or microservice becomes hard to manage and expensive from a cost standpoint. A running a VM adds a lot of overhead as each VM has its own operating system. So let's bring Docker into our discussion. Docker uses the Linux container technology to provide an isolation of applications running inside a container. In plain terms, containers allow for multiple different applications to run on the same machine, but allowing each of them to have their own configuration and requirements in a similar way. How a virtual machine will do it, but with so much less overhead. Docker can also run on windows and macOS using lightweight virtual machines or windows native Hyper-V containers to facilitate Linux containers. So what's the real difference between a virtual machine and a container? A virtual machine has its own operating system, which requires additional resources to run. A container runs as a process in an isolated way inside the host operating system, and only consumes the resources that the respective application running inside the container needs. So the feeling inside the container is very similar to the one inside the virtual machine. I won't get into the details of what makes containers inside Linux possible, but remember the main takeaway from this. A container is simply an isolated process inside the host operating system, and it is very efficient to run an application inside the container. Another key difference comes from the virtualization technology, which is no longer needed. This leads to a much more effective usage of the hardware resources, and this is a very important factor that has led to the widespread use of containers that we know today. While container technology is not new, it has been traditionally quite complex and hard to use. Docker has introduced a container system that allows us to package the application and its dependencies. In our Jenkins pipeline, we have used Docker containers as they offer a very clean and reusable configuration. We don't need to worry about installing NodeJS or Python or any other programming languages and tools. We can just use a suitable Docker image which comes pre-configured. When talking about Docker, there are a few terms you need to be familiar with. The image holds the application files configuration and the dependencies. A container is created from a Docker image and is the running process which contains the application. Let me make an analogy so that you can better understand the two terms. Did you ever buy furniture from Ikea? An image is like the package with all the parts required and the instructions needed to build something. The container is the final product after you have followed the instructions given and assembled the furniture. So you can think of the image like a blueprint, and the container is what comes when you follow that blueprint and actually construct something. When using Docker. We also use a registry to store and share Docker images and to be able to retrieve them when needed. We do a push when we upload an image and a pull when we download an image. There can be private or public registries. The most well known public registry is Docker Hub, and is the registry from where Docker pulls images by default. So Docker provides a way to package, distribute and execute an application. It does not do the process isolation itself, but builds on top of the Linux container technology and makes it easier to use. Applications running inside the container don't affect other applications running on the same host and are isolated. Docker itself is a complex topic, and we cannot fully cover Docker in only a few lectures. To use Docker locally and learn a bit more, you need to have Docker desktop installed. I think Docker is one of the technologies that you need to start experimenting with and to add more know how about Docker as you go along?
