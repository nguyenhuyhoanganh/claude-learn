# Introduction to Orchestration Saga Pattern

Inside this new section, we are going to implement our Orchestration Saga pattern. Before we try to implement the code changes very quickly let's try to understand what is this Orchestration Saga pattern, how it is different from the Choreography Saga pattern. Inside Choreography Saga pattern we know all the microservices they are going to play a role inside the Saga pattern implementation, whereas inside the Orchestration Saga pattern, there will be a saga manager who is going to control the flow of the Saga pattern. I will explain these flow in few seconds, but before that, let's try to understand how the name Orchestration came for this pattern. This Orchestration name is inspired from the way an orchestra is conducted by a conductor who directs and coordinates the performance of the musicians to play in a harmony. Any time if you attend an orchestra, there will be a coordinator in the center of the stage who is going to move his hands continuously, which gives the instructions to the magicians who are playing the music very similarly here inside the Orchestration Saga pattern. A centralized orchestrator is going to present who will direct and sequence the calls of different services to ensure that the overall transaction is completed successfully, or rollback in the case of failure. In case if you are also wondering how the term Saga came, you can refer to this point. Saga term originally comes from storytelling and refers to a long story of heroic exploits, whereas inside the computing, the Saga pattern was named to represent a long running transaction composed of a sequence of operations. I hope you are clear how these names are derived, though they are not important technically, but sometimes people may ask you why this pattern is called as Orchestration or Choreography. I hope with all the details that we discussed right now, you are in a better position to answer these kind of questions. Next let's try to understand what we have inside these technical flow. As of now inside our bank application we know there are four different microservices, namely customer account, card and loans. Whenever we want to update the mobileNumber, the mobile number has to be updated in all the four microservices. Otherwise it has to be rolled back in all the four microservices in case of any runtime exception. While implementing these Orchestration Saga pattern, there will be a saga manager will be present. The Saga manager can be outside of the four microservices, or it can also present inside the service, which receives the initial request from the client. In our scenario, the customer microservice is going to expose an API to the client applications to accept the request that is going to update the mobileNumber in all the microservices. Since these customer microservice is accepting the original request inside this customer service only, I'm going to build a Saga manager. If you build the Saga manager outside of the customer service, then again there will be some network latency issues. While these customer service is trying to communicate with the Saga manager. What the Saga manager it is going to do is based upon the events that are being published. The Saga manager it is going to give instructions to the other microservices, whether to perform the update operation or whether to initiate an compensation transaction. We are going to implement the Saga manager with the help of Axon Framework. Since we already implemented CQRS and event sourcing with the help of Axon inside the same framework, we also have flexibility to implement Saga Manager, which is going to help us to implement the Orchestration Saga pattern. I hope you are clear about the Orchestration Saga pattern and what we are going to implement throughout this section. This time the complexity is more because inside our application we already implemented CQRS with event sourcing. So they'll be a writer database which stores the information using event sourcing. Similarly, there is a reader database anytime if an microservice want to update the mobileNumber first, it has to update inside the writer database, followed by inside the reader database. In the case of compensation transaction, the rollback needs to happen inside both of these databases. So inside the read database and write database, we need to roll back the mobileNumber to the previous mobileNumber. In the example of Choreography Saga pattern, since we don't have any CQRS pattern implemented, it was pretty easy because there is only a single database. But while implemented CWRS pattern, since we are going to have two separate databases, we need to be more cautious to make sure that the updates and rollbacks are happening in both of these databases.
