The very first approach is by implementing API composition pattern. We also saw the demo on how to implement this pattern with the help of Spring Cloud Gateway. And the next option that we have is, with the help of CQRS. By implementing CQRS pattern inside our microservices, we should be able to solve this challenge of cross service queries. Now inside this lecture, let's try to understand the next two challenges that we may face inside microservices environment whenever we adopt database per service pattern. These two challenges are around data consistency and complex transactions. I know you may have too many questions around these challenges. I'll try to explain you about these challenges by taking an example. Let's take the same e-commerce microservices as an example scenario. As part of my e-commerce microservices, I have three different microservices, namely Order microservice, Payment microservice and Inventory microservice. Let's imagine one of the end user trying to create an order inside my e-commerce web application, based upon the actions performed by the end user inside the UI my order service. It is going to receive a request to create a new order as part of the very first step. As a next step, you know what is going to happen. The business logic inside the createOrder API is going to be executed, and eventually the order will be created and stored inside the database. Once the createOrder operation is completed, this order service, it is going to call the next dependent microservice, which is paymentService. Because we can't create an order without payment, we need to make sure along with the order, the payment also is completed successfully. So that's why as part of the third step, as soon as my payment service receives this request from the order microservice, it is going to process the business logic present inside the deduct payment API. With that, the amount will be deducted from the end user wallet and the same will be stored inside my payment related database. Once the payment is successful as a next step, the payment service can call the inventory service to update the inventory details. Because the end user might have purchased some products, the same details needs to be updated inside the inventory or inside the stock, so that for all the future customers, we're going to show the correct availability details around the product. That's what it is going to happen as part of the step five and step six, the inventory details will be updated inside the database, which is owned by the inventory microservice. So this is a very happy scenario and you might be assuming what are the challenges that we have this seems to be very simple to me, but let me try to ask you some questions so that you will be clear more about these data consistency and the complex transactions challenge. Next let's try to understand what is complex transactions, any transaction that is going to travel, multiple microservices, such transactions, we can call them as complex transactions. If the request only processed within the single microservice, then it is not a complex transaction because we are always going to deal with a single database. But the moment you have a scenario where your request is going to travel across multiple microservices, then you're going to bring all the multiple databases into the picture. This kind of scenarios, we can call them as complex transactions. I'll explain in a few minutes why we are calling them as complex transactions. But before that, let me try to take one more example and try to explain the challenges. This time, let's imagine a bank application and my bank application has two different microservices the very first one is accounts followed by cards microservice. Think of a scenario where an end user has an account and a card inside this bank, and this end user received an email saying that so-and-so is your credit card bill. This end user, what he is going to do is, he will try to log in into the UI application and try to make the payment for the credit card bill during this operation first, the amount has to be deducted from these accounts service database. Once the amount is deducted as a third step, the request will be forwarded to the Card service to process the payment and adjust the outstanding amount inside the Cards database. If all these four steps they are completed successfully, then there is no problem. But think of a scenario the amount is deducted from the end user account. Maybe $100 has been deducted from the end user account, and during the third step or fourth step, the operation failed. And these $100, which is deducted from the accounts it never reached to the cards database. This is another classic example of data consistency issues, and we can also consider this as a complex transaction, because more than one service and more than one database is involved as part of this client request. I hope you are clear about these challenges, why these challenges are not going to arise inside a monolithic or a normal web application. It's a great question. Let's try to understand the same. Let's imagine you have a monolithic application. This can be a bank application or e-commerce application. It can be anything usually inside our monolithic application, we write all our business logic using multiple layers, like there'll be a UI layer, there will be service layer, and there will be a database layer. This entire application, it is going to point to the single database. Inside this single database we'll have various tables related to accounts and cards. This way we can have any number of tables. Think of a scenario where the request received to these monolithic application to make the payment of the credit card bill. Here what is going to happen? First, the balance will be deducted from the accounts table, followed by the same balance will be credited into the cards table. The beauty of monolithic application is since we are pointing to the same database, and since all the logic is being executed inside the same code base, we are going to execute the entire request or entire transaction as part of a single database transaction. So since all these multiple tables, they are present inside a same database, it is perfectly possible and easy to execute this entire flow as part of a single database transaction. Inside Spring Boot Framework, we have an annotation which @Transactional. As a developer, what we can do is we can mention this annotation on top of a service() method. Inside this service() method, I can make DB transactions related to accounts, for example, as part of the step one, I'm going to execute the operation on the accounts table. Similarly, as part of the second step, I'm going to execute the operation on top of a cards table. Since both these tables belongs to the same database. Whatever annotation that I have mentioned on top of the service() method, it is going to take care of executing both these two operations as part of a single database transaction. If there is an exception arises while updating the cards table, then the rollback operation is going to take care of rollback the data updates inside the accounts table as well. And it is going to be easy as well, because the commit operation is not going to happen until this entire method is executed. What Spring Boot is going to do is, since we have mentioned this annotation on top of the service() method, until all the DB operations completed inside the service() method, the database commit is not going to happen. But in the scenario of microservices, all the story is not going to be valid because accounts and cards database, they are completely different databases. Apart from different databases, the code also is being executed completely in different servers or in different nodes, or are in some scenarios it will be in different locations as well. That's why traditional style of transaction management for the database operations is not going to work inside microservices whenever we are adapting that database per service pattern, let's try to understand how to overcome this challenge. To overcome the challenges around the data consistency and complex transactions we have a pattern with the name Saga pattern you might have heard this many times from other senior developers. I'm not going to discuss in detail about this pattern right now, because there is a separate dedicated section for this pattern in the coming sections. For now, I'm going to give a brief introduction about this Saga pattern. What this Saga pattern is going to do is whenever a developer implemented the Saga pattern, it is going to give you the assurance to the developers that any data updates that we are going to make as part of the multiple distributed transactions, either they are successful altogether, even if one of the transaction in one of the microservices failed, then the rollback of the data is going to happen in all the previous microservices where the data has been updated.
