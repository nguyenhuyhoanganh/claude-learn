# Understanding microservices code and setup - Part 2

Let's try to explore what code I have written inside these microservices. If you have completed my microservices course, then most of the code will look familiar to you because I have copied the same code into these microservices. I'm not going to discuss anything around the Eureka server and gateway server, because I already discussed about them inside my microservices on how to create them and how to setup them. Let's try to understand what business logic I have written inside my microservices. If you go and open my accounts microservice, you'll be able to see these are all the packages under the audit package. I have audit related logic, which is going to help me to audit the created date, updatedDate, createdBy and updatedByColumns inside my database table. We have constants file, we have a controller. Inside the AccountsController there are multiple APIs. The create API is going to help me to create a brand new account inside my database, and fetch API it is going to help me to load the account details from the database based upon the mobile number. We are doing some validations also on top of the mobile number that we receive, it has to be a ten digit numerical value. After the fetch we have update API as well, which is going to help you to update the account details inside the database. At last we also have delete API, which is going to help you to solve to delete the account details inside the database. All the actual business logic we have written inside the ServiceImpl. I will explain the same logic in a few minutes. Before that, let's explore other packages under the DTO package you have all the Dtos, AccountsDto, ResponseDto under the entity you have BaseEntity and AccountsEntity which represents the accounts table inside the database. Under the exception package, you have the logic to handle the exceptions globally with the help of ControllerAdvice. This class, it is going to handle all the runtime exceptions and the business exceptions like ResourceNotFoundException and AccountAlreadyExistsException . Under the mapper, I have the logic to to transfer the data from DTO to entity and entity to DTO, repository you already know this is the repository, which is going to help me to perform the Crud operations on top of the accounts table. Now let me open the AccountServiceImpl. This is an important class. Inside this class only I have written all the business logic. During the create account operation. We are going to receive the mobile number from this mobile number first, we will try to check if there is an account already with the given mobile number and active switch as true. If there is such account, we are going to throw an business exception. Otherwise we are going to create a brand new account. I'm trying to invoke an private method with the help of this mobile number. Here I'm trying to generate a random account number followed by I'm trying to set the account type as savings so and so as branch address and active switch as true. At last, this entire new entity will be passed to the save method, which is going to insert a new record inside the database table. Next we have the fetchAccount() method, which is going to help you to fetch the account details based upon the given mobileNumber. But to whatever account you are trying to fetch, it has to be in active status. If someone deleted it and marked it as inactive, then you're going to get ResourceNotFoundException. After the fetchAccount(), I have updateAccount() method which is going to help us to update already existing account details. So first we are going to load the existing account from the database using the mobileNumber and the ACTIVE_SW as true. Once the account record is loaded, I'll try to map the new data from D2 to account inside this mapToAccounts() method. Inside this method, if you see I'm not giving flexibility to the end user to update whatever he wants, the end user cannot update the account number because account number once generated should not be updated and the mobileNumber also, I'm not allowing to update through this method, but in the coming sections we are going to write a separate method to update the mobile number. The reason is, we are going to maintain the same mobile number across all the microservices. If someone want to update the mobile number, they have to invoke a separate API, which is going to take care of updating the same mobile number across all microservices. With that, I'm going to allow only to update the account type and branch address as part of the update operation. Now let me go back to the AccountServiceImpl. Once this account mapper logic is executed, we are passing the same to the save() method which is going to take care of update operation. At last I have deleteAccount() method which is going to accept the account number as an input. This method first it will check if there is any such account number. If yes, it is going to mark the ACTIVE_SW as false so that the account details will be soft deleted. In real projects we should never try to hard delete the entire record. We should always soft delete them with the help of columns like Active Switch. I hope you are clear with the entire logic present under the AccountServiceImpl . Now let me go back to the package structure. Under the accounts application you'll be able to see this is a spring boot application. I have annotated the same. Next I also mentioned this annotation which is enabled JPA auditing to enable the auditing process. Under the Application.Yml, you'll be able to see port configuration, data source configuration. Here I want to share you about an interesting configuration around the h2 database, since I am going to use the h2 database, which is an in-memory database to store all my details. By default, whenever I perform restarts or shutdown, all the data that I stored will be wiped off. But if I want to retain that data even after restarts and shutdown, then I need to tell to my h2 database library to store all the data inside a file inside my system, so that whenever the restart is happening, the data will be read from the file that is stored inside my local hard disk. So this is the property that you need to give jdbc:h2:~file and this is the file name accounts semicolon followed by AUTO_SERVER = true. And this is the username and password. And I am trying to enable the h2 console. And there are some Eureka specific properties actuator related properties. And at last I also have a logging related property, which is going to help me to log the details onto the console by using various colors and the format that I want. I'll show you the benefit of these pattern in the coming lectures when we start all our microservices.
