# Course Introduction

Welcome to the course on event driven microservices with CQRS, Saga and Event Sourcing patterns. As microservices continue to gain popularity, we are facing many new challenges, especially around the data consistency, managing complex transactions across microservices and adapting to the high scalability demands. In traditional microservices handling all these issues can lead to tight coupled systems, and often we end up sacrificing the flexibility we are looking in microservices. This is where event driven microservices come into play and shine By designing our microservices to respond to the events, we allow each service to react independently and only when required. You might be thinking building event driven microservices is very easy. You just have to publish an event to the Kafka topic or to a RabbitMQ. And on the other side, you need to have a consumer microservice to consume these events. But believe me, my friend, building event driven microservices is more complex than you think. That's why inside this course we are going to focus on how to build the event driven microservices by taking various design patterns around the event driven architecture. My name is Madan and I'm thrilled to guide you throughout this course. Let's try to understand what exactly I'm going to cover throughout this course. Inside this course, we are going to focus on various design patterns. First, we are going to start with database per service pattern. Using this pattern, we are going to adopt a separate database for each microservice. Followed by we are going to focus on API composition pattern. Using this pattern, we can compose or aggregate the data from multiple microservices before sending back to the client application. After this pattern, we are going to move on to the one of the advanced pattern available inside the event driven microservices with the name CQRS. CQRS means command query responsibility segregation. By adopting this pattern inside our microservices, we are going to separate the write and read operations so that they can scale independently. Once we implemented CQRS pattern, we are going to explore the event sourcing pattern as well. Using event sourcing pattern, we are going to store each change to that data as an event, preserving a complete history inside that database. After event sourcing, we will move on to the materialized view pattern, which is going to help us to read that data from multiple microservices efficiently. After the materialized view pattern, we will move on to the transactional outbox pattern, which you can use in the scenarios where you are looking for reliable event publishing. Towards the end of the course, we are going to look into my favorite design pattern, which is Saga pattern. Inside Saga pattern we have two flavors, namely orchestration followed by choreography. Both these flavors we are going to discuss along with the examples. I hope you are clear with all the patterns that we are going to discuss throughout this course.
