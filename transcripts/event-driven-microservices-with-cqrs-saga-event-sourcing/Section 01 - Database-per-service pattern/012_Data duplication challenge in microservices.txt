# Data duplication challenge in microservices

The last challenge that we need to discuss is data duplication. Let's try to understand this challenge by taking an example. Let's imagine you have a bank application. And inside this bank application we have four different type of microservices namely Customer service, Account service, Card service and Loan service. In this kind of setup, the customer related data like the customer mobile number, customer address, customer email details, all the personal information of customer they are going to be stored and warned and maintained by the customer service inside the customer specific database. But sometimes other microservices like account service, card service, loan service, they also need some customer data to perform their day to day operations. For example, let's imagine a customer performed a transaction using his or her account. In such scenarios, the basic expectation is to trigger an email or to trigger an SMS to the end user, saying that you have performed so-and-so so transaction. And these are the details. But my accounts microservice does not know about the customer email and mobile number to trigger the notifications. The same kind of scenario will exist in cards and loans microservice as well. Let's imagine for one of the existing customer, they want to send new renewed credit card to his or her house. So to initiate this operation, definitely this card's microservice it needs the address details of the customer which are stored inside the customer database. You can imagine similar kind of scenario for loans as well. Here this problem can be solved in two different ways. The very first basic way is, whenever the accounts microservice are cards microservice or loans microservice. If they are looking for the customer specific data, what they can do is, they can ask the customer microservice by invoking its API to give the customer data that they are looking. But do you think this is a feasible option? It is going to bring lot many performance issues. Let's imagine this accounts microservice it is going to handle ten millions of transactions every month. So for all these 10 million transactions, if these account microservice end up calling the customer microservice, it is going to bring a lot of performance issues. Apart from performance issues, we are going to waste a lot of infrastructure and we are going to get lot of cloud bill as well. That's why to solve these kind of problem, what usually organizations they will do is, they will try to maintain the same data across multiple microservices. Let's imagine this customer data has lot of customer information. But these other microservice they only need three informations. The very first one is, mobileNumber followed by email and address details only these three details are required by all my microservices and they don't need any other customer data. So what these microservices they can do is, they can try to have these data stored locally inside their own database. So here inside the accounts database they can have a customer data and similarly inside cards and loads also they can maintain the customer related data. So here what is happening. We are duplicating the same customer data across multiple databases. Why we are doing this because right now we are having different databases for different microservices. Inside a monolithic application there is no problem since it is only a single database code or methods they can always look into the customer related table inside the same database and they can fetch the data. But it's not going to be the same story for the microservices scenario. That's why they will end up maintaining the duplicate data across all the databases. But the story not ended there let's imagine all these databases they have the customer mobile number as one, two, three. So I'm just trying to keep the mobile number to a simple number After a few days, the customer, they want to update their mobileNumber. So what they are going to do using the UI of the bank application, they are going to update their mobile number and that request it is going to handle by the customer service. And eventually the updated mobile number which is four, five, six, will be stored inside the customer database. But the same needs to be communicated to the other microservices as well. Otherwise you will end up having a lot of problems. The notifications will go to the old mobile number which is not correct. That's why let's try to understand how this updated mobile number will be communicated to all the places where the same mobile number is duplicated. We can solve this challenge by adopting the event driven architecture inside our microservices. When we implement the event driven architecture inside our microservices, what is going to happen is, whenever the customer microservice, it is trying to update the mobile number of the end user, it is going to publish the event to an message queue or to a Kafka topic. All the consumers who registered to this queue are to this Kafka topic they can read the message and they can try to update the mobile number inside their local copy. That's how the mobile number will be migrated to the latest value, which is four, five, six. off course, in this scenario the data will not be updated instantly. It is going to take few milliseconds or few seconds of time to be updated in all the remaining microservices. But here you may have a question, which is why can't we achieve this with the help of synchronous API calls? Why can't my customer microservice can trigger the APIs exposed by the other microservices to update their local mobile number. This can be a question, but it has some drawbacks, which is we are going to make our end user to wait unnecessarily for more time until this mobile number is updated in all the remaining microservices. It is going to bring some usability issues inside the bank application. That's why in most of the scenarios where the data needs to be duplicated in multiple microservices, developers will end up using the event driven architecture. Few companies they'll also leverage caching mechanism to avoid data duplication. What they're going to do is, they're going to write all these customer data into the cache, and these cache will be pointed by the individual microservices from where they are always going to read the customer data. But here also we have the challenge where we need to always keep the cache up to date based upon the updates happening on the customer database side. Regardless whether you store the data inside a cache or inside a actual database of the microservice, it is always advisable to use the event driven architecture to handle the challenge around the data duplication.
