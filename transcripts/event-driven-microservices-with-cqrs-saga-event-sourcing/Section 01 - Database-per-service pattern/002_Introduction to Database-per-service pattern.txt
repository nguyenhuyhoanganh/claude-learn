# Introduction to Database-per-service pattern

Inside this course, we are going to focus a lot around the event driven microservices patterns like CQRS, Saga, Event Sourcing and many others. But before we try to discuss and understand all these patterns in detail, first, as a microservice developer, you need to be very clear on why these design patterns will be used only in the microservices environment, but not in the traditional applications or monolithic applications. Have you ever saw someone using these patterns inside traditional web applications or monolithic applications? off course not. There is a very good reason on why these patterns will be used only in the microservices environment. Let's try to understand the same. This is the pattern which is the main reason or the culprit, which forces the microservice developers to adopt various event driven microservices patterns. What we are going to do is first, we'll try to understand what is this pattern, Why we can't avoid this pattern inside the microservice. And when we adopt this pattern, how we will end up using the event driven patterns. So we'll go in a flow so that it is going to be clear for you. What is a Database-per-service pattern? The name itself is self-explanatory. As part of this pattern, the microservice developers, they will end up using separate or individual databases for their microservices. As you can see on the screen, we have three different microservices. Let's imagine these three microservice belongs to an e-commerce application. Let's consider this Microservice1 belongs to the customer, and similarly microservice2 belongs to the order. And the third microservice belongs to the products. This way, e-commerce application can have any number of microservices focusing various components. To keep things simple, I have considered only three microservices focusing customer, order and products. Each of the microservice will have its own service logic. We all know inside the microservices, the business logic will be developed independently by different teams. That's the whole purpose of adopting microservices in any organization. If you closely observe behind the service logic each of the microservices, they have their own individual database. Whenever someone has this style of microservices where they have their own individual databases, then we can safely assume such organization they are adopting these Database-per-service pattern. 99% of the projects who are adopting the microservices, they will end up using these pattern. There are only some special scenarios or rare scenarios where an organization will try to use shared database between multiple microservices. Even if they are using shared database, they are not going to use the same database for all of their microservices. For example, if an organization has ten Microservices, maybe for three of them they are going to have the same database, whereas for the other seven microservices they are going to have separate, separate database. With this explanation, you may have a question which is what are the advantages that developers or organizations are going to get by adopting these Database-per-service pattern? Let's try to understand the same. By adopting these pattern organizations, they can enjoy all these benefits. The very first benefit is around the loose coupling. The whole reason of developing microservices is to achieve the loose coupling between the business components. If you take the same e-commerce application scenario when they build the microservices, which are having their own individual database, it is going to give flexibility to the team to develop and deploy the microservices individually without worrying about other business components. And the next advantage is around the independent scaling. Since each of the microservice is going to have its own database. It is going to give the flexibility to the microservice to independently scale the database based upon their data demands. If you take the e-commerce application scenario, we have three different services focusing customer, product, and order. And each of these services, they have their own database. Do you think all these microservices, they are going to have the same volume of data? off course not. The orders microservice it has to process huge volume of data because inside an e-commerce application, what is the most frequent activity an end user can create an order. After the order the products, they are going to have good amount of data. And at last customer microservice will have very less data. In this kind of scenario, what order microservice can do is it can scale its individual database to meet the demands of the end user. If all these three microservices, if they have a single database, then obviously it will be a very complex process to take a decision when to scale the database or when to scale down the database. But with the Database-per-service pattern, it will be always an easy decision around the scaling of the database. And the next advantage is around the faster development. Since each microservice will have its own database, the development team, they don't have to coordinate with other microservice developers around the database changes or the schema changes that they are going to do. Whereas if someone is not using this pattern and if they're using a shared database among all the microservices, then any time if someone wants to make a schema change and code change in such scenarios, they have to coordinate with all the other microservice developers. The same we can avoid here that's why we have this benefit, which is faster development. The next benefit is around the resilience and fault tolerance. Since each of the microservices have their own database, even if one of the database is not working properly or it is having some performance issues, it is not going to have impact on other microservices. For example, if the customer database is down, then other microservices like product and order, they are not going to have any impact. By following these database per service pattern, developers will have a freedom to choose different databases based upon the data that they are trying to handle. We have various kinds of databases inside the industry like SQL, NoSQL databases. In this scenario, we know that product microservice it has to store lot many images related to various products. To store all those images inside a database then obviously someone can choose the NoSQL database compared to the relational database, whereas for the customer and order they can go happily with the relational database like MySQL, Oracle, Postgres. Apart from the type of the data developers, they can also choose the database based upon the type of operations that they are going to perform. If one microservice, it is going to do a lot of read operations compared to the write operations. Then developers, they are free to choose a database which is more optimized for the read operations. I hope you are clear. And the last advantage is around the unauthorized access by other microservices. With this kind of setup, whenever the customer microservice want to read the orders information, it can't directly read the data from the orders database. What it has to do, it has to invoke the APIs exposed by the order microservice to fetch or read the orders related data. With this, we are always going to have some authentication and authorization mechanisms to enforce proper security while a service is trying to read another service data. With this, you might be assuming wow, this is a great pattern it has lot many advantages. Why anyone want to avoid this pattern? This can be your assumption, but don't go with that assumption. Because while this pattern brings a lot of advantages onto the table, it also brings some serious challenges or disadvantages onto the table. Let's try to understand what are those challenges. Here I have mentioned the most important challenges that any microservice developer will face while adopting the Database-per-service pattern. These challenges, including cross service queries, data consistency, complex transactions, and data duplication. We are going to discuss all these challenges in detail in the coming lectures. For now, I just wanted to give you a clue around these challenges. Inside a traditional web applications or monolithic applications, all your data is going to be stored in a single database. But now inside microservices where they are trying to adopt these database per service pattern, we will end up storing the data across multiple databases. Since we are trying to store and read the data from the multiple databases. We are going to get a good amount of challenges around the transaction management. Have you ever heard about distributed transactions? By adopting these database per service pattern, we will end up having distributed transactions inside our microservice. Whenever you want to write a data for a specific scenario, there is a very good chance that you will end up writing the data across multiple microservices database. Since your transaction is being distributed among the multiple databases, it is going to bring some serious challenges around the transaction management, rollback, exception handling, committing of the transaction. As a developer, we need to understand all these challenges very clearly. Once we understand all these challenges, we are also going to adopt and implement various design patterns to handle these challenges. I hope you are clear. Thank you and I'll catch you in the next lecture, bye.
