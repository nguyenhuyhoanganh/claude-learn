# Introduction to API composition pattern

As I said, API composition pattern handles the query operations by calling the services that own the required data and then combining their responses. Here you need to make a very important point. You need to use this pattern only during the read or query operations. This pattern is not suitable for the data manipulation operations, like if you are trying to update or delete or insert the data across multiple services, then this service is not recommended. We should go for the complex or advanced pattern which is CQRS. That's why I have highlighted here. This pattern handles the query operations, but not any data manipulation operations. So this pattern is nothing to do with the event driven microservices. But I'm trying to give some information here. off course there is a better pattern which is CQRS which we can use, but CQRS is very complex compared to the API composition pattern. If you are going to work in a smaller project or a smaller organization where they don't have flexibility to implement a CQRS pattern, then in such scenarios, you can safely implement API composition pattern for the read operations. That's why I'm trying to put some efforts here to make you understand about all the possible options that you have. Inside this pattern, there will be two components, namely API composer and the Provider Service. This API composer is a component which owns the operation. If some client application, if they want to read the data from multiple microservices, what they are going to do is, they are going to send the request to these API composer, this API composer component, it is going to take the responsibility of interacting with all the dependent services or provider services. Who owns a pieces of data? Once all these provider services respond with the given data, this API composer, it is going to compose the entire response and send back to the client application. After the API composer, we have another component with the name provider service. This is the service responsible for managing part of the data returned by the query. For example, in this bank scenario, all these microservices, we can assume them as provider services because they own a part of data which is required by the client application. With this introduction, you may have a question which is who is going to act as a API composer? Let's try to answer the same. Whenever you are trying to implement this API composition pattern, you will end up having three different common options. The very first one is, the client itself will act as a API composer here. For example, there can be an UI application. This UI application can make individual microservice calls and the client will have the logic of composing or aggregating the entire data before displaying onto the UI page. But I don't recommend this approach because we should never expose our actual microservice endpoint details to the client application. We should always have a single entry point into the microservice, which we can build with the help of API gateways. Inside my microservices course, there is a dedicated section about Spring Cloud Gateway, using which we can easily build an edge server or an API gateway inside microservices environment. I hope you recall all the Spring Cloud Gateway discussion. The second option is the API gateway can act as an API composer, and the last option is a standalone service or an existing service can act as an API composer. I know you may have question around the option three. Let me share more details. We have four different microservices inside the bank application. Here what a client can do is, a client can call the API exposed from the customer microservice. This API name can be customer summary. So this customer service, what it is going to do is, it is going to make the individual calls to all the dependent microservices and the data aggregation or the composition is going to happen at the customer service level. Here the API gateway is not playing the role of API composer. One of the microservice itself is playing the role of API composer. I hope now you are clear about the third option from these three options my recommendation is always to go with the second option. It is always a best practice to use API gateway as an API composer. Inside Spring ecosystem, we can build the API gateway with the help of Spring Cloud Gateway. So these gateway server, it is going to act as an edge server. When we implement this pattern with the help of Spring Cloud Gateway, we are going to get a lot of performance advantages. Since Spring Cloud Gateway is built with the help of reactive programming model, we can make the dependent service invocations parallelly with the minimum number of threads and less memory. I'm going to show you this in the demo in the coming lectures. I hope you are clear with this pattern, but this pattern is not going to be your silver bullet. This pattern also has some disadvantages. Let's try to understand these disadvantages so that you can always take an informed decision based upon your business requirements inside your project. The very first drawback is increased latency, since a single component is going to make a call to all the remaining microservices or the network to fetch the data, the more number of services you are going to invoke to fetch the data, the more latency you are going to introduce inside your microservices network. And the next disadvantage is the logic of composing the results can become complex. Sometimes clients may want data in a different format compared to what you get from the microservices. In such scenarios, you will end up writing a lot of complex logic to convert the data or to transform the data as per the client requirements. Error handling will be the next challenge when you are making multiple microservices invocations from ,a single component, few of them will fail, few of them will not respond, few of them will respond slowly. You need to build retry logic. All these error scenarios you need to handle inside your API composer. The next disadvantage is around the data consistency. Since the data of a customer is spread across multiple microservices, there can be some data mismatches. For example, think like inside your customer, accounts, loans and cards microservice you are storing the customer mobile number separately whenever a customer updates a mobile number. The mobile number may get updated quickly in few of the microservices, and it may get delayed in other microservices. So all such data consistencies will come into picture whenever you try to adopt these ,API composition pattern. After data consistency, we have limited scalability disadvantage since API composer is an entry point to fetch the data from multiple microservices, regardless of how many instances of accounts, loans and cards, and customer microservices you have. Sometimes your API composer will become a performance bottleneck, which will bring some limited scalability issues. And the next disadvantage is, self-explanatory. You're going to have dependency on service availability. If you are looking to read the data from four different microservices, if one of the microservice is not available, then you need to handle such scenarios to send either some null information or empty information. All such native scenarios you should be handling and at last, testing also is going to be a challenge whenever you try to implement these API composition pattern. If you are okay to handle all these challenges, then you can happily use this pattern. I would recommend this pattern only for the smaller applications or low severity applications for enterprise applications, or for the high severity applications. The recommendation is to use the CQRS which you are going to discuss in the coming sections. As you can see, we have the API gateway or Spring Cloud Gateway, which is going to act as an edge server for our microservices. The client applications, they will always call an API,  which we exposed on the gateway. And this gateway is going to take care of invoking all the microservices individually and combining the data here. And the combined or the aggregated data will send back to the client application. These invocations to the multiple microservices can be made parallelly based upon your business requirements. If there is no dependency between these microservices data, then your gateway can make these calls parallelly with the help of reactive programming. When you make the API calls parallelly, you're going to get some performance benefits. As you can see here, I have changed the title of the slide. Whenever you are implementing the API composition pattern with the help of API Gateway or Spring Cloud Gateway, such pattern we can call it as Gateway Aggregator Pattern or Gateway Composition Pattern.
