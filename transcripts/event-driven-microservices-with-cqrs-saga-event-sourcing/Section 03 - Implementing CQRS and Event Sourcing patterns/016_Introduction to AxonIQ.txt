# Introduction to AxonIQ

We want to implement CQRS and Event sourcing patterns inside our microservices. Building these patterns from scratch is a very complicated task, but inside this course, we are going to take a much simpler approach because at the end of the day, how you implement these patterns does not matter. The knowledge that you gain on these patterns is going to matter inside your real projects. Different organizations may have different approaches in implementing these patterns. That's why my humble request is try to understand these patterns. What are the differences, what are the advantages that these patterns are bringing onto the table inside our microservices environment? Let's try to break down on how we are going to implement these patterns inside our microservices. As of now, inside our microservices, all type of operations we are trying to carry out with a single database. But when we try to implement CQRS pattern, the very first task that we need to complete is we need to separate our APIs into write APIs and read APIs so that anyone who wants to alter the data, they are going to invoke the write APIs, and the change will eventually go back to these right databases. Whereas if someone wants to read the data, they're going to invoke the read APIs, and the data will be fetched from the read database. When we want to write that data into the right database, we are going to follow the Event sourcing, which means as part of the step two, we need to enable Event sourcing. While we are trying to write the data into the right database, all the transactions that we are going to perform on a single record, they are going to be stored in an historical style or in an audit style. So this is the second step as part of the next step, whenever a change happens on the right database side, we need to make sure we are triggering an event to the event bus. So there is an event bus which is going to sit in the middle as soon as an event is published into this event bus. From the read side, we need to consume the events. Once this event is consumed, we need to update the data inside the relational database. The same will be used by the read APIs whenever someone is trying to query the data. So these are the most important steps that we need to take care while implementing the CQRS pattern and the event sourcing pattern. It may sound very complex, but with the approach that we are going to take, it is going to be super simple. Before I try to introduce the approach that we are going to take, first, let me share a real life analogy that you can always relate with the access pattern. Have you ever went to a restaurant, for example, inside a pizza restaurant will be having two kinds of people. The very first kind of person you will be able to see inside the restaurant is chef. Chef is the person who is responsible to prepare the pizza. The chef is going to make the pizzas. Whereas on the other side, we are going to have persons who are going to take order from the customers. This person is going to take order from customers. If you closely look at this restaurant operation, the responsibilities are segregated into two different kinds. One is making the pizzas, the other one is delivering the pizzas to the customer by taking their order once they enter into the restaurant, off course, for a very small restaurant where they are going to get very limited number of customers, then definitely the same single person can play the both roles, like making the pizza and taking the order. But for bigger restaurants, it is really impossible by single person to manage everything. The same applies for our enterprise microservices as well. Sometimes we'll be having scenarios where implementing traditional microservices may not be an solution. So for such kind of scenarios, we are going to follow the CQRS approach, just like how we have two different kinds of persons inside a restaurant. Similarly, inside our microservices we are going to have command side and query side. I hope you are clear. Please always remember this analogy to remember CQRS forever.
