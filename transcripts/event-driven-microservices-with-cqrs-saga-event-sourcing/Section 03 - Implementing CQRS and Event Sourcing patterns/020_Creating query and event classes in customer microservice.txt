# Creating query and event classes in customer microservice

Just like how we have created three different classes on the command side very similarly, we need to create a class on the query side as well. On the query side, in our microservice scenario, we always going to have a single read operation. In case if you have multiple read operations, then you need to create multiple query related classes. For now, I'm going to create only a single class. Since we have only a single read operation. The class name I'm going to mention is FindCustomerQuery. Here also, we need to follow a standard which is first you need to mention verb. The verb can be find, read, get followed by noun. And at last we need to mention query. Inside this class I'm going to create a single string field which is private final string mobileNumber. Because anyone who want to know the customer details, they are going to send the mobile number as a request to param based upon the same, we are going to fetch the data from the read database to support the same functionality I have declared only a single field inside this class. On top of this class, let me mention the annotation, which is @Value. So @Value annotation is going to generate getter method, but not the setter method. Since I will not be having any scenario to set a new value once this object is created, I am trying to avoid that data annotation. Instead, I'm trying to leverage @Value annotation. I hope it makes sense. Now if you closely look under the command package, we have three different classes and on the query side we have a single class. And each of these class represent a API operation that an end user can invoke. And whatever data that end user is sending, we are going to transform that data into these command or query classes. As a next step, we should also create the event classes. Like I said before, whenever the command is processed by the Axon server and the data is stored inside the right database. As a next step, the Axon server, it is going to trigger an event for the query side to update the read database to support the same functionality what I have to do, I need to create the classes related to the event. That's why under this command, I'm going to create a new sub package with the name event, because the command side only is responsible to trigger these events, whereas the query side, they are only responsible to process those events. To me, it is making sense to create these event classes under the command package instead of query. But feel free to create these event package wherever you want. Under this event package, I'm going to create a new Java class with the name CustomerCreatedEvent. In the case of event scenario, you need to follow a different naming convention. First, we need to mention the noun followed by verb in past tense, so it will be past tense followed by event always. The event should be created by using this naming convention. And please make sure your event represents a past tense. By the time the command executes its logic, the customer will be created inside the right database. That's why the event also should convey the same. The customer created. The customer updated. The customer deleted. By following this pattern only, we need to create all these event classes very similarly. Let me create one more class with the name CustomerUpdatedEvent. At last I can create a class with the name CustomerDeletedEvent. Inside this customer created event I'm going to create multiple fields. For example, I'm going to first create customerId followed by name, followed by email, mobileNumber and activeSw. I didn't mark any of these fields as final because in the case of events, sometimes once the object of event is created and before we try to publish it to eventbus, we may have some requirements to change some data. That's why I don't want to mark these fields as final. Next, on top of these class, I'm going to mention @Data annotation. Inside these classes, we don't have to mention any of the annotation like target, aggregate, identifier because event classes, they are only data carriers. For example, once a command is completed, the event object will be published into the event bus, and on the read side, this event will be read and the data will be processed. Since there is no DB or storage involved, we don't have to mention any of the annotations. Next, let me copy the same fields into the customer updated event. Here also, we are going to mention five different fields followed by inside CustomerDeleteEvent. We are going to mention only two fields which are private string customerId followed by private boolean activeSw. I hope you are clear on why we have created these event classes.
