I'll come to this picture, but before that, let me try to highlight inside the Axon framework, event processes come roughly in two forms. The very first one is, subscribing and the next one is streaming. Let's try to understand the differences between subscribing and streaming event process, so that we can choose that best suits our customer microservice. Inside the subscribing event processor, everything is going to be carried out with a single thread. When I say everything right from command dispatching, after the command dispatching, event handling, event storing, so everything they are going to be handled using a single thread. To leverage these subscribing event processor, you should make sure that both your query side and command side, they are built under a single JVM. If you look at these technical flow as of now inside our customer microservice, we built both these command side and query side under a single microservice are under a single JVM. When both these command and query are under a single JVM it is perfectly possible to execute this entire flow in a single thread because they are deployed inside a same JVM, but few projects they are going to build this command side and the query side as two different microservices, where they are going to deploy them in separate jvms are into separate Docker containers. In such scenarios, the entire flow cannot be executed in a single thread because they are deployed in a completely two different jvms. So based upon how you build your CQRS pattern, you need to choose one of the event processes available. In our scenario, since we deployed everything into a single JVM, it is going to make sense to use the subscribing event processor because it is also going to give some flexibility on how we are handling the exceptions. When we execute the entire flow in a single thread, we can perform the rollbacks on the command side also, whenever there is an exception on the event handling side. We will explore on how to implement the same in few minutes. But before that, let's try to understand how the streaming event processor is going to work. Inside the streaming event processing every operation, they are going to be executed using multiple threads. So there will be thread one which is going to take care of command dispatch, and there will be thread two, which is going to take care of event handling. And these event processor is well suited for the applications where they have implemented command side and query side in different microservices are in different jvms. Apart from the thread perspective differences, these event processors, they also have another important difference subscribing event processing it is going to follow sequential event processing. So when you have multiple events published all the events they are going to be processed sequentially when we are leveraging the subscribing event processor. Whereas under the streaming event processor, all the events that are published from a command, they can be processed parallelly. So as of now, inside our microservices, a single command is publishing a single event. But I will explain you on how to dispatch or publish multiple events from a single command. Overall, I would recommend you to use the subscribing when you have built CQRS in a single JVM, whereas if you have built command and query on two different Jvms in such scenarios, using subscribing is not going to make any sense. You need to use the streaming event processor, the streaming event processor it is also going to help you to process a good amount of events compared to the subscribing event processor, because inside the streaming event processor, all the events that are going to be processed parallelly instead of sequentially.
