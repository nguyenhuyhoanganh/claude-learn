# Different approaches to implement CQRS

When someone is trying to implement CQRS pattern, they will end up choosing various flavors based upon their business needs. Inside this lecture, I want to explain about all the possible flavors available for the developers to implement the CQRS pattern. The very first flavor is, someone can implement the CQRS with a single read write model and a single database. So when I say single database, both write and read operations are going to be performed on top of a single database. With this single database, they are not going to get any great advantages. The only advantage they are going to get is, they are separated their APIs into write side and read side by following the CQRS pattern. Some people, they may think there is no need of separating the write to model and read model into separate components. That's why they are going to use the same read write model. When I say read and write model, I'm referring to the repository interfaces that we leverage to store and read that data from the database here. I also clearly highlighted the key benefit of this approach is clean, modular code with clear separation of concerns like command and query, making the system flexible and easy to test. But in this approach, the scalability and the performance are limited, and it is ideal for smaller to medium apps or for those new to CQRS pattern. I don't recommend this approach for any enterprise applications. Now if you go to the next flavor, this flavor is very similar to the previous flavor, but if you see inside this flavor, the models are separated. For example the command side, it is going to have its own model classes. Similarly for the read side also it is going to have separate read model. So what is the advantage that they are getting with this approach compared to the previous approach, developers can use a heavier ORM for writing and a lightweight model for querying. Since you have separated these two models now, developers are free to write some complex code on the write side, whereas simple code on the read side. So this is the only advantage that they are getting from the previous flavor that we have discussed. And since the previous flavor and this current flavor is using the single database for the command and read APIs, all the changes that are going to be committed in a single atomic transaction, consistency is always going to be ensured, allowing for better query performance without added complexity. Now let's go to the third flavor. Inside this third flavor, both command side and read side they are going to use separate databases. As you can see on the command side we have the write database. And on the query side we have the read database and the data synchronization between these two database it is going to happen with the help of Event bus. This event bus can be an Kafka or RabbitMQ or Axon server. It can be any product that is capable of handling the events between the command side and the read side. Inside this approach there are some trade offs to implement these flavors it requires high complexity of code and it brings eventual consistency. The data from the write and read it is not going to be instantly synced. It is going to take some milliseconds or nanoseconds of time because the event has to be published here. And from here the event has to be consumed by the query side. But this architecture is powerful because it is giving you some flexibility on choosing the different technologies for read and write databases. The scalability is also going to be independent in this flavor, but inside this flavor, the event sourcing pattern is not being used. When event sourcing is not being used there are some disadvantages which I have highlighted here. Since you can't commit the both sides in a single transaction, write changes are typically propagated to the read models via asynchronous events. This will sometimes lead to the challenges like out of order events are last messages are failed read updates. Sometimes this write database might have published E1 and E2 very quickly, but these read database might first process E2, followed by E1. In this kind of scenarios, the read database will end up having some invalid data. That's why to handle these kind of situations and to get more advantages, we need to leverage the next flavor where CQRS with event sourcing with different read and write databases. As you can see here, the write database is an event store which is capable of storing the events using event sourcing pattern. So this is the pattern that we have implemented inside our microservices. With the help of Axon framework, we are going to get lot of advantages when we implement this flavor. But it also has some challenges like consistency will be eventual consistency. Complexity will be high, but with the help of accent Axon framework we're able to implement it very easily. But coming to the advantage, which is around the performance and scalability, it is going to be high. And the last flavor that we have is, CQRS with event sourcing, but with a single database. So this single database usually will be a NoSQL database. What developers they will do is, they'll have a separate schema, a separate table to store the events using event sourcing pattern. But for the read operations they are going to have separate table or schema inside the same database. The advantage with this is the consistency is going to be strong, since we are leveraging the same database, and the complexity is also going to be medium as there is no event source involved. But on the performance and scalability also it is going to be medium. So these are the typical flavors that are going to be used by the developers. But very few organizations, they use CQRS with the CDC technology. I'll come to the point. What is a CDC technology? As you can see here, there will be a write database and the read database whenever some transaction happened on the right database, the CDC technology, it is going to publish an event about the change to the Kafka topic. From this Kafka topic the read database it is going to process the events and update the data. So here the data synchronization between the write and read database. It is going to completely happen with the help of CDC technology. In this approach the developers will end up writing very less code. The developers, they don't have to publish the events from the command side, and they also don't need to write the code to consume the events from the Kafka. Everything is going to be handled by the databases itself. So let's try to understand what is the CDC technology? CDC means Change Data Capture. It is a technique that monitors and captures changes in a database, and propagate these changes to the other systems in real time. I have more details in the next slide. Let me go there. As you can see here, every database, it can be a SQL or no SQL database whenever they are executing a query that results into the data change, they usually write into transaction logs. For example MySQL it is going to maintain a log with the name Binlog Postgres it is going to maintain write ahead logging and MongoDB they are going to track the changes with the help of app log. Since all these databases, they are trying to write the transaction details into these logs after committing the data into the database. What the CDC technology it is going to do is, it is going to keep on monitoring these transaction logs. For example, there is a source database where the insert, update and delete operations can perform. So inside these transaction log will be having details like shown here I indicates insert, u indicates update and d indicates delete. Off course there will be other details as well, whereas for the read operation, this transaction log is not going to be updated because read operation is not going to change any data. So from these transaction logs, these CDC technology, it is going to read the data and it is going to eventually publish those data changes into the target systems. These target systems can be a database or data warehouse or an analytics product or it can be a caching database. The target systems can be anything. How they are going to publish is the CDC technology products they are going to leverage the event streaming products like Kafka, where they are going to publish the events into them. And from these events, these target systems are going to read the data and update the data inside them. Though I don't recommend this approach from developer perspective. But still, there are some projects that are going to leverage these flavor because they team might be heavy on the infrastructure side and less on the developer side. So for such teams, it is going to make sense. There are some advantages when we implement the CDC technology. Obviously we are going to get loose coupling microservices. They don't need to directly communicate for data synchronization, and the communication is going to happen asynchronously with the help of tools like Kafka, RabbitMQ, and everything is going to be event driven. And at last we are going to get the advantage of scalability as well. The CDC technology, particularly useful in the use cases where we are trying to look to implement the data synchronization inside the event driven microservices that adopt event sourcing. So CDC is a technology, but if you look for an implementation product, there is a popular open source product with the name Debezium. So this is an open source CDC tool built on top of Apache Kafka. It captures the changes from the database like MySQL, Postgres, MongoDB by monitoring their transaction logs. This is the website of Debezium.
