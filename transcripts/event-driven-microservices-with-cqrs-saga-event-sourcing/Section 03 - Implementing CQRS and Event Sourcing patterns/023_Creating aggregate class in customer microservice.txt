# Creating aggregate class in customer microservice

To create an Aggregate class first I need to create a sub package. Under the command package, the package name is going to be Aggregate. Inside this Aggregate package let me create a new Java class with the name CustomerAggregate. On top of this class we need to mention an annotation which is @Aggregate itself. This @Aggregate is an annotation from the Axon framework. By mentioning this annotation, we are telling to the Axon framework that this class is going to act as an aggregator, which is responsible to process all the commands that we are going to trigger from the Controller layer. From the CustomerCommandController class, we are just trying to publish the command with the help of CommandGateway. So this command we need to handle inside the CustomerAggregate class. Apart from handling the commands, this class is also responsible to store the given data using event sourcing pattern. Once the storing of the data is completed, it is also responsible to trigger the event so that the data will be synced under the database side as well. Since this class is also responsible to store the data inside the event store database, we need to create the fields that are related to all the data that we want to store inside the right database. If you can go to the CreateCustomerCommand class, we have these fields, all these fields we want to store them inside the event sourcing database or right database. That's why I'm going to copy these fields and paste it here. But we should not mention them as final because every time someone try to update the customer data, we want to leverage these fields to update the data inside the right database. That's why we can't mark these fields as final as a next step on top of these customerId, I need to mention an annotation which is @Aggregate identifier. We need to tell to the Axon server what is the field name that the framework can use whenever it wants to make updates to the existing record related to a customer. We know very first time the brand new record is going to be created on the right database. But from the next time onwards, whenever someone is trying to perform some update operation or delete operation, the framework, what it is, going to do is first, it is going to try to load the existing data from the right database. Once it understands the existing database, then only it is going to write the new data in an event sourcing style. What is going to happen inside event sourcing pattern? We are not going to have a single record, which is going to be updated every time. Instead, every time a write operation happened on the given customer, the framework is going to make a new entry. Whenever it is trying to make a new entry, it has to understand what are the previous entries already made accordingly, it is going to increment the sequence number so that using the sequence number at any point of time, the framework will understand the sequence of the events happened on top of a customer. To help the framework on loading the existing records, we should choose a field which is not going to change throughout the life cycle of a customer. We know customerId is such field, which is going to be created during the creation time and the same will never change. Whereas if you try to choose the mobileNumber as the aggregate identifier, then you will be having some challenges. For example, if a customer initially he created the record with the mobileNumber as one, two, three and later. If they want to change the mobileNumber to 456, then in such scenarios, the whole event sourcing pattern is not going to work because you have few events with the mobileNumber one, two, three and the other events with the mobileNumber four, five, six. That's why it is always important to choose a field that is never going to change as an aggregate identifier. Once you mention these fields, you need to create an default constructor of this class.
