# Technical flow of CQRS & ES

It is going to take a lot of effort to implement CQRS and event sourcing pattern inside any microservices. We just started this journey and as we continue this journey, we will end up creating lot many components inside our microservices that are going to support CQRS and event sourcing pattern. I know this is going to be complex to understand at first, because there are too many concepts and components. To make your life easy what I have done is, I have built these technical flow diagram. We are going to refer these flow at multiple places throughout this course. For now, let's try to understand what is there inside this flow. At the top we have client application. This client can be an UI application or it can be some other microservice as well. The client has two different paths that they can choose. Whenever they want to initiate a write operation. They need to invoke the write APIs that we are exposing on the command side, whereas if they are looking to query the data or read the data, they need to invoke the read APIs that we are going to expose on the query side. As of now, on the command side, we have built the write APIs, and we also built the equivalent command classes. Using these command classes, we are trying to create the objects of them, and with the help of CommandGateway, we are trying to trigger these commands so that the action server can start processing these commands. Whenever a command is being processed by the Axon server, it is going to look for the business logic on how to process a specific command. All such logic, we are going to write inside the aggregate classes based upon the logic that we are going to write inside this Aggregate class, the data is going to be stored inside an event sourcing database. Once the data is stored inside the event sourcing from the same Aggregate class, we are going to trigger the event corresponding to the command that we are trying to process. The event will reach to the event bus and from the event bus, the events will be read by the Projection class. On the query side, projection is very similar to the Aggregate. Just like how we have written business logic on the command side with the help of aggregate very similarly, using the Projection objects, we need to write the business logic on how a particular event needs to be handled on the query side. By using the logic inside this Projection class, we are going to store that data inside the read database. Once the data is stored inside the read database, it can be consumed by the read APIs. As you can see here, if some client is going to invoke the read API, first we are going to convert the given input into a query object, just like how we have Command objects here. On the query side, we are going to have a query object representing the Query criteria details and this query object we are going to publish to the framework. Since framework does not have any clue on what needs to be done when a query is triggered, we need to provide all our business logic on how a particular query needs to be handled using these QueryHandler classes. From these QueryHandler classes only, we are going to read the data from the read database so that the data can be sent back to the client applications. I hope you are clear about these technical flow diagram for your reference I also mentioned all the important annotations that we are going to use while implementing all these components related to the CQRS and event sourcing. I know this may look very complicated, but don't worry, with the help of Axon Server and Axon framework, building all these components is going to be easy. As of now, we have developed the write APIs and the commands, and using these command objects, we are trying to trigger the commands with the help of CommandGateway. As a next step, we need to build this Aggregate class. If you can recall our previous discussion around the event sourcing, I have highlighted about these key concepts. Here you will be able to see aggregate and projections. Aggregate is a component inside event sourcing, which is going to encapsulate the state and behavior of the given record. These aggregates are going to process the events, and they are also going to produce the events for the query side. Similarly we have projections. Projections are the read only views or the models built from the sequence of events that are often used to query the current state of the record. You're going to get more clarity about this aggregate and projections when we start implementing them inside our microservices. Let's start implementing the aggregate from the next lecture. Thank you, bye.
