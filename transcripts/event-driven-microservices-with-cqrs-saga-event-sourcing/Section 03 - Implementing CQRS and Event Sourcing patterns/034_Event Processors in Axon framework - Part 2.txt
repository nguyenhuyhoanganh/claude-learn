Let's try to understand this image. As I was saying in the previous lecture, multiple events can be published from a single command. First, let's try to understand how to publish multiple events. If you go to the events and click on this dispatching, you will be able to see how to dispatch events from an aggregate. So as of now we are using apply() method available inside the aggregate lifecycle, which is going to take care of dispatching the event. So with the help of this apply() method we are only able to dispatch a single event. So this is a story that we know already. Let's scroll down to know more advanced information. As you can see here, the apply() method returns an implementation of the ApplyMore interface. This interface provides the following methods. By using the return type of this apply() method, you can invoke any of these four methods. The very first method is, andThenApply() all these four methods they are going to allow you to apply our dispatch more number of events. So this is how we can publish our dispatch multiple events. I'm assuming you got some clarity. If needed, we can also dispatch the events from a Non-aggregate class as well. If you are inside a Non-aggregate class, you can't use these aggregate lifecycle. In such scenarios, you can use Event Gateway to publish the event. You just have to inject this event gateway bean as a dependency to your normal Non-aggregate class. And with the help of this publish() method, you should be able to publish any number of events. So there is a good chance that as part of a single command, we will end up into situations where we want to trigger our publish or dispatch multiple events. Now with this understanding, let's go to the event process and here try to understand this diagram. Think like one of the command trigger three different events and these three different events they are going to be handled by three different event handler methods because at the end of the day, a single event can be handled by a single event handler method. So let's assume this is e1, e2, and e3. When you have these kind of multiple events, it is recommended to group all related events into a processing group. So what is the advantage that we are going to get when we group multiple events under a processing group? With the help of processing group, we can provide our implement non-functional configurations like on how we want to error handling or how we want to process these events, whether we want to process them sequentially or concurrently. So all such configurations we can make with the help of processing group. So this way you can define any number of processing groups inside your application. But please note a single event handler can be assigned to a single processing group. So once these processing group layer is established, we can assign how a particular processing group we want to execute, whether we want to use subscribing style or whether we want to use streaming style of event processing. That's why you will be able to see at the end we have event processor. So what is an event processor inside Axon we have only two options. One is streaming, the other one is subscribing. So based upon your requirements you need to assign one of the event processor implementation. So with the event processor implementation we will be able to control threading mode whether there needs to be executed in a single thread or in a multiple threads. And we can also implement error handling logic as well. In the case of streaming event processor you are going to get one advantage, which is if you have multiple events, all of them, they are going to be processed parallelly. But there is one disadvantage with the streaming processing. If there is an runtime exception happens inside one of the event handler, it is not going to stop the execution inside the other event handlers, or it is not going to roll back the transactions that are committed by the other event handlers. If you are looking to execute all your events and commands inside a single thread, then subscribing event to processing is the correct event processor that you need to choose. Because inside the subscribing event processor, you will have a flexibility to execute all your events in a single thread. Since it is a single thread under a single JVM, you are going to get a flexibility to roll back the transactions and to control the error handling behavior but the disadvantage with the subscribing is all your events are going to be processed sequentially one by one. For our case, where we have implemented everything inside a single microservice, it is going to make sense to leverage the subscribing event  processor. But to leverage the event processor, first we need to assign our events to a processing group, followed by to this processing group we need to assign the event processor. So let's try to understand how to do the same in the next lecture thank you, bye.
