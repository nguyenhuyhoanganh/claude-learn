# Benefits and drawbacks of Saga Pattern

Right now we have some good understanding on the Saga patterns. We also learned that there are two flavors of Saga patterns that we can follow in real projects. What I'm going to do is, I'll try to implement and show you a demo around the Choreography Saga pattern inside this section. In the coming sections, I will show you how to implement the Orchestration Saga Pattern along with the CQRS pattern. Before we try to implement this Choreography Saga Pattern, first let's try to understand what are the advantages and disadvantages of Saga patterns. Saga Pattern is not a silver bullet to solve all your problems. When you are implementing Saga pattern, you should also be aware about the challenges and ready to handle them. First, let's try to understand the benefits of Saga pattern. The very first and the greatest advantage or benefit that anyone is going to get with the help of Saga pattern is data consistency. This benefit is the whole reason for anyone to implement the Saga pattern. When you implement this pattern, it is going to help you to maintain consistency across the distributed services. You don't have to use a single database for all your microservices. You can go ahead and adopt the separate databases for your microservices. And the challenge around the distributed transaction and the data consistency is going to be solved with the help of the Saga pattern. The next advantage is around the scalability, since each transaction or each step inside a Saga is an independent transaction, services can scale independently, allowing for better performance and fault tolerance. For example, if you have a specific microservice which is going to receive a lot of traffic as part of a Saga flow, what you can do is, you can try to scale that specific microservice without scaling up other microservices. After the scalability, the next advantage is around the resiliency. Since we are going to use a compensating transactions as part of the Saga pattern, it is going to allow the system to recover from the failures gracefully. If there is some runtime exception, or if there is an error happen at the runtime. With the help of the Saga pattern, we can run the compensation transactions, which are going to take care of ensuring the system remains in a consistent state. In the scenario of vacation booking app, with the help of Saga pattern, what we can make sure is either all the components like hotel, car and flight are booked or all of them are canceled. We will not be having any scenarios where the car is booked, but other components like hotel and flight is not booked, so that kind of inconsistent states can be avoided with the help of Saga pattern. The next advantage is around the flexibility. Like we are discussing in the previous lecture, Saga pattern allows you to implement in two flavors one is Choreography and the other one is Orchestration. The developers has flexibility to choose any of these approach based upon their project  set up based upon their expertise. If your project don't want to implement access with the help of frameworks like axon, then you have a flexibility to go with the Choreography pattern. Whereas if your project is already using CQRS pattern, then it would be a wise decision to go ahead with the Orchestration approach. We are going to see the demo of these two approaches in the coming lectures. These are the advantages, but we should not be super happy by looking at the advantages alone. We should also aware about the drawbacks. The very first drawback is complexity based upon our discussion around the Saga pattern by this time, I'm assuming you have some basic understanding around the complexity. It's not going to be very easy to build the coordination between multiple microservices to trigger the compensation transactions in a specific order when a runtime exception happens. So to build that coordination, either using the Choreography or using the Orchestration approach, it requires some a good amount of time from the developers. We are also going to put some couple of hours efforts to build the Saga pattern inside our microservices. The next drawback is around the eventual consistency. Majority of the projects they are going to implement the Saga pattern inside event driven microservices, because if your business transaction spreads across multiple microservices, it can be 4 or 5 microservices. It is not going to make sense to make the end user to wait until all these five microservices execute their business transactions or compensation transactions. It may take 30s or 40s or 20s based upon the business logic that you have written inside all these microservices. That's why majority of the time projects will use event driven microservices. So they will just try to trigger the event and the microservice one. And from the microservice one, all the subsequent events will be triggered in a sequenced manner based upon the logic that we have written. So from the UI application or from the client perspective, they can keep polling about the status based upon the overall status that they have received. They will come to no business transaction is successful or failed. Since the Saga pattern is going to build majorly with the help of event driven microservices, that data updates that you are going to make across multiple microservice databases. They are going to be updated eventually, but not instantly. So there will be a gap of few seconds or milliseconds to update the data across all microservices. Even in the scenario of Compensation transaction, once a Cn  is completed, C n-1 is going to be invoked, followed by C3, C2, C1. So if you see here when the compensation transaction is being updated at this level, it is going to take some time to process and eventually come to the C1. So that time difference can be few seconds or milliseconds based upon the business logic that you have inside your microservices. If you are looking for the instant updates inside the database, then Saga pattern may not be an option. You should either be ready for the eventual consistency, or you need to let your clients aware about the data is not going to be update instantly. They need to keep polling for few seconds or few milliseconds. The last drawback is around the testing and the debugging. When we implement the Saga pattern, there will be normal transactions and they will be associated compensation transactions as well. So if you want to test them or debug them, since the transactions are going to spread across multiple microservices, it will be a very cumbersome process to test them and debug them. Based upon your microservices setup, you need to think about some workarounds or some hacks to test and debug your microservices efficiently. When Saga pattern is implemented. I hope you are clear with all the benefits and drawbacks of Saga pattern. Thank you, bye.
