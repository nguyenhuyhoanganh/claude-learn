# Problem that Saga pattern solves

For the next few hours, we are going to focus on Saga pattern. This is an important design pattern for every microservice developer, because it is going to solve a big problem inside microservices environment. Before I introduce this pattern to you, first let me explain on what is a problem that the Saga pattern is going to solve. Let's imagine there is a travel booking company, and this company behind the scenes has lot many microservices. For example, they have a car, microservice, hotel microservice and flight microservice which are going to take care of booking car, vehicle and flight for an end user. Let's imagine one of the end user trying to book the vacation. When a request comes from the UI application, the request will travel through multiple microservices. For example, the very first microservice that is going to be invoked is car microservice. So as part of the transaction one, we are going to reserve a car for this end user based upon the date selected on the UI. For example, if the end user is planning for a vacation of five days for the whole five days we are going to book a car for him with the help of this reserve car API. Once the logic present inside this microservice is completed, we are going to make an entry into the database about the car reservation details along with the dates information. Once the car microservice is done with its business logic, it is going to forward the request to the hotel microservice. Here the communication might be using synchronous communication or might be asynchronous communication with the help of patterns like SQS. Regardless of what is the type of communication is happening between the microservices. Let's imagine from the car microservice, the request is forwarded to the hotel microservice. Now this hotel microservice, what it is going to do as part of that transaction two, it is going to execute all the business logic present inside the reserve hotel API. Based upon the user selected hotel details, one of the hotel room will be reserved and the details will be saved into this database. Once the hotel service logic is executed, the request will be forwarded to the flight microservice. Inside this flight microservice, the transaction three is going to be initiated. As part of this T3 transaction, we are going to execute the business logic present inside the reserve flight API. Based upon the date selected by the end user the flight details also will be booked and the details will be stored inside this database. If you closely observe from the end user perspective, it is a single transaction or it is a single request that he is trying to make. But the same request is being traveled across multiple microservices where inside each service we are trying to create an a transaction to commit the data into the database. Since the request is being traveled through the multiple microservices and during its journey, since it is creating multiple transactions, this kind of request processing inside microservices, we can call them as distributed transactions because the request is being traveled across multiple microservices, which are distributed across various containers, nodes and regions. So this is a very happy path. Let's imagine an end user submitted a request and all the services they executed the business logic without any issues. The end user also will get a confirmation through SMS or through email, or on the UI page saying that your vacation itinerary is confirmed. But let's imagine while executing the business logic inside this flight service, we got a runtime exception since there is an runtime exception happened inside the service, all the data which might have committed inside this database will be rolled back automatically by the code, because inside the code we usually use the annotations like @Transactional, which are going to take care of rollback of the transactions that are committed as part of a single transaction. But what about all the data that we saved inside this hotel service and car service? We can't rollback them because the data that we have written into these databases, they are executed as part of another transactions where this flight service does not have any control. And we can't make these transactions also to wait until we get a successful response from other microservices because you are going to lock your keyword database row for two long time, which will create some performance issues. That's why usually inside microservices, as soon as they're done with their business logic, they are going to commit the transaction and release the lock on the database table row. Similarly, the runtime exception can happen inside the hotel service as well, whenever the runtime exception happens inside the hotel service, since it is not at invoke the flight service. We are good from the flight service because there is no data returned inside this database. But what about the data that is already committed inside the car database? With this setup, what is going to happen is even though we have not booked anything on the flight side, but we end up booking a hotel room and a car for the end user. This is going to bring data consistency issues. We should build our microservices in such a way that either all transactions in a single request are successful, or in the scenarios of failures. We need to make sure we are role backing all the data that we have previously committed. So that's where the Saga pattern is going to help us.
