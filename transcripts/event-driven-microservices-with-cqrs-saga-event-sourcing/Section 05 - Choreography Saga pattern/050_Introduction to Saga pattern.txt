# Introduction to Saga pattern

Let's try to understand what is a Saga pattern and how to implement this Saga pattern. Like I said before Saga pattern is a design pattern used to maintain data consistency in distributed systems like microservices. When a business request or a business transaction spans across multiple microservices, ensuring that data consistency across all these services, it is going to become challenging, especially when errors happens or runtime exceptions happens at the runtime inside the microservices. The Saga pattern it is going to address this problem by breaking down a large transaction into a series of smaller, independent transactions that can be managed by individual microservices. I know this definition will not provide any clarity. That's why let's try to take some example and try to understand the meaning of this definition. Please remember from the definition you need to remember these two keywords one is smaller and independent transactions. With that now let let's go to the example. Let's imagine you have a business request that span across five different microservices. In this kind of scenarios what Saga pattern recommends is, consider whatever transactions that you are going to commit inside an a single microservice as a single and independent transaction. Here, let's assume all the transactions that we are going to do as part of the microservice one, we can mark that as a T1 transaction. Similarly T2 and so on and this can proceed to any number of transactions. In our scenario since we have five microservices they will be T1, T2, T3, T4 and T5. If you see here, the whole business transaction is divided into smaller independent transactions and all these independent transactions. They are going to take care by the individual microservices. Now let's try to understand what should happen when an runtime exception or when an error arised during the execution of this independent and smaller transactions. What Saga pattern recommends during the exception flow or error flow is the compensation transactions needs to be executed in the reverse order to undo the changes made by the original transactions, like T1, T2. So every transaction should have an associated compensation transaction. For example, for this reserveCar transaction, the compensation transaction is going to be cancelCar. . Whatever logic that we are going to write inside these cancelCar, it is going to take care of canceling the car bookings that we have made as part of these reserveCar API. Similarly, for the reserve hotel transaction, the compensation transaction can be cancel hotel. When someone initiate this cancel hotel compensation transaction, it is going to cancel the hotel bookings. Now coming to the reserveFlight, we don't need any compensation transaction, because this is the last transaction that is going to be executed as part of the business request. If there is any runtime exception happens inside this flight microservice, whatever data that is going to be committed here will be rolled back with the help of that transactional annotation. Once the roll backing of the data inside the service is completed, it is the responsibility of this flight service to trigger the compensation transaction and the previous microservice. Here what it is, it is a cancelled hotel. Once these cancel hotel event, our business logic is handled and once the hotel booking details are roll backed or canceled, it is the responsibility of this hotel service to trigger the compensation transaction and the car microservice. Inside these cancelCar compensation transaction, what we are going to do, we are going to roll back the data or compensate the transaction. When I say roll back data, I'm not talking about the database transaction rollback. I am talking about the role backing of the data to a meaningful state. For example, when the carService initially booked a car, the status in the database can be booked. But when these cancelCar compensation transaction executed, what it is going to do is, it can update the status from book to cancelled. With this, what is, going to happen is there won't be any business impact since the status is cancelled at the ground level who are monitoring the car bookings, they are not going to make any arrangement for these end user because the booking has been cancelled. I hope you are clear. What are these compensation transactions they are going to do? They are going to roll back the data to a meaningful state so that there won't be any impact to the business in this picture also, I have highlighted that the compensation transactions needs to be executed in the reverse order to undo the changes made by that T1 to TN transactions. For example, let's imagine there is a runtime exception at that Tn+1 transaction. In this scenario, this Tn+1 transaction, it needs to take care of invoking the compensation transaction of this Tn, which is Cn. Once the Cn is completed, we need to take care of invoking the next compensation transaction which is associated to that Tn-1 which is Cn-1. This way we need to travel up to the C1, which is the compensation transaction of the T1. In the scenarios where all the transactions are successful, then there is no need of invoking the compensation transactions. The compensation transactions will only come into the picture when an exception arise, or when an error arises inside the distributed transaction flow. All the microservices that are involved inside the Saga pattern it is their responsibility to provide the logic that needs to be executed as part of the normal transaction logic and the compensation transaction logic. The logic that we write inside this T1 and C1, they will be present inside the same microservice, pointing to the same database. So this C1 will be a local compensation transaction for the T1. Similarly, for the T2 they will be a local compensating transaction with the name C2, which points to the same database inside an microservice. I hope you are getting some clarity. If there is some confusion, don't worry while we try to implement the Saga pattern, it is going to make more clear for you. Now coming to the implementation part, whenever we are trying to implement a Saga pattern, we end up having two different options. The very first option is choreography Saga pattern. The second option is OrchestrationSagaPattern. As part of the ChoreographySaga pattern, what is going to happen is all the microservices that are involved inside the business flow. They are going to be involved. So every microservice that is part of the business transaction, they need to be aware what is the next microservice that they need to invoke in the case of successful scenario, or in the case of compensation scenario, they should know what is a previous microservice they need to invoke. Since inside the choreography Saga pattern all the microservices, they are going to be involved. In other words, we can call this approach as a decentralized approach where microservices autonomously execute their part of the transaction and then trigger the next step in the flow. Whereas inside the orchestration flow, a centralized orchestra or a centralized Saga manager is going to coordinate the execution of the transaction. This central component, as Saga manager, what it is going to do is first, it will forward the request to the microservice one, which is going to track the request as AT1. Similarly, it is going to forward the request to T2, T3 and T4 and so on. If there is an exception occurs during the T4. This Saga manager, it is going to take care of triggering the compensation transaction C3, C2 and C1 in an reverse order. So here the entire control will be present in a single place, which is a Saga manager. The individual microservices, they don't have any clue. What should they do when an exception arises or when an transaction is processed successfully? Only the central Saga manager aware about what needs to be done as a next step. That's why in this approach, the orchestrator sends commands to each service dictating the flow of the entire transaction. Usually, this orchestration option is used when the CQRS pattern is implemented. So when you implement CQRS pattern by using frameworks like Axon in such scenarios, you can happily use these Orchestration Saga Pattern. Otherwise, you can go with the Choreography Saga Pattern. There is no good and bad here. It's all going to be depends upon your project requirements, project technical stack, both of them. They are going to give you the same result.
