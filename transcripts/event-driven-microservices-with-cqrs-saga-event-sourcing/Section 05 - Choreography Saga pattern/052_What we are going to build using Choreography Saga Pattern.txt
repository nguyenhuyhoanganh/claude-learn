# What we are going to build using Choreography Saga Pattern

As a next step, I want to show you the demo of Choreography Saga Pattern. Before I try to implement this pattern and show you the demo. First, let me explain you about the scenario where we are going to implement the Saga design pattern. As you know, inside our microservices environment we have four different types of microservices, namely customer, accounts, cards and loans. All these microservices, they have their individual database. The thumb rule right now that we are following is, for a customer we need to make sure we are using the same mobileNumber across all the microservices so that any time, if someone wants to see the customer profile summary or customer summary details using the APIs that we have implemented with the help of API composition pattern or with the help of materialized view pattern in all such scenarios, all the data related to an end user will be fetched from all these microservices and will be sent as a response to the client applications. As an extension to this functionality, I am going to build an rest API inside the customer microservice, which is going to give flexibility to the end user to update their mobileNumber. Whenever an end user wants to update their mobileNumber behind the scenes, I'll make sure I am updating the mobileNumber across all these four microservices. Once the mobileNumber is updated in the customer microservice with the help of T1, then these customer microservice is going to forward the request to the account microservice, where the mobileNumber will be updated using the transaction T2. After accounts, it will go to the cards microservice where we have a transaction with the name T3. And at last the loans microservice will be invoked where with the help of T4 transaction, we are going to update the loans database. So if you see here the same business request is being travelled across multiple microservices. So this is a classic example of distributed transaction. For some reason at any point if there is an exception arises, then as part of a Saga pattern, we are going to initiate the compensating transactions. As you can see if there is an exception happens in the loans microservice this loans microservice will roll back any data that is committed in its own database, followed by it is going to initiate the compensation transaction on the previous microservice which is Carts microservice. So we need to come in the reverse order. That's why we need to make sure we are first initiating the C3 compensation transaction, which is present inside the Cards microservice. As part of this compensation transaction, what we are going to do is, we are going to update the mobile number to the old mobile number itself. Once this is completed, this C3 compensation transaction will initiate the C2 compensation transaction. Similarly, C2 is going to initiate the C1 compensation transaction. So what we are going to achieve here by using this Choreography Saga pattern, we will make sure either new mobileNumber is updated in all microservices or in the scenarios where an exception arises in one of the microservice. Then we will roll back to the old mobile number across all the microservices. So here T indicates normal transaction and C indicates compensation transaction. As you can see here, inside the Saga pattern, all the microservices are being involved. So the customer microservice know after T1 it has to trigger the T2. After T2 the account microservices know that it has to trigger T3. And the same applies for T4. And in the case of exception scenario, all the microservices know what is the compensation transaction that they need to trigger. So since all the microservices are being involved, we are going to call this as Choreography Saga pattern. Do you know why this name comes? Just like how we saw a group dance where everyone is going to dance and play their role very similarly here, all these microservices, they are going to dance based upon my Saga pattern implementation. That's why this approach we call it as Choreography Saga Pattern. I hope you are clear. So this is what we are going to implement inside our microservices. What I'll do is, while I'm trying to implement this Choreography Saga Pattern, I will not use any CQRS pattern because it is not mandatory to use CQRS pattern while Saga pattern is being implemented. That's why I'll show you both the approaches with CQRS and without CQRS. And the beauty of this Choreography Saga pattern is while we are trying to implement these, we don't have any dependency on any of the framework. Like you don't have to depend on the frameworks like Axon.
