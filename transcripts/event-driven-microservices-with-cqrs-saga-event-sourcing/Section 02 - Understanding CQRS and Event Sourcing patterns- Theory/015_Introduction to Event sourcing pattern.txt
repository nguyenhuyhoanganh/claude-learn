# Introduction to Event sourcing pattern

Let's try to understand what is an Event sourcing pattern. With the help of this pattern, we can store every change that is happening to a record as an event inside a storage system. Instead of storing only the final state of the record, we can store all the entire history of the changes that happened to and record all these events that are going to be stored in an special type of database, which, in other words, we can call them as event log or event store. Inside these event log or event store, all the events will be maintained in their sequence to capture the entire history. I know it may not be clear for you based upon this definition. Let me take an example and try to explain the same. We all have saving bank accounts. Usually when you log in to your bank account, you'll be able to see all the transactions that you have performed. For example, the very first event here you can see $2,000 has been credited into my account. So this event happened at so and so date and time. The next event or change that happens to my account, it is going to be stored as a second event. So here $120 has been debited from my account. Similarly I have this third event, fourth event, fifth event and sixth event inside my saving account all these six events happened and each of the event is associated to a date and time value. All these events are stored inside an event store or inside an event log. Apart from storing these events history, the bank application should also display the what is the current effective balance after processing all these events. As of now, you can see my current balance is $2,150. Usually when we implement an event sourcing with the help of CQRS pattern, all these event store mechanism will be handled by the right side or the command side, whereas the most current state value like the current balance, this will be handled by the reader side or query side of the CQRS. By combining these event sourcing pattern with the CQRS, we're trying to get the benefits from two different worlds, usually without CQRS and Event sourcing pattern. We can either store the most current state value, or we can store the history of the events. If someone looking for the most current state value, they need to have a business logic to process all this history of the events, then only they will be able to know what is the current state value. But with the help of event sourcing and CQRS on the command side or writer side, we are going to store all these events as a history of events, whereas the reader side, at the query side, it is always going to maintain the most current state value. Using event sourcing pattern with the CQRS is not mandatory if you don't use Event sourcing pattern. On the writer side, you're not going to maintain any history of the events. It is always going to be a single record, which is going to be updated based upon the latest data. But most of the organizations, they are going to leverage event sourcing along with the CQRS, because it is incredibly useful for the scenarios where we need a complete audit trail of all the changes happened over time inside e-commerce applications also, you will have these kind of scenarios. For example, you may want to show the list of events happened to the end user after he placed an order. So initially order has been placed next payment completed after payment shipping happened after the shipping delivery happened. This kind of history of the events you need to show on the UI, sometimes all these kind of functionality we should be able to easily achieve by adopting the event sourcing pattern along with the CQRS. I hope you are clear. What is event sourcing pattern and when to use it. When we see the demo of this pattern along with the speakers, it is going to be more clear for you. For now, let's try to understand what are the key concepts of event sourcing. Inside event sourcing, a command represents an intent to perform an action in the system, such as debit money or credit money. Whenever an end user initiate a command on the UI, it is going to trigger a change which is going to be recorded as event. In other words, a command will always result into an event. The next question you will have is what is an event? An event represents a change in the state of the system. It is an immutable record of something that has occurred in the past, such as money debited or money credited. If you see here, the command will always use the present tense like debit money or credit money, whereas the event they are going to use the past tense like money debited or money credited. So by the time the event occurred, the command might have completed that's why the standard is always to follow the past tense while we are trying to give the names to our events. After the event, we have event store. Like I said before, event store is a specialized database where we can store the events. Events are appended to the store in the order they occur, and can be replayed any time to recreate the state of the system. The next important concept inside event sourcing is, aggregate. Aggregates are the primary objects that encapsulates the state and behavior of the record. For example, if you consider the same bank example that we have discussed previously, in such scenarios, we can assume transaction as an aggregate, which is going to be responsible to process the events like money debited or money credited, that will be resulted from the command. So whenever a command initiates an event, definitely we should write business logic on what should happen. All such logic, we are going to write inside the aggregate classes or aggregate objects. These aggregates they are capable of processing the events and producing the events as well. When a command triggers an event, these events will be processed by an aggregate object. Once the event is processed and stored inside the event store, it needs to communicate about the state, change to the read side of the CQRS or to the query side of the CQRS. To handle such scenarios, this aggregate is also capable of producing the events on the query side or on the reader side of the actors. There will be a component which will be listening to the events that are produced by the aggregate, so that it can update the most current state value based upon the most recent event occurred. After the aggregate we have projections. We know with the help of aggregate all the events we are going to store in a sequence, but we'll be having certain scenarios where we don't want to read all the events, but instead I want to read the most and the current state value of the record. For example, in this scenario I only want to read the current balance value, but I don't want to read all these events. Such a current state values we can build and process with the help of projections. Projections are the read only views or models built from the sequence of events. They are often used to query the current state of the system or to generate the reports, so these projections can be updated in real time as the new events occur. In simple words, you can construct this aggregate as the command side or the right side of your CQRS. Whereas projection you can consider as query side or read side of the CQRS pattern. Aggregates they are going to help you to write the events into the event store, whereas projections they are going to help you to update the most current state value into the Read database. I hope you are clear with these key concepts of event sourcing. There are many advantages that we get when we implement event sourcing pattern. Let's try to understand the same. The very first and the greatest advantage is Auditability. Since you are going to store each state change as an event inside your event store, it is going to provide a complete audit trail of all the changes happening inside your system. The next advantage is Replayability. Anytime if you find some bug or an issue inside your system where the most current state value has been corrupted, what you can do is you can leverage the Replayability capability of event sourcing. All the events can be replayed in separate layer environments, and this is going to be useful for debugging or restoring the data. Basically, with the help of event sourcing, you are providing developers with a time machine, you can travel to the past, and you can try to process your business logic based upon all the past events happened. If you don't store all these events, and if you are always storing only the current state value, then obviously you never know what happened one year back or two years back to a given record. The next advantage is around the Scalability. Since we are separating both write and read side with the help of CQRS and event sourcing, the systems can scale more effectively based upon your business requirements, and the last advantage is around the Flexibility. This is my favorite benefit. Projections can be created on the fly for new reporting requirements without modifying the core data model. For example, here, if we store all the events inside an event store in future, if you have a requirement to calculate a new report, or if you want to perform some analytics, you can always use these past events. For example, if I am looking for a report to understand on which dates the most number of transactions are happening, then I can leverage all the records present inside the event store to build the report. Especially these kind of storage is going to help a lot of e-commerce applications, bank applications, social platforms to understand the behavior of the end user. Since each and every event we are storing inside the event store, they can run a lot of analytics to understand the behavior of the end user, and accordingly, they can try to fine tune their product so that they can attract more number of end users towards their application.
