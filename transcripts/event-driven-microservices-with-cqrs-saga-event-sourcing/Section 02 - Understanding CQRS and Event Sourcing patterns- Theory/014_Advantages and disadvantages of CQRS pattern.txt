# Advantages and disadvantages of CQRS pattern

CQRS pattern has both benefits and drawbacks. It is not going to be your silver bullet to solve all your distributed transactions problems. There are some trade offs which you need to consider while choosing or implementing these skills pattern. Inside this lecture, let's try to understand both the benefits and drawbacks of CQRS pattern so that you can take a well informed decision about the CQRS pattern in your real projects. The very first advantage or benefit will be around the scalability. By separating the read and write databases, we can independently scale each side based upon our project specific needs. This is going to be a major benefit for organizations who process millions and billions of records on day to day basis. For example, you take Facebook, you take LinkedIn, Twitter, all these organizations, they are going to process humongous amount of data for them. It is very important to have flexibility to independently scale both on the write and read side. Here I also mentioned an example where for an organization, if read operations are more frequent than write operations, you can scale the read side database alone accordingly based upon your traffic. The next advantage will be around the optimized data model. The right model or the right database can be optimized for data consistency and transactional integrity, while the read model can be optimized for the faster data retrieval. This also allow you to use different data storage mechanisms for each model. On the write side, you can have NoSQL database, whereas on the read side you can have either relational database or you can also have some caching mechanism based upon your business requirements. The next advantage will be around the flexibility CQRS it is going to allow us to implement different query models, such as projections, to accommodate various views of the data without affecting the right side. We have discussed about this example in the previous lecture. As you can see here, this is all my write side this is all my read side. This is one of the projection are the query side that I have built for my UI page, where it is looking for an information from all the microservices. So these kind of custom projections are queries we can build based upon our business requirements. Here I only have one example. What if you have hundreds of different pages inside your web application, where they are expecting the data from multiple microservices? For all such 100 different UI pages, you can have 100 different projections or 100 different read operations where you are only going to focus on reading the specific data that is required by your UI application. I hope it is clear the last advantage will be around the event sourcing CQRS pattern is often used in conjunction with the event sourcing pattern as part of these event sourcing pattern on the write side, every change that is happening to the data we are going to capture as an event. This provides a complete audit trail of all the changes happening on your write side, whereas on the read side, you can only try to have the latest or the most updated data for the reading purpose. For example, if you take the bank application as an example, usually inside a bank application, whenever we make a transaction, there will be an entry happening on the statement, like $100 I spent and $50 has been credited. Whereas if you have a requirement to only display the current balance inside the account, we don't have to display our process all these transactions we can store the current account balance on the read side, and from these read database, the UI application can read the most updated current account balance and the same will be displayed to the end user. So instead of processing all these transactions every time to calculate the current balance, what we can do, we can store the current balance separately inside the read database. From this read database, UI applications can read the data. So that kind of advantage we are going to get when we implement the event sourcing pattern. More details around the event sourcing pattern I'm going to discuss in the coming lectures. So these are the benefits. You might be happy by hearing about all these benefits, but there are some challenges or drawbacks. When we adopt CQRS inside our project, let's try to understand the same. The very first challenge is increase the complexity. Since we are separating both read and write models, more complexity will be added into the system, particularly in terms of maintaining that data consistency between the write and read database. That's why this pattern is only suitable for the organizations or for the projects where they deal with the lots of amount of transactions. For simpler applications or for smaller organizations, this pattern is not going to make any sense. And the next challenge will be around the eventual consistency. Like I said, whenever the updates are happening on the write side, the data will not be copied onto the read side instantly. It is going to take few seconds or few milliseconds time there will be some delay to sync the data from the write side to the read side. This eventual consistency can be challenging to manage in certain scenarios. For example, if you have a scenario where you want to show the updated data immediately on the UI page, then in such scenarios, this CQRS pattern may not make sense. Such operations you need to make with the help of normal synchronous calls or are normal rest API calls. So this is the trade off that you need to consider for most of the project this should be fine because the delay will be only in milliseconds or at a max a second, and the last disadvantage will be around the development overhead. So to implement this pattern, someone has to write all the logic where to separate the command side and the query side, and having their own databases. So this is going to be a development overhead for the developers. And it usually requires a lot of learning curve to implement the CQRS pattern in any organization. Bigger organizations, they are going to implement this CQRS pattern from scratch. But we don't have to develop this pattern from scratch. There are some good frameworks like Axon Framework, so this framework is going to help us in implementing the CQRS pattern, Saga pattern, Event sourcing pattern with the very minimum efforts.
