# Introduction to CQRS pattern

We are into a new section, and inside this new section we are going to talk in detail about the CQRS design pattern and event sourcing design pattern. These two design patterns, they are going to complement each other throughout this section we'll only try to understand these patterns from theory perspective. Once we are clear with these patterns. From the next section, we are going to start implementing these patterns inside our microservices. So for the next one two hours, don't worry about the demos. First, let's focus on understanding these patterns. And at a later point of time I'm going to show you a demo around these patterns. So what is a CQRS pattern? The full form of the CQRS is, Command Query Responsibility Segregation. Anytime if you want to build a business logic inside your microservice or inside any web application, we usually end up writing logic either to update the data or read the data. When I say update data, it can be insert or delete or update or modify the data. Whereas when I say read the data, we can only query the data without modifying it. So all the operations or the business logic that we write inside microservices will become either into the update operation or the read operation. What this pattern recommends is, to separate the write API's and read APIs into two different individual components. Apart from building two different individual components, handling the update operations and read operations this pattern it also recommends to have separate databases for write and read purpose. So that's why we have this name. Command means an operation that can alter the data or modify the data, or create the data or delete the data. So all such operations we can call them as Command, whereas Query means we can only perform reading of the data or the query of the data from the database. So the command and query responsibility it is recommending to segregate separately. You can see here this is the command part and this is the query part. Anytime if someone invokes the API that alters the data it will be handled by this command component. Once the business logic is executed, the data will be written into the right database or command database. And on the other side, if anyone wants to read the data, they are going to invoke the APIs exposed by the read component or the query component. And this query component, it is going to read the data from the read database here since we are trying to maintain two separate databases, one for writing purpose and the other one for read purpose. Obviously the question will arise, which is how you are going to sync these two database whenever the new data or updation of the data or deletion of the data is happening for the same purpose we're going to have a middle component, which is event bus. This event bus can be any component like Kafka or RabbitMQ or any other product that is available inside the market. Whenever the data updation is happening inside the right database. This command component, it is going to publish an event into this event bus. As soon as the event is published into the event bus, the read component or the query component, it is going to read the event from the event bus, and accordingly, it is going to update the read database in future. If someone is trying to read this data through the read APIs, then obviously the data will be fetched from the read database. And since both these databases are in sync, they will not be any issues. off course, the sync between these two database is not going to be instant. The instant updation is not going to happen. They'll be some nanoseconds or milliseconds or seconds difference will be there to sync these two database. So basically this kind of sync mechanism where the sync is not happening instantly, we'll call it as eventual consistency. So the data will be eventually updated on the other database maybe within a nanosecond or within a millisecond or a second. I know here you may have lot many questions what advantages we get by separating command and query responsibilities and having different databases. We are going to talk in detail about these advantages in the coming lectures. For now, I want to highlight few of the most important advantages that we get. The very first advantage is if your organization is going to perform a lot of read operations, then you can choose the read database that is optimized for read operations. And if needed, you can also scale this read database to any extent based upon your read operations. And you can also choose the right database that is suitable for the minimum write operations. One more advantage that you get while implementing this pattern is, you can have your write database as no SQL database, or any document based database, whereas the read database can be a relational database. Maybe your application process lot of documents, lot of Jsons, all these documents on Json's you want to store inside the right database. But for someone to read the data, you don't want to send that data present inside these right database in a JSON format or in a document format. Eventually, by the time you store the data inside the read database, you can write some transformation logic where you're going to store all the data in a relational database style. With this setup, you're going to make read operations much simpler. And the next advantage that we have with the CQRS pattern is, it is going to help you to handle the Cross-service queries efficiently. Long back if you can recall, we resolved the cross-service query challenge by using the API composition pattern. The same challenge can be efficiently resolved by adopting CQRS design pattern. Let's imagine you have multiple microservices like customer accounts, loans and cards. All these microservices, they store their own data. For example, accounts microservice stores account specific information, and similarly, cards will store card specific information. But you have a UI page where for an end user or where for a customer, you want to show a subset of data from all these microservices. For example, from customer service, you want to display the profile information like what is the name, email photo of the end user, mobile number, address details and from the account service you want to display only the current balance inside the account and from the loans you want to display loan summary and from cards you want to display the card summary. If you don't have access CQRS pattern implemented, what is the approach that you're going to follow? You're going to follow the API composition pattern where the gateway is going to call all these services individually, and it is going to aggregate the data, and the aggregated data will be sent to the UI application. But this has some drawbacks like performance issues. And you're going to increase the traffic inside your microservices network unnecessarily. But the same problem can be solved very efficiently with the help of CQRS. Let's try to understand how we can solve the same. All these microservices they are going to have their own individual databases. Once the data has been written into their individual databases, a event will be published to the event bus saying that so-and-so customer data is updated, so-and-so account data is updated, so-and-so loans data is updated, and cards data is updated. Now on the other side, we have a query component which is responsible for the query operations. So what this component is going to do is it is going to read the events from the event bus and update the data received into a read database. So this database is going to act only as a reader database, where we have a dedicated table that is going to adhere the requirements of the customer summary UI page. So from this UI page, the call will go to these fetchCustomerService API and these API. Instead of reaching out to all these microservices, what it is going to do, it is going to simply perform the read operation from the customer summary view database. With this, we are trying to avoid many network calls at runtime from the UI. And here you may have a question which is, where is this query component is going to be deployed. It can be deployed anywhere, it can be deployed as a separate component, or it can be deployed as part of the customer or accounts or loans or cards microservice. Basically, with this pattern, what we are trying to do is, we are separating the write operations and the query operations. Anytime you want to read the data, they should reach out to the query component. Whereas for the command operations they can reach out to the individual microservices. I hope you are clear with this introduction of CQRS pattern this pattern is mandatory for enterprise applications where they are processing millions of transactions on day to day basis. If you try to have a same database for both write and read, then definitely we are going to get some performance issues. That's why it is always recommended to use CQRS pattern to separate command APIs and query APIs. When you combine the CQRS with the event sourcing, then it is going to be even more powerful. Will be discussing what is event sourcing in the coming lectures. By this time, I'm assuming you might have got some clarity that these CQRS pattern can be implemented only with the help of event driven architecture, because there is a mediator here, which is Event bus, due to which the syncing between the two different databases is always going to be eventually, but not instant. off course, there are some options where we can avoid these event bus and implement these skills without event driven architecture, but it is not going to be super helpful and it is going to have some challenges.
