# What happens in ES when a new event being stored

In the previous sections, we saw the demo of event sourcing pattern, and we also discussed on what are the advantages that we are going to get when we implement event sourcing, along with the CQRS pattern. So it is a very good combination. But whenever you are implementing the event sourcing pattern on the command side or on the write database, you're going to get a performance problem. Inside this lecture, I'm going to explain you how these performance problem is going to arise, and what is a workaround that we can use to overcome these performance problem. To understand why this performance problem arises inside the event sourcing pattern, first we need to understand what is going to happen behind the scenes whenever a new event is being saved into the event sourcing database. As of now, we know whenever we are adapting the event sourcing pattern, all the events related to a specific record or a specific account are related to a specific person. They are going to be stored sequentially in an historical style. Here I have an example where we have the events of an account being stored inside the event sourcing database. Whenever a transaction is being performed on an account, the corresponding event details we are trying to store here one by one in a sequence manner. As of now, we did not discuss what is going to happen when a new event is being stored inside the event store. Let's try to understand the same. Whenever we are trying to insert a new event, the Aggregator object or the Aggregator class, where we have defined all the business logic to store the event inside the event sourcing database, what it is going to do is it is, going to load and apply all past events to accurately rebuild the current state of an entity. Here, when I'm trying to insert the new event, my Aggregator object or my Aggregator class, it is going to load all the past events and it is going to replay all these events one by one to calculate the current state value before it is trying to store this new event. Here you may have a interesting question, which is why Aggregator needs to replace all the events? It's a great question. Let's try to understand as part of the event sourcing pattern. There are multiple reasons why aggregator do this. The main reasons are to ensure the consistency, support to complex calculations and maintain a full audit log. Let's imagine you have a business logic inside your aggregator to calculate and update the current balance whenever a new transaction is being made inside an account. Since we have adopted the event sourcing pattern behind the scenes, what is going to happen is, to calculate this account balance every time when we try to save a new event, all the past events, they will be replayed so that the current account value will always accurate and there is no tampering has been done by anyone. So that's a guarantee that you are going to get due to the event sourcing pattern and its character of replaying all the events every time a new event is being stored. The replay functionality of event sourcing enables accurate decision making even when logic Evolves over time and provides traceability for debugging and auditing purposes. I hope you are clear why all the events that are going to be replayed by the Aggregator. Now let's try to understand under which scenarios we are going to face performance problems. Let's imagine there is an e-commerce application as part of this e-commerce application developers they are trying to store the order details as events inside the event sourcing database. So basically for an order there will be multiple milestones. Very first event will be order placed followed by confirmed. After the confirmed next stage will be shipped and at last delivered. So these are the important milestones that every order is going to have inside an e-commerce application. If you see at a max for a given order, there will be four events placed, confirmed, shipped and delivered. Worst case, there will be a canceled event as well, which will be triggered in the place of confirmed. So at any point of time, every order record will have only the four events, so there won't be any performance issues to replay these four events every time a new event is being stored inside the event store database for a given order. But let's try to imagine another scenario. This time, let's take an bank application. Inside this bank application, the developers, they are trying to store the transaction details of an account inside event sourcing database. So whenever a new transaction is being on top of an account, a new event entry will be made into the event sourcing database. During the process of inserting the new entry this bank application. It also has a logic to calculate currentAccountBalance. Let's imagine there is a person who does a lot of transactions on his account, and this person has the account with this bank from last ten years. Can you imagine how many entries will be there inside the event sourcing database? There will be 1000 10,000 or there is also a good chance that the person might have made 100 K transactions over the last ten years. Any time this person is making a new transaction to calculate the currentAccountBalance all the 100 K events they needs to be replayed by the event sourcing or by the Aggregator object, because it has to identify the current state of the entity, and also it needs to execute the logic of calculating the current account balance. This is a perfect scenario where performance issues may arise. Replaying all the 100 K events and executing all the logic of aggregator definitely will have some performance issues to overcome these kind of performance issues many event sourcing systems they use a concept called snapshots. For the next few lectures, we are going to understand how to implement snapshots in our microservices as well. I hope you are clear what happens inside event sourcing pattern when a new event is being stored. Thank you and I'll catch you in the next lecture, bye.
