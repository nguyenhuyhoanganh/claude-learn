# Snapshots in Event sourcing pattern - Theory

Snapchats in event sourcing are a important tool for improving performance and scalability by reducing the need to replay all events from the beginning of an aggregator's history. When we adopt the snapshots, they provide balance, maintaining a detailed event history and ensuring that the system remains efficient and responsive, particularly in the scenarios of volume of events grows. When we adopt the snapshots concept to get the current value, the aggregator, it don't have to replay all the events right from the very first event. Instead, the application only has to retrieve the snapshot and the events that occur after it. Basically, as a developer, what you can do is, you can define after how many events you want a snapshot to be triggered. For example, if I configure inside my code that a snapshot needs to be created for every 100 events behind the scenes, what is going to happen is, whenever the 100 events are stored inside the event sourcing database, a snapshot object will be created and stored. This snapshot will have current state of the entity after replaying all the previous 100 events. Once the snapshot is created and whenever a new event is being published in the future, the aggregator will not consider all the previous 100 events. Instead, it is going to start from the snapshot, followed by all the subsequent events that are published. So basically this snapshot is going to be present at the location one, not one. Once your number of events it reaches 200 then again a new snapshot will be created. The snapshot events and details are also going to be stored inside the same event sourcing database. Or we can store in a separate storage system as well. Here I also highlighted the steps explaining how snapshots are going to work. First, a snapshot will be created based upon the configurations that we have defined. Once a snapshot is created in future, whenever a new event is being published, the restoration of the state of the entity will start from a given snapshot. By ignoring all the previous events. You need to use the snapshots in the scenarios of high event volumes, performance concerns, and when you are looking for an option to have a periodic consistency checks while implementing snapshots, you need to consider some trade offs.
