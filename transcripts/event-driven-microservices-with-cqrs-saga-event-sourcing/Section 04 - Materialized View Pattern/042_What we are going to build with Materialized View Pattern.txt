# What we are going to build with Materialized View Pattern

Let's try to implement this materialized view pattern inside our microservices. As of now we have four different microservices, namely customer, which holds the customer information followed by accounts. After accounts we have loans and cards microservices. Let's imagine we have an UI page which is responsible to display the data that is spread across all these four microservices. We're going to build these new API with the help of these materialized view pattern. We already saw an example with the help of GatewayAggregatorPattern. But this time we're going to follow this better approach. To make things clear around what we are going to build. I have this slide. As you can see here. First we have customer related table. So this table is present inside the customer database that is going to be used by the customer microservice. Similarly we have account table followed by card followed by loan table. We know all these tables are being used by the respect to microservices. For example, inside the customer table, which is going to be maintained by the customer microservice, it is going to store the information related to the customer like customerId, mobileNumber, name, email, activeSw and other audit related columns. Similarly under account we have mobileNumber, accountNumber, accountType, branchAddress, activeSw and audit column. Under the card we have mobileNumber, cardNumber, cardType, totalLimit amountUsed, availableAmount, activeSw and audit column. At last, under the loan we have mobileNumber, loanNumber, loanType, totalLoan, amountPaid, outstandingAmount, activeSw and audit column. Now inside all these four tables you might have observed there is a common column which is mobileNumber. Initially, when the customer registered with the bank application, it is going to make an entry inside the customer table using the same mobileNumber the end user can create a new account, card and loan. Now what we want to do? We want to build a new API inside a new microservice. Let's imagine this microservice name is profile. What this profile microservice it is going to do is it is going to maintain an materialized view with a table name as profile. Inside this profile table we are going to maintain the mobileNumber, which is a common column for all the microservices followed by name. This name is going to get from the customer table and the accountNumber it is going to come from the accounts table. Similarly, cardNumber comes from the card table and loanNumber comes from the loan table, and the activeSw indicates whether the given profile record is active or not. And at last, it is also going to have some audit columns. I'm trying to keep this table very simple just to make our lives easy, but in real applications, your table may have hundreds of columns or your profile microservice may have multiple tables. It is all going to depend on your business requirements. For now, let's keep things simple and let's assume this is the table or the materialized view that we want to maintain under the profile microservice. Initially this table, it is going to be an empty table as soon as a customer is created, an event is going to be published by the customer microservice for the profile microservice. As part of this event processing, a new record will be inserted inside this profile for a given mobile number along with the name details. And all other columns they are going to be null values because we are yet to create the account card and loans for a specific user, and there is also no guarantee that end user will always have these three kinds of information. Few of the customers, they will only have account number, whereas few customers they are going to have both account and card since it is optional for the end user initially. So these fields can be nullable fields. During the very first insertion operation, we are going to keep these columns null. Later on if the end user is end up creating an account or card or loan, an event is going to be published for this microservice and as part of this event processing these individual fields will be updated inside this profile table based upon the mobileNumber. Once we have this materialized view pattern, what we can do is, we we're going to invoke an profile API, which is exposed by this profile microservice itself using a postman. For now we are going to mimic these API invocation using postman. But in real applications you can assume this invocation will happen from UI applications or from the websites. As soon as someone invoke this profile API based upon the mobile number that we provide, all this information is going to be loaded from the database, and the same we are going to send to the postman our client application as a response. With that, what we are going to achieve, we are going to overcome all the challenges that we have with the API gateway aggregator pattern. Whenever we are looking to read the profile information, we are never going to call these individual microservices. We are always going to call these profile microservice, which is going to read the data by using these materialized view table that we are going to maintain behind the scenes. I hope you are clear what we are trying to build with the help of materialized view pattern from the next lecture, let's make the code changes inside our microservices to adopt this pattern.
