# Transactional Outbox Pattern

Very similar to materialized view pattern we have another pattern with the name Transactional Outbox Pattern. Inside this lecture, let's try to understand what is this pattern? While I am trying to explain this pattern, you may find very similarities between the materialized view pattern, and you may also confused between these two patterns because they are going to look very similar. But both these patterns, they are going to solve different use cases. Towards the end of this lecture, I am going to explain you the differences between these two patterns so that it is going to be more clear for you. For now, let's try to understand what is this Transactional Outbox Pattern and how it is going to work. To understand more details about this pattern, let's imagine you are working in an bank based microservices application, and inside this bank application, you have a scenario where whenever a new account data is created inside the account microservices, The data science team, they want to perform analytics on these new data that accounts microservice just now inserted. In this kind of scenario the development team, they will end up choosing these most commonly used approaches. The very first approach is, you can grant a direct DB access to that data science team. Basically, you're going to store the accounts data into an account specific database to the same accounts database you can grant direct access to the data science team so that the data science team, they can connect to the accounts database, and they can run required analytics based upon their own tools or based upon their business logic. But there is a problem with this approach. The very first problem is, you are creating tight coupling, requiring coordination, and the schema changes whenever you want to change some schema changes inside the accounts microservice, you need to coordinate with the data science team also to make sure they are updating their business logic. The second problem with this first approach is, if data science team, if they are running lot many analytics and the accounts database, then your accounts database is going to receive lot of traffic and it may introduce performance issues inside your accounts microservice. That's why this approach is not a valid approach to consider. The second approach is, you are going to use two phase commit approach. 2PC means two phase commit. Inside this approach, what you are going to do is you're going to have two different databases, one for the accounts microservice and the other one for the data science team. Whenever you are inserting a new data inside the accounts microservice, you also going to insert new data inside that data science database as well. But since these are two different databases, you can't run using a single transaction. So to handle the rollback scenarios or error scenarios, you're going to approach that two phase commit approach. Two phase commit is a approach that is followed by very few folks inside the industry, basically inside these two phase commit what is going to happen is the code is going to coordinate with the both the databases. First it will ask both databases, are you guys ready to perform a transaction? If both databases respond yes, then it is going to write the database in both the databases. Even if one of the database has a problem writing the data, then the data inside another database also is going to be rolled back. But it is a very complicated approach to implement two phase commit because it is slow and blocking due to node coordination. Since your business logic is going to coordinate with the two different databases to perform the transaction at the same time, or to roll back the transaction, it is usually slow and it will bring some challenges as well while implementing. That's why most of the people they don't use this approach. Instead, inside the industry there is a pattern that we follow, which is Saga pattern. We are going to discuss about the Saga pattern in the coming sections. For now, what I want to highlight here is this two phase commit also is not going to be a worthy solution to update the data inside the accounts database and inside the Data science team database. With that, we left with the third approach. Let's see what is this third approach? Inside this third approach, we are going to push the data into the Kafka. So whenever a new record is being inserted into the accounts database, the account microservice, it is also going to publish an event into a Kafka topic or into the RabbitMQ queue, so that from these Kafka topic or RabbitMQ queue the data science team, they can read the event details and they can execute the business logic on their side. With this approach we are getting lot many advantages. The very first advantage is we are decoupling the services. So there is no direct link or direct communication between the analytics team and the accounts microservice. Since that data team, they are going to consume from the Kafka, they are going to store their own data independently and run analytics based upon their own requirements. This third approach, it looks decent and promising, but it has some limitations which are going to be addressed with the help of transactional outbox pattern. First, let's try to understand what are these limitations whenever we are going with this third approach, the account microservice, it has to perform two important jobs. The very first job is it needs to insert the data in its own database. And the second job is once the insertion of the data into its own database is completed, it needs to publish an event to the Kafka or RabbitMQ. While doing these two important jobs, the account microservice may end up into some scenarios. The very first scenario is the account service crashes after the DB write. But before sending an event to the Kafka. In this scenario, what is happening? The new data is written into the database, but the Kafka event is not published and the vice versa also is possible where accounts microservice sent an event to the Kafka. But DB write fails. If you closely analyze these two scenarios, the account microservice may end up doing half job, where it can make sure either to insert data into the account microservice, or it is going to publish an event to the Kafka. But ideally, we are expecting the account microservice to achieve both these two jobs at any cost. This is the place where the Transactional Outbox Pattern come into picture. This pattern what it recommends is the account microservice. It has to write the data into an account table and an outbox table. Basically, inside that database of account, there will be two tables. Account table and outbox table. Whenever the account microservice is making an entry into the account table, it is also going to make an entry into another table, which we call it as outbox table. And since these two tables are present inside the same database as part of a single transaction, these two records can be inserted. If there is a scenario where the data inside the accounts table is inserted, but outbox table is not inserted due to an runtime exception. Then, since both of them are running as part of a single transaction, the entire transaction will be rolled back and whatever data is written into the accounts table, it is also going to be rolled back with this approach. The job of accounts microservice is very simple. It will be either write into both tables or it will roll back the complete transaction. And it is easily achievable because both tables are present inside the same database server. There won't be any possibility where accounts microservice inserts the data only in one of the table. So we are good from the accounts microservice perspective because it is doing its job perfectly. Now let's understand what is going to happen once the data is written into the outbox table. The data science team, what they are going to do is, they're going to write a batch, or they are going to have a polling process, which is going to keep on polling the outbox table to scan for any new records that are inserted. If this polling process identifies any of the new records inside the outbox table, it is going to immediately publish the events corresponding to the new records that are available in the outbox table. Here, what polling process is going to do is, if it receives an acknowledgment from the Kafka that the event has been received by the Kafka, then only it is going to either delete the record inside the outbox table, or it is going to update some status flag inside the outbox table saying that so and so record has been dispatched to the Kafka event. In the scenarios where the Kafka does not give any successful acknowledgement to this polling process, it is going to perform the retry operation because this polling process never deletes its corresponding entry inside the outbox table. Let's imagine at 9 a.m. this polling process executed, and during this 9 a.m., the Kafka was very busy, and it has not given any acknowledgements for all the events that this polling process is trying to publish. The polling process is not going to worry much here. What it is going to simply do is, it is not going to delete the new entries inside the outbox table, or it is not going to update any of the status flags, maybe in the next schedule when it is trying to run this polling process, it will again try to perform the same operation. If you configured this polling process maybe for every 30 minutes, then at 9:30 a.m., the same process is going to try again and the Kafka event this time will be successful. And with that, the outbox table also will be updated. So from these Kafka events, the analytics service, they can read the events. Once the event details are reached to Kafka, we can safely assume at some point of time this analytic service it is going to read the events and process the data. This is what Transactional Outbox Pattern recommends. By this time, I am assuming you are clear why this outbox name came here. This name came because this pattern recommends you to maintain a separate outbox table, which can be read by a separate asynchronous process to communicate the data from this outbox table to the analytic service or any other target service. The same outbox pattern can be also implemented using CDC technology as well. We discussed about CDC technology in the previous lectures. Whenever the transaction log is updated, the CDC technology, it is going to make sure to publish the events to the Kafka based upon the new entries made inside this outbox table. Once the event has been reached to the Kafka this analytic service, they can consume the events and execute their own business logic. It's up to you and your organization whether you want to follow the complex CDC approach, or you want to build some polling process with the help of a spring scheduler, batch, or any other approach. But both these approaches, they are going to have the outbox table and the accounts microservice side. I hope you are clear with this pattern. Let's try to understand what are the advantages of this pattern. The very first advantage is Decoupling. This pattern promotes loose coupling, letting microservices focus on their own logic without worrying about message transmission details my account microservice. It is only going to worry writing the data into outbox table as part of the same transaction, where it is trying to write the data into the accounts table. Apart from writing the data, it is not going to worry how these message is going to be transmitted to the data science team. The second advantage is Reliability. This is the main advantage of transactional outbox pattern. Messages are saved transactionally with the business data ensuring delivery even during the failures as well. If there is a failure, then the accounts data also will be rolled back. That's the beauty we have here. The next advantage is around the scalability. It handles high message volumes efficiently as messages are stored in the database and consumed by the brokers at a controlled pace. Once the data is returned by the accounts microservice, the data science team, or any other team who are trying to read the data from the outbox table, They can do their job at their own pace. They can have the polling process run maybe every one day, or every 30 minutes or every one hour based upon their requirements. And the last advantage is around the performance. This is going to improve the speed by separating the business transactions from the message sending, allowing asynchronous processing. With this pattern right now my account microservice, it does not have to worry about publishing an event to the Kafka or RabbitMQ. Its job is only to store the data inside the outbox table. I hope you are clear with the advantages. Before I try to close this lecture, I want you to explain about the differences between the materialized view pattern and Transactional Outbox Pattern. There are very good amount of similarities between these two patterns, but the use cases that they handle are very different. Let's try to understand what are those. Materialized view pattern it is going to be used in the scenarios where we are looking to optimize read operations by creating a query friendly view. For example, if you are looking to fetch the data from multiple microservices in such scenarios, you need to go for the materialized view pattern because it is going to make the view ready in a friendly manner so that anyone who are looking to fetch the data from multiple microservices, they can directly query these friendly view that is built with the help of materialized view pattern. On the other side, Transactional Outbox Pattern ensures reliable event publishing between database and message broker. This pattern you need to consider if you are looking for an option to publish an event between database and message broker at any cost. The focus of the materialized view pattern is to improve the query performance, whereas the focus of the transactional outbox pattern is to maintain the data consistency and reliability in event publishing. Coming to the data synchronization inside the materialized view pattern data is replicated or updated asynchronously for read efficiency in a separate view table, whereas inside the Transactional Outbox Pattern, the data is going to be recorded in an outbox table and later published as an event by an asynchronous process. Typical use cases of materialized view pattern is, query heavy systems needing aggregated data from multiple sources. We already discussed about this point, whereas the outbox pattern use case is event driven systems that requires reliable inter service communication. At last coming to the challenges inside the materialized view pattern, achieving consistency is challenging and additional storage requirements is also challenging. Whereas in the outbox pattern, outbox table management and processing overhead may be challenging. I hope you are clear with these two patterns. Please select one of these two patterns based upon your business requirements. Thank you, bye.
