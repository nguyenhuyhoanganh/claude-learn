# Why we need to consider Materialized View Pattern

Inside this new section, we are going to talk about a new pattern which is materialized view pattern. Before I try to share more details about this pattern, first let me show you some information regarding under which scenarios we need to use this pattern. If you recall our discussion from the section1, we talked in detail about cross service queries. Since each of the microservice, it is going to have its own database. Whenever we have a scenario where we need to fetch the data from multiple services, such scenarios can be called as cross service queries, because you are going to execute the query across multiple microservices, and you are going to aggregate or collate the data before sending to the client application. We also saw the demo on how to overcome the cross service query challenge by using the API composition pattern with the help of gateway. So here I have my Spring Cloud Gateway. In this API gateway only we exposed an API to the client applications. So whenever my client application call to this API internally, my gateway is going to invoke all the services to fetch the data from their respective database, and the collating of the data is going to happen at the gateway level, and the response will eventually reach to the client application. But there is a drawback with this approach, even though from client perspective it is a single call. But internally gateway is going to make the individual calls to the respective microservices here, since you have a scenario to fetch the data from four different microservices. So the gateway is going to make four different API calls to their respective microservices. If each of the microservice is going to take 200 milliseconds to process and send the data, then obviously it is going to take total 800 milliseconds to load the data from all the microservices if they are invoked to sequentially. Let's consider a scenario where the gateway server invoked all these APIs one by one sequentially. In this kind of scenarios, the total time is going to be 800 milliseconds. And here inside the gateway for the aggregation of the data, let's consider it is going to take 200 milliseconds more time. So in total the client has to wait for one second. So as the number of microservices increases, the waiting period or the performance time to load, the response to the client is going to increase. Off course, we can try to overcome this challenge to some extent by invoking the APIs parallelly from the Spring Cloud Gateway. But still there will be some network latency which will add to the overall response time to the client application. So performance is the very first problem that we have in this approach. The second problem is, these individual microservices they are going to store the data inside their respective database based upon their business logic. But for some reason, if the client is expecting the data in a different format, then the formatting or the transformation logic, we need to write inside this gateway. In this gateway also, we will end up writing some good amount of complex business logic, which is going to take some time to execute and return the response to the client application. So that's a second drawback. And the third drawback is, whenever the client application want to load the data using the Spring Cloud gateway, then these gateway has dependency on all these microservices. If one of the microservice is down or if it is responding very slowly in such scenarios, obviously the client application, they are going to get the null information or empty information from that respective service. So these are all the challenges that we have with these gateway aggregator or API composition pattern. This pattern can be useful for a smaller applications who has some limited number of microservices or limited number of scenarios around the cross service queries. But we have a better approach to have any clue what is that approach? This better approach is Materialized View Pattern. Let's try to understand about this pattern in the next lecture. Thank you, bye.
