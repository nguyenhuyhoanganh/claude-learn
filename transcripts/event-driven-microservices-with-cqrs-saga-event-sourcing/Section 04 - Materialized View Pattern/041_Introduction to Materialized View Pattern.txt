# Introduction to Materialized View Pattern

Let's try to understand what is this pattern? The materialized view pattern in microservices is a design approach used to optimize the read operations by maintaining a pre-computed denormalized copy of data for efficient querying. This pattern is particularly useful when data needs to be aggregated or queried from multiple microservices. Instead of querying the microservices individually in real time, this pattern recommends you to maintain a separate copy of data from all the microservices in an ready made format, so that anyone who wants to query, they can simply look into this new table and read that data with a single API operation. I know you may have too many questions. That's why let's try to explore more details about this pattern. I'll come to these key concepts in a few minutes, but before that, let's imagine we have an and bank website. As soon as the end user login into this bank website, they're going to have a dashboard. Inside this dashboard they're going to have their profile information, their account balance, their card details, and the last transactions and other details regarding loans as well. As you can see in this scenario, the back end of the bank application is built with the help of microservices. Then getting all these information from a single microservice is not possible because the data is spread across multiple microservices. So this account related information will be present inside the accounts microservice, whereas this profile information and profile picture information will be present inside the user or customer microservice. And similarly this card information will be present inside the cards microservice and loans information will be present inside the loans microservice. One of the approach that these bank application can follow is, they can build an API inside the API gateway with the help of API composition pattern, and this API inside the API gateway is going to query all the dependent microservices and respond to the client application with the aggregate or the combined data. But we don't want to do that because it has its own challenges. Instead, we want to follow the materialized view pattern. What this pattern recommending is, whenever a change is happening inside these individual microservices, publish those changes as events, and these data change, events can be consumed by other new microservices. Let's imagine this microservice name is CustomerBankViewService. So these service responsibility is only to consume the events related to the data change happening on the customer accounts and loans and cards information. So whenever a new data are existing, data is being updated inside these microservices. They are going to publish an event which is going to be consumed by these CustomerBankViewService. So these service, what it is going to do is, it is going to have a singular table where it is maintaining all the required information related to these dashboard page. For example, here it is going to have the customer name, profile picture, what is the account balance, what is the credit card details, outstanding balance. What are the transaction details. So all these information is going to present inside this new microservice. This new microservice can have a single database to store all this information or it can have multiple tables as well based upon the business requirements. So whenever the change is happening at the source microservice, the event will be published. And the data here inside this new microservice is going to be updated with this what is going to happen? We are maintaining a separate copy of the required information for this dashboard page in a separate table. Whenever the dashboard page is looking to query the data, it can simply invoke this new microservice, which is customer bank view. And this microservice, it is going to read that data from the new table, where it is maintaining all the required information, and the same will be returned back to the dashboard UI page. With this, we are able to overcome all the challenges that we have with the API composition pattern. Now there is a single API call and the data also will be readily available based upon the client requirements. I hope you are clear. Now let's try to understand these key concepts that we have. The very first key concept inside this pattern is Denormalized View. Inside this pattern we are going to maintain an materialized view. Materialized view is nothing but the new table that we are trying to maintain in the new microservice. This materialized view it is going to contains a denormalized version of the data, which means it is going to store the data in a form that is optimized for the reading, rather than normalization, which is optimized for the storage efficiency. This denormalized view will have the data from multiple microservices anytime the source data is being changed, the same will be updated inside this Denormalized view as well. So how the updates are going to happen with the help of event driven updates, the responsibility of the individual microservices is, whenever a data change is happening, like updating a customer profile or adding a transaction, or making a card transaction. In all such scenarios, they are going to publish an event and whatever service which is maintaining the materialized view, it is going to listen and update the view accordingly. And the third important key concept is, this materialized view pattern it is optimized for the read operations. Why it is an optimized approach because it is going to speed up the read operations, especially in scenarios where data is spread across multiple microservices. Instead of querying each service in real time, the materialized view provides a pre-computed results, significantly reducing the time required to fetch data. I hope you are clear. Based upon your business requirements, you need to choose between the API composition pattern and materialized view pattern. API composition pattern you can go for a smaller applications where they have very less traffic, but if your application is expecting a huge amount of traffic, then going with the API composition pattern may not be a feasible option, because every time an end user is trying to perform an action, if you want to read the data from multiple microservices and once the reading operation is completed, you want to transform the data and send back to the client application. So doing all these operations in real time, it is going to bring some performance issues. And it is also make your microservices super busy always. They will be already busy with the write operations, but for the read operations also, if you are making them busy then you can imagine what is going to happen. That's why for heavy load traffics go with these materialized view pattern. One more beauty of this pattern is, whenever the read operations are happening, there is no dependency on the actual microservices, even if the actual microservice is down or if it is performing slowly. Still, the read operations will have zero impact. In the case of these easy bank scenario obviously, bank will have thousands and millions of customers when these customers as soon as they log in if these dashboards needs to be displayed, then it is always recommended to maintain a separate materialized view focusing this scenario. I also mentioned a sequence flow that is going to happen whenever these materialized view pattern is implemented. As you can see here, there will be a separate service with the name CustomerBankViewService. And inside these service we are going to have a separate database which is going to have the tables responsible to maintain these materialized view. Any time the data change is happening inside the actual individual microservices, they are going to publish that data, change events, these events, they are going to be handled and processed by this customer bank view service. So as part of this event handling logic, the data inside these materialized view is going to be updated with that what is going to happen any time clients they can make a API invocation to this service, which is going to be handled by the query handler. And this query handler is going to read the data from the materialized view database. And with that the UI application it is always going to get the response within few milliseconds. Off course, there are some drawbacks with this pattern as well. The very first drawback is, the sync between the actual microservice database and these materialized view database it is not going to be real time. The sync is not going to happen instantly. It is going to take some milliseconds or few seconds time to process these events and to update the data eventually into these materialized view database. So this is one of the drawback. The other drawback is, you are going to unnecessarily maintain a separate microservice, and you are also bringing more event handler logic. That's why choose a pattern that works for you. Inside microservices environment, there is no single pattern that you can take it as your silver bullet. Every pattern has its own pros and cons. At the end of the day, you need to take a decision based upon your project requirements, based upon the traffic that you handle, based upon the data that your microservice handle. So there are multiple factors that comes into picture. Always do a brainstorming inside your project before adopting a design pattern. I hope you are clear.
