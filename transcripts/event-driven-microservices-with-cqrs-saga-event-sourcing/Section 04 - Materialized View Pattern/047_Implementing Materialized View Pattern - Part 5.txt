Before I try to close this section, let me share a few more use cases under which you can use these materialized view pattern hapily. As of now, we saw one use case where we can use the materialized view pattern, which is cross service queries. Similarly, there are good amount of scenarios where you can use materialized view pattern. It is not going to be possible for me to cover all of them. What I'm going to do is, I'll try to cover the most common scenarios where we can use the materialized view pattern with this knowledge. In future, whenever you have some problem inside the microservices, think for a second whether we can solve the problem with the help of materialized view pattern. If yes, you can happily adopt the same. The second common scenario where you can use the materialized view pattern, is sharing the data across microservices. We can say this as a sharing or duplicating data. To explain about this scenario, let's consider I have two different microservices. The very first one is, account microservice followed by customer microservice. Let's imagine inside this customer microservice I have customer details, address details and many other details. Now inside the accounts microservice, there are some scenarios where they are going to look for the address details of the customer. For example, think like this account microservice has an API which is dispatch checkbook, CB means checkbook to dispatch the checkbook. Definitely accounts microservice they are going to need the address detail of the customer. So at runtime inside these microservice, what they can do is, they can make a call to the customer microservice to read the address details. This is one basic approach. The other approach is, inside the account microservice itself we can maintain the copy of address details for all the customers. This way this account microservice don't have to rely on the customer microservice to fetch the address details. So these address details we are going to maintain with the help of materialized view pattern. Whenever a change around the address details are happening inside the customer microservice, it is going to dispatch an event which can be handled on the account microservice and update these materialized view table. With this account microservice never have to invoke the customer microservice to fetch the address details. And the third most common scenario where these materialized view pattern can be used is during the reporting or analytics scenarios. Let's imagine you have a requirement to generate a daily report or weekly report or monthly report based upon the data present inside the multiple microservices. There are two ways on how we can achieve this. The very first basic approach is, inside these reporting or analytics processing they can invoke the actual microservices. The data can be transformed, aggregated, all the complex business logic you can try to execute in real time, but it is going to demand a lot of computation power and you're also going to make your microservices busy. Because this reporting and analytic process, they are trying to invoke the APIs exposed by the microservices. So these are very basic approach which I don't recommend you to follow. Instead what you can do is, you can tell all your individual microservices to publish that data that are interested by the reporting and analytics processes into a materialized view pattern table. So just like how we have maintained profile table very similarly, we can maintain a separate table for daily reporting or weekly reporting or monthly reporting and for the analytics purpose. This way we can add any number of MVP tables which are going to have readymade data in the required format. When the data is readily available, the reporting and analytics process, they are going to be super quick with very less computation required. They can directly read the data and generate the reports. I hope you are clear with this third approach. So these are the most common scenarios that I can think of. But like I said, materialized view pattern is used at many places with the knowledge that you have gained. I am assuming right now you are in a better position to decide when to use this materialized view pattern inside your projects. Thank you, bye.
